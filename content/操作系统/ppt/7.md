# 操作系统 Lecture 7：进程同步与互斥实现方法

## 1. 核心概念与准则

### 1.1 临界区问题的四个准则
并发进程进入临界区（Critical Section, CS）必须遵循以下原则：
1.  **互斥使用 (Mutual Exclusion)**：不能同时有两个进程在临界区内执行。
2.  **让权等待 (Wait with Yielding)**：等待进入临界区的进程，应释放处理机后阻塞等待（避免忙等待）。
3.  **有空让进 (Progress)**：在临界区外运行的进程不可阻止其他进程进入临界区；若无进程在CS内，允许一个请求进入的进程立即进入。
4.  **有限等待 (Bounded Waiting)**：不应使要进入临界区的进程无限期等待在临界区之外。

### 1.2 解决方法的分类
*   **软件实现方法**：Dekker算法、Peterson算法等。
*   **硬件实现方法**：屏蔽中断、Test_and_Set指令、Swap指令。
*   **高级同步机制**：信号量（Semaphore）、管程（Monitor）。

## 2. 软件实现方法 (Software Algorithms)

软件算法需遵循：
1. 不依赖特定硬件指令/处理机数目
2. 不假设进程相对速度
3. 非临界区不阻塞他人
4. 有限时间内选出进入者。

### 2.1 Dekker 算法 (针对两个进程 P0, P1)
通过逐步改进解决互斥问题：

*   **初步设想 (仅用 turn 变量)**：
    *   `turn` 指示谁可以进入。
    *   **缺点**：严格交替，若一方不想进入，另一方也无法进入（违背“有空让进”）；若一方在CS内失败，另一方永久阻塞。
*   **改进1 (仅用 flag 数组)**：
    *   `flag[i]=true` 表示 Pi 想进入。先检查对方 flag，再置自己 flag。
    *   **缺点**：若两者同时检测到对方 flag 为 false 并置自己为 true，则都进入 CS（违背“互斥”）；若都在 CS 内失败，对方永久阻塞。
*   **改进2 (先置 flag 再检查)**：
    *   先 `flag[i]=true`，再 `while(flag[j])` 等待。
    *   **缺点**：若两者同时置 flag 为 true，则陷入死锁（互相等待，违背“有空让进”和“有限等待”）。
*   **最终成功的 Dekker 算法**：
    *   结合 `flag` (意愿) 和 `turn` (礼让)。
    *   若冲突发生，由 `turn` 决定谁退让（暂时将 flag 置 false 等待 turn 变化）。

### 2.2 Peterson 算法
*   **特点**：代码更简洁，同样使用 `flag[2]` 和 `turn`。
*   **逻辑**：
    ```c
    // P0
    flag[0] = true;
    turn = 1; // 礼让给 P1
    while (flag[1] && turn == 1); // 当 P1 想进 且 轮到 P1 时，P0 等待
    <CS>
    flag[0] = false;

    // P1
    flag[1] = true;
    turn = 0; // 礼让给 P0
    while (flag[0] && turn == 0); // 当 P0 想进 且 轮到 P0 时，P1 等待
    <CS>
    flag[1] = false;
    ```
*   **原理**：只有当对方也想进入 (`flag[j]==true`) 且 轮到自己让对方先进 (`turn==j`) 时才等待。

## 3. 硬件实现方法 (Hardware Support)

利用处理器提供的特殊指令实现原子操作。

### 3.1 屏蔽中断 (Disable Interrupts)
*   **方法**：进入临界区前关中断，退出后开中断。
*   **优点**：简单。
*   **缺点**：
    *   将特权交给用户进程不安全（若用户忘记开中断，系统崩溃）。
    *   限制了 CPU 交叉执行能力。
    *   **不适用于多处理机系统**（只影响当前 CPU）。

### 3.2 Test_and_Set 指令
*   **功能**：原子地读取目标值并设置为 true。
    ```c
    boolean Test_and_Set(boolean &target) {
        boolean rv = target;
        target = true;
        return rv;
    }
    ```
*   **实现互斥**：lock初值为false
    ```c
    do {
        while (Test_and_Set(lock)); // 忙等待直到 lock 变为 false 并被设为 true
        critical section;
        lock = false; // 释放锁
        non-critical section;
    } while (1);
    ```

### 3.3 Swap 指令
*   **功能**：原子地交换两个布尔变量的值。
    ```c
    void Swap(boolean &a, boolean &b) {
        boolean temp = a;
        a = b;
        b = temp;
    }
    ```
*   **实现互斥**：每个进程拥有局部变量 `key`。
    ```c
    do {
        key = true;
        while (key == true)
            Swap(lock, key); // 如果 lock 是 false，key 变 false 退出循环；否则继续循环
        critical section;
        lock = false;
        non-critical section;
    } while (1);
    ```

## 4. 信号量机制 (Semaphores)

由 Dijkstra (1965) 提出。

### 4.1 定义与类型
*   **定义**：整型变量 `S`，只能通过初始化及 P、V 原语访问。
*   **类型**：
    *   **公用信号量**：用于互斥，初值通常为 1。
    *   **私用信号量**：用于同步，初值通常为 0 或 n。

### 4.2 P、V 操作 (原语)
*   **P(S) (Wait/Down)**：申请资源。
    *   `S = S - 1`
    *   若 `S < 0`，进程阻塞，挂入等待队列。
*   **V(S) (Signal/Up)**：释放资源。
    *   `S = S + 1`
    *   若 `S <= 0`，从等待队列唤醒一个进程。
*   **原子性实现**：通过屏蔽中断或硬件锁保证 P/V 操作本身不被中断。

### 4.3 信号量的具体数据结构实现
```c
typedef struct {
    int value;              // 资源数量
    struct process *L;      // 等待队列指针
} semaphore;
```
*   **消除忙等待**：操作系统结合进程调度，P操作失败时挂起进程，V操作时唤醒进程。

### 4.4 信号量的应用实例

#### A. 互斥 (Mutex)
*   设置信号量 `mutex = 1`。
*   结构：
```c
do{
	P(mutex);
	CS;
	V(mutex);
	Non-CS;
}while(1)
```

#### B. 同步 (Synchronization)
*   场景：P1 执行 S1 后，P2 才能执行 S2。
*   设置信号量 `synch = 0`。
*   P1: `S1; V(synch);`
*   P2: `P(synch); S2;`

#### C. 生产者-消费者问题 (有限缓冲区)
*   **信号量**：
    *   `mutex = 1`：互斥访问缓冲池。
    *   `empty = N`：空缓冲区数量。
    *   `full = 0`：满缓冲区数量。
*   **生产者**：
    ```c
    produce item;
    P(empty);   // 申请空位
    P(mutex);   // 申请互斥锁
    add item to buffer;
    V(mutex);   // 释放互斥锁
    V(full);    // 增加满位计数
    ```
    *   **注意**：两个 P 操作顺序不能颠倒！若先 `P(mutex)` 再 `P(empty)`，当缓冲区满时，生产者持有锁并等待空位，消费者无法获取锁来消费，导致**死锁**。
*   **消费者**：
    ```c
    P(full);    // 申请满位
    P(mutex);   // 申请互斥锁
    remove item from buffer;
    V(mutex);   // 释放互斥锁
    V(empty);   // 增加空位计数
    consume item;
    ```

#### D. 读者-写者问题 (Readers/Writers)
*   **第一类 (读者优先)**：只要有读者在读，新读者可直接进入，写者需等待。
*   **信号量**：
    *   `wrt = 1`：保护共享数据 A 的互斥访问。
    *   `mutex = 1`：保护 `readcount` 变量的互斥访问。
    *   `readcount`：记录当前读者数量。
*   **Writer**:
    ```c
    P(wrt);
    write data;
    V(wrt);
    ```
*   **Reader**:
    ```c
    P(mutex);
    readcount++;
    if (readcount == 1) P(wrt); // 第一个读者封锁写者
    V(mutex);
    read data;
    P(mutex);
    readcount--;
    if (readcount == 0) V(wrt); // 最后一个读者释放写者
    V(mutex);
    ```

#### E. 哲学家就餐问题
*   **描述**：5个哲学家，5只筷子。需同时拿起左右两只筷子才能吃饭。
*   **naive 实现 (可能导致死锁)**：
    ```c
    // 哲学家 i
    P(chopstick[i]);       // 取左
    P(chopstick[(i+1)%5]); // 取右
    Eat;
    V(chopstick[i]);
    V(chopstick[(i+1)%5]);
    ```
    *   若所有哲学家同时拿起左边筷子，所有人都在等待右边筷子，形成环路等待，导致死锁。

## 5. 管程 (Monitors)

### 5.1 引入原因
P-V 机制分散在各个进程中，导致：
1.  易读性差。
2.  不利于修改和维护。
3.  正确性难以保证。

### 5.2 定义与组成
*   **定义** (Hoare & Brinch Hanson, 1973)：一个管程定义了一个数据结构和能为并发进程所执行的一组操作，这组操作能同步进程和改变管程中的数据。
*   **组成**：
    1.  共享数据结构（局部于管程）。
    2.  一组过程/函数（对数据进行操作）。
    3.  初始化程序。
*   **特点**：
    *   **模块化**：可单独编译。
    *   **抽象数据类型**：封装数据和操作。
    *   **信息掩蔽**：外部只能调用定义的过程，不能直接访问内部变量。
    *   **互斥性**：系统保证任一时刻只有一个进程在管程内执行。

### 5.3 条件变量 (Condition Variables)
*   用于进程间的阻塞和唤醒。
*   `wait(condition)`: 进程阻塞在该条件变量的等待队列上，并释放管程互斥锁。
*   `signal(condition)`: 唤醒等待在该条件变量上的一个进程（若无等待进程则无操作）。

### 5.4 管程示例：生产者-消费者
```c
monitor prod_conshow {
    char buffer[n];
    int k = 0; // 产品计数
    int nextempty = 0, nextfull = 0;
    condition nonempty, nonfull;

    procedure put(product) {
        if (k == n) wait(nonfull); // 满则等待
        buffer[nextempty] = product;
        nextempty = (nextempty + 1) % n;
        k++;
        signal(nonempty); // 唤醒可能等待的消费者
    }

    procedure get(product) {
        if (k == 0) wait(nonempty); // 空则等待
        product = buffer[nextfull];
        nextfull = (nextfull + 1) % n;
        k--;
        signal(nonfull); // 唤醒可能等待的生产者
    }
}
```
*   **调用方式**：
    *   Producer: `prod_conshow.put(item);`
    *   Consumer: `prod_conshow.get(item);`
    *   管程自动保证 `put` 和 `get` 的互斥执行。

### 5.5 管程 vs 进程
| 特性 | 管程 (Monitor) | 进程 (Process) |
| :--- | :--- | :--- |
| **目的** |管理共享资源同步 | 资源分配和执行的基本单位 |
| **数据结构** | 公用数据结构 | 私有数据结构 |
| **存在方式** | 静态模块，无生命周期 | 动态实体，有生命周期 |
| **执行方式** | 被调用，无并发性 (内部互斥) | 并发执行 |

### 5.6 局限性
*   需要编程语言支持（如 Java 有 synchronized/methods，C/Pascal 原生不支持）。
*   编译器需特殊处理以保障互斥。

## 6. 总结
本章主要介绍了进程同步与互斥的四种实现层次：
1.  **软件算法**：理解逻辑，但复杂且易错 (Dekker, Peterson)。
2.  **硬件指令**：提供原子操作基础 (Test_and_Set, Swap)。
3.  **信号量**：通用的同步互斥工具，灵活但需注意 P 操作顺序以防死锁。
4.  **管程**：高级抽象，集中管理同步逻辑，更安全、易维护，但依赖语言支持。
