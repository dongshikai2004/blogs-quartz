## 3. 并发需求
程序设计需要利用操作系统对并发的支持（进程或线程），安排可并行事务并发执行。操作系统核心程序也要尽可能地并发运行。

### 两个层次的并发需求：
1.  **计算任务存在可并行成分：** 例如 S1 和 S2 两个子任务可并发执行，之后执行 S3。
2.  **并发编程的三种方法：**
    *   程序员写顺序程序，用自动识别工具识别可并行成分，组织使用操作系统的进程或线程实现并发。
    *   由程序员识别可并行成分，用并发程序设计语言设计并发程序，由编译系统安排使用进程或线程。
    *   在传统语言基础上，利用操作系统的进程或线程系统调用设计并发程序。

## 4. 并发程序设计语言 --- 并发语句
*   **定义：** 在传统高级语言基础上增加描述并发的语句。
*   **语法形式：**
    ```text
    Parbegin S1; S2; ... Sn; Parend;
    ```
    *   `Si` (i=1, 2, ... n) 是单个语句。
    *   `Parbegin` 和 `Parend` 之间的语句可以并发执行。
*   **示例：**
    *   任务结构：S1 和 S2 并行，完成后执行 S3。
    *   描述：
        ```text
        Parbegin
            S1; 
            S2; 
        Parend;
        S3
        ```

### 并发语句描述手段的优缺点：
*   **优点：** `Parbegin/Parend` 的结构化特征非常好。
*   **缺点：** 描述能力不强。存在用 `Parbegin/Parend` 语句无法描述的并发优先关系（例如复杂的依赖图，如 S1->S2/S3, S2->S4, S3->S5/S6, S4/S5->S7 等复杂拓扑结构）。
*   **改进：** 若能辅以其它手段（如信号量机制），并发语句可以大大增加其描述能力。

## 5. 并发执行实现
实现并发执行需要通过操作系统支持的**进程**或**线程**机制。操作系统提供进程（线程）创建、结束和同步的系统调用。

### 与进程相关的系统调用 (Linux)
1.  **`fork()`：创建一个新进程。**
    *   执行完成后，系统创建了一个子进程。
    *   子进程逻辑复制（共享）了父进程的程序，复制了父进程的数据段和栈段。
    *   **返回值：**
        *   **父进程：** 返回子进程的进程标识 `pid`。
        *   **子进程：** 返回 `0`。
    *   **执行点：** 父进程和子进程在占有处理机后，都从系统调用的返回点开始运行。子进程从 `trap` 指令后一条指令开始运行。
2.  **`exit(status)`：结束进程。**
    *   发出后，操作系统将从系统中删除调用 `exit` 的进程。
3.  **`wait(&status)`：等待子进程结束。**
    *   当有多个子进程时，任一个子进程结束即将控制返回调用者。
    *   将子进程调用 `exit(status)` 时的 `status` 值送到 `&status` 指针所指单元中。
    *   同时将所等到的子进程 `pid` 作为 `wait()` 系统调用函数的返回值。
4.  **`waitpid(pid, ...)`：等待 `pid` 所指定的特定进程结束。**

### 多进程实现前述计算任务并发示例
假设任务为：S1 和 S2 并行，S3 在它们之后执行。
```c
pid = fork();
if (pid == 0) then {
    S2;
    exit(0);
} else {
    S1;
    wait(); // 等待子进程(S2)结束
    S3;
}
```

## 6. 进程的互斥与同步

### 基本概念
*   **同步 (Synchronization)：**
    *   指完成同一任务的伙伴进程间，因需要在某些位置上协调它们的工作而等待、传递信息所产生的制约关系。
    *   亦称**直接制约关系**。
*   **互斥 (Mutual Exclusion)：**
    *   指进程间因相互竞争使用独占型资源（互斥资源）所产生的制约关系。
    *   亦称**间接制约关系**。

### 典型问题示例

#### 例1：同步问题
*   **场景：** 进程 P1 执行 S1, S3；进程 P2 执行 S2。
*   **约束：** P1 在执行 S3 之前必须等待 P2 执行完 S2。
*   **图示关系：** S1(P1) -> [等待 S2(P2)] -> S3(P1)。

#### 例2：互斥问题（独占资源）
*   **场景：** P1、P2 两进程使用同一打印机。
*   **问题：** 如果不互斥使用，输出会交叉混乱。
*   **结构：**
    *   Entry code (进入区)
    *   使用打印机 (临界区)
    *   Exit code (退出区)

#### 例3：互斥问题（共享数据）
*   **场景：** 存取钱对共享变量 `balance` 的访问。
*   **代码示例：**
    ```text
    Parbegin 
        A(amount) { 
            R1 = balance; 
            R2 = amount; 
            R1 = R1 + R2; 
            balance = R1; 
        }; 
        B(amount) { 
            R1 = balance; 
            R2 = amount; 
            R1 = R1 - R2; 
            balance = R1; 
        }; 
    Parend;
    ```
*   **问题：** 如果 A 和 B 并发执行且没有互斥保护，`balance` 的计算结果可能错误（竞态条件）。

#### 例4：有限缓冲区的生产者/消费者问题
*   **场景：** 生产者和消费者共享一个产品缓冲链（N个缓冲区）。
*   **数据结构：**
    ```c
    typedef struct { ...... } item; // 消息类型
    typedef struct {
        struct buffer *next;
        struct item inst;
    } buffer; // 缓冲类型
    
    struct buffer *P, *C, *First;
    struct item nextp, nextc;
    First = null;
    ```
*   **伪代码逻辑：**
    ```text
    Parbegin 
        Producer() {
            do {
                produce an item in nextp;
                new(P); # 获得一空缓冲区
                P.inst = nextp;
                P.next = First;
                First = P;
            } while(1);
        };
        
        Consumer() {
            do {
                while(first == null); # 空，循环等待
                C = First;
                first = first.next;
                nextc = C.inst;
                dispose(C); # 释放缓冲区
                consume the item in nextc;
            } while(1);
        };
    Parend;
    ```
*   **并发错误示例（缓冲区丢失）：**
    若不加锁/同步，可能发生以下交错导致数据丢失：
    *   T0: consumer `C = First`
    *   T1: producer `P.next = First`
    *   T2: producer `First = P`
    *   T3: consumer `First = First.next` (此时 First 指向的是旧链表的下一个，导致 T2 中加入的 P 丢失)

## 7. 关键术语定义
*   **临界资源 (Critical Resource)：** 一次仅允许一个进程使用的资源。如打印机、共享的变量等。
*   **临界段 (Critical Section)：** 各进程中访问临界资源的那段代码，必须互斥执行的程序段。
