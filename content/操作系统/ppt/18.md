# Linux 存储器管理 (Linux Memory Management)

## 1. 概述
*   **目标**：了解 Linux 系统的物理内存管理、虚拟存储空间的管理策略和方法。
*   **内存用途**：
    *   存储内核映像。
    *   其它可动态加载内核模块的内存需求。
    *   进程用户页面的需求。
    *   缓冲需求（Buffer/Cache）。
*   **功能要求**：
    *   快速响应请求。
    *   尽可能利用内存，同时减少内存碎片。
*   **核心解决方案**：
    *   基于区域的 **伙伴系统 (Buddy System)**。
    *   **Slab 分配器**。

## 2. 物理内存的管理

### 2.1 基本概念与结构
*   **基本单位**：**页帧 (Page Frame)**，大小固定为 **4KB**。
*   **管理数据结构**：硬件使用 `mem_map[]` 数组，数组大小为页帧总数。
*   **地址空间划分 (以32位系统为例)**：
    *   **0 - 3G**：用户虚空间。
    *   **3G - 4G**：内核空间。
    *   **物理内存区域划分**：
        1.  **ZONE_DMA** (< 16MB)：
            *   DMA 方式能够访问的物理内存。
            *   一般用于 I/O 缓冲区。
        2.  **ZONE_NORMAL** (16MB - 896MB)：
            *   直接被内核虚地址映射。
            *   用途：内核程序及常数、动态表格、用户空间页。
        3.  **ZONE_HIGHMEM** (> 896MB)：
            *   高端内存。
            *   用于存放动态映射页面（不能直接线性映射到内核空间）。

### 2.2 区域分配优先级
在内存分配时，根据访问优先级别决定分配次序：
*   **DMA 请求**：只能在 `ZONE_DMA` 分配。
*   **普通内存分配**：
    1.  优先尝试 `ZONE_HIGHMEM`。
    2.  若不满足，尝试 `ZONE_NORMAL`。
    3.  若仍不满足，最后尝试 `ZONE_DMA`。

### 2.3 伙伴算法 (Buddy Algorithm)
*   **作用**：对每个区域内的空闲内存页帧进行管理，主要目的是**减少外部碎片**。
*   **应用场景**：为进程用户空间页面、系统 I/O 缓冲区、系统存储对象（如表格、动态模块）分配页帧时调用。
*   **算法思想**：
    *   维护多个链表，分别对应不同大小的连续空闲块（如 1, 2, 4, 8, 16... 个页面）。
    *   **分配过程**：
        1.  寻找合适大小的空闲块。如有，摘除并返回。
        2.  若无，寻找更大的连续空闲块，将其**剖分**为两个大小相等的“伙伴”。
        3.  将其中一个伙伴插入合适大小的链表，另一个继续检查是否满足需求，若不满足继续剖分。
    *   **释放过程**：
        1.  检查释放的块是否有空闲的“伙伴”。
        2.  若有，则**合并**成更大的连续块。
        3.  重复检查新合并的块是否有伙伴，直到无法合并或达到最大块。
        4.  将最终块插入合适位置的链表。
    *   **“伙伴”的定义条件**：
        1.  两个块具有相同的大小 $b$。
        2.  物理地址连续。
        3.  第一块的第一个页帧编号必须是 $2b$ 的倍数。
*   **优点**：
    *   分配时尽量动用小的连续页面。
    *   回收时尽可能合并，形成大的连续页面，有效减少外部碎片。
*   **相关操作函数**：
    *   申请：`__alloc_pages()`
    *   释放：`__free_pages_ok()`
    *   **kswapd 内核线程**：当发现页面短缺时被唤醒，负责腾出空间以满足要求。

### 2.4 Slab 分配器
*   **存在的问题**：伙伴系统以页帧（4KB）为基本单位，对于小对象（如几十字节）容易造成严重的**内部碎片**。
*   **解决思路**：基于伙伴系统之上，建立 Slab 分配器。
*   **基本思想**：
    *   为经常使用的小对象建立缓冲池。
    *   小对象的申请与释放由 Slab 分配器管理，Slab 分配器再与伙伴系统交互。
*   **好处**：
    1.  充分利用空间，减小内部碎片。
    2.  管理局部化，减少与伙伴系统的交互频率，提高效率。
*   **构成**：
    *   **缓冲 (Cache)**：为不同的常用对象生成不同的缓冲，每个缓冲存储相同类型的对象。
    *   **Slab**：某种对象的缓冲区由一连串的 slab 构成。每个 slab 由一个或多个连续的物理页帧组成，包含若干同种类型的对象。
    *   **数据结构**：
        *   `slab_t`：描述 slab 对象。
        *   `kmem_cache_t`：特定对象建立的 slab 队列的队头。
        *   `cache_chain`：系统全局的 slab 队列头，管理所有对象的 `kmem_cache_t`。
    *   **通用缓冲**：除了特定对象，Linux 还提供 13 种通用缓冲，大小分别为 32B, 64B, ..., 128KB。采用级数增长保证内部碎片率不超过 50%。
*   **相关操作函数**：
    1.  `kmem_cache_create()`：创建特定对象的 `kmem_cache_t` 结构并加入 `cache_chain`。确定最佳 slab 构成（每 slab 页数、对象数、控制结构位置等）。
    2.  `kmem_cache_alloc()` / `kmem_cache_free()`：从专用 slab 队列中分配/释放对象。
    3.  `kmem_cache_grow()`：当队列无空闲 slab 时，向伙伴系统申请空间创建新 slab。
    4.  `kmem_cache_reap()`：由 kswapd定时调用，回收缓冲区队列中的页帧。
    5.  `kmalloc()` / `kfree()`：从**通用**缓冲区队列中申请和释放空间。

## 3. 进程空间的管理 (虚拟内存)

### 3.1 基本概念
*   每个进程有独立的虚拟地址空间。
*   虚拟内存以**页**为单位，大小等同于物理页帧 (4KB)。
*   通过进程的**页目录**、**页表**实现与物理内存的映射。
*   进程通过**按需分页 (Demand Paging)** 获得物理内存。
*   **核心结构**：`mm_struct` 描述进程空间，包含页目录指针和 `vm_area_struct` 指针。

### 3.2 页表机制 (以 x86/i386 为例)
*   **线性地址结构 (32位)**：
    *   **高 10 位**：页目录索引 (Page Directory Index)。
    *   **中 10 位**：页表索引 (Page Table Index)。
    *   **低 12 位**：页内偏移 (Offset)。
*   **地址转换过程**：
    1.  从寄存器 **CR3** 获取页目录基址。
    2.  用线性地址的高 10 位在页目录中找到**页表基址**。
    3.  用线性地址的中 10 位在页表中找到**物理页帧号**。
    4.  物理页帧号 + 线性地址的低 12 位偏移 = **物理地址**。
*   **页表项标志位** (共32位，高20位存物理地址，低12位存控制信息)：
    *   **存在位 (Present)**：物理页是否存在。
    *   **读/写位 (R/W)**：只读或可写，起保护作用。
    *   **访问位 (Accessed)**：该页是否被访问过。
    *   **脏标志位 (Dirty)**：该页是否被写过。
*   **多级页表优势**：节省存储空间（未使用的页表无需分配物理内存）。
*   **TLB (Translation Lookaside Buffer)**：CPU 内部的专门缓冲，加速地址转换。
*   **注意**：Linux 采用三级页模型（页目录-页中间目录-页表），但在 32 位机器上，页中间目录层实际不起作用。

### 3.3 进程空间结构管理
*   **地址空间布局**：
    *   **用户空间 (0 - 3G)**：进程可直接访问。
    *   **内核空间 (3G - 4G)**：内核专用。
*   **vm_area_struct (VMA)**：
    *   绝大多数程序不会用完全部 3G 空间。
    *   进程空间由多个不连续的区间组成，Linux 用一系列 `vm_area_struct` 结构描述这些区间。
    *   进程页面必定属于某个 VMA，但属于 VMA 的页面不一定在页表中有数据项（可能尚未分配物理页）。
    *   `task_struct` -> `mm_struct` -> `vm_area_struct` 链表/树。

### 3.4 相关系统调用
*   `exec`：抛弃原地址空间，根据可执行文件生成新地址空间。
*   `fork`：
    *   拷贝父进程的地址空间描述 (`vm_area_struct`, 页目录, 页表)。
    *   **写时拷贝 (Copy On Write, COW)**：
        *   父子进程共享同一物理页帧。
        *   页表项标记为**只读**。
        *   写入时触发页面异常，此时才复制物理页。
        *   **好处**：节省开销；“懒惰策略”，分散复制压力，响应更快。
*   `brk()`：改变堆的大小，增大进程空间。
*   `exit()`：结束进程，销毁进程空间。
*   `mmap()` / `munmap()`：创建/解除文件的内存映射，扩展/收缩进程空间。
*   `shmat()` / `shmdt()`：连接/分离共享内存区。

## 4. 页面异常处理 (Page Fault Handling)

*   **处理函数**：`do_page_fault()`
*   **参数**：
    1.  指向异常发生时寄存器值的指针。
    2.  错误码 (3位二进制)：
        *   Bit 0: 物理页是否存在 (0=不存在, 1=存在但保护错误)。
        *   Bit 1: 读/写/执行错误 (0=读, 1=写)。
        *   Bit 2: 运行模式 (0=内核态, 1=用户态)。
*   **执行流程**：
    1.  从 **CR2** 寄存器获取导致异常的线性地址。
    2.  检查是否发生在中断或内核线程中，若是则出错处理。
    3.  检查该线性地址是否属于进程的某个 `vm_area_struct` 区间。
        *   若不属于，检查是否为栈的合理可扩展区间。
        *   若是栈扩展，调用 `expand_stack()` 扩展 VMA（此时仍未分配物理页）。
    4.  根据错误码确定下一步：
        *   **写错误**：
            *   若区间不允许写 -> 出错处理。
            *   若允许写 -> **写时拷贝 (COW)** 处理。
        *   **页面不存在** -> **按需分页 (Demand Paging)** 处理。

### 4.1 写时拷贝 (COW) 处理
1.  修改页表项访问标志位（标记为已访问，避免调度时优先换出）。
2.  若页帧仅被当前进程独占 -> 直接将页表项置为**可写**。
3.  若页帧被多进程共享 ->
    *   申请新物理页。
    *   复制原页面内容到新页。
    *   修改当前进程页表项指向新页，并置为可写。
    *   原物理页帧的共享计数减一。

### 4.2 按需分页 (Demand Paging) 处理
*   **情况 A：页面从未被访问 (页表项全0)**
    *   (a) 若 VMA 的 `vm_ops->nopage` 不为空（映射到文件）：调用该函数装入页面。
    *   (b) 若 `vm_ops` 或 `nopage` 为空（匿名映射，如堆/栈）：调用 `do_anonymous_page()` 申请一个新页帧（通常清零）。
*   **情况 B：页面曾被访问，但已被交换到磁盘 (Swap)**
    *   页表项存在位为 0，但其他位记录了 Swap 分区信息。
    *   调用 `do_swap_page()` 从交换分区调入页面。
