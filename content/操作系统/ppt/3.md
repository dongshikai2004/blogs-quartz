## 1. 中断/异常机制 (Interrupts/Exceptions)

### 1.1 基本概念
*   **定义**：指系统发生某个异步/同步事件后，处理机暂停正在执行的程序，转去执行处理该事件程序的过程。
*   **引入目的**：
    *   **中断**：为了开发 CPU 和通道（或设备）之间的并行操作。I/O 完成后通知 CPU。
    *   **异常**：用于表示 CPU 执行指令时出现的错误（如算术溢出、除零、越界）或执行“陷入指令”（实现系统调用）。
*   **历史演变**：早期不区分，统称中断；后因发生原因和处理方式差异明显而区分。

### 1.2 分类与区别
| 特征 | 中断 (狭义/外中断) | 异常 (内中断) |
| :--- | :--- | :--- |
| **来源** | 来源于外设等 | 来源于应用程序/CPU内部 |
| **相关性** | 与正执行指令无关 | 与正执行指令有关 |
| **屏蔽性** | 可以屏蔽 | 不可屏蔽 |
| **典型例子** | I/O中断、时钟中断 | 系统调用、缺页异常、断点指令、算术溢出 |

> **注意**：广义的中断包含中断和异常。

### 1.3 处理过程
一般过程分为四步：
1.  **中断/陷入进入**
2.  **保存现场** (Save Context)
3.  **分析原因，转中断(陷入)处理程序**
4.  **恢复现场** (Restore Context)

## 2. 操作系统的结构 (OS Structure)

### 2.1 内核 (Kernel) 的概念
*   **定义**：作为可信软件，提供支持进程并发执行的基本功能和基本操作的一组程序模块。
*   **特性**：
    *   驻留在内核空间，运行于**核心态**。
    *   具有访问硬件设备和所有主存空间的权限。
    *   是唯一能执行**特权指令**的部分。
*   **作用**：扩展机器功能，改善进程运行环境，保证安全性，提高系统效率。

### 2.2 操作系统核心功能模块
1.  **系统初始化模块**：准备运行环境，为每个终端创建进程运行命令解释程序。
2.  **进程管理模块**：处理进程类系统调用（创建/结束/通信），负责调度、同步。
3.  **存储管理模块**：分配进程空间，处理存储类系统调用，处理缺页异常。
4.  **文件管理模块**：存储空间管理、目录管理、读写管理和保护。
5.  **外设管理模块**：驱动、中断处理、缓冲管理、设备分配。

### 2.3 结构设计类型

#### (1) 简单结构 (Simple Structure)
*   **特点**：整体结构，过程的集合，相互调用。
*   **缺点**：难于调试、阅读和理解。
*   **代表**：MS-DOS, 早期 UNIX。

#### (2) 分层结构 (Layered Structure)
*   **设计原则**：
    *   功能模块排列成若干层。
    *   **单向调用**：每一层仅使用其底层提供的功能。
    *   同层模块间无调用。
    *   第0层为硬件，第N层为用户接口。
*   **优点**：易于实现，增加可靠性。
*   **缺点**：难以确切定义每一层，效率较低。

#### (3) 微内核结构 (Microkernel)
*   **基本思想**：将更多 OS 功能放在核心之外，作为独立的服务进程运行。内核保持最小，只实现基本功能（如消息中继、中断处理、调度）。
*   **模式**：客户机/服务器 (Client/Server) 模式。
*   **优点**：
    *   良好的扩充性（添加服务进程即可）。
    *   可靠性好（模块隔离，调用关系明确）。
    *   便于网络服务和分布式处理。
*   **缺点**：消息传递比直接调用效率低（可通过硬件性能补偿）。
*   **代表**：Mach, QNX, OpenHarmony/HarmonyOS 部分架构。

#### (4) 单内核/宏内核 (Macrokernel/Monolithic Kernel)
*   **基本思想**：大部分 OS 功能都在内核中，模块间通过函数调用通信。
*   **优点**：运行效率高，组织相对简单。
*   **缺点**：内核庞大，移植困难，耦合度高。
*   **代表**：Linux (总体上是单内核，但引入了微内核的设计方法，如可加载模块)。

#### (5) 模块结构 (Modular Structure)
*   **特点**：用面向对象技术生成模块化内核；有一组核心部件，支持动态链接附加服务。
*   **优势**：类似于微内核，但不需要消息传递，效率更高。
*   **代表**：Solaris, Mac OS X。

#### (6) 混合结构 (Hybrid)
*   **趋势**：现代 OS 多采用单内核和微内核的混合结构，取长补短。
*   **示例**：Linux, Windows NT。

#### (7) 虚拟机结构 (Virtual Machine)
*   **基本思想**：将整个系统在逻辑上虚拟为若干个相同的机器。
*   **技术基础**：分时系统 + 多道程序设计。
*   **代表**：IBM VM/370, VMware。

## 3. 操作系统的运行模型 (OS Execution Model)

OS 本身也是程序，其运行方式主要分为三种模型：

1.  **非进程内核模型 (Independent Kernel)**
    *   用户程序与核心程序在分离的环境中运行。
    *   核心程序有独立的运行栈。
    *   用户进程通过中断/陷入机制启动核心程序（以请求包方式传递请求）。

2.  **OS 功能在用户进程内执行的模型 (Embedded in User Process)**
    *   **本课程重点/实用 OS 常用模式**。
    *   OS 核心程序通过中断/陷入机制启动。
    *   **运行于被打断进程的核心栈上**。
    *   内核程序执行并发性好。

3.  **OS 功能作为独立进程执行的模型 (Microkernel Mode)**
    *   核心程序只包含中断处理、系统调用总控、进程调度等最少功能。
    *   其他功能由用户态运行的系统进程实现。
    *   开销较大。

## 4. 用户接口 (User Interfaces)

### 4.1 接口分类
1.  **编程接口 (Programming Interface)**
    *   即**系统调用 (System Calls)**。
    *   提供给程序设计人员，通过代码请求 OS 服务。
2.  **操作接口 (Operational Interface)**
    *   提供给最终用户，控制作业流程。
    *   包括：命令行 (CLI)、图形界面 (GUI)。

### 4.2 系统调用 (System Calls)

#### 基本原理
*   **状态转换**：
    *   **用户态 (User Mode)**：运行用户程序，无权执行特权指令。
    *   **核心态 (Kernel Mode)**：运行 OS 核心，可执行特权指令。
*   **转换机制**：
    *   **访管指令 (Trap/SVC)**：一条特殊的非特权指令，功能是“自愿进管”，引起访管异常。
    *   过程：执行 Trap -> 保护现场 -> 置为管态 -> 执行系统子程序 -> 恢复现场 -> 返回用户态。

#### 系统调用 vs 普通过程调用
| 特征 | 系统调用 | 普通过程调用 |
| :--- | :--- | :--- |
| **运行状态** | 调用者(user)与被调用者(kernel)状态不同 | 通常在同一状态下 |
| **调用方法** | 必须通过软中断(trap)进入核心 | 直接跳转 |
| **返回问题** | 可能涉及重新调度 (Preemptive) | 直接返回继续执行 |

#### 实现过程 (以 UNIX 为例)
1.  **用户程序**调用系统调用库函数 (如 `write`)。
2.  库函数将参数放入寄存器，执行 **Trap 指令**。
3.  **CPU** 保护现场，转入内核总控。
4.  **内核总控** 根据 Trap 类型转至**系统调用处理总控**。
5.  **系统调用处理程序** 根据**系统调用号**查表，找到对应处理程序入口。
6.  执行具体的系统调用服务 (如 Write 处理程序)。
    *   若涉及 I/O，可能启动驱动程序，进程阻塞，CPU 切换。
    *   I/O 完成后产生中断，唤醒进程。
7.  服务结束，结果存入寄存器。
8.  返回内核总控，进行可能的进程调度。
9.  恢复现场，返回用户程序继续执行。

#### 常见系统调用类别 (UNIX)
*   进程控制 (Process Control)
*   文件管理 (File Management)
*   设备管理 (Device Management)
*   信息维护 (Information Maintenance)
*   通信 (Communication)

#### Windows 系统调用支持组件
*   **KERNEL32.dll**: 内存管理、程序装入、任务调度。
*   **GDI32.dll**: 图形文字输出、图像操作、窗口管理。
*   **USER32.dll**: 声音、时钟、鼠标键盘输入管理。

### 4.3 命令解释器 (Command Interpreter / Shell)

*   **作用**：解释执行用户输入的命令。
*   **存在形式**：
    *   DOS: OS 内核的一部分。
    *   UNIX/Linux: 特殊程序 (Shell)，登录时运行。
    *   Windows: Explorer.exe (图形化程序管理器)。

#### UNIX Shell 工作流程
1.  系统启动，1号进程为每个终端生成 tty 进程，运行登录程序。
2.  验证用户 ID/口令。
3.  执行 Shell。
4.  Shell 循环：
    *   读取输入 (`read` 系统调用)。
    *   解析命令。
    *   若是内置命令，直接处理。
    *   若是外部命令，在 `PATH` 目录下查找可执行文件。
    *   **创建子进程** (`fork`) 去运行该程序。
    *   等待子进程结束 (`wait`)。
    *   取下一条命令。

#### 用户命令与系统调用的关系
1.  **运行状态不同**：用户命令对应的实用程序在**用户态**运行；系统调用处理程序在**核心态**运行。
2.  **依赖关系**：实用程序在运行过程中可能会调用系统调用。
3.  **接口性质**：用户命令是**操作接口**；系统调用是**编程接口**。

### 4.4 图形用户界面 (GUI)
*   用图符、菜单表示命令。
*   用对话框提示参数选择或输入。
*   命令解释器演变为图形化的程序管理器。

## 5. 总结
*   **操作系统结构**：从简单结构向分层、微内核、模块化及混合结构发展，旨在平衡效率、可靠性、可维护性和可扩展性。
*   **运行模型**：现代 OS 多采用内核嵌入用户进程执行的模式（基于当前进程的核心栈）。
*   **用户接口**：
    *   **系统调用**是内核与用户程序交互的桥梁，通过 Trap 机制实现态切换。
    *   **命令解释器 (Shell)** 是用户与 OS 交互的操作界面，它本身也是一个用户程序，通过调用系统调用来完成功能。
