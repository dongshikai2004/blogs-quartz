# 操作系统 Lecture 13：文件与文件目录

## 1. 引言与文件系统概述

### 1.1 文件管理系统 (File System)
*   **定义**：操作系统中实现文件统一管理的一组软件和相关数据的集合，简称文件系统。
*   **核心目标**：
    *   实现“按名存取”（用户只需提供文件名）。
    *   提供方便的操作和统一的调用接口，屏蔽底层硬件细节。
*   **主要功能**：
    1.  **空间管理**：组织、分配、回收文件的存储空间。
    2.  **数据管理**：负责文件的存储、检索、共享和保护。
    3.  **映射转换**：将文件的逻辑块定位到外存的物理块。
    4.  **安全控制**：支持对文件访问的安全检查及读写权限控制。

### 1.2 与页式存储管理的对比
*   **页式存储**：进程分页存放于**主存**，**页表**记录逻辑页到物理页的映射。
*   **文件系统**：文件分块存放于**外存**，**文件控制块 (FCB)** 包含文件的定位信息。

## 2. 基本概念

### 2.1 文件 (File)
*   **定义**：记录在外存上的、具有符号名的、在逻辑上具有完整意义的一组相关信息项的集合。
*   **视角**：从用户角度看，文件是逻辑外存的最小分配单元。
*   **组成部分**：
    1.  **文件体**：文件真实的内容（数据）。
    2.  **文件说明 (属性/FCB)**：操作系统为了管理文件所用的信息，包括：
        *   文件名、文件类型
        *   文件存储位置（物理地址）
        *   文件大小
        *   访问权限、口令
        *   时间（创建、修改、访问）、日期
        *   用户标识 (UID/GID)

### 2.2 文件的分类
*   **按保存期限**：临时文件、档案文件、永久文件。
*   **按保护方式**：只读文件、读写文件、可执行文件、不保护文件。
*   **按性质和用途**：系统文件、库文件、用户文件。
*   **按后缀名区分**：
    *   `.exe, .com, .bin`: 可执行文件
    *   `.c, .cpp, .java`: 源代码
    *   `.txt, .doc`: 文本数据
    *   `.zip, .rar`: 压缩文件
    *   `.mpeg, .mov`: 多媒体文件

### 2.3 UNIX 系统中的文件分类
1.  **普通文件**：通常的数据或程序文件。
2.  **目录文件**：由目录项构成，用于维护文件系统结构。
3.  **特别文件**：
    *   字符型特别文件（输入/输出设备，如终端）。
    *   块特别文件（磁盘等块设备）。
    *   管道文件。
    *   *特点*：操作与普通文件类似（查找目录、验证权限、读写），但系统将其转换为对特定设备的操作。

### 2.4 常见文件系统类型
| 文件系统 | 描述 |
| :--- | :--- |
| **FAT** | MS-DOS使用，8.3文件名格式，分区限制2GB。 |
| **FAT32** | Win95+支持，突破2GB限制，支持长文件名(255字符)。 |
| **NTFS** | Windows NT+使用，安全性强，具备恢复 a功能。 using位示图管理簇。 |
| **Ext2/Ext3** | Linux使用，支持256字符文件名，最大4TB。使用块组+位示图。 |
| **HPFS** | OS/2使用，改善大目录访问时间，支持2TB。
| **UDF** | 符合ISO966标准，支持CD-ROM。 |

## 3. 文件的结构

文件结构分为**逻辑结构**（用户视角）和**物理结构**（存储视角）。文件系统的作用是在二者之间建立映射。

### 3.1 逻辑结构 (Logical Structure)
独立于物理特性，是用户直接处理的数据组织形式。
1.  **有结构的记录式文件**：由若干记录组成。
    *   **定长记录**：每个记录长度相同。
    *   **变长记录**：记录长度不同。 
2.  **无结构的流式文件**：由一串字符流构成，无内部结构。
    *   *注*：**UNIX系统**将所有文件视为流式文件，系统不进行格式处理。

### 3.2 文件的存取 (Access) 方式
1.  **顺序存取**：按顺序依次读写（适合磁带、流式文件）。
2.  **随机存取**：
    *   **直接法**：随意存取任意物理记录。
        *   对于定长记录，第 $i$ 个记录的位置：$Pointer = Offset + i \times Length$。
    *   **按键存取**：根据记录中的某个数据项（键，Key）来存取，如 Hash 结构。

### 3.3 物理结构 (Physical Structure)
文件在外存上的存放方法。假设物理块大小为 512 Byte 或 1 KB。

#### A. 连续存储 (Sequential/Contiguous Allocation)
*   **机制**：逻辑上连续的信息存放在编号连续的物理块上。目录项记录：`起始块号` + `长度`。
*   **优点**：
    *   实现简单。
    *   顺序访问效率高（寻道次数少）。
    *   支持定长记录的直接计算访问。
*   **缺点**：
    *   **外部碎片**严重。
    *   文件动态增长困难（需要预分配或移动文件）。
    *   不支持高效的随机查找（若不定长）。

#### B. 链接结构 (Linked Allocation)
*   **机制**：逻辑连续的信息存放在不连续的物理块中，每个块包含指向下一个块的指针。目录项记录：`起始块号`。
*   **优点**：
    *   无外部碎片，空间利用率高。
    *   易于文件动态扩充、插入和删除。
*   **缺点**：
    *   **不支持随机/直接存取**（必须从头遍历链表）。
    *   指针占用存储空间。
    *   可靠性低（指针损坏导致文件断裂）。
    *   随机访问时寻道次数多，效率低。

#### C. 索引结构 (Indexed Allocation)
*   **机制**：为每个文件建立一个**索引表**，记录逻辑块号到物理块号的映射。目录项记录：`索引表地址`。
*   **优点**：
    *   既支持顺序存取，也支持直接存取。
    *   无外部碎片，支持动态增长。
*   **缺点**：
    *   索引表本身占用存储空间。
    *   增加了一次访盘开销（读取索引表）。

#### D. 索引表的组织方式（针对大文件）
当文件很大，单个索引块放不下所有索引项时：
1.  **链接文件方式**多个索引块通过指针串联。
2.  **多重索引方式 (Multi-level Indexing)**：
    *   **一级索引**：索引表项指向数据块。
    *   **二级索引**：一级索引表项指向二级索引表，二级索引表项指向数据块。
    *   **三级索引**：以此类推。

#### E. UNIX 混合索引结构 (inode)
UNIX 使用 **inode** (Index Node) 存储文件控制信息（不含文件名）。`di_addr[40]` 采用混合索引：
*   **直接寻址 (Direct)**：前10-12个条目直接指向数据块。适用于小文件，速度快。
*   **一级间接寻址 (Single Indirect)**：指向一个索引块，该块包含多个数据块指针。
*   **二级间接寻址 (Double Indirect)**：指向一级索引块，再指向二级索引块，最后指向数据块。
*   **三级间接寻址 (Triple Indirect)**：进一步扩展。

*   **容量计算示例**（假设块大小1KB，索引项3字节，每块约341个索引项）：
    *   直接：~10 KB
    *   一级：~341 KB
    *   二级：~341 * 341 KB ≈ 116 MB
    *   三级：~341 * 341 * 341 KB ≈ 40 GB
    *   *总支持文件大小可达数十GB。*

*   **为什么 inode 中没有文件名？**
    *   为了实现**硬链接 (Hard Link)**。多个目录项可以指向同一个 inode，从而实现文件共享。文件名存储在目录文件中，而非 inode 中。

## 4. 文件的目录

### 4.1 文件控制块 (FCB) 与目录
*   **OS 观点**：文件 = FCB + 文件体。
*   **目录**：是 FCB 的集合。
*   **目录的功能**：
    1.  实现“按名存取”。
    2.  提高检索速度。
    3.  实现文件共享。
    4.  解决重名问题。

### 4.2 目录结构分类

#### 1. 一级目录结构 (Single-Level)
*   **结构**：整个系统只有一个目录表，包含所有文件的 FCB。
*   **优点**：简单。
*   **缺点**：
    *   查找速度慢（需遍历整个表）。
    *   **不允许重名**。
    *   不便于共享。
*   **适用**：单用户环境。

#### 2. 二级目录结构 (Two-Level)
*   **结构**：
    *   **主目录 (MFD, Master File Directory)**：记录每个用户的用户目录指针。
    *   **用户目录 (UFD, User File Directory)**：记录该用户所有文件的 FFCB。
*   **路径名**：`用户名/文件名`。
*   **优点**：解决了不同用户间的重名问题。
*   **缺点**：同一用户下仍不能重名；文件共享较困难。

#### 3. 多级目录结构 (Tree-Structured / Hierarchical)
*   **结构**：树形结构。根目录 -> 子目录 -> ... -> 文件。
*   **特点**：
    *   利于分类。
    *   允许不同目录下重名。
    *   **绝对路径**：从根目录开始（如 `/usr/lib/man`）。
    *   **相对路径**：从**当前目录 (Working Directory)** 开始。PCB 中保存当前目录信息，提高检索效率。
*   **优点**：检索速度快，解决重名，层次清晰。
*   **缺点**：路径名可能过长；跨目录共享文件较复杂。

#### 4. 无环图目录结构 (Acyclic Graph)
*   **目的**：更好地支持**文件共享**。
*   **机制**：允许一个文件同时属于多个目录（多条路径）。
*   **实现难点**：多个目录项指向同一文件，需保证 FCB 一致性。
*   **解决方案**：
    *   **硬链接 (Hard Link)**：
        *   基于 **inode**。
        *   多个目录项包含相同的 **inode 号**。
        *   inode 中有一个 `link count` (链接计数)。
        *   删除文件时，仅当 `count == 0` 时才真正释放空间。
    *   **符号链接 (Symbolic Link / Soft Link)**：
        *   创建一个特殊的文件（符号链文件），其内容是**目标文件的路径名字符串**。
        *   访问时，系统读取该字符串，重新从根目录查找目标文件。
        *   *缺点*：若原文件被删除，符号链接失效（悬空链接）；访问速度稍慢（需额外解析路径）。

## 5. 文件存储器空间布局与管理

### 5.1 存储布局
*   **文件卷 (Volume)**：可以是整个物理盘，也可以是分区，或多个盘组成的逻辑卷 each。
*   **分区**：
    *   **管理信息区**：存放超级块、位示图、inode 表、目录等。
    *   **文件数据区**：存放实际文件数据。

### 5.2 空闲空间管理方法

#### 1. 空闲区表 (Free Space Table)
*   **机制**：记录所有连续空闲区的`起始块号`和`长度`。
*   **适用**：连续分配方式。
*   **缺点**：表项可能很多，管理开销大。

#### 2. 位示图 (Bitmap / Bit Map)
*   **机制**：用一位 (bit) 表示一个物理块的状态（0=空闲，1=占用）。
*   **计算**：
    *   磁盘相对块号 = 字节号 $\times$ 8 + 位号
    *   字节号 = 块号 / 8 (商)
    *   位号 = 块号 % 8 (余数)
*   **优点**：
    *   结构简单，易于查找连续空闲块。
    *   尺寸固定，可常驻内存，分配/回收速度快。
*   **应用**：Linux Ext2 (块组内), Windows NTFS。

#### 3. 空闲块链 (Free Block Chain)
*   **机制**：将所有空闲块通过指针链接成一个链表。头指针存放在特定位置。
*   **优点**：不需要额外的分配表空间。
*   **缺点**：
    *   分配/回收时需读写磁盘以修改指针，开销大。
    *   不适合频繁操作。

#### 4. 成组链接法 (Grouped Linking) —— **UNIX/Linux 常用**
*   **机制**：
    *   将空闲块分成若干组（例如每组50块）。
    *   每组的**第一个空闲块**中存放：
        1.  本组空闲块的总数。
        2.  下一组所有Í空闲块的块号。
    *   **专用表 (Superblock area in memory)**：内存中维护当前可用的一组空闲块信息。
*   **分配过程**：
![[Pasted image 20260615211854.png]]
    1.  从内存专用表中取出一块。
    2.  若专用表为空 (`s-free == 0`)，则读取当前栈顶块号对应的磁盘块（即下一组的索引块）载入内存，更新专用表，再分配。
*   **释放过程**：
![[Pasted image 20260615211940.png]]
    1.  将回收块号填入内存专用表。
    2.  若专用表已满 (`s-free == 组大小`)，则将专用表内容写入刚回收的这个块中（使其成为新的索引块），重置专用表。
*   **优点**：
    *   大部分操作在**内存**中进行，效率高。
    *   不占用额外的磁盘空间存储空闲表（借用空闲块自身存储索引信息）。

### 5.3 总结对比
*   **UNIX**：采用 **inode** + **混合索引** + **成组链接法**管理空闲块。
*   **Linux Ext2**：采用 **块组** + **位示图** 管理空闲块。
*   **Windows NTFS**：采用 **位示图** (Cluster Bitmap) 管理空闲簇。
