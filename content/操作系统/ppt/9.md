根据提供的《操作系统 Lecture 9: 进程的死锁》课件内容，以下是对死锁相关知识的详尽整理：

### 1. 死锁的定义与性质

#### 1.1 定义
在一个进程集合中，若每个进程都在等待某些事件（指：释放资源）的发生，而这些事件又必须由这个进程集合中的某些进程来产生，就称该进程集合处于**死锁状态**。

#### 1.2 死锁产生的四个必要条件
系统出现死锁必须**同时满足**以下四个条件：
1.  **互斥 (Mutual Exclusion)**：必须存在需要互斥使用的资源
2.  **占有等待 (Hold and Wait)**：一定有占有资源而又等待其它资源的进程。
3.  **非剥夺 (No Preemption)**：系统中进程占有的资源在未主动释放时不可以被强行剥夺。
4.  **循环等待 (Circular Wait)**：存在一个进程集合 $\{P_0, P_1, ..., P_n\}$，其中 $P_i$ 等待 $P_{i+1}$ 占有的资源，而 $P_n$ 等待 $P_0$ 占有的资源，形成环路。

*逻辑关系*：$D \rightarrow C_1 \land C_2 \land C_3 \land C_4$。若要防止死锁 ($\neg D$)，只需破坏其中一个条件即可 ($\neg C_1 \lor \neg C_2 \lor \neg C_3 \lor \neg C_4 \rightarrow \neg D$)。

### 2. 解决死锁的策略

#### 2.1 置之不理法（鸵鸟政策）
*   **思想**：忽略死锁问题。
*   **优点**：简单，简化系统设计，节约成本。
*   **缺点**：安全性欠佳，可能导致系统崩溃或资源利用率严重下降。

#### 2.2 积极防御法（死锁预防）
*   **思想**：通过静态或动态策略，破坏产生死锁的四个必要条件之一，从根源上杜绝死锁。
*   **具体方法**：
    1.  **破坏互斥条件**：让资源共享使用（但许多资源如打印机必须互斥，此法局限性大）。
    2.  **破坏占有等待条件**：
        *   **静态预分配**：进程创建时一次性分配所有所需资源，否则不执行。缺点：资源浪费严重，并发性低。
        *   **释放再申请**：进程在申请新资源前，必须先释放已占有的所有资源。
    3.  **破坏非剥夺条件**：
        *   当进程 $P_i$ 申请资源 $r_i$ 得不到满足时，剥夺 $P_i$ 已占有的所有资源。
        *   或者剥夺占有 $r_i$ 的其他进程的资源分配给 $P_i$（需考虑被剥夺进程的状态）。
    4.  **破坏循环等待条件**：
        *   **资源顺序分配法**：给每类资源编号，进程只能按序号由小到大的顺序申请资源。若违反顺序则拒绝分配。

#### 2.3 事后处理法（死锁避免、检测与恢复）
*   **思想**：允许死锁产生的条件存在，但在运行时进行动态管理。
*   **分类**：
    *   **死锁避免**：在分配前判断是否安全，避免进入不安全状态。
    *   **死锁检测**：允许死锁发生，定期检测系统是否处于死锁状态。
    *   **死锁恢复**：检测到死锁后，采取措施解除。

### 3. 死锁避免 (Deadlock Avoidance)

#### 3.1 核心思想
*   在资源分配之前，系统判断若满足该请求，系统是否仍处于**安全状态**。
*   如果会进入不安全状态，则拒绝分配；否则予以分配。
*   **优点**：比预防策略更灵活，资源利用率更高。

#### 3.2 关键概念：安全状态
*   **安全状态**：指在某个时刻，存在一种进程执行序列（安全序列），使得系统按照该顺序逐次为每个进程分配其所需的最大资源后，每个进程都能顺利完成并释放资源。
*   **不安全状态**：不存在这样的安全序列。**注意：不安全状态不一定导致死锁，但死锁一定是不安全状态。**

#### 3.3 银行家算法 (Banker’s Algorithm)
由 Dijkstra提出，适用于多类资源系统。

**数据结构**：
*   $n$: 进程数, $m$: 资源种类数。
*   `Available[m]`: 可用资源向量。
*   `Max[n][m]`: 最大需求矩阵。
*   `Allocation[n][m]`: 已分配矩阵。
*   `Need[n][m]`: 需求矩阵，$Need[i][j] = Max[i][j] - Allocation[i][j]$。
*   `Request_i[m]`: 进程 $P_i$ 的请求向量。

**算法步骤**：
1.  **合法性检查**：若 $Request_i \le Need_i$，转步骤2；否则出错（超过最大声明值）。
2.  **资源充足性检查**：若 $Request_i \le Available$，转步骤3；否则进程等待。
3.  **试探分配**：修改数据结构：
    *   $Available = Available - Request_i$
    *   $Allocation_i = Allocation_i + Request_i$
    *   $Need_i = Need_i - Request_i$
4.  **安全性检测**：执行安全算法检查系统是否安全。
    *   若**安全**：正式分配。
    *   若**不安全**：撤销试探分配，恢复原状，让进程等待。

**安全性检测算法**：
1.  初始化：$Work = Available$, $Finish[i] = false$。
2.  寻找进程 $P_i$ 满足：$Finish[i] == false$ 且 $Need_i \le Work$。
    *   若找到：$Work = Work + Allocation_i$, $Finish[i] = true$, 返回步骤2。
    *   若找不到：转步骤3。
3.  若所有 $Finish[i] == true$，则系统安全；否则不安全。

### 4. 死锁检测 (Deadlock Detection)

#### 4.1 检测算法
类似于安全性算法，但针对的是当前已发生的状态。

**步骤**：
1.  初始化：$Work = Available$。
    *   若 $Allocation_i \neq 0$，则 $Finish[i] = false$；否则 $Finish[i] = true$。
2.  寻找进程 $P_i$ 满足：$Finish[i] == false$ 且 $Request_i \le Work$。
    *   若找到：$Work = Work + Allocation_i$, $Finish[i] = true$, 返回步骤2。
    *   若找不到：转步骤3。
3.  若存在 $Finish[i] == false$，则 $P_i$ 为死锁进程，系统处于死锁状态。

#### 4.2 检测工具：资源分配图 (Resource Allocation Graph)
*   **节点**：圆圈表示进程，方块表示资源类（方块内圆点表示资源实例）。
*   **边**：
    *   资源 $\rightarrow$ 进程：分配边（资源已分配给进程）。
    *   进程 $\rightarrow$ 资源：请求边（进程正在申请资源）。

**化简算法（死锁定理）**：
*   **定理**：系统处于死锁状态 $\iff$ 资源分配图是**不可完全化简**的。
*   **化简步骤**:
    1.  找一个既非阻塞（请求能被满足）又非独立（有连接边）的进程 $P_i$。即 $P_i$ 的所有请求边对应的资源都有足够空闲实例。
    2.  若找到，假定 $P_i$ 运行完毕并释放资源，删去与 $P_i$ 相连的所有边（$P_i$ 变为孤立节点）。
    3.  重复上述过程。
    4.  若最终所有进程都成为孤立节点，则无死锁；否则，剩余未化简的部分即为死锁。

*   **特殊情况**：若每类资源只有一个实例，则图中存在**环路**即意味着死锁。

### 5. 死锁恢复 (Deadlock Recovery)

一旦检测到死锁，可采取以下措施解除：

1.  **重新启动**：
    *   粗暴方法，损失大，工作白费。
2.  **撤消进程 (Process Termination)**：
    *   **一次性撤消**所有死锁进程。
    *   **逐个撤消**：根据优先级、已运行时间、剩余运行时间、占用资源多少等指标，选择代价最小的进程撤消，直到死锁解除。
3.  **剥夺资源 (Resource Preemption)**：
    *   从死锁进程中剥夺资源分配给其他进程，直到打破循环等待。
    *   需考虑被剥夺进程的后续处理（如回退）。
4.  **进程回退 (Process Rollback)**：
    *   利用**检查点 (Checkpoint)** 机制。
    *   将死锁进程回退到之前的某个安全状态（检查点），重新执行。
    *   广泛应用于数据库管理系统 (DBMS)。

---

### 6. 死锁的综合处理

在实际操作系统中，通常结合多种策略：
*   **资源分类处理**：
    *   主存、CPU：采用**剥夺法**（破坏非剥夺条件）。
    *   辅存：采用**预分配法**（破坏占有等待条件）。
    *   其他资源：采用**检测后处理**。
*   **资源顺序分配**：整体上采用资源有序分配法破坏循环等待。
*   **设立资源阈值**：当剩余资源低于阈值时，只允许老进程申请，禁止新进程申请，以防资源耗尽导致死锁。

### 总结图示逻辑
*   **预防**：破坏4个必要条件之一（静态/动态）。
*   **避免**：银行家算法，确保始终处于安全状态。
*   **检测**：资源分配图化简或检测算法，发现死锁。
*   **恢复**：撤消进程、剥夺资源、回退进程。
