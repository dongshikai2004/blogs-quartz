## 1. 回顾：进程同步与互斥基础

### 1.1 核心问题
*   **并发带来的问题**：多进程/线程并发执行时，由于共享数据或资源，可能导致数据不一致、丢失等问题。
*   **关键关系**：
    *   **互斥 (Mutual Exclusion)**：同一时刻只能有一个进程访问临界资源。
    *   **同步 (Synchronization)**：进程之间存在先后顺序的依赖关系（如生产者必须先生产，消费者才能消费）。

### 1.2 解决方案演进
1.  **软件方法**：通过算法逻辑控制（复杂，易出错）。
2.  **硬件指令**：利用原子指令（如Test-and-Set），通常对上层应用不直接开放，由OS底层使用。
3.  **信号量机制 (Semaphores)**：操作系统提供的简化并发设计的机制。
    *   **定义**：一种特殊的数据结构（整型变量），代表可用资源的数量或状态。
    *   **操作限制**：只能通过 **P操作 (Wait/Down)** 和 **V操作 (Signal/Up)** 改变其值，严禁加减乘除等其他运算。
    *   **P操作**：申请资源。若资源不足（S<=0），进程阻塞等待。
    *   **V操作**：释放资源。唤醒等待该资源的进程。

> **注意**：上节课遗留的同步节点遗漏问题，请同学们课后自行修正代码，确保同步逻辑完整。

## 2. PV操作的经典应用场景

### 2.1 案例：大数据累加（线程协作）
**场景**：两个线程并行处理数组累加，最后汇总结果。

#### 方案 A：互斥访问（粗粒度）
*   **逻辑**：两个线程完全并行，但访问共享变量（`sum`, `data[]`, 下标 `i`）时必须互斥。
*   **实现**：将临界区代码包裹在 `P(mutex)` 和 `V(mutex)` 之间。
*   **特点**：保证了正确性，但并发度低（本质上是串行执行临界区）。

#### 方案 B：同步协作（细粒度/分治）
*   **逻辑**：
    *   线程A：负责前500个数累加 -> 等待线程B完成 -> 汇总两者结果。
    *   线程B：负责后500个数累加 -> 完成后通知线程A。
*   **关系变化**：从纯互斥变为 **同步关系**。
*   **实现**：
    *   线程B完成后执行 `V(sync_sem)`。
    *   线程A在汇总前执行 `P(sync_sem)`。
*   **启示**：根据任务特性，灵活选择互斥或同步，甚至组合使用。

### 2.2 经典问题一：生产者-消费者问题 (Producer-Consumer)

#### 问题分析
*   **角色**：一组生产者进程（生成数据），一组消费者进程（处理数据）。
*   **缓冲区**：大小为 N 的共享缓冲池。
*   **约束关系**：
    1.  **互斥**：对缓冲区的读写操作必须互斥（防止数据混乱）。
    2.  **同步**：
        *   缓冲区满时，生产者必须等待（有空位才能写）。
        *   缓冲区空时，消费者必须等待（有数据才能读）。

#### 信号量定义
```c
semaphore mutex = 1;      // 互斥信号量，保护缓冲区访问
semaphore empty = N;      // 同步信号量，表示空闲缓冲区数量
semaphore full = 0;       // 同步信号量，表示已填充缓冲区数量
```

#### 伪代码实现
**生产者 (Producer):**
```c
while (true) {
    produce an item;
    P(empty);   // 1. 申请空位 (同步)
    P(mutex);   // 2. 申请进入临界区 (互斥)
    
    add item to buffer;
    
    V(mutex);   // 3. 离开临界区
    V(full);    // 4. 增加满位计数，唤醒消费者 (同步)
}
```

**消费者 (Consumer):**
```c
while (true) {
    P(full);    // 1. 申请数据 (同步)
    P(mutex);   // 2. 申请进入临界区 (互斥)
    
    remove item from buffer;
    
    V(mutex);   // 3. 离开临界区
    V(empty);   // 4. 增加空位计数，唤醒生产者 (同步)
}
```

#### ⚠️ 关键考点：P操作的顺序
*   **问题**：生产者中的 `P(empty)` 和 `P(mutex)` 能否交换顺序？
*   **答案**：**绝对不能**。
*   **原因（死锁风险）**：
    *   若先执行 `P(mutex)`：生产者获得了缓冲区的互斥锁。
    *   接着执行 `P(empty)`：如果缓冲区已满，生产者阻塞等待空位。
    *   **后果**：生产者持有锁并阻塞，消费者无法获取锁 (`P(mutex)`) 来取出数据以释放空位。**互相等待，形成死锁**。
*   **原则**：**同步P操作在前，互斥P操作在后**。
*   **V操作顺序**：可以交换，因为V操作是释放资源，不会导致阻塞等待。

### 2.3 经典问题二：读者-写者问题 (Readers-Writers)

#### 问题分类
1.  **读者优先 (Reader Preference)**：追求并发极大化。只要有读者在读，后续读者可直接进入，写者需等待所有读者读完。
2.  **写者优先 (Writer Preference)**：追求数据时效性。一旦有写者请求，后续读者需等待，防止写者“饿死”。

#### 读者优先的实现思路 (第一类)
*   **目标**：允许多个读者同时读，但读写互斥，写写互斥。
*   **信号量**：
    *   `wrt` (mutex for writing): 初值1，用于读写互斥、写写互斥。
    *   `mutex` (for read_count): 初值1，用于保护 `read_count` 变量的互斥访问。
    *   `read_count`: 整数，记录当前正在读的读者数量，初值0。

**写者 (Writer):**
```c
P(wrt);
    write data;
V(wrt);
```

**读者 (Reader):**
```c
P(mutex);
    read_count++;
    if (read_count == 1) 
        P(wrt);  // 第一个读者负责锁定写者
V(mutex);

    read data;   // 多个读者可并发执行此段

P(mutex);
    read_count--;
    if (read_count == 0) 
        V(wrt);  // 最后一个读者负责释放写者锁
V(mutex);
```
*   **思考题**：为何 `read_count` 的增减需要单独的 `mutex` 保护？因为 `read_count` 本身也是共享变量。为何分为两段 `P/V(mutex)`？为了缩小临界区，提高并发度（如果在整个读取过程都持有 `mutex`，则读者之间也变成了串行）。

### 2.4 经典问题三：哲学家就餐问题 (Dining Philosophers)

*   **场景**：5个哲学家，5根筷子。每人需左右两根筷子才能吃饭。
*   ** naive 解法导致的死锁**：所有人同时拿起左边的筷子，等待右边的筷子 -> **循环等待** -> 死锁。
*   **解决思路**（课后作业）：
    1.  限制最多4人同时进餐。
    2.  奇数号哲学家先拿左筷，偶数号先拿右筷（破坏循环等待）。
    3.  仅当左右筷子都可用时才拿起（原子操作）。

## 3. 高级同步机制：管程 (Monitor)

### 3.1 为什么引入管程？
*   **PV操作的缺陷**：
    *   P/V操作分散在各个进程中，代码可读性差。
    *   容易出错（如忘记V操作导致永久阻塞，P操作顺序错误导致死锁）。
    *   维护困难，修改同步逻辑需改动多处代码。
*   **管程的优势**：
    *   **封装性**：将共享数据结构及其操作封装在一起（类似面向对象中的 Class）。
    *   **自动互斥**：管程保证同一时刻只有一个进程能执行管程内的过程。
    *   **易用性**：开发者只需调用管程提供的接口，无需关心底层的P/V细节。

### 3.2 管程的组成
1.  **共享数据**：管程内部定义的局部变量（如缓冲区、计数器）。
2.  **操作过程 (Procedures)**：对共享数据进行访问的一组函数（如 `put()`, `get()`）。
3.  **条件变量 (Condition Variables)**：
    *   用于进程间的同步（阻塞与唤醒）。
    *   操作：`wait(condition)` (阻塞当前进程), `signal(condition)` (唤醒等待该条件的进程)。

### 3.3 基于管程的生产者-消费者实现
```java
monitor ProducerConsumer {
    condition notFull, notEmpty;
    int count = 0;
    buffer[N];

    procedure put(item) {
        if (count == N) wait(notFull); // 缓冲区满，等待
        add item to buffer;
        count++;
        signal(notEmpty); // 唤醒可能等待的消费者
    }

    procedure get() {
        if (count == 0) wait(notEmpty); // 缓冲区空，等待
        remove item from buffer;
        count--;
        signal(notFull); // 唤醒可能等待的生产者
    }
}
```
*   **调用方代码**：极其简洁，只需调用 `PC.put(item)` 或 `PC.get()`。

## 4. 进程间通信 (Inter-Process Communication, IPC)

### 4.1 通信层级
1.  **低级通信**：传递控制信息（如信号量、状态标志）。效率低，数据量小，编程复杂。
2.  **高级通信**：传递大量数据。效率高，对用户透明。

### 4.2 高级通信方式

#### A. 共享存储 (Shared Memory) - 间接通信
*   **原理**：在内存中开辟一块公共区域。
*   **特点**：
    *   速度最快（无需内核拷贝数据）。
    *   **必须配合同步互斥机制**（如信号量、管程）以防止竞态条件。
    *   发送者和接收者无需知道对方身份（解耦）。
*   **类比**：快递柜。快递员存，用户取，彼此不见面。

#### B. 消息传递 (Message Passing) - 直接/间接通信
*   **直接通信**：
    *   明确指定接收者ID (PID)。
    *   API示例 (Unix): `send(pid, message)`, `receive(&message)`.
    *   点对点，耦合度高。
*   **间接通信 (信箱/Mailbox)**：
    *   消息发送到中间实体（信箱/消息队列）。
    *   接收者从信箱取消息。
    *   **优点**：支持一对多、多对多通信；解耦。
    *   **底层实现**：每个进程PCB中维护一个消息队列指针。操作系统通过 `mutex` 和 `semaphore` 管理信箱的互斥访问和数据有无。

#### C. 管道通信 (Pipe)
*   **原理**：基于文件系统，以数据流形式传输。
*   **特点**：
    *   **单向通信**：半双工。双向需建立两个管道。
    *   **匿名管道**：主要用于父子进程间通信（`fork`后继承文件描述符）。
    *   **命名管道 (FIFO)**：可用于任意相关或不相关进程间通信。
    *   操作方式：像读写文件一样使用 `read()` / `write()`。

## 5. 死锁 (Deadlock)

### 5.1 定义与现象
*   **定义**：一组进程中的每一个进程都在等待仅由该组进程中其他进程才能引发的事件，导致所有进程无限期阻塞。
*   **生活实例**：交通路口堵塞、哲学家就餐、系统卡死。
*   **危害**：系统吞吐量下降，资源浪费，严重时系统崩溃。

### 5.2 死锁产生的四个必要条件 (缺一不可)
1.  **互斥条件 (Mutual Exclusion)**：资源一次只能被一个进程占用。
2.  **占有并等待 (Hold and Wait)**：进程持有至少一个资源，并等待获取其他被占用的资源。
3.  **非剥夺条件 (No Preemption)**：资源不能被强制从持有者手中抢占，只能由持有者自愿释放。
4.  **循环等待 (Circular Wait)**：存在一个进程等待链 $P_1 \to P_2 \to ... \to P_n \to P_1$。

### 5.3 死锁处理策略

#### 策略一：鸵鸟算法 (Ostrich Algorithm)
*   **做法**：忽略死锁，假设它很少发生。
*   **适用**：个人PC、通用操作系统（重启成本低）。
*   **缺点**：不安全，不适合关键任务系统。

#### 策略二：死锁预防 (Prevention) - 静态/积极
*   **核心思想**：破坏四个必要条件中的至少一个。
    1.  **破坏互斥**：将独占资源改为共享（如SPOOLing技术虚拟打印机），但很多资源本质上不可共享。
    2.  **破坏占有并等待**：
        *   **静态分配**：进程运行前一次性申请所有所需资源。
        *   **缺点**：资源利用率低，可能导致饥饿。
    3.  **破坏非剥夺**：允许强行剥夺资源（适用于CPU、内存，不适用于打印机等）。
    4.  **破坏循环等待**：**有序资源分配法**。给资源编号，进程只能按序号递增顺序申请资源。

#### 策略三：死锁避免 (Avoidance) - 动态/积极
*   **核心思想**：允许必要条件存在，但在资源分配时进行判断，确保系统始终处于**安全状态**。
*   **关键概念**：
    *   **安全状态**：存在一个**安全序列** $\langle P_1, P_2, ..., P_n \rangle$，使得系统按此顺序为进程分配资源，所有进程都能顺利完成。
    *   **不安全状态**：不存在安全序列。**注意：不安全状态 $\neq$ 死锁，但不安全状态有可能导致死锁。**
*   **著名算法：银行家算法 (Banker's Algorithm)**
    *   **灵感来源**：银行借贷规则（客户声明最大需求，银行确保剩余资金能满足至少一个客户完成项目并还款）。
    *   **数据结构**：
        *   `Available[j]`: 可用资源数量。
        *   `Max[i][j]`: 进程 $i$ 对资源 $j$ 的最大需求。
        *   `Allocation[i][j]`: 进程 $i$ 已分配的资源 $j$。
        *   `Need[i][j]`: 进程 $i$ 还需要的资源 $j$ ($Need = Max - Allocation$).
    *   **算法流程**：
        1.  进程发起请求 `Request`。
        2.  检查 `Request <= Need` 且 `Request <= Available`。
        3.  **试探性分配**：修改 `Available`, `Allocation`, `Need`。
        4.  **安全性检查**：运行安全性算法，寻找是否存在安全序列。
            *   若存在：正式分配。
            *   若不存在：恢复原状，进程等待。
    *   **优缺点**：比预防更灵活，资源利用率更高；但开销大，需预先知道最大需求。

#### 策略四：死锁检测与恢复 (Detection & Recovery)
*   **检测**：定期运行检测算法（构建资源分配图，检测环路）。
*   **恢复**：
    1.  **终止进程**：杀死所有死锁进程，或逐个杀死直到打破死锁。
    2.  **资源剥夺**：从某些进程抢占资源分配给其他进程（需考虑回滚代价）。
