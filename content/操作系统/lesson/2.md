## 2. 核心概念回顾：中断与异常
> **助教提示**：中断和异常是操作系统内核运行的触发机制，是理解 OS 如何响应事件的基础。

### 2.1 定义与目的
- **定义**：CPU 暂停正在执行的程序，转去处理异步（中断）或同步（异常）事件。
- **引入目的**：
  - 实现 **CPU 计算与 I/O 操作的并行处理**。
  - 提高系统资源利用率（例如：通道完成 I/O 后通过中断通知 CPU，CPU 无需轮询）。
- **处理流程**：
  1. 保存当前程序现场（Context）。
  2. 转入中断/异常处理子程序（分析原因、执行处理）。
  3. 恢复现场。
  4. 继续执行原程序。

### 2.2 分类与区别
| 特性 | **中断 (Interrupt)** | **异常 (Exception)** |
| :--- | :--- | :--- |
| **别名** | 外中断 | 内中断、例外 (Exception)、陷入 (Trap) |
| **来源** | **外部硬件设备** (如 I/O 完成、时钟信号) | **当前运行程序内部** (如指令错误、系统调用) |
| **与指令关系** | 与当前执行指令无关 | 与当前执行指令直接相关 |
| **可屏蔽性** | **可屏蔽** (Maskable) | **不可屏蔽** (Unmaskable) |
| **常见场景** | 键盘输入、网卡数据到达、定时器 | 除零错误、越界访问、缺页异常、Debug 断点、**系统调用** |

## 3. 操作系统的系统架构
### 3.1 操作系统的核心功能模块
操作系统作为核心系统软件，必须包含以下模块：
1.  **系统初始化**：启动计算机，准备上层应用运行环境。
2.  **进程管理 (CPU 管理)**：创建进程、进程切换、同步通信、保护现场。
3.  **存储管理**：内存分配、加载程序、避免内存串扰。
4.  **文件管理**：文件的存储、命名、检索、资源管理器底层支持。
5.  **外设管理**：管理键盘、显示器、网卡、磁盘等 I/O 设备。

### 3.2 操作系统内核 (Kernel)
- **定义**：操作系统中驻留内存、运行在**核心态**、具备访问所有硬件资源权限的可信软件部分。
- **基本任务**：
  1.  资源抽象（将硬件/软件资源抽象为实体）。
  2.  资源分配。
  3.  资源共享。
- **内核特性**：
  - **中断驱动**：被动响应上层请求。
  - **不可抢占**：内核执行时，其他应用需等待。
  - **屏蔽中断**：部分内核代码执行时关闭中断以保证原子性。
  - **特权指令**：可执行访问硬件的特权指令。

### 3.3 操作系统结构设计演变
| 结构类型 | 特点 | 优点 | 缺点 | 代表系统 |
| :--- | :--- | :--- | :--- | :--- |
| **简单结构** | 整体结构，模块间函数直接调用，耦合紧密。 | 开发简单，早期易理解。 | 结构不清晰，难调试，可靠性低。 | DOS, 早期 UNIX |
| **分层结构** | 类似网络七层协议，单向调用，不能跨层。 | 结构清晰，易于调试和维护。 | 效率较低（层层传递开销）。 | THE 系统 |
| **单内核 (Monolithic)** | 核心功能全部在内核空间，模块间直接函数调用。 | **运行效率高**。 | 内核庞大，移植性差，扩展复杂。 | 传统 UNIX, Linux (主要) |
| **微内核 (Microkernel)** | 内核仅保留最基本功能 (中断、调度)，其他服务移至用户态。 | 内核小，易扩展，易移植，模块隔离好。 | **执行效率低** (需 Client-Server 通信)。 | Minix, Mach, 现代趋势 |
| **模块化结构** | 核心部件加载，扩展服务作为动态链接按需加载。 | 兼顾效率与灵活性。 | 管理复杂度适中。 | Solaris, Linux (部分机制), macOS |
| **虚拟机结构** | 通过分时和多道程序设计，将物理资源虚拟为多个逻辑系统。 | 支持多系统共存，资源隔离。 | 有一定性能开销。 | VMware, IBM VM |

> **助教提示**：没有绝对完美的架构，设计是在**效率、空间、可维护性**之间做平衡 (Balance)。现代硬件速度提升，微内核架构因安全性和稳定性优势成为趋势之一。

### 3.4 操作系统设计规范
- **高效率**：底层服务响应快。
- **高可靠性**：作为地基，假定 OS 本身不犯错（可信软件）。
- **可维护性/可用性**：具备故障恢复能力。
- **可移植性**：适配不同硬件平台。
- **可扩展性**：适应新硬件和新应用。
- **安全性**：保护资源不被非法访问。

## 4. 操作系统的运行模式与接口
### 4.1 运行模式
1.  **非进程内核模型**：内核独立运行，用户进程通过中断/陷入启动内核。
2.  **用户进程内执行模型**（本课程重点）：操作系统功能代码嵌入到用户进程中执行（内核态代码在用户进程地址空间运行）。
3.  **微内核模型**：核心程序仅含中断、调度等，其他功能作为独立用户态进程执行。

### 4.2 用户接口 (User Interface)
操作系统通过两类接口提供服务：
1.  **操作接口**（面向普通用户）：
    - **命令行 (CLI)**：通过 Shell 解释器交互（Linux/UNIX 常用）。
    - **图形界面 (GUI)**：所见即所得（Windows 常用，如 `explorer.exe` 进程负责界面翻译）。
2.  **编程接口**（面向开发人员）：
    - **API (Application Programming Interface)**：系统调用的集合。
    - **作用**：上层应用通过 API 请求 OS 服务，无需直接操作硬件。
    - **差异性**：不同 OS (Windows vs Linux) 的 API 不同（函数名、参数、二进制格式不同），因此应用软件需区分平台安装包。

### 4.3 系统调用 (System Call) 机制
- **为什么需要系统调用？**
  - 硬件访问权限必须控制在 OS 内核手中（安全性、可靠性）。
  - 上层应用通过封装好的 API 间接访问硬件（如 `printf` 底层调用 `write`）。
- **实现原理**：
  1.  **用户态 vs 核心态**：用户程序运行在用户态，系统程序运行在核心态。
  2.  **Trap 指令 (陷入指令)**：
      - 编译器在编译系统调用时自动嵌入 `TRAP` 指令。
      - 执行 `TRAP` 触发状态切换：**用户态 (1) → 核心态 (0)**。
  3.  **执行流程**：
      - 保护现场。
      - 切换 CPU 状态位。
      - 根据**系统调用号**查表进入对应内核处理程序。
      - 参数通过**寄存器**传递。
      - 执行完毕后恢复现场，切换回用户态。
  4.  **异步性**：如磁盘 I/O，CPU 发出请求后可转去执行其他任务，完成后通过中断通知。

## 5. 进程管理引入 (Process Management)
> **助教提示**：进程是操作系统最核心的概念，后续内存、文件、设备管理均围绕进程展开。

### 5.1 为什么要引入进程？
- **程序的顺序执行特性**：
  - 顺序性、封闭性、结果可再现性。
  - 适合单道程序设计，但资源独占，效率低。
- **多道程序设计的需求**：
  - **并发性**：宏观并行，微观串行。
  - **共享性**：资源被多个程序共享。
- **并发执行带来的问题**：
  - 失去封闭性。
  - **结果不可再现性**（例如：两个进程共享变量 N，执行顺序不同导致结果不同）。
- **解决方案**：引入**进程 (Process)** 概念，描述程序执行的**动态过程**。

### 5.2 进程的定义与特性
- **定义**：
  - 程序的一次执行过程。
  - 系统进行资源分配和调度的基本单位。
- **特性**：
  1.  **动态性**：有生命周期（创建→运行→消亡）。
  2.  **独立性**：独立运行的基本单位。
  3.  **并发性**：多个进程可交替执行。
  4.  **结构性**：由 PCB、程序段、数据段组成。
- **进程 vs 程序**：
  - 程序是**静态**的（磁盘上的代码），进程是**动态**的（内存中的运行实体）。
  - 一个程序可对应多个进程（如打开多个浏览器窗口）。
  - 资源分配对象是进程，而非程序。

### 5.3 进程的组成
1.  **进程控制块 (PCB, Process Control Block)**：
    - **进程存在的唯一标识**（类似身份证）。
    - 存放于**系统空间**（内核区），由 OS 统一管理。
    - **包含信息**：
      - 标识信息（PID、用户 ID、父进程 ID）。
      - 现场信息（寄存器内容、程序计数器、栈指针）。
      - 控制信息（状态、优先级、队列指针、资源占用）。
    - *不同 OS 实现不同*：UNIX (`proc`), Linux (`task_struct`), Windows (`EPROCESS`)。
2.  **进程映像 (Process Image)**：
    - 包含程序代码 (Code)、数据 (Data)、工作区 (Stack)。
    - 存放于**用户空间**。
3.  **PCB 的组织方式**：
    - 线性表、链表、树状结构等（存放在系统空间）。

### 5.4 进程的状态与转换
- **三种基本状态**：
  1.  **就绪态 (Ready)**：已获得除 CPU 外的所有资源，等待 CPU 调度。
  2.  **运行态 (Running)**：正在 CPU 上执行。
  3.  **阻塞态 (Blocked/Waiting)**：因等待某事件（如 I/O 完成）而暂停，即使有 CPU 也无法执行。
- **状态转换**：
  - 就绪 → 运行：进程被调度。
  - 运行 → 就绪：时间片用完（分时系统）。
  - 运行 → 阻塞：请求 I/O 或等待事件。
  - 阻塞 → 就绪：I/O 完成或事件发生。
- **生命周期**：
  - **创建**：分配 PCB、分配内存、加载代码、初始化、插入就绪队列。
  - **终止**：回收资源、关闭文件、释放内存、撤销 PCB。
  - **系统调用支持**：`fork`, `wait`, `exit`, `sleep` 等（Linux 示例）。
