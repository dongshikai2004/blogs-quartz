## 一、课程回顾：进程调度 (Process Scheduling)

### 1. 调度核心概念
- **目的：** 在多任务操作系统中，合理、有序地分配有限的 CPU 资源，保障多个任务共享 CPU 并正常运行。
- **调度时机：** 操作系统底层决定何时进行调度（如进程创建、终止、阻塞、时间片用完等）。
- **常见调度算法：**
  - FCFS (先来先服务)
  - SPF/SPN (短进程优先)
  - 优先级调度
  - 多级反馈队列 (Multilevel Feedback Queue)

### 2. 通用操作系统的调度策略
- **主流策略：** 现代通用操作系统（如 Windows, Linux）多采用 **多级反馈队列 + 动态优先级** 策略。
- **Linux 调度策略调研分享（课堂作业反馈）：**
  - **普通调度 (Normal)：** 默认采用 **CFS (Completely Fair Scheduler，完全公平调度)**。
    - **核心机制：** 使用 **红黑树** 管理进程。
    - **关键指标：** `vruntime` (虚拟运行时间)。
      - 进程运行时，`vruntime` 增加；不运行时，保持不变。
      - 优先级越高，`vruntime` 增加得越慢。
      - 调度器总是选择红黑树中 `vruntime` 最小（即执行时间最少）的进程执行。
    - **设计理念：** 通过虚拟方式让 CPU 表现为“完美 CPU"，N 个进程每个理论上分到 1/N 的资源。
  - **实时调度 (Real-time)：**
    - `SCHED_FIFO` (先进先出，实时静态优先级)
    - `SCHED_RR` (时间片轮转，实时静态优先级)
    - `SCHED_DEADLINE` (实时任务模型，有明确时间约束)
  - **扩展调度：** 策略灵活，可兼容，不同任务类型选择不同策略。

> **助教提示：** 学习操作系统不应仅限于记忆知识点，更要理解背后的设计机制（如：为什么要分多种调度策略？是为了针对不同进程类型优化系统整体性能）。这种分析问题的能力可迁移至科研及 AI 应用（如 Prompt 设计、问题拆解）。

## 二、线程 (Thread)

### 1. 引入线程的目的
- **减小并行粒度：** 进程间并行粒度太大，线程允许 **同一个进程内部** 的不同任务之间并行。
- **提高并发度：** 不仅进程间可并行，进程内线程也可并行。

### 2. 线程与进程的关系
- **包含关系：** 线程从属于进程。一个进程可包含若干个线程。
- **资源分配：**
  - **进程：** 资源分配的基本单位（拥有独立地址空间、代码段、数据段、PCB 等）。
  - **线程：** CPU 调度的基本单位。
- **共享与独立：**
  - **共享：** 同一进程内的线程共享进程的资源（内存、文件描述符等）。
  - **独立：** 每个线程拥有独立的 **TCB (Thread Control Block)**、寄存器组、栈空间、程序计数器 (PC)、线程 ID。

### 3. 引入线程的优势
1.  **开销小：** 线程切换只需保存少量上下文（TCB），比进程切换（PCB + 内存映射等）开销小。
2.  **切换快：** 上下文信息少，切换效率高。
3.  **通信效率高：** 同一进程内线程通过共享内存通信，无需内核介入。
4.  **并发度高：** 进程内部可多线程并发。

### 4. 线程实现机制
- **用户级线程 (User-Level Thread)：** 由用户程序管理调度，内核 unaware。
- **内核级线程 (Kernel-Level Thread)：** 由操作系统内核管理调度（如 Windows, Linux 默认）。
- **混合式：** 结合两者（如 Solaris）。

> **注意：** 同一进程内的线程切换很快；不同进程的线程切换等同于进程切换，开销大。多线程共享资源带来了 **数据一致性** 问题（后续重点）。

## 三、并发执行 (Concurrent Execution)

### 1. 并发的需求层面
1.  **程序设计层面：** 利用 OS 提供的并发机制（多进程/多线程）提高应用执行效率。
2.  **操作系统内核层面：** 内核程序自身也需并发（如 I/O 与 CPU 计算并发）。

### 2. 并发带来的问题
并发执行主要引出两类制约关系：
1.  **同步 (Synchronization)：**
    - **定义：** 为共同完成任务，伙伴进程间在某些位置上需要协调工作的先后顺序（直接制约关系）。
    - **例子：** 任务 S1、S2 完成后，才能开始 S3。S3 必须等待 S1、S2。
2.  **互斥 (Mutual Exclusion)：**
    - **定义：** 进程竞争使用独占资源（间接制约关系）。
    - **例子：** 多个进程竞争打印机、共享变量、共享缓冲区。同一时刻只允许一个进程访问。

### 3. 并发程序设计方法
- **伪代码表示：** 使用 `parbegin` ... `parend` 表示并发块。
  ```text
  parbegin
      S1;
      S2;
  parend
  S3; // S1, S2 并发完成后执行 S3
  ```
- **实现方式：**
  1.  顺序程序 + 自动并行化工具。
  2.  并发程序设计语言（编译器支持）。
  3.  传统语言 + OS API（如 `fork`, `pthread` 等）。

### 4. 经典并发问题模型
1.  **临界段访问问题：** 对临界资源的互斥访问。
2.  **生产者 - 消费者问题 (Producer-Consumer)：**
    - **场景：** 缓冲区 (Buffer) 的读写。
    - **生产者：** 写数据入缓冲区。
    - **消费者：** 从缓冲区取数据。
    - **问题：** 对缓冲区链表指针（如 `first`）的访问必须互斥，否则数据丢失或链表断裂。

## 四、临界区与互斥实现

### 1. 基本概念
- **临界资源 (Critical Resource)：** 一次仅允许一个进程使用的资源（如打印机、共享变量）。
- **临界区 (Critical Section, CS)：** 进程中访问临界资源的那段代码。

### 2. 临界区调度准则
1.  **互斥进入：** 同一时刻最多一个进程在临界区。
2.  **让权等待：** 无法进入时，释放 CPU，进入阻塞态（避免忙等）。
3.  **有空让进：** 临界区空闲时，允许申请进程进入。
4.  **有限等待：** 进程申请进入临界区后，应在有限时间内获得批准（避免饥饿）。

### 3. 互斥实现方案

#### (1) 软件实现方法
*要求程序员在代码中额外添加控制逻辑，复杂且易错。*
- **Dekker 算法：** 结合 `turn` (令牌) 和 `flag` (意愿)。解决了互斥和死锁，但逻辑复杂。
- **Peterson 算法：** Dekker 的简化版，代码更简洁。
  ```c
  // 伪代码逻辑
  flag[i] = true;       // 我想进
  turn = j;             // 让你先
  while (flag[j] && turn == j); // 忙等，直到对方不想进或轮到我
  // Critical Section
  flag[i] = false;      // 我出来了
  ```
  > **缺点：** 仅适用于双进程，多进程扩展难，且存在忙等（不满足让权等待）。

#### (2) 硬件实现方法
- **关中断 (Disable Interrupts)：**
  - **原理：** 进入 CS 前关中断，退出后开中断。
  - **缺点：** 仅适用单 CPU；权限过大，用户程序滥用会导致系统瘫痪；多核无效。
- **原子指令 (Atomic Instructions)：**
  - **Test-and-Set (TSL)：** 硬件指令，原子性地读取并修改锁变量。
  - **Swap：** 交换变量值。
  - **优点：** 适用于多核，由硬件保证原子性。
  - **缺点：** 通常实现为“忙等”，不满足让权等待（除非结合阻塞机制）。

## 五、信号量机制 (Semaphore Mechanism)

### 1. 概述
- **提出者：** Dijkstra (迪杰斯特拉)。
- **定义：** 一种特殊的数据结构，用于实现进程间的同步与互斥。
- **组成：**
  - **整型值 (Value)：** 代表资源数量。
  - **等待队列 (Queue)：** 存放因等待该资源而阻塞的进程 PCB。
- **操作：** 仅能通过 **P 操作** 和 **V 操作** (原语) 修改。

### 2. PV 操作原语
*注：PV 操作本身是原子操作，底层通过硬件指令或关中断实现。*

- **P 操作 (Wait/Down)：** 申请资源
  ```c
  P(S) {
      S.value--;
      if (S.value < 0) {
          // 资源不足，当前进程阻塞，加入等待队列
          block(S.queue); 
      }
  }
  ```
- **V 操作 (Signal/Up)：** 释放资源
  ```c
  V(S) {
      S.value++;
      if (S.value <= 0) {
          // 有进程在等待，唤醒一个
          wakeup(S.queue);
      }
  }
  ```

### 3. 信号量类型
- **互斥信号量 (Mutex Semaphore)：**
  - **用途：** 实现互斥。
  - **初值：** 通常为 **1** (表示临界资源数量为 1)。
  - **用法：** P 操作在临界区前，V 操作在临界区后。**P/V 在同一进程内**。
- **同步信号量 (Sync Semaphore)：**
  - **用途：** 实现同步（先后顺序）。
  - **初值：** 通常为 **0** (表示初始事件未发生)。
  - **用法：** 前驱进程执行完 V 操作，后继进程执行前 P 操作。**P/V 在不同进程间**。

## 六、信号量应用实例

### 1. N 个进程互斥访问临界区
```c
semaphore mutex = 1; // 互斥信号量

Process Pi() {
    P(mutex);        // 申请进入
    Critical Section;
    V(mutex);        // 释放
}
```

### 2. 进程同步 (P1 执行完 S1 后，P2 才能执行 S2)
```c
semaphore sync = 0; // 同步信号量，初值 0

Process P1() {
    S1;
    V(sync); // 通知 P2，S1 已完成
}

Process P2() {
    P(sync); // 等待 P1 通知
    S2;
}
```

### 3. 复杂任务图同步 (课堂练习 S1-S7)
**任务依赖关系：**
- S1, S2, S4, S5, S7 存在串行逻辑。
- S3 与 S2 可并发（但 S3 需等 S1）。
- S6 与 S5 可并发（但 S7 需等 S6）。

**解决方案示例 (多进程 + 信号量)：**
- **进程划分：**
  - P1: S1 -> S2 -> S4 -> S5 -> (等待 S6) -> S7
  - P2: (等待 S1) -> S3
  - P3: (等待 S5 或其他前置) -> S6
- **信号量定义：**
  - `sem1`: 控制 S3 等待 S1 (初值 0)
  - `sem2`: 控制 S6 等待前置 (初值 0)
  - `sem3`: 控制 S7 等待 S6 (初值 0)
- **逻辑实现：**
  ```c
  // P1
  S1; V(sem1); 
  S2; S4; S5; V(sem2); 
  P(sem3); S7;

  // P2
  P(sem1); S3; 

  // P3
  P(sem2); S6; V(sem3);
  ```
*(注：具体进程划分方案不唯一，核心是正确表达依赖关系)*

## 七、总结与下节预告

### 1. 核心知识点总结
- **线程：** 轻量级调度单位，共享进程资源，TCB 独立。
- **并发问题：** 同步（协作顺序）与互斥（资源竞争）。
- **临界区：** 必须满足互斥、让权等待、有空让进、有限等待。
- **信号量：** 解决同步互斥的标准机制，P 操作申请/阻塞，V 操作释放/唤醒。

### 2. 信号量底层实现细节
- P 操作导致阻塞时，进程状态由 **运行态 -> 阻塞态**，并挂入该信号量的等待队列。
- V 操作唤醒时，进程状态由 **阻塞态 -> 就绪态**，从等待队列移入就绪队列。
- PV 操作本身需保证原子性（通过硬件指令或屏蔽中断）。
