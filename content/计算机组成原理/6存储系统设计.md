## 容量、速度和价格的⽭盾  
## 容量、速度和价格分析  
### 容量  
$S_M = W * I * m$  
- $W$ 存储体字长（位或字节）  
- $I$ 每个存储体的字数  
- $m$ 并行工作的存储体 个数  
![[Pasted image 20251222211156.png]]  
### 速度  
访问时间 $T_a$ 、存储周期 $T_m$ 和频宽 $B_m$   
- $T_a$ 存储器从接到访存读申请，到数据被读到数据总线上所需要的时间  
	- 是启动⼀个访存读操作后, CPU **必须等待的时间**  
- $T_m$ 连续启动⼀个存储体所需要的时间  
	- 存储器进⾏⼀次存/取所需要的时间, 比 $T_a$ 大  
- $B_m$ 存储器可以提供的数据传输率  
	- 每秒钟传送的位数或字节数  
	- 分为  
		- 最大（极限）频宽  
		- 实际频宽  

单体存储器 $B_m=W/T_m$  
多体存储器 $B_m=m\times W/T_m$  
与存储器器件有关，也与设计有关  
### 价格  
总价格 $C$   
位价格 $c=C/S_M=C/(W\times I\times m)$  
与存储器器件有关，也与设计有关

## 存储体系的性能参数  
- 每位价格 $c$ - 命中率 $H$ - 等效访问时间 $T_A$ 二级存储层次为例  
![[Pasted image 20251222222805.png]]  
### 每位价格  
![[Pasted image 20251222222830.png]]  
为使 $c$ 接近于 $c_2$ ，应使 $S_{M1}$ << $S_{M2}$ ### 命中率  
CPU产⽣的逻辑地址能在 $M_1$ 中访问到的概率  
可由实验或模拟方法获得  
若 逻辑地址流 在 $M_1$ 中访问到的次数为 $R$ ，在 $M_2$ 中访问次数为 $R_2$ ,则  
$$  
H=\frac{R_{1}}{R_1+R_2}  
$$  
不命中率（失效率）= $1-H$ 影响命中率的因素  
- 程序的地址流  
- 地址预判算法  
- $M_1$ 的容量  
### 等效访问时间 $T_A$  
#### 假设访问是同时启动的  
$$  
T_A=H\times T_1+(1-H)\times T_2  
$$  
#### 假设访问 不是同时启动的  
$$
T_A=H\times T_1+(1-H)\times (T_1+T_2)
$$  

$$
T_A=T_1+(1-H)\times T_2
$$
$T_A$ 越接近 $T_1$ 最好  

#### 访问效率  
$$  
e = \frac{T_{A1}}{T_A} = \frac{T_{A1}}{H T_{A1} + (1-H) T_{A2}} = \frac{1}{H + (1-H) \times \frac{T_{A2}}{T_{A1}}}  
$$  
访问效率 主要与 **命中率** 和 **两级存储器的速度之比** 有关  
速度差不要太⼤，命中率越⾼越好  

### 提高命中率的方法：程序局部性原理  
不命中时，将 $M_2$ 中相邻的几个单元中的数据⼀起取出送入 $M_1$ 预取后的命中率 $H'=\frac{H+n-1}{n}$ - $H$ 原命中率  
- $n$ 数据块大小×数据重复使⽤次数  
不命中率降低 n 倍

# 并行存储系统  
## 交叉访问存储器  
在一个存储周期中 可以访问到多个数据，从而提高主存 **频宽**  
类型  
- 单体多字  
- 多体单字交叉存储器  
- 多体多字交叉存储器  
### 单体单字存储器  
![[Pasted image 20251223135714.png]]  
一个字长为 $W$ 位的存储器，一次可以访问一个存储字  
若存储器字长与CPU字长相等，最大频宽为  
$$  
B_M=W/T_M  
$$  
### 单体多字存储器  
![[Pasted image 20251223135849.png]]  
增加存储器的字长（ $m$ 倍），可以在一个周期内读出多个 CPU 字，最大频宽为  
$$  
B_M=m\times W/T_M  
$$  
缺点  
- 需要位数足够多的寄存器  
- 多次访问总线  
### 多体单字存储器  
![[Pasted image 20251223140044.png]]  
由多个 **相同存储器** 芯片组成  
每个芯片都有自己的 地址译码、读写驱动等外围电路  
每个存储器均为 1个CPU字的宽度，让 $m$ 个字长为 $W$ 的存储体 并行工作  
一次可访问多个存储器字  
最大频宽  
$$  
B_M=m\times W/T_M  
$$  
优点  
- 实际频宽 比单体多字高，总价和器件数量差不多  
- 并行访问不同的存储体  

缺点  
- 访问冲突  
	- 取指  
	- 读操作数  
	- 写数据  
	- 读写  

#### 减少分体冲突：按 模 $m$ 交叉编址  
- 高位交叉  
	- 用地址码的高位部分区分存储体号  
	- 扩大存储容量  
- 低位交叉  
	- 用地址码的高位部分区分存储体号  
	- 提⾼存储器访问速度

多体单字存取与单体多字存取结合

## 双端⼝存储器  
双口RAM：同⼀个存储器 有两组 独立的 读写控制电路，是高速工作的存储器  
有两个独立的端口，分别有  
- 地址线  
- 数据线  
- 控制线  
可对存储器 任何位置的数据进行存取操作  
### 核心  
用于 **数据存取** 的 **存储器阵列**，可被 左右两个端口共用  
两个端口 地址不同时，不会发生冲突，任一端口被选中驱动时，就可对整个存储器进行存取，每一个端口都有独立的 **片选控制** 和 **输出驱动控制**  
两个端口 地址相同时，会因数据冲突造成数据存储或读取错误  
两个端口的存取共四种情况  
1. 不同时 对 同一地址单元 存取数据  
	1. 正常  
2. 同时 对 同一地址单元读出数据  
	1. 正常  
3. 同时 对 同一地址单元写入数据  
	1. 写入错误  
4. 同时 对 同一地址单元，一个写入数据，一个读出数据  
	1. 读出错误  

为避免 $3$ 、$4$  
双口 RAM 设计有硬件 $\overline{BUSY}$ 输出，对同一地址单元存取时，  
- $\overline{BUSY_R}=H,\overline{BUSY_L}=H$ 时，可正常存储  
- 有一个 $\overline{BUSY}=L$ ，禁止数据的存取  
	- 哪个存取请求信号 出现在前，则对应的 $\overline{BUSY}=H$ ，允许写入，另一个禁止  

两端⼝间的存取请求信号出现时 间要相差在5ns以上，否则仲裁逻辑⽆法判定哪⼀个端⼝的存取请求信号在前  
两根控制线不会**同时**为 **低电平**  
# Cache存储系统  
## 程序局部性  
程序在执行时所用到的指令和数据的地址分布不是随机的，而是相对簇聚的。它包括  
- 时间局部性  
	- 最近的未来要用到的信息可能就是当前正在使用的信息  
- 空间局部性  
	- 最近的未来要用到的信息可能就是当前信息的相邻信息  

结论  
-  $M_1$ 不必放整个程序，只需存放 最近使用的块或页（时间局部性）  
- 调用时，一并把数据所在的 块或页 一起调入（空间局部性）  

**预判的准确性** 存储层次设计好坏的主要标志  
取决于 使用的 **算法** 和 **地址映像与变换方式**  
##  高速缓冲存储器  
Cache  
小容量   
由快速的SRAM 组成  
直接制作在CPU芯片内  
访问速度与CPU速度匹配  
用于存放 当前最急需处理的程序和数据  
### CPU-主存瓶颈  
![[Pasted image 20251223150435.png]]  
性能受 **主频带宽** 和 **响应时间** 的限制  
- 响应时间：一次访存所需要的时间  
	- 主存访存时间>>处理器机器周期  
- 带宽：单位时间访存次数  
	- 假设每条指令需要一个机器周期，一条指令需要访问主存 $m$ 次，意味着每个机器周期需要访存 $m$ 次  
#### 解决速度问题的方法  
在主存 和 CPU设置速度快，容量小的Cache  
依据 **程序局部性原理** ，将未来要用到的指令或数据 从低速主存预存取到高速 Cache中  
减少 平均响应时间

## Cache存储系统基本结构  
Cache和主存都被分成若干 **大小相等的块** ，每个块由若干字节组成  
Cache容量远小于主存，块的数量也是  
Cache保存的信息只是 主存中最急需执行的若干块的副本  
如果Cache命中，就把主存地址换成Cache地址，直接访问Cache  
如果Cache失效（未命中）,用主存地址访问主存，从主存中度一个字送往CPU，同时把包括这个字在内的 一整块 送入Cache

### Cache特点  
1. 与CPU工艺相同  
2. 地址映象、变换、替换、调度由专门的 **硬件** 实现  
3. 靠近 CPU 或在CPU 中，减少传输延迟

5. Cache 访问主存的优先级 最高  
6. Cache 与CPU 有直接的通路，主存与CPU 也有直接的通路，读直达，写直达  

## 地址映像与变换  
三个问题  
1. 定位  
	1. 将主存数据放在Cache的哪里  
	2. 直达主存中的数据已经装入 Cache  
	3. 如果命中，如何形成Cache地址并访问主存  
2. 替换  
	1. 未命中或失效，将数据从主存调入Cache  
	2. Cache 满，如何将Cache中的数据替换出去  
3. 数据一致性  
	1. Cache内容和主存内容一致  

**地址映像**：把存放在主存中的程序按照某种规则 装⼊到Cache中，并建⽴主存地址与Cache地址之间的对应关系  
**地址变换**：程序已经装⼊到Cache之后，在实际运⾏过程中，把主存地址变换成Cache地址  

![[Pasted image 20251223205325.png]]  
地址映像的选择依据  
- 地址变换的硬件要容易实现  
- 地址变换的速度要快  
- 主存空间利⽤率要⾼  
- 发⽣块冲突的概率要⼩  

**块冲突** ：主存中的块要调入 Cache中的某个位置，但该位置已经被其他主存块占用  
### 地址映像与变换方法  
#### 全项联映像和变换  
映像规则：主存中的任意一块 都可以装入Cache中的任意一个块位置  
地址变换：采用相联存储器构成的 **目录表**，以便硬件方式实现  
![[Pasted image 20251223205733.png]]  
优点：  
- 块冲突概率低  
- Cache空间利用率高  

缺点：  
- 需要一个 **相联存储器**，造价高  
- 相联比较时间 会影响Cache的访问速度  
#### 直接映像及其变换  
映像规则：主存中的每⼀块只能装⼊到Cache内唯 ⼀⼀个指定的块位置  
只要从主存地址中取出低位部分，即可形成Cache的地址  
![[Pasted image 20251223210019.png]]  
地址变换：设置一个按地址访问的区表存储器（称之为 区号表），存放Cache中每一块目前被主存中哪个区的对应块所占用

优点：  
- 硬件实现简单，不需要 相联访问存储器  
- 访问速度快，实际上不做地址变换  

缺点  
- 块冲突概率高

### 组相联映像及其变换  
将主存和Cache按同样⼤⼩划分成块，Cache空间划分成 **大小相同** 的 **组**，组里有若干个 块  
让 主存中的任意一个块 只能被放置到 Cache中唯一的一个 指定组，用全相联映像 装入Cache中对应组的一块位置上  
即 组间采用直接映像，组内采用全相联映像  

组内块数 $S$ =Cache块数时，组相联 变成 全相联  
组内块数 $S$ =1时，组相联 变成 直接映像  
$S$ 越大，冲突越少，地址变换 越复杂  
$S$ 越小，冲突越多，地址变换 越简单  

假设 Cache空间分成 $C_g=2^g$ 组，每组 $G_b=2^b$ 块  
主存地址分为三部分：标记、组号、块内地址  
Cache地址分为三部分：组号、组内块号、块内地址

Cache分组，主存不分组

映像规则：主存块到Cache组set之间采⽤直接映像⽅式（一个主存块 只能到 唯一一个Cache 组）  
在对应的组内采用全相联映像方式（一个主存块可以放入 指定Cache组内 任意位置）  
![[Pasted image 20251223222252.png]]  
地址变换：根据 区内块号，按地址访问 **块表**，取得目录表内容，按内容（区号）访问 **目录表**

## Cache替换算法  
**块失效** ：该块未装入Cache,需要从主存中调块  
**块冲突** ：两个以上的主存块 想要进入Cache中同一个块位置  
块失效 不一定块冲突，块冲突 一定是块失效引起  

确定 替换算法的规则：  
- 高命中率  
- 易于实现，硬件成本低  

五种典型替换算法  

| 算法           | 描述                         | 优点             | 缺点      |                       |                        |  
| ------------ | -------------------------- | -------------- | ------- | --------------------- | ---------------------- |  
| 随机(RAND)     | 随机数生成器 形成被替换的块号            | 简单，易于实现        | 命中率低    | 没有利用Cache使用历史，反映不了局部性 |                        |  
| 先进先出(FIFO)   | 选最早装入Cache的块作为被替换的块        | 简单，易于实现        | 反映不了局部性 | 最先被调用的块可能是经常被使用的      | 命中率随Cache块数增加可能上升，可能下降 |  
| 近期最少使用(LFU)  | Least Frequently Used      | ⽐较正确地反映了程序的局部性 | 实现困难    | 需要为每个块配置很长的计数字段       |                        |  
| 近期最久未使用(LRU) | Least Recently Used,LFU的变形 | 实现方便           |         | 将LFU的多少，转换成有无         | 命中率随Cache块数增加上升        |  
| 优先替换(OPT)    | 将未来的近期内不使用的块替换出去           |                |         | 实现不现实，需要运行程序两次，是理想化算法 |                        |  
**颠簸**：下次要使用的块 本次被替换出去，而发生的连续块 失效的现象  

命中率与 **块地址流** 和 分配给该程序的 **Cache块数** 有关  

#### 堆栈型替换算法  
对任意⼀个程序的块地址流作两次Cache块数分配， 分别分配 m 个Cache块和 n 个Cache块，并且有 m ≤ n。 如果在任何时刻t， Cache块集合 $B_t$ 都满⾜关系  
$$  
B_t(m)\subseteq B_t(n)  
$$  
堆栈型替换算法的命中率随分配的Cache块数的增 加⽽单调上升，⾄少不会下降  
LRU,OPT 是堆栈型算法  
FIFO 不是堆栈型算法  

可以利用 **堆栈** 实现 堆栈型算法  
堆栈容量 $2^b$  
过程：检查要访问的Cache块号是否在堆栈中  
- 是：将该块调置栈顶  
- 否：新块压入堆栈，其余弹出  

结果：  
- 栈顶 恒为最近访问过的 页号  
- 栈底 恒为最久没有访问过的 页号  

堆栈要求  
- 有相联比较功能  
- 全上推 或 全下推功能  
- 从中间抽走一项的功能  

### Cache替换算法与实现  
Cache 通常使用 组相联映像 或 直接映像，不使用 全相联映像  
块冲突时，必须进行块替换  
两种方法  
- 堆栈法：使用硬堆栈  
- 比较对法：使用逻辑电路、触发器

## Cache更新策略  
Cache内容可能与主存内容不一致  
- CPU 更新，写了Cache而未更新主存  
- I/O 更新了 主存而未更新 Cache  
### Cache透明性分析  
![[Pasted image 20251223225428.png]]  
#### 指令比例  
- Load: $26$ %  
- Store: $9$ %  
“写” 比例： $9$ /( $26$ + $9$ )= $25$ %  

“写”问题：读出标识，确认命中后，对Cache写；确保Cache内容和主存一致  
写策略：解决何时更新 **主存** 的问题  
Cache一致性算法：2种写策略  
### 写策略
#### 写直达 Write-through
CPU 在执行写操作时，利用 **直接通路**，把数据同时写入Cache和主存  
#### 写回法 Write-Back
抵触修改法  
CPU 数据只写入 Cache，不写入主存  
为 每个Cache块设置 “修改位”  
当替换时，才把修改过的Cache块 写回 主存  
![[Pasted image 20251224111103.png]]
![[Pasted image 20251224111112.png]]

|         | 写直达法              | 写回法          |
| :------ | :---------------- | :----------- |
| 可靠性     | 好于写回法             | 块替换前仍存在一致性问题 |
| 与主存的通信量 |                   | 少于写直达法达10多倍  |
| 控制复杂性   | 比写回法简单            |              |
| 硬件实现代价  |                   | 比写直达法低得多     |
| 优点      | 易于实现，容易保持不同层次的一致性 | 速度快，减少访存次数   |
| 缺点      | 速度慢               | 一致性问题        |
### “写”调块
“写”操作 必须在 确认命中后进行  
当 写不命中：
- 不按写分配法：当Cache不命中，把所有要写的字 直接写入主存，不调块
- 按写分配法：当Cache不命中，写入主存，并把单元所在的块调入Cache

### 写策略与调块
一般：
- 写直达法：不按写分配
- 写回法：按写分配

单处理机系统：大多 写回法  
共享主存的多处理机：大多 写直达法，多Cache一致性算法  
![[Pasted image 20251224113518.png]]

#### “写”缓冲器
写回法 和 写直达法 都需要少量缓冲器  
写回法 存放将要写回的块，不必等待 写回主存后才进行Cache取  
写直达法 存放 将要写回主存的内容，减少CPU等待写主存的时间  
缓冲器 对Cache-主存透明  
### Cache 预取算法  
| 预取策略       | 描述                                                                 |
| :------------- | :------------------------------------------------------------------- |
| **按需取**     | 在出现Cache不命中时，把一个块取到Cache中来                           |
| **恒预取**     | 无论Cache是否命中，都把下一块取到Cache中                             |
| **不命中预取** | 当Cache不命中，把本块和下一块一起取到Cache中                         |
主要考虑的因素  
- 命中率提高
- Cache与主存之间通信量的增加

# 虚拟存储系统  
由 主存储器(DRAM) 和 联机工作的辅助存储器(磁盘存储器)共同组成  
主存使用DRAM，容量小，速度快  
磁盘容量大，速度慢  
应用程序员看成一个存储器，虚拟空间大  
需要考虑 交换块的大小，地址映像，替换问题，写一致性问题  

|                      | Cache存储系统 | 虚拟存储系统    |
| :------------------- | :-------- | :-------- |
| 目的                   | 弥补主存速度的   | 弥补主存容量的不足 |
| 存储管理实现               | 由专用硬件实现   | 软件、硬件实现   |
| 访问速度的比值<br>(第一级和第二级) | 几比一       | 几百比一      |
| 典型的块(页)大小            | 几十个字节     | 几百到几千个字节  |
| CPU对第二级的访问方式         | 可直接访问     | 均通过第一级    |
| 失效时CPU是否切换           | 不切换       | 切换到其他进程   |
| 透明性                  | 对所有程序员透明  | 仅对应用程序员透明 |

虚拟存储器 是一个大容量存储器的 **逻辑模型** ，指 主存-辅存层次  
![[Pasted image 20251224120421.png]]
### 虚拟存储器
- 依赖 程序的局部性原理，提高主存利用率
- 对 应用程序员 透明，对系统程序员不透明
- 程序不再受 物理内存空间的限制，每个程序有独立的程序空间（虚拟存储空间），更容易编程，程序的执行 独立与存储器的容量和配置
- 多用户 共享存储器
- 提供 主存保护机制

## 虚拟存储器的管理方式
虚拟空间 比 实存空间 大得多，无法一一对应  
虚拟存储器 采用 **地址映像表机构** 实现程序的 **动态定位** ，根据地址映像算法的不同，分为  
- 页式管理
- 段式管理
- 段页式管理
### 页式管理
基本思想：将 **主存空间** 和 **程序空间** 分成 **页**，按页进行调入调出和管理
