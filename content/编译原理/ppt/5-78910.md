# 第五章：语法分析 (Ch5 Syntax Analysis)

本章主要讨论自下而上（Bottom-Up）的语法分析方法，重点在于 **LR分析法** 的不同变体（LR(0), SLR(1), LR(1), LALR(1)），以及如何处理二义文法和错误恢复，最后介绍了自动生成工具 YACC。

## 5.1 - 5.3 概述与基础
*   **5.1 移进-归约分析法 (Shift-Reduce Parsing)**：LR分析的基础机制。
*   **5.2 算符优先分析法**：一种特定的自下而上分析方法（本章后续未详细展开，重点在LR）。
*   **5.3 LR分析鸟瞰**：
    *   **核心思想**：利用“句柄”（Handle）进行归约。
    *   **关键概念**：活前缀（Viable Prefix）、项目（Item）、识别活前缀的DFA。
    *   **LR(k)**：L表示从左到右扫描，R表示最右推导的逆过程，k表示向前看k个符号。

## 5.4 - 5.7 LR分析法的主要类型

这四种方法在**分析能力**和**存储开销/状态数**之间进行了权衡。

### 1. 能力与开销对比
*   **分析功能（能力）**：$LR(1) \supset SLR(1) \supset LR(0)$
    *   $LALR(1)$ 的能力介于 $SLR(1)$ 和 $LR(1)$ 之间，通常认为 $LALR(1) \subseteq LR(1)$，但在实际应用中能解决绝大多数SLR(1)无法解决的问题，且状态数远少于LR(1)。
*   **存储开销（状态数）**：
    *   $LR(1)$ 状态最多（因为搜索符不同导致状态分裂）。
    *   $LR(0)$ 和 $SLR(1)$ 状态最少（基于LR(0)项目集）。
    *   $LALR(1)$ 状态数 = $LR(0)$ 的状态数（通过合并同心集实现）。

### 2. 各类分析法详解

#### **LR(0) 分析**
*   **基础**：构造识别活前缀的DFA，项目不带搜索符。
*   **缺点**：很多文法存在冲突（移进-归约或归约-归约），实用性低。

#### **SLR(1) 分析 (Simple LR)**
*   **改进**：在LR(0)基础上，利用 **FOLLOW集** 来解决冲突。
*   **构造**：若状态中存在 $A \to \alpha \cdot$ 和 $B \to \beta \cdot a \gamma$，仅当 $a \notin FOLLOW(A)$ 时才移进，否则归约。
*   **局限**：FOLLOW集过于粗略，可能无法解决某些LR(1)可解决的冲突。

#### **LR(1) 分析**
*   **改进**：项目中包含具体的**搜索符（Lookahead Symbol）**，形式为 $[A \to \alpha \cdot \beta, a]$。
*   **优点**：分析能力最强，无冲突的文法范围最广。
*   **缺点**：状态数量巨大（相比LR(0)可能增加许多倍），因为同一核心项目集因搜索符不同而分裂成多个状态。

#### **LALR(1) 分析 (Look-Ahead LR)**
*   **核心思想**：合并 LR(1) 中的**同心项目集**。
*   **定义（同心项目集）**：两个LR(1)项目集，如果它们的**LR(0)项目核心相同**，仅**搜索符不同”，则称为同心项目集。
*   **构造步骤**：
    1.  构造文法的 LR(1) 项目集规范族 $C$。
    2.  找出所有同心项目集并合并（搜索符取并集），得到 LALR(1) 项目集规范族 $M$。
    3.  基于 $M$ 构造分析表。
*   **特性**：
    *   状态数与 LR(0)/SLR(1) 相同。
    *   分析能力优于 SLR(1)，略弱于 LR(1)（极少数情况）。
    *   **重要定理**：合并同心集后，**不会引入新的“移进-归约”冲突**。如果原LR(1)无冲突，合并后若有冲突，只能是**“归约-归约”冲突**。

### **例5.17 & 例5.18：文法分类判断流程**
判断一个文法属于哪类LR文法的步骤：
1.  **LR(0)**：构造LR(0) DFA。若无冲突 $\rightarrow$ LR(0)。
2.  **SLR(1)**：若有冲突，用SLR(1)方法（检查FOLLOW集）解决。若解决 $\rightarrow$ SLR(1)。
3.  **LR(1)**：若SLR(1)失败，构造LR(1)项目集。若无冲突 $\rightarrow$ LR(1)。
4.  **LALR(1)**：合并LR(1)同心集。若无冲突 $\rightarrow$ LALR(1)。
5.  **非LR**：若LALR(1)仍有归约-归约冲突 $\rightarrow$ 非LR文法。

> **例5.18 分析**：
> 文法：$S \to Aa | bAc | Bc | bBa, A \to d, B \to d$
> *   LR(0): 状态1中有 $A \to d \cdot$ 和 $B \to d \cdot$，归约-归约冲突。非LR(0)。
> *   SLR(1): $FOLLOW(A)=\{a,c\}, FOLLOW(B)=\{a,c\}$。交集非空，无法区分。非SLR(1)。
> *   LR(1): 构造后发现无冲突（因为搜索符不同，如 $A \to d \cdot, a$ 和 $B \to d \cdot, c$ 在不同语境下区分开）。是LR(1)。
> *   LALR(1): 合并同心集后，状态中出现 $A \to d \cdot, a/c$ 和 $B \to d \cdot, c/a$，产生归约-归约冲突。**因此该文法是 LR(1) 但不是 LALR(1)。**

## 5.8 LR分析对二义文法的应用

*   **定理**：任何一个二义文法都不是严格的LR文法（因为LR要求确定性，二义性导致冲突）。
*   **应用价值**：二义文法通常更简洁（如表达式文法 $E \to E+E | E*E$）。
*   **解决方法**：
    1.  **改写文法**：消除二义性（如引入优先级层次 $E, T, F$），但这会增加文法规模和状态数。
    2.  **使用二义文法 + 优先级/结合性规则**：
        *   在构造LR分析表时，虽然会出现冲突，但可以通过人为规定的**优先级**和**结合性**来解决冲突。
        *   **移进-归约冲突**：通常由运算符优先级决定（高优先级移进，低优先级归约）。
        *   **归约-归约冲突**：通常由产生式顺序或特定规则决定。

> **例5.19 表达式文法**：
> $E \to E+E | E*E | (E) | i$
> *   直接构造LR分析表会在状态7和8等处出现移进-归约冲突。
> *   规定 `*` 优先级高于 `+`，且均为左结合。
> *   在冲突状态下，根据优先级表填写 Action 表（例如在 `E+E·` 遇到 `*` 时移进，遇到 `+` 时归约）。

> **例5.20 悬空Else问题**：
> $S \to iSeS | iS | a$
> *   这是一个经典的二义文法。
> *   规则：**Else 与最近的 Then 匹配**。
> *   在LR分析表中，当处于状态4（$S \to iS \cdot$）且输入为 `e` (else) 时，存在移进（$S \to iS \cdot eS$）和归约（$S \to iS$）的冲突。
> *   解决：选择**移进**（Shift），从而保证 else 与最近的 if 结合。

## 5.9 LR分析的错误处理与恢复

*   **目标**：准确报告错误位置/类型；迅速恢复以继续检测后续错误。
*   **策略**：
    1.  **紧急方式 (Panic Mode)**：丢弃输入符号，直到遇到同步符号（如 `;`, `}`, `end`）。
    2.  **短语级恢复 (Phrase-level Recovery)**：在出错点进行局部修正（插入、删除、替换符号），使分析能继续。
    3.  **全局纠正**：寻找最小代价的修正序列（计算量大，少用）。
    4.  **出错产生式**：在文法中增加错误产生式。

*   **LR分析表中的错误处理实现**：
    *   在Action表中，原本填“error”的地方，填入特定的错误处理程序入口（如 `e1`, `e2`...）。
    *   **例5.19 带错误处理的表达式分析表**：
        *   **e1 (缺少运算对象)**：在期望操作数时遇到运算符。动作：压入假设的 `i` 和对应状态。
        *   **e2 (括号不配对)**：在不该出现 `)` 时出现。动作：删除 `)`。
        *   **e3 (缺少运算符)**：在期望运算符时遇到操作数。动作：压入假设的 `+` 和对应状态。
        *   **e4 (缺少右括号)**：在表达式结束但未闭合时。动作：压入假设的 `)` 和对应状态。

## 5.10 语法分析器的自动生成器 — YACC

### 5.10.1 YACC 简介
*   **全称**：Yet Another Compiler-Compiler。
*   **功能**：LALR(1) 分析器自动生成器。
*   **工作流程**：
    1.  编写 YACC 源文件 (`.y`)。
    2.  运行 `yacc file.y` $\rightarrow$ 生成 C 代码 `y.tab.c` (包含分析表和驱动逻辑)。
    3.  编译 `cc y.tab.c -ly` $\rightarrow$ 生成可执行文件 `a.out`。
*   **内部实现**：
    1.  构造 LR(1) 项目集。
    2.  合并同心集得到 LALR(1) 项目集。
    3.  构造 LALR(1) 分析表。

### 5.10.2 YACC 语言结构
YACC 源程序分为三部分，由 `%%` 分隔：

1.  **说明部分 (Declarations)**：
    *   `%{ ... %}`: C 代码块（头文件、宏定义等）。
    *   `%token`: 声明终结符。
    *   `%left`, `%right`, `%nonassoc`: 声明运算符的结合性和优先级（用于解决二义性）。
    *   `%start`: 指定开始符号。

2.  **翻译规则部分 (Rules)**：
    *   格式：`非终结符 : 候选式1 { 语义动作 } | 候选式2 { 语义动作 } ;`
    *   **语义动作**：嵌入在产生式后的 C 代码。
    *   **变量引用**：
        *   `$$`: 左部非终结符的属性值。
        *   `$1, $2, ...`: 右部第1、第2...个文法符号的属性值。
    *   **示例**：
        ```yacc
        expr : expr '+' expr { $$ = $1 + $3; }
             | NUMBER         { $$ = $1; }
             ;
        ```

3.  **辅助过程部分 (Subroutines)**：
    *   C 语言函数，如 `main()`, `yyerror()` (错误处理), `yylex()` (词法分析，通常由LEX生成)。

### 5.10.3 YACC 处理二义文法
*   YACC 允许使用二义文法，通过**优先级**和**结合性**声明来解决冲突。
*   **优先级声明**：
    ```yacc
    %left '+' '-'      /* 低优先级，左结合 */
    %left '*' '/'      /* 高优先级，左结合 */
    %right UMINUS      /* 最高优先级，右结合（用于负号） */
    ```
*   **特殊处理**：
    *   `%prec`：用于改变某个产生式的优先级。
    *   例如：`'-' expr %prec UMINUS { $$ = -$2; }` 告诉YACC将单目负号的优先级视为 UMINUS。

## 总结：LR分析法家族对比表

| 特性 | LR(0) | SLR(1) | LALR(1) | LR(1) |
| :--- | :---: | :---: | :---: | :---: |
| **搜索符** | 无 | FOLLOW集 | 合并后的Lookahead | 精确Lookahead |
| **状态数** | 最少 | 最少 | 少 (同LR0) | 最多 |
| **分析能力** | 最弱 | 较弱 | 强 (实用标准) | 最强 |
| **冲突类型** | S/R, R/R | S/R, R/R | 仅 R/R (若LR1无冲突) | 无 (对于LR1文法) |
| **典型应用** | 理论教学 | 简单文法 | **YACC/Bison默认** | 复杂手写分析器 |

**核心结论**：
*   **LALR(1)** 是性价比最高的选择，也是 YACC 等工具的基础。
*   **LR(1)** 能力最强但开销大。
*   **二义文法** 可通过优先级规则在 LR 框架下有效处理。
*   **错误恢复** 可通过修改分析表中的错误入口实现短语级恢复。
