# 第3章 词法分析 (Lexical Analysis) 知识点整理

## 1. 词法分析程序的功能 (3.1)

### 1.1 核心定义
*   **词法分析器 (Scanner/Lexer)**：编译程序的第一个阶段，也是必要阶段。
*   **输入**：字符串形式的源程序。
*   **输出**：属性字流 (Token Stream)，即 `(单词类别, 内码值)` 的二元组序列。
*   **任务**：从左至右扫描源程序，按照语言的**词法规则**识别具有独立意义的**单词**，并生成对应的属性字。

### 1.2 词法分析器作为独立子程序的优点
1.  **模块化**：只要接口不变，修改词法分析器不影响编译器其他部分，易于维护。
2.  **结构清晰**：使整个编译器结构简捷、清晰。
3.  **效率与工具**：可以采用更有效的方法和自动化工具（如 LEX/FLEX）进行处理。

### 1.3 实现途径
*   **手工生成**：直接编写代码。
*   **自动生成**：使用工具（如 LEX）根据正规式描述自动生成。

## 2. 词法分析器的设计与实现 (3.2)

### 2.1 单词与属性字

#### A. 单词 (Token)
*   **定义**：语言中具有独立意义的**最小语法单位**。
*   **分类**：
    1.  **关键字 (Keywords/Reserved Words)**：语言系统定义的固定字符串（如 `int`, `if`, `for`, `while`）。通常关联语句性质。
    2.  **标识符 (Identifiers)**：用户定义的名字（变量名、函数名、数组名等）。
        *   正规式示例：`(a|...|z)(a|...|z|0|...|9)*`
    3.  **常量 (Constants)**：
        *   整型 (Integer): `-26`, `0x123`, `037`
        *   实型 (Real): `-1.6`, `2e-6`
        *   布尔型 (Boolean): `TRUE`, `FALSE`
        *   字符/字符串: `'a'`, `"abc"`
    4.  **运算符 (Operators)**：算术、逻辑、位运算等（如 `+`, `-`, `*`, `/`, `++`, `+=`, `&&`）。
    5.  **界限符 (Delimiters)**：逗号、分号、括号、引号等（如 `,`, `;`, `(`, `)`, `{`, `}`）。

*   **注意**：
    *   同一个字符开头可能跨越多个单词类（如 `0` 可能是整数开始，`0.` 是实数，`0x` 是十六进制）。
    *   **非单词成分**：注释、空格、换行符（通常由预处理或扫描器过滤）。
    *   **预处理指令**：`#define`, `#include` 等通常在预处理阶段处理，但也涉及词法层面的识别。

#### B. 属性字 (Attribute Word)
*   **结构**：通常表示为二元组 `(Code, Value)` 或 `(Token, Attribute)`。
    *   **Code (类别码)**：刻画单词的性质（如：标识符、关键字、运算符）。
    *   **Value (内码值)**：单词的具体值或指向符号表/常量表的指针。若无需额外信息（如关键字、单字符运算符），该项可为空。

*   **各类单词的属性字设计策略**：
    1.  **标识符**：
        *   Code: 标识符类码。
        *   Value: 指向**符号表**中该标识符入口的指针。
    2.  **关键字**：
        *   Code: 每个关键字对应唯一的类别码。
        *   Value: 通常为空。
    3.  **常量**：
        *   Code: 常量类码（可按类型细分，如整型、实型）。
        *   Value: 指向**常量表**中该常数值的指针，或直接存储数值。
    4.  **运算符和界限符**：
        *   Code: 每个符号对应一个类别码。
        *   Value: 通常为空。
        *   *优化技巧*：优先级相同的运算符可共用一个类别码，或通过编码值反映优先级。

### 2.2 源程序的输入与预处理

#### A. 输入缓冲区技术
为了减少磁盘I/O次数，提高扫描效率，采用**双缓冲区机制**。
*   **结构**:两个大小相等（通常为2的整次幂 $n$）的半区（First Half, Second Half）。
*   **指针**:
    *   `B` (Begin): 当前识别单词的起始位置。
    *   `F` (Forward): 当前扫描指针。
*   ** eof 标志**: 在每个半区的末尾设置特殊标志 `eof`。
*   **互补装载算法**:
    *   当 `F` 指向某半区的 `eof` 时，触发另一个半区的重新装载。
    *   若 `F` 在前半区 eof -> 装载后半区，`F` 移向后半区首。
    *   若 `F` 在后半区 eof -> 装载前半区，`F` 移向前半区首。

#### B. 预处理程序 (Preprocessor)
*   **作用**：
    1.  减少内存占用。
    2.  减轻扫描器负担，使其只处理“纯净”的代码。
*   **主要任务**：
    1.  **滤除非单词成分**：删除注释、多余空格、换行符、制表符。
    2.  **宏替换**：展开 `#define`。
    3.  **文件包含**：嵌入 `#include` 的文件内容。
    4.  **条件编译**：处理 `#ifdef` 等指令。

### 2.3 扫描器设计与实现技术

核心工具：**有限自动机 (FA)**。

#### A. 状态转换图的构造步骤
1.  **构造单个单词的状态图**：为每类单词（标识符、常数、运算符等）画出对应的 NFA/DFA。
2.  **合并状态图**:
    *   将所有单词状态图的**初态合并**为一个唯一的初态。
    *   调整状态编号以避免冲突。
    *   结果是一个能识别该语言所有单词的大状态图（通常是 NFA，需确定化）。

#### B. 状态转换图的实现方法

##### 方法一：数据中心法 (Table-Driven)
将状态转换图视为数据结构（状态表），由总控程序驱动。
*   **数据结构**：二级目录表。
    *   **主表**：索引为当前状态，内容为分表地址或终态处理子程序入口。
    *   **分表**：索引为输入字符，内容为下一状态。
*   **算法流程**：
    1.  初始化状态 $B=$ 初态。
    2.  查主表。若是终态，执行相应动作并重置；若非终态，获取分表地址。
    3.  读入字符 $W$，查分表得到新状态 $B$。
    4.  重复直到识别出一个单词。

##### 方法二：程序中心法 (Direct Coding)
将状态转换图视为流程图，为每个状态编写代码（通常使用 `switch-case` 或 `if-else`）。
*   **特点**：
    *   每个状态对应一段代码。
    *   利用 `nextchar()` 读取下一个字符。
    *   利用 `ungetchar()` (或 `untread()`) 回退多读的字符（因为识别单词时往往需要超前看一个字符才能确定结束）。
*   **示例逻辑**：
    ```c
    while(1) {
        char1 = nextchar();
        switch(state) {
            case 0: // 初态
                if (isLetter(char1)) state = 1;
                else if (isDigit(char1)) state = 3;
                else if (char1 == '+') state = 7;
                // ...
                break;
            case 1: // 标识符/关键字中间态
                while (isLetterOrDigit(nextchar())) ; // 继续读
                ungetchar(); // 回退最后一个非字母数字字符
                state = 2; // 进入终态处理
                break;
            case 2: // 标识符终态
                lookup_and_return_token();
                state = 0;
                break;
            // ... 其他状态
        }
    }
    ```

## 3. 词法分析器的自动生成 (3.3)

### 3.1 自动生成思想
从**正规式 (Regular Expression)** 到 **扫描器代码** 的过程：
1.  **正规式 $\rightarrow$ NFA** (算法 X, 如 Thompson 构造法)。
2.  **NFA $\rightarrow$ DFA** (算法 Y, 子集构造法/Subset Construction)。
3.  **DFA 最小化** (算法 Z, 划分算法/Hopcroft算法)。
4.  **生成代码**：基于最小化 DFA 生成表驱动或直译代码，并加上总控程序。

### 3.2 LEX / FLEX 工具

*   **LEX**: 1 the Bell Labs (1972), UNIX标准工具。
*   **Flex**: GNU工程 (1984), Fast Lexical Analyzer Generator, 兼容LEX且更快。

#### A. LEX 工作流程
1.  编写 LEX 源程序 (`.l` 文件)。
2.  运行 `lex source.l` $\rightarrow$ 生成 C 代码 `lex.yy.c`。
3.  编译 `cc lex.yy.c -ll` $\rightarrow$ 生成可执行文件 `a.out` (即词法分析器)。

#### B. LEX 源程序结构
LEX 源程序由三个部分组成，用 `%%` 分隔：

```lex
%{
/* 第一部分：说明部分 (Definitions) */
/* C语言头文件包含、全局变量声明、宏定义等 */
#include <stdio.h>
#define IF 1
#define ID 2
%}

/* 正规式辅助定义 (可选) */
digit   [0-9]
letter  [a-zA-Z]
id      {letter}({letter}|{digit})*

%%
/* 第二部分：规则部分 (Rules) */
/* 模式(Pattern) { 动作(Action) } */
{ws}        { /* 忽略空白，无动作 */ }
if          { return(IF); }
{id}        { yylval = install_id(); return(ID); }
"+"         { return(PLUS); }
"++"        { return(INC); }

%%
/* 第三部分：用户子程序部分 (Subroutines) */
/* 辅助函数，如 install_id(), install_num() 等 */
int install_id() {
    /* 将 yytext 中的字符串填入符号表，返回指针 */
}
```

#### C. LEX 中的关键概念
*   **模式 (Pattern)**: 使用扩展正规式表示。
    *   `x`: 匹配字符 x
    *   `[abc]`: 匹配 a, b, c 中任一
    *   `[^abc]`: 匹配除 a, b, c 外的字符
    *   `r*`: 0个或多个 r
    *   `r+`: 1个或多个 r
    *   `r?`: 0个或1个 r
    *   `.`: 匹配除换行符外的任意字符
*   **动作 (Action)**: C 语言代码片段。
    *   `yytext`: 全局指针，指向当前匹配的单词字符串。
    *   `yyleng`: 当前匹配单词的长度
    *   `yylval`: 全局变量，用于传递属性值给语法分析器 (YACC/Bison)。
    *   `return(Token_Code)`: 返回 token 给语法分析器。

### 3.3 最长匹配原则 (Maximal Munch) 与优先匹配

*   **问题**: 当输入串可以匹配多个规则，或者一个规则的前缀也是另一个规则时，如何确定边界？
    *   例：输入 `a+++b`。
    *   可能的切分：`a`, `+`, `++`, `b` 或 `a`, `++`, `+`, `b`。
*   **LEX 的处理策略**:
    1.  **最长匹配 (Longest Match)**: 扫描器总是试图匹配**最长**的可能字符串。
        *   对于 `a+++b`，扫描器读到 `a` (匹配 id)，然后读 `+`，再读 `+` (此时 `++` 匹配运算符)，再读 `+` (此时 `+++` 不匹配任何单一运算符，但前缀 `++` 已匹配)。
        *   实际上，LEX 会一直读直到无法匹配任何规则的前缀为止，然后回溯到最后一个合法的接受状态。
    2.  **优先匹配 (Priority)**: 如果两个规则匹配了**相同长度**的最长字符串，则选择在 LEX 文件中**先出现**的规则。
        *   例：关键字 `if` 和标识符 `id` (`[a-z]+`)。
        *   输入 `if`，两者都匹配长度为2。
        *   如果在 LEX 文件中 `if {return(IF);}` 写在 `{id} {return(ID);}` 之前，则识别为关键字 `IF`。

    *   *修正思考*: 文档中选择题答案为 **A**。

### 判断题解析
1.  **Lex是典型的词法分析程序。**
    *   **错 (×)**。Lex 是词法分析器**生成器** (Generator)，它生成词法分析程序，而不是词法分析程序本身。
2.  **词法分析的依据是源语言的文法规则。**
    *   **错 (×)**。词法分析的依据是**词法规则** (通常用正规式描述)。文法规则 (上下文无关文法) 是语法分析的依据。
3.  **源程序中的单词是具有独立意义的短语。**
    *   **错 (×)**。单词是**最小**语法单位，不是短语。短语由多个单词组成。
4.  **单词的属性字一般应该包括单词类别和单词内码。**
    *   **对 (√)**。
5.  **编译的预处理程序的处理对象是源程序。**
    *   **对 (√)**。预处理在词法分析之前（或作为其一部分），直接处理源代码文本。

## 总结
本章重点在于理解词法分析器如何将字符流转换为 Token 流。
*   **理论基础**：正规式、有限自动机 (NFA/DFA)、状态转换图。
*   **实现技术**：双缓冲区输入、状态表驱动 vs 直接编码。
*   **自动化工具**：LEX/FLEX 的使用及其内部原理 (Regex -> NFA -> DFA -> Minimized DFA -> Code)。
*   **关键细节**：最长匹配原则、优先匹配规则、属性字的设计。
