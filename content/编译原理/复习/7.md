# 第7章 运行环境 (Runtime Environment)

## 7.1 关于程序运行环境与存储组织

### 1. 基本概念
*   **运行环境**：指目标计算机的寄存器及存储器结构，用来管理存储器并保存执行过程所需的信息。
*   **环境 (Environment) vs 状态 (State)**：
    *   **环境**：名字（标识符）到存储位置的映射。
    *   **状态**：存储位置到值的映射。
    *   **区别**：赋值操作改变的是**状态**（值变了），但不改变**环境**（变量所在的内存地址没变）。

### 2. 运行环境的分类
根据语言特性，运行环境分为三类：
1.  **完全静态环境 (Full Static Environment)**：
    *   代表语言：**FORTRAN**。
    *   特点：所有数据对象的大小和位置在编译时确定。
2.  **基于栈的环境 (Stack-based Environment)**：
    *   代表语言：**C, C++, Pascal, Ada**。
    *   特点：支持递归，局部变量通过栈动态分配。
3.  **完全动态环境 (Full Dynamic Environment)**：
    *   代表语言：**LISP**。
    *   特点：名字的性质只有在运行时才能知道。

> **静态与动态的定义**：
> *   **静态**：名字的性质通过说明语句的隐式或显式规则在编译时定义。
> *   **动态**：名字的性质只有在程序运行时才能确定。

### 3. 存储分配策略
*   **原则**：尽可能对数据对象进行**静态分配以提高效率**。

#### (1) 静态存储分配 (Static Allocation)
*   **定义**：在编译时能够确定目标程序运行时所需的全部数据空间大小。编译器将名字直接关联到具体的存储单元。
*   **适用场景**：数据长度固定、无递归、无动态数据结构。

#### (2) 动态存储分配 (Dynamic Allocation)
*   **定义**：在编译时不能确定全部空间，需在运行时动态确定。
*   **两种方案**：
    1.  **栈式动态分配 (Stack-based)**：
        *   在内存中开辟一个栈区。
        *   进入函数/过程时，在栈顶分配空间；返回时释放。
        *   适用于具有嵌套调用和递归特性的语言。
    2.  **堆式动态分配 (Heap-based)**：
        *   在内存中开辟一个堆区（自由区）。
        *   程序运行时按需申请，使用后需显式或隐式释放，归还给自由区。
        *   适用于生命周期不确定的数据对象。

### 4. 运行时内存布局
典型的内存划分如下：

| 区域            | 内容                              | 特点                 |
| :------------ | :------------------------------ | :----------------- |
| **代码区**       | 生成的目标代码（函数入口、指令等）               | 固定长度，编译确定，只读       |
| **静态数据区**     | 全局变量、静态局部变量、常量等                 | 固定长度，编译确定          |
| **栈 (Stack)** | 活动记录（AR）、局部变量、临时变量、参数、控制信息      | 动态增长/缩小，由编译器管理     |
| **自由空间**      | 未分配的内存剩余部分                      | -                  |
| **堆 (Heap)**  | 动态分配的数据对象（如 `malloc/new` 产生的对象） | 动态变化，通常由程序员或GC管理  |

### 5. 活动记录 (Activation Record, AR)
*   **定义**：一块连续的存储区，用于存放过程或函数**一次调用执行**所需要的信息。
*   **重要性**：是运行时存储分配的重要单元。
*   **基本内容**：
    1.  **返回值/结果空间**
    2.  **实际参数 (Actual Parameters)**
    3.  **控制链 (Control Link / DL)**：指向调用该过程前正在运行的过程的活动记录基地址（即返回后恢复的上下文）。
    4.  **存取链 (Access Link / SL)**：指向定义该过程的**直接外层过程**运行时最新活动记录的基地址（用于访问非局部变量）。
    5.  **机器状态信息**：保存调用中断时的寄存器状态等。
    6.  **局部临时变量空间**
    7.  **局部数据空间**

## 7.2 静态运行时环境与存储分配

### 1. 特点
*   编译时确定所有名字对应的存储位置。
*   FORTRAN 是典型代表。

### 2. 示例分析 (FORTRAN)
*   主程序和函数的所有变量都在编译时分配了固定的静态地址。
*   即使函数被多次调用，它们共享同一块存储空间（除非使用 `SAVE` 语句保留状态，但本质上仍是静态地址绑定）。

### 3. 静态分配的局限性
1.  **数据对象限制**：数据对象的长度和在内存中的位置必须在编译时已知。
2.  **不支持递归**：因为一个过程的所有活动实例必须使用同样的局部名字结合（即同一块内存），递归调用会覆盖自身数据。
3.  **不支持动态数据结构**：没有运行时的存储分配机制来创建链表、树等动态结构。

## 7.3 基于栈的运行时环境的动态存储分配

### 1. 核心机制
*   **运行时栈 (Run-time Stack)**：也称为调用栈。
*   **生长与缩小**：
    *   **调用 (Call)**：在栈顶压入一个新的活动记录 (AR)。
    *   **返回 (Return)**：弹出栈顶的 AR，释放空间。
*   **栈指针**：
    *   **SP (Stack Pointer)**：通常指向当前活动记录的底部或顶部（视架构而定）。
    *   **TOP**：指向栈顶空闲位置。

### 2. 关键问题：非局部量的引用 (Non-local Reference)
在允许**过程嵌套定义**的语言（如 Pascal, Ada）中，内层过程可能需要访问外层过程定义的变量。
*   **挑战**：如何找到外层过程当前的活动记录？
*   **解决方案**：
    1.  **静态链/存取链 (Static Link / Access Link, SL)**
    2.  **Display 表 (嵌套层次显示表)**

*(注：C语言不允许函数嵌套定义，因此C语言中通常不需要复杂的SL或Display来访问非局部变量，全局变量直接通过静态地址访问，局部变量通过SP偏移访问。但在递归调用中，DL用于返回。)*

### 3. 方案一：存取链 (Static Link / SL)
*   **定义**：每个活动记录中包含一个指针 `SL`。
*   **指向**：指向**定义**该过程的**直接外层过程**的**最新**活动记录的基地址。
*   **作用**：形成一条链，沿着这条链可以逐级向上查找外层过程的变量。
*   **构建规则**：
    *   如果过程 P 嵌套定义在过程 Q 中，当 P 被调用时，P 的 AR 中的 SL 指向 Q 的当前 AR。
*   **查找效率**：访问第 $k$ 层外层的变量需要沿 SL 链跳转 $k$ 次。

### 4. 方案二：Display 表 (Nested Depth Display Table)
*   **定义**：一个数组（指针数组），通常存放在寄存器或静态存储区，或者作为 AR 的一部分。
*   **结构**：
    *   主程序设为第 0 层。
    *   第 $i$ 层过程中定义的过程为第 $i+1$ 层。
    *   Display 表 $D$ 有 $K+1$ 个单元（假设当前层数为 $K$）。
    *   $D[i]$ 存放着第 $i$ 层过程**最新**活动记录的基地址。
*   **构建规则**（假定从 $i$ 层进入 $j$ 层）：
    1.  **若 $j = i + 1$** (调用当前块的局部过程)：复制第 $i$ 层的 Display 表，将新过程 AR 的地址填入 $D[j]$。
    2.  **若 $j \le i$** (调用外层或同层的全程/非局部过程)：复制来自 $i$ 层过程 AR 中 Display 区的前 $j$ 个记录，将新过程 AR 的地址填入 $D[j]$。
*   **查找效率**：访问第 $k$ 层外层的变量，直接查表 $D[k]$ 即可得到基地址，速度比 SL 快（O(1)）。

### 5. 实例分析：递归与非局部变量

#### 案例 A：C语言递归 (gcd函数)
```c
int gcd(int u, int v) {
    if(v==0) return u;
    else return gcd(v, u%v);
}
```
*   **栈结构**：
    *   `main` AR
    *   `gcd(15, 10)` AR -> DL 指向 main
    *   `gcd(10, 5)` AR -> DL 指向 gcd(15,10)
    *   `gcd(5, 0)` AR -> DL 指向 gcd(10,5)
*   **特点**：C语言无嵌套定义，所以没有 SL 或复杂的 Display 需求用于变量访问，主要靠 DL 维持调用链以正确返回。

#### 案例 B：嵌套过程与非局部变量 (Pascal风格)
```pascal
program main;
VAR x: Integer;
PROCEDURE one; BEGIN Writeln(x); END; // 访问 main 的 x
PROCEDURE p(PROCEDURE a); BEGIN a(); END;
PROCEDURE q;
    VAR x: Integer;
    PROCEDURE two; BEGIN Writeln(x); END; // 访问 q 的 x
    BEGIN
        x := 2;
        p(one);  // 调用 p，传入 one
        p(two);  // 调用 p，传入 two
    END;
BEGIN
    x := 1;
    q;
END.
```

**嵌套层级关系**：
*   `main` (Level 0)
*   `one` (Level 1, defined in main)
*   `p` (Level 1, defined in main)
*   `q` (Level 1, defined in main)
*   `two` (Level 2, defined in q)

**问题解析：**

1.  **活动记录栈（第一次调用 p(one) 中调用 a(one) 之后）**：
    *   栈底 -> 栈顶：
        *   `main` AR (x=1)
        *   `q` AR (x=2, SL指向main)
        *   `p` AR (SL指向main，因为p定义在main中)
        *   `one` AR (SL指向main，因为one定义在main中)

2.  **活动记录栈（第二次调用 p(two) 中调用 a(two) 之后）**：
    *   栈底 -> 栈顶：
        *   `main` AR (x=1)
        *   `q` AR (x=2, SL指向main)
        *   `p` AR (SL指向main)
        *   `two` AR (SL指向q的当前AR，因为two定义在q中)

3.  **程序打印出什么？为什么？**
    *   **第一次调用 `p(one)`**：执行 `one`，引用 `x`，通过 `one` 的 SL 找到 `main` 的 AR，`main` 中 `x = 1`。**输出：1**
    *   **第二次调用 `p(two)`**：执行 `two`，引用 `x`，通过 `two` 的 SL 找到 `q` 的 AR（因为 `two` 定义在 `q` 中），`q` 中 `x = 2`。**输出：2**
    *   **结论**：程序依次打印 **1** 和 **2**。
    *   **原因**：这体现了**静态作用域 (Static Scoping / Lexical Scoping)** 的规则。变量引用的查找依赖于过程定义的文本嵌套结构，而不是调用栈的动态顺序。`one` 定义在 `main` 中，所以它访问 `main` 的 `x`；`two` 定义在 `q` 中，所以它访问 `q` 的 `x`。SL (静态链) 正是为了实现这一机制而存在的。

## 总结

1.  **三种环境**：静态(FORTRAN)、栈式(C/Pascal)、动态(LISP)。
2.  **存储分配**：
    *   静态：编译时定址，无递归。
    *   动态：运行时定址。
        *   栈：管理过程调用、局部变量。
        *   堆：管理生命周期不定的数据。
3.  **活动记录 (AR)**：包含局部数据、控制信息(DL)、访问信息(SL/Display)。
4.  **非局部变量访问**：
    *   **SL (静态链)**：指向定义处外层过程的 AR。查找需遍历链（O(n)），空间开销小。
    *   **Display 表**：数组索引直接指向各层最新 AR。查找快（O(1)），维护开销稍大（进出过程时需更新表）。
5.  **C语言特例**：无嵌套函数，故无需 SL/Display 处理非局部变量，仅需 DL 处理递归返回。
