# 第6章 语义分析与中间代码生成

## 6.1 语义分析综述

### 1. 地位与作用
*   **位置**：位于编译器前端（分析）与后端（综合）之间，是编译程序最实质性的工作。
*   **意义**：第一次对源程序的语义作出解释，引起源程序质的变化。
*   **任务**：按照语法分析器识别的语法范畴进行语义检查和处理，产生相应的中间代码或目标代码。

### 2. 中间代码 (Intermediate Code)
*   **定义**：介于源语言和目标代码之间的一种代码。
*   **引入目的**：
    1.  方便生成目标代码；
    2.  便于优化；
    3.  便于移植（使编译器前端与后端分离）。

## 6.2 语法制导翻译与属性文法

### 1. 基本概念
*   **语法制导翻译 (Syntax-Directed Translation)**：为文法的每一个产生式配一个相应的语义子程序（或语义规则描述的语义动作），并在语法分析的同时调用它。
*   **属性文法 (Attribute Grammar)**：把语义引入文法，给产生式中的文法符号附加"属性"，构成涉及语义的翻译文法。
    *   **形式定义**：$A = (G, V, F)$
        *   $G$：一般为二型文法（上下文无关文法）。
        *   $V$：属性的有穷集。
        *   $F$：关于属性的断言或谓词的有穷集。

### 2. 属性类型
*   **综合属性 (Synthesized Attribute)**：
    *   分析树中，如果一个结点的属性值是通过其**子结点**的属性值计算得到的，则称为综合属性。
    *   *示例*：表达式求值。
        ```text
        E -> E1 + T   { E.val = E1.val + T.val }
        T -> F        { T.val = F.val }
        F -> digit    { F.val = digit.lexval }
        ```
*   **继承属性 (Inherited Attribute)**：
    *   分析树中，如果一个结点的属性值是由该结点的**父结点**和(或)**兄弟结点**的属性定义的，称为继承属性。
    *   *示例*：变量声明中的类型传递。
        ```text
        D -> T L      { L.in = T.type }
        T -> int      { T.type = integer }
        L -> id       { addtype(id.entry, L.in) }
        ```

## 6.3 中间语言
语义程序的输出
中间语言的设计需考虑从源语言到目标语言的翻译跨度及目标机指令集特点。主要形式包括：N元式、逆波兰式、图（树）。

### 1. 逆波兰式 (后缀式 / Reverse Polish Notation)
*   **形式定义**：$e_1 e_2 ... e_n \theta$ ($n \ge 1$)，即运算对象在前，运算符在后。
*   **特点**：适于栈式存储的计算机（堆栈机）。
*   **示例**：
    *   $a * b \rightarrow ab*$
    *   $-a \rightarrow a@$ (@表示单目负)
    *   $a+b*(c+d)*(e+f) \rightarrow abcd+*ef+*+$
*   **控制流处理**：引入操作符 `JF` (Jump if False) 和 `J` (Jump)。
    *   `IF <expr> then <s1> else <s2>` 的逆波兰结构：
        1. `<expr> <4>` $J_F$
        2. `<s1>`
        3. `<5> J`
        4. `<s2>`
        5. ` `

### 2. N元式 (N-tuples)
N个域的记录结构 $(D_1, D_2, ..., D_n)$，包含操作符域和操作对象域。

#### A. 三元式 (Triples)
*   **形式**：`NO. (OP, ARG1, ARG2)`
*   **特点**：
    *   `ARG1` 和 `ARG2` 可以是常数、变量名，也可以是前面某一个三元式的**编号**（代表该三元式的计算结果）。
    *   **缺点**：优化时若移动三元式顺序，引用该结果的编号需要全部修改，维护困难。
*   **示例**：$X = A + B * C$
    1. `(*, B, C)`
    2. `(+, A, (1))`
    3. `(=, X, (2))`

#### B. 间接三元式 (Indirect Triples)
*   **结构**：由**间接码表**（指向三元式表的指针列表）和**三元式表**组成。
*   **优点**：优化时只需调整间接码表的顺序，无需修改三元式表内部的引用，时空效率类似于四元式。
*   **示例**：$(a+b)^2 + (a+b)^3$
    *   **三元式表**：
        1. `(+, a, b)`
        2. `(**, (1), 2)`
        3. `(+, a, b)`  *(可复用(1))*
        4. `(**, (1), 3)`
        5. `(+, (2), (4))`
    *   **间接码表**：按执行顺序排列的指针序列。

#### C. 四元式 (Quadruples)
*   **形式**：`NO. (OP, ARG1, ARG2, Result)`
*   **特点**：
    *   显式指定结果存放位置 `Result`（通常是临时变量 $T_i$）。
    *   **优点**：便于优化，因为操作数直接指向变量或临时变量，不依赖指令序号。
    *   **缺点**：比三元式多占用空间。
*   **示例**：$X = A + B * C$
    1. `(*, B, C, T1)`
    2. `(+, A, T1, T2)`
    3. `(=, T2, _, X)`

### 3. 图/树 (Graphs/Trees)
*   **结构**：
    *   一个三元式对应一棵子树。
    *   `OP` 为根节点。
    *   `ARG1` 为左子节点。
    *   `ARG2` 为右子节点。
*   **关系**：
    *   语法分析的结果通常为树。
    *   树的后序遍历 $\rightarrow$ 逆波兰式。
    *   树的线性化表示 $\rightarrow$ 三元式/四元式。

### 4. 各种表示法的比较综述
*   **逆波兰式**：适于栈式机器。
*   **四元式**：便于优化，常用。
*   **间接三元式**：优化时的时空效率类似四元式，且节省空间。
*   **三元式**：优化困难。

## 6.4 符号表 (Symbol Table)

### 1. 作用
*   存放源程序中有关标识符的属性信息的数据结构。
*   **语义检查依据**：如类型检查、作用域检查。
*   **代码生成依据**：如地址分配、存储空间大小。
*   **动态性**：整个编译过程中动态地采集、记录、变更、引用。

### 2. 标识符的主要属性
1.  **名字 (Name)**：用于查重（考虑作用域）。
2.  **类型 (Type)**：决定存储空间分配及可施加的运算。
3.  **存储类别 (Storage Class)**：如 static, auto, register 等。
4.  **作用域/可视性 (Scope/Visibility)**：所在层次Level。
5.  **其他**：
    *   变量：地址 (Addr)。
    *   数组：维数 (Dim)、每维大小、体积。
    *   过程/函数：入口地址、形参表。
    *   标号：定义标志、地址。

### 3. 结构
通常包含 **名字域** 和 **属性信息域**。对于复杂类型（如结构体、数组），可能需要建立额外的信息向量表（内情向量表）。

## 6.6 语句翻译与中间代码生成

### 设计要点
1.  根据语义确定语句的目标结构（控制流图布局）。
2.  确定中间代码形式（如四元式）。
3.  构造语义子程序。
4.  关键技术：**拉链-返填技术** (Backpatching)。

### 6.6.1 说明类语句的翻译
*   **特点**：一般不产生目标代码，主要作用是填充符号表。
*   **常量说明**：将常量值填入符号表或常量表。
*   **简单变量说明**：
    *   文法：$D \rightarrow \text{int } id \mid \text{float } id \mid D, id$
    *   动作：调用 `fill(ENTRY(id), type)` 将类型填入符号表，并分配相对地址。
*   **复合类型说明 (Struct)**：
    *   结构成员连续存放。
    *   需记录每个成员的偏移量 (Offset) 和类型。
    *   结构整体不能直接引用，只能引用成员。
    *   **内情向量表**：记录结构的总长度、成员信息等。

### 6.6.2 赋值语句与表达式翻译

#### 1. 赋值语句 $A \rightarrow V = E$
*   **目标结构**：先计算表达式 $E$，再将结果存入 $V$。
*   **语义规则**：
    *   $A \rightarrow i = E$: 生成 `(=, E.PLACE, _, ENTRY(i))`
    *   `E.PLACE`：存放表达式值的变量或临时变量名。
    *   `ENTRY(i)`：左部变量在符号表中的入口。

#### 2. 表达式翻译
*   **文法**：$E \rightarrow E \ op \ E \mid op \ E \mid id$
*   **语义规则**：
    1.  $E \rightarrow E_1 \ op \ E_2$: 生成新临时变量，生成 `(op, E1.PLACE, E2.PLACE, E.PLACE)`。
    2.  $E \rightarrow op \ E_1$ (单目): 同上，生成 `(op, E1.PLACE, _, E.PLACE)`。
    3.  $E \rightarrow id$: `E.PLACE = ENTRY(id)`。
*   **示例**：`a = b + c + d`
    1.  `(+, b, c, t1)`
    2.  `(+, t1, d, t2)`
    3.  `(=, t2, _, a)`

### 6.6.3 控制流类语句翻译

控制流语句用于改变程序的执行顺序（向前或向后发生跳转）。在编译时，控制流程的变换依靠生成的代码中的**跳转指令**与对应的**语句标号**来实现。

#### 1. 语句标号与拉链-返填技术 (Backpatching)
*   **语句标号的分类**：
    *   **显式标号**：位于源语句之前，如 `L1: S;`
    *   **隐式标号**：包含在源语句控制流逻辑中，未显式标识（如 `if` 结构中跳过 `else` 分支的目标）。
*   **面临的公共问题**：**先引用后定义**。在单遍扫描的编译器中，经常会遇到产生跳转代码时，目标语句还未被扫描到，目标地址未知的情况（例如向后跳转的 `goto L`）。
*   **拉链-返填技术 (Backpatching)**：
    *   **基本思想**：当遇到未定义的标号时，暂时将跳转指令的目标地址留空（或填0），并将所有跳转到该相同标号的指令链接成一个链表。当该标号真正定义（即确定了目标地址）时，再沿着这个链表把正确的地址"返填"回去。
    *   **实现机制**：建立**语句标号表 (LT)**，记录标号名、定义状态（已定义/未定义）、以及地址（若未定义则存放链表头指针）。拉链时新跳转指向旧跳转（链表尾→链表头）；返填时从链表头遍历到尾，将所有地址域替换为真实地址。

#### 2. 条件语句的翻译
条件语句依赖于对条件表达式 $E$ 的真假测试，通常会生成隐式标号来控制执行流。
*   **`if (E) then S1`**：
    *   计算 $E$，若真执行 $S1$；若假跳转到 $S1$ 之后的 Next。
*   **`if (E) then S1 else S2`**：
    *   增加 `else` 分支。$S1$ 执行完毕后，必须无条件跳转到 $S2$ 之后的 `Next` 处，以跳过 $S2$。
    *   对应的四元式结构：
        ```text
             计算 E.code
             测E: (jT, _, _, E.true)
                  (jF, _, _, E.false)
        E.true:  S1.code
                 (j, _, _, Next)    // S1结束，跳出条件语句
        E.false: S2.code
        Next:    ...
        ```

#### 3. 循环语句的翻译
*   **`while (E) S1`**：
    *   需要在每次循环体 $S1$ 执行完毕后跳回计算 $E$ 的地方。
    *   代码结构：
        ```text
        begin:   E.code
                 测E (jT -> E.true, jF -> E.false)
        E.true:  S1.code
                 (j, _, _, begin)   // 回到循环条件判断
        E.false: ...                // 退出循环
        ```
*   **`for (e1; e2; e3) S`**：
    *   执行顺序为：`e1` -> `e2`真 -> `S` -> `e3` -> `e2`...
    *   代码结构与跳转关系：
        ```text
             e1.code
        L1:  e2.code
             测e2 (jT -> L2, jF -> L3)
        L4:  e3.code
             (j, _, _, L1)          // e3执行完跳回e2判断
        L2:  S.code
             (j, _, _, L4)          // 循环体S执行完跳去执行e3
        L3:  ...                    // 循环结束点
        ```

#### 4. 多分支语句的翻译 (switch-case)
*   **`switch(e) { case c1: S1 break; ... }`**
    *   通常将**测试表集中放置在代码块的末尾**，首先计算 $e$ 的值，然后无条件跳转到测试表。
    *   代码结构：
        ```text
               e.code
               (j, _, _, test)       // 跳往底部的测试表
        L1:    S1.code
               (j, _, _, next)       // 对应 break
        L2:    S2.code
               (j, _, _, next)
               ...
        test:  If e=c1 goto L1       // 测试表集中进行条件分支跳转
               If e=c2 goto L2
               ...
               goto Ln+1             // 对应 default
        next:  ...
        ```

### 6.6.4 数组说明与数组元素引用的翻译

#### 1. 数组说明的翻译
*   **格式**：`type A[d1][d2]...[dn]`。
*   **语义处理**：
    *   记录公共、等长信息（如数组名、类型）到**符号表**。
    *   记录与计算数组元素地址有关的、不等长信息（如维数、各维大小、体积等）到**信息向量表（内情向量表）**。
    *   **动态数组**：若维度大小是变量（即体积在运行时才能确定），内情向量表在编译时仅分配空间，实际的体积、元素偏移量计算以及存储空间分配必须产生中间代码，推迟到**运行时**计算并填写。

#### 2. 数组元素引用的翻译
*   数组元素不能整体引用，必须对单一数组元素进行地址计算。
*   **存储方式**：按行存储（C/C++等主流语言采用）或 按列存储（Fortran等采用）。
*   **地址计算公式**（以一维数组为例，设基址为 $a_0$，元素大小为 `byte`）：
    *   默认下界为0：$Addr = a_0 + i \times byte$
    *   若下界不为0：计算时需要拆分出**不变部分（编译时可算）**和**可变部分（运行时可算）**。
*   **多维数组寻址的优化展开**（按行）：
    *   $A[i_1][i_2]...[i_n]$ 的地址：$Addr = a_0 + Variable\_Part - Constant\_Part$
    *   **可变部分 $V$**：与下标 $i_k$ 相关的项（需在运行时产生中间代码计算）。
    *   **常数部分 $C$**：仅与下界和各维大小相关的项（可在编译时计算出常量）。
    *   令 **$CUN = a_0 - C$**（编译时常量），则运行时只需计算 $T = CUN + V$，大大提高代码运行效率。

### 6.6.5 过程、函数说明和调用的翻译

#### 1. 函数说明（定义）的翻译
*   **语义处理任务**：
    1.  将函数名、类型、形参信息及局部量信息登录到**符号表**中。
    2.  记录函数中局部变量的**作用域 (Level/层次)**，支持运行时的动态环境（如活动记录机制）。

#### 2. 函数调用的翻译
*   **文法**：$S \rightarrow \text{call } id \ (Elist);Elist\rightarrow Elist,E|E$
*   **语义处理任务**：
    1.  **语义检查**：检查被调用的过程或函数是否已定义；实参的数量、顺序、类型是否与形参一致。
    2.  **分配空间**：给被调过程或函数分配**活动记录 (Activation Record)** 所需的存储空间。
    3.  **传参并计算**：计算实参表达式的值，并将其传递给对应的形参（值传递、引用传递等）。
    4.  **保存现场**：加载调用结果和返回地址，以便被调用过程结束后恢复主调用过程的继续执行。
    5.  **跳转**：生成无条件转移指令，将控制流转向被调过程的入口地址。

#### 3. 函数调用返回的语义处理
*   当被调函数执行完毕（遇到 `return`）：
    1.  **返回值处理**：将一个或多个结果值存放到指定的位置（如特定的寄存器或活动记录的特定偏移处）。
    2.  **恢复现场**：恢复调用过程的活动记录（释放被调函数的栈帧）。
    3.  **生成转移指令**：跳转回主调用函数保存的返回地址处，继续向下执行。

### 第6章 总结：语义分析与中间代码生成
*   **核心逻辑**：
    *   **说明性语句** $\Rightarrow$ 登录编译信息，丰富符号表，为后续的可执行语句提供语义检查和代码翻译的信息依据。
    *   **可执行语句** $\Rightarrow$ 结合动态语义和语义动作，将其映射为目标代码结构（即中间代码）。
*   编译过程依赖于源程序文本扫描的顺序性，通过**属性文法**和**语法制导翻译**机制，保证了源程序语义向中间代码映射的准确性，为后端的代码优化与目标机代码生成奠定了坚实的基础。
