## 5.1 传输层提供的服务

### 1. 传输层的核心作用
- **逻辑通信**：为运行在**不同主机上的应用进程**之间提供逻辑通信（进程到进程）。
  - 对比：网络层提供的是**主机到主机**的逻辑通信。
- **协议运行位置**：传输层协议运行在**端系统**（如主机）中，**不运行在路由器中**。
- **与网络层关系**：传输层依赖并强化网络层服务（如尽力而为的IP服务），在其基础上增加可靠性、流量控制、拥塞控制等功能。

> 🏠 **家庭类比**：假设有两个家庭（主机），各有 12 个孩子（进程）。孩子之间通信（写信）→ 信封中的信 = 应用报文；Ann 和 Bill = 传输协议（负责封装/拆封、投递）；邮政服务 = 网络层/IP。

---

### 2. 传输层的功能

#### （1）复用（Multiplexing）与分解（Demultiplexing）
- **复用（发送方）**：
  - 收集来自多个应用进程的数据；
  - 为每块数据添加传输层首部（含源/目的端口号）形成段（segment/datagram）；
  - 交给网络层。
- **分解（接收方）**：
  - 根据段的**源IP、目的IP、源端口、目的端口**等信息；
  - 将段正确交付给对应的应用进程（套接字）。

#### （2）传输层寻址：端口（Port）
- **目的**：标识本机上的不同应用进程。
- **端口号长度**：16 位（范围 0~65535）。
- **端口分类**：

| 类型 | 范围 | 说明 |
|---|---|---|
| **熟知端口**（Well-known） | 0 ~ 1023 | 周知服务使用（如 HTTP=80，FTP=21，DNS=53） |
| **登记端口**（Registered） | 1024 ~ 49151 | 无熟知端口的应用登记使用，需避免冲突 |
| **短暂/客户端端口**（Ephemeral） | 49152 ~ 65535 | 由客户进程临时选用；通信结束后可复用 |

#### （3）连接管理方式
- **无连接服务**（如 UDP）：
  - 无握手过程；
  - 每个段独立处理；
  - 不保证可靠、有序。
- **面向连接服务**（如 TCP）：
  - 通信前需**三次握手**建立连接；
  - 连接中维护状态（序号、窗口等）；
  - 通信后需**四次挥手**释放连接。

---

### 3. 因特网传输层协议对比

| 特性 | **UDP** | **TCP** |
|------|--------|--------|
| 连接方式 | 无连接 | 面向连接（三次握手/四次挥手） |
| 可靠性 | ❌ 不可靠（可能丢包、失序） | ✅ 可靠、按序交付 |
| 流量控制 | ❌ 无 | ✅ 有（滑动窗口） |
| 拥塞控制 | ❌ 无 | ✅ 有（慢开始、拥塞避免、快重传、快恢复） |
| 服务模型 | 尽力而为（Best-effort） | 全功能（可靠流） |
| 首部开销 | 小（8 字节） | 较大（通常 20~60 字节） |
| 典型应用 | DNS、SNMP、视频/语音流、实时游戏 | HTTP、FTP、SMTP、Telnet |

> ⚠️ 注意：因特网传输层**不提供**的服务：
> - 时延保证（如 ≤ 50 ms）
> - 带宽保证（如 ≥ 2 Mbps）

## 5.2 UDP协议（User Datagram Protocol）

### 1. UDP 的核心定位与设计哲学
- **RFC 标准**：[RFC 768](https://datatracker.ietf.org/doc/html/rfc768)
- **服务模型**：  
  - “尽力而为”（Best-effort）传输  
  - **不提供**：可靠性、有序性、流量控制、拥塞控制  
- **设计目标**：
  - ✅ 极简轻量：最小开销、最短延迟  
  - ✅ 无连接（connectionless）：**无握手过程**，每个 UDP 段独立处理  
  - ✅ 适用于容忍丢包、敏感于延迟的应用  

> 📌 **类比**：UDP ≈ 明信片投递，TCP ≈ 挂号信+签收回执

---

### 2. UDP 段格式（8 字节固定首部）

```
 0      7 8     15 16    23 24    31
+--------+--------+--------+--------+
| 源端口 | 目的端口 |   长度   | 校验和 |
+--------+--------+--------+--------+
|           应用数据（报文）         |
+------------------------------------+
```

| 字段 | 长度 | 说明 |
|------|------|------|
| **源端口** | 16 bit | 可选（可为 0），用于接收方回传；若全 0 表示不需回复 |
| **目的端口** | 16 bit | 必须，标识接收应用进程 |
| **长度** | 16 bit | UDP 段总长度（**首部 + 数据**），单位：字节；最小值 = 8（仅首部） |
| **校验和** | 16 bit | 可选（但强烈建议启用），用于检测传输中比特错误 |

> ⚠️ 注意：  
> - UDP 长度字段 ≠ IP 总长度字段（含 IP 首部）  
> - 若应用数据非偶数字节，计算校验和时需填充一个 **全 0 字节**（仅用于计算，**不实际传输**）

---

### 3. UDP 校验和（Checksum）

#### ✅ 目的  
检测 UDP 段在传输过程中是否发生**比特翻转/错误**。

#### 🔧 计算方式（发送方）  
采用**反码求和**（1’s complement addition）：
1. 将 UDP 段视为 **16-bit 字序列**（必要时填充 0 字节凑偶数）；
2. 加上**伪首部**（pseudo header），仅用于计算校验和，**不发送**：

```
伪首部（12 字节）：
  0      7 8     15 16    23 24    31
+--------+--------+--------+--------+
|        源 IP 地址（32 bits）       |
+--------+--------+--------+--------+
|      目的 IP 地址（32 bits）       |
+--------+--------+--------+--------+
|  0  | 协议=17 |   UDP段长度        |
+--------+--------+--------+--------+
```

3. 所有 16-bit 字相加（**回卷**：最高位进位加到低位）；
4. 对最终和**取反码**（bitwise NOT），作为校验和填入 UDP 首部；
5. 若校验和字段为 **0**：表示未计算校验和（不推荐）。

#### 🔍 验证方式（接收方）
1. 将收到的 UDP 段 + 伪首部所有 16-bit 字相加（含校验和字段）；
2. 正确情况下，结果应为 **全 1（即 0xFFFF）**；  
   - 若结果 ≠ 0xFFFF → 出错，**通常直接丢弃该段**（UDP 不重传）。

#### 📐 校验和例子（示意）
```
两字相加：
  11110011 00110011  (0xF333)
+ 10101010 10101010  (0xAAAA)
= 1 10011101 11011101 → 进位1回卷 → +1
=   10011101 11011110  (0x9DDE)
校验和 = 反码 = 01100010 00100001 = 0x6221
```

> ❗ 注意：UDP 校验和可检测多数单比特或多比特错误，但**不能保证 100% 检出**（例如偶数位翻转可能抵消）。

---

### 4. 应用场景与典型协议  
| 应用类型 | 特点 | 协议/应用举例 |
|----------|------|----------------|
| **实时音视频流** | 可容忍少量丢包，极低延迟要求 | VoIP (RTP/UDP), 视频会议、直播 |
| **查询类协议** | 请求-响应短交互、低开销 | DNS（域名解析）、NTP（时间同步） |
| **网络管理** | 简单可靠机制由上层实现 | SNMP（简单网络管理协议） |
| **游戏、IoT** | 快速、轻量、状态常驻应用层 | 多人在线游戏、传感器上报 |

> ⚙️ **重要补充**：  
> - DNS 通常默认使用 UDP（端口 53），**若响应 > 512 字节则转用 TCP**  
> - 某些应用（如 QUIC）在 UDP 上**自己实现可靠传输/拥塞控制**，以突破 TCP 的队头阻塞等限制

## 5.3 TCP协议  
> RFCs: 793, 1122, 1323, 2018, 2581  
> 核心目标：**可靠、有序、全双工的字节流传输**

### 一、TCP 核心特性概览

| 特性 | 说明 |
|------|------|
| **面向连接** | 三次握手建立连接，四次挥手释放；连接状态维护于端系统（非路由器） |
| **点到点** | 一个发送方 ↔ 一个接收方（不支持多播/广播） |
| **可靠、按序交付** | 基于序号/确认/重传；无“报文边界”，视为**连续字节流** |
| **全双工通信** | 连接双方可同时收发数据 |
| **流量控制** | 通过接收方通告窗口（rwnd）防止发送过快导致接收缓存溢出 |
| **拥塞控制** | 通过拥塞窗口（cwnd）动态调整发送速率，避免网络过载 |
| **流水线传输** | 使用滑动窗口机制支持多个未确认段并发传输 |

> 📌 **重要概念辨析**：  
> - **MSS（Maximum Segment Size）**：TCP 报文段**数据字段**的最大长度（不含 TCP/IP 首部）  
> - **MTU（Maximum Transmission Unit）**：链路层帧中 IP 数据报的上限  
> - 关系：`MSS = MTU - IP首部(20) - TCP首部(20)`（典型值 1460 字节）

---

### 二、TCP 报文段格式

```
 0      7 8     15 16    23 24    31
+--------+--------+--------+--------+
| 源端口 | 目的端口 |   数据偏移 | 保留 | URG|ACK|PSH|RST|SYN|FIN |
+--------+--------+--------+--------+
|         序号（Sequence Number）        |
+----------------------------------------+
|       确认号（Acknowledgment Number）   |
+----------------------------------------+
|  窗口大小  |  校验和  | 紧急指针 |      |
+----------+--------+--------+----------+
|               选项（可变长）            |
+----------------------------------------+
|                填充（使首部4字节对齐）    |
+----------------------------------------+
|               应用数据（字节流）         |
+----------------------------------------+
```

#### 🔑 关键字段详解

| 字段 | 长度 | 作用与说明 |
|------|------|------------|
| **源/目的端口** | 各 16 bit | 用于复用/分解，标识应用进程（套接字） |
| **序号（Seq）** | 32 bit | 本报文段**第一个数据字节**在字节流中的编号（按字节编序） |
| **确认号（Ack）** | 32 bit | 期望收到的**下一个字节**的序号 → 累积确认（Cumulative ACK）<br>❗仅当 `ACK=1` 时有效 |
| **数据偏移** | 4 bit | TCP 首部长度（单位：32 bit = 4 字节）；最小值 5（20 字节） |
| **保留** | 6 bit | 全 0 |
| **控制位** | 6 bit | 各 1 bit：<br>• `URG=1`：紧急指针有效（高优先级数据）<br>• `ACK=1`：确认号有效<br>• `PSH=1`：接收方尽快交付上层（不等缓存满）<br>• `RST=1`：重置连接（严重错误）<br>• `SYN=1`：连接请求/应答（建连阶段）<br>• `FIN=1`：发送方数据发完，请求释放连接 |
| **窗口大小（Window）** | 16 bit | 接收方可接收的**字节数**（即 `rwnd`），用于流量控制<br>❗单位是**字节**，非报文段！ |
| **校验和（Checksum）** | 16 bit | 检验范围 = **TCP伪首部 + TCP首部 + TCP数据**（同 UDP，但必选） |
| **紧急指针** | 16 bit | `URG=1` 时有效，指出紧急数据**长度**（位于数据字段开头） |
| **选项** | 可变 | 常见：<br>• **MSS**：协商最大报文段长度<br>• **窗口扩大因子（Window Scale）**：移位值 `S`，新窗口 = 原窗口 << S<br>• **时间戳（Timestamps）**：RTT 估算、防旧报文<br>• **SACK（Selective ACK）**：选择性确认（减少重传） |
| **填充** | 可变 | 使首部长度为 4 字节整倍数 |

> 📌 **关于失序处理**：  
> TCP 规范**未强制要求**实现方式；常见策略：  
> - 缓存失序段，等待缺失段到达后按序交付  
> - 不缓存，直接丢弃（依赖重传）

### 三、TCP 连接管理

#### （1）三次握手（Three-way Handshake）建连

| 步骤 | 发送方 → 接收方 | 关键字段 | 状态变迁（客户 A / 服务 B） |
|------|-----------------|---------|---------------------------|
| 1 | `SYN=1, seq=x` | 主动打开 | `CLOSED` → `SYN_SENT` / `LISTEN` |
| 2 | `SYN=1, ACK=1, seq=y, ack=x+1` | 被动打开 | `SYN_SENT` → `SYN_RCVD` / `LISTEN` → `SYN_RCVD` |
| 3 | `ACK=1, seq=x+1, ack=y+1` | 确认应答 | `SYN_RCVD` → `ESTABLISHED` / `SYN_RCVD` → `ESTABLISHED` |

> ✅ 目的：  
> - 双方确知对方存在  
> - 协商参数（如 MSS）  
> - 分配资源（缓存、连接表项）

#### （2）四次挥手（连接释放）

> 假设 A 主动关闭：

| 步骤 | 发送方 → 接收方 | 关键字段 | 状态 |
|------|-----------------|---------|------|
| 1 | `FIN=1, seq=u` | 主动关闭 | A: `ESTABLISHED` → `FIN_WAIT_1` |
| 2 | `ACK=1, ack=u+1, seq=v` | 被动确认 | A: `FIN_WAIT_1` → `FIN_WAIT_2`；B: `ESTABLISHED` → `CLOSE_WAIT` |
| 3 | `FIN=1, ACK=1, seq=w, ack=u+1` | B 主动关闭 | B: `CLOSE_WAIT` → `LAST_ACK` |
| 4 | `ACK=1, ack=w+1, seq=u+1` | A 最终确认 | A: `FIN_WAIT_2` → `TIME_WAIT`；B: `LAST_ACK` → `CLOSED` |

> ⏳ **A 为何需 `TIME_WAIT` 等待 2MSL？**  
> - 确保 A 的最后一个 ACK 到达 B（若丢失，B 重传 FIN，A 可再确认）  
> - 使本连接所有报文段在网络中“自然消亡”，防止**已失效连接请求**干扰新连接  

#### （3）有限状态机（FSM）要点
- **状态总数 11 种**：如 `CLOSED`, `LISTEN`, `SYN_SENT`, `ESTABLISHED`, `FIN_WAIT_1/2`, `TIME_WAIT`, `CLOSE_WAIT`, `LAST_ACK`, `CLOSING` 等  
- **异常事件**：如收到 `RST` → 直接回 `CLOSED`  
- **同时打开/关闭**：双方同时发 SYN / FIN，有专用路径

---

### 四、TCP 可靠传输机制

#### （1）以字节为单位的滑动窗口

- 每端维护：**发送窗口** & **接收窗口**
- 窗口动态变化，由对方通告的 `rwnd` 控制发送上限  
- 发送窗口三段划分：
  - 已发送且已确认（可释放缓存）  
  - 已发送未确认（等待 ACK）  
  - 可发送未发送（可用窗口 = `rwnd - (已发未确认)`）

#### （2）累计确认 + 超时重传

- **累计确认**：Ack = N 表示“期望收到字节 N”，即 0~N-1 已正确接收  
- **超时重传时间（RTO）动态估算**：

  ```math
  \text{EstimatedRTT} = (1 - \alpha) \cdot \text{EstimatedRTT} + \alpha \cdot \text{SampleRTT} \quad (\alpha = 0.125)
  ```
  ```math
  \text{DevRTT} = (1 - \beta) \cdot \text{DevRTT} + \beta \cdot |\text{SampleRTT} - \text{EstimatedRTT}| \quad (\beta = 0.25)
  ```
  ```math
  \text{TimeoutInterval} = \text{EstimatedRTT} + 4 \cdot \text{DevRTT}
  ```

#### （3）快速重传（Fast Retransmit） & 快速恢复（Fast Recovery）

| 机制 | 触发条件 | 动作 |
|------|---------|------|
| **快速重传** | 收到 **3 个重复 ACK**（表明某段丢失，后续段已到） | **立即重传**丢失段，**不等超时** |
| **快速恢复**（Reno版） | 在快速重传后：<br>① `ssthresh = cwnd / 2`<br>② `cwnd = ssthresh`<br>③ 进入 **拥塞避免**（线性增长） | 避免慢启动导致吞吐骤降<br>（Tahoe 版：重置 `cwnd=1`，已淘汰） |

---

### 五、TCP 流量控制（Flow Control）

- **目的**：防止发送方压垮接收方缓存  
- **机制**：接收方在 ACK 中携带 **`rwnd`（接收窗口）**  
- 若 `rwnd = 0`：
  - 发送方**暂停发送新数据**
  - 启动 **持续计时器（Persistence Timer）**
  - 超时后发**1 字节探测段**，诱使对方回最新 `rwnd`

#### 📈 流量控制示例（见原图）：
```
A 发 seq=1~100    → B 回 ack=101, rwnd=300
A 发 seq=101~200  → B 回 ack=201, rwnd=200
A 发 seq=201~300  → B 回 ack=301, rwnd=100
A 发 seq=301~400  → B 回 ack=401, rwnd=0   → A 停发新数据
A 超时 → 重传旧数据（但不发新）
B 回 ack=501, rwnd=100 → A 继续发 seq=501~600
```

> 🔁 **发送窗口上限**：  
> ```
> SendWindow = min(rwnd, cwnd)
> ```

---

### 六、TCP 拥塞控制（Congestion Control）

> 全局性问题，涉及**所有主机+路由器**  
> ⚠️ 流量控制 ≠ 拥塞控制！

#### （1）核心变量

| 变量 | 含义 |
|------|------|
| `cwnd`（Congestion Window） | 发送方根据网络拥塞状况估算的**可用发送窗口** |
| `ssthresh`（Slow Start Threshold） | 慢开始门限，单位：字节 |

#### （2）四大算法组合使用

##### ① 慢开始（Slow Start）

- `cwnd ← 1 MSS`（建连/超时时重置）
- 每收到一个 ACK → `cwnd += 1 MSS`
- 每 RTT → `cwnd` **指数翻倍**（1, 2, 4, 8…）
- 直至：`cwnd ≥ ssthresh`

##### ② 拥塞避免（Congestion Avoidance）

- `cwnd ≥ ssthresh` 时进入
- 每 RTT → `cwnd += 1 MSS`（**线性增长**）
- 口诀：“**加法增大**”

##### ③ 拥塞发生（超时）

- `ssthresh ← max(2, cwnd/2)` // 乘法减小
- `cwnd ← 1`
- 重新**慢开始**

##### ④ 快速恢复（见前文）

> 📊 **经典过程图**（轮次 vs cwnd）：  
> 慢开始（指数）→ 拥塞避免（线性）→ **拥塞（超时）→ 重置 cwnd=1 + ssthresh÷2** → 慢开始...  
> 或  
> 拥塞避免中→**3 DUP ACK → 快重传+快恢复（cwnd=ssthresh，继续拥塞避免）**

#### （3）发送速率调节逻辑

| 网络状况 | 操作 |
|----------|------|
