## 6.1 网络应用模型

### 一、核心模型分类
| 模型类型 | 特点简述 | 代表应用 |
|----------|-----------|-----------|
| **客户/服务器模型**（Client/Server, C/S） | 服务器长期在线、固定IP；客户机按需连接、动态IP；客户之间不直接通信 | Web、FTP、电子邮件 |
| **对等模型**（Peer-to-Peer, P2P） | 无中心服务器；任意端系统可直接通信；节点动态上线/下线、IP变化频繁 | Gnutella、BitTorrent（纯P2P） |
| **混合模型**（Hybrid） | 结合 C/S 与 P2P 优点：控制/索引用中心服务器，数据传输用 P2P | Napster（文件搜索集中、传输P2P）、即时通信（如QQ：好友在线状态集中管理，聊天P2P） |

---

### 二、客户/服务器模型（Client/Server）

#### ✅ 服务器特点
- **总是打开（Always-on）**：长期运行，提供服务。
- **永久性 IP 地址**：便于客户定位。
- **可扩展性**：可通过服务器集群/负载均衡扩展为“服务器池”。

#### ✅ 客户机特点
- 按需发起连接（**间歇性连接**）。
- IP 地址通常**动态分配**（如 DHCP）。
- **彼此不直接通信**：所有通信需经服务器中转。

#### 📌 优缺点
| 优点 | 缺点 |
|------|------|
| 管理集中、安全可控 | 单点瓶颈、扩展成本高、服务器故障影响大 |

---

### 三、纯 P2P 模型

#### ✅ 核心特征
- **无中心服务器**：所有节点平等（peer）。
- **任意端系统直接通信**：数据直接在 peer 间交换（如文件上传/下载）。
- **高度动态**：
  - 节点间歇连接（如家用电脑仅开机时在线）；
  - IP 地址频繁变化；
  - 节点既是**消费者（client）**又是**服务提供者（server）**（如 Alice 下载同时上传片段给他人）。

#### ✅ 实例说明（见讲义 p.8）
> Alice 使用 P2P 客户端搜索 “Hey Jude”，系统列出拥有该文件的其他 peer（如 Bob）。  
> Alice 向 Bob 发起下载（底层使用 HTTP），**同时她自己也成为临时服务器**，向其他 peer 提供已下载部分 → **高度可伸缩性**（用户越多，服务能力越强）。

#### 📌 优缺点
| 优点 | 缺点 |
|------|------|
| 高可扩展性、成本低（利用用户资源） | 难管理、安全性差、可靠性低（依赖不稳定节点） |

---

### 四、混合模型（Hybrid）

#### ✅ 典型代表 1：Napster（早期音乐共享）
- **文件传输：P2P** → 直接 peer-to-peer 下载。
- **文件搜索：集中式**：
  - 各 peer 启动时向**中心服务器注册**自己共享的文件列表；
  - 查询时也向中心服务器发请求 → 获得目标 peer 地址后直接 P2P 传输。

#### ✅ 典型代表 2：即时通信（如早期 MSN/QQ）
- **聊天通信：P2P**（两人直连，低延迟）；
- **在线状态/定位服务：集中式**：
  - 用户上线 → 向中心服务器注册自身 IP；
  - 想找好友？→ 向中心服务器查询其 IP → 建立 P2P 会话。

#### 📌 优势
- 兼顾**可扩展性**（P2P传数据）与**可控性**（中心管理元数据）；
- 降低纯 P2P 的管理难度，避免纯 C/S 的服务器瓶颈。

## 6.2 DNS系统

### 一、DNS 的作用与必要性

- **人 vs 机器标识方式不同**：
  - 人偏好**易记名字**（如 `www.yahoo.com`）；
  - 网络设备使用**32位 IP 地址**进行寻址和路由。
- **核心问题**：如何实现 **主机名 ↔ IP 地址** 的映射？
- **解决方案**：**域名系统（Domain Name System, DNS）**
  - 是一个**分布式数据库**；
  - 由**层次化**的众多**域名服务器**实现；
  - 同时也是应用层协议：主机、路由器、名字服务器之间通信以完成**名字解析**（Name Resolution）。

> 📌 **注意**：DNS 是因特网核心功能，但却作为**应用层协议实现**（非底层）。

---

### 二、为什么不用集中式 DNS？  
（→ 说明分布式设计的必要性）

| 缺陷 | 说明 |
|------|------|
| **单点故障** | 中心服务器宕机，全网解析失败 |
| **通信量过大** | 所有查询集中处理，难以承载全球查询量 |
| **延迟高** | 全球用户访问同一远程中心，RTT 高 |
| **维护困难** | 需频繁更新海量记录，运维成本极高 |
| ❌ **结论** | **集中式不具可扩展性！** |

---

### 三、DNS 提供的服务（不止是 A 记录！）

| 服务类型 | 说明 | 示例/用途 |
|----------|------|-----------|
| **主机名 → IP 地址转换** |最基本功能| `www.example.com` → `93.184.216.34` |
| **主机别名（Canonical & Alias）** |为同一主机提供多个名字；<br>• **规范名（Canonical Name）**：真实唯一主机名<br>• **别名（Alias）**：便于记忆/服务切换 | `www.ibm.com` 实际是 `east.backup2.ibm.com`（CNAME） |
| **邮件服务器别名** |将邮件域名映射到具体邮件服务器 | `mail.example.com` → `mx1.example.net`（MX 记录） |
| **负载分配** |为高访问量服务提供多个 IP，实现均衡 | 对 `www.google.com` 返回多个 A 记录，客户端轮询或就近选择 |

---

### 四、DNS 的体系结构：分布式、层次化数据库

#### 1. 四类关键服务器（自顶向下）

| 类型 | 功能 | 维护者/示例 |
|------|------|-------------|
| **根 DNS 服务器**（Root） | 最高层；<br>• 当下级服务器无法解析时被联系；<br>• 若自己未知映射，则指引到下一层（如 `.com` 服务器）；<br>• **全球 13 组逻辑根（物理镜像数百个）** | Verisign、US DoD、NASA、ISC 等运营 |
| **顶级域 DNS 服务器**（TLD） | 负责顶级域（Top-Level Domain）：<br>• 通用 TLD：`.com`, `.org`, `.net`, `.edu` …<br>• 国家码 TLD：`.cn`, `.uk`, `.jp`, `.ca` … | `.com`：Network Solutions； `.edu`：Educause |
| **权威 DNS 服务器**（Authoritative） | 为特定组织/域提供**权威解析**（即“官方”记录）；<br>• 存储该域内主机与 IP 的映射；<br>• 可由组织自建或托管给 ISP/云商 | `dns.cs.umass.edu` 是 `cs.umass.edu` 的权威服务器 |
| **本地 DNS 服务器**（Local / Default） | ≠ 严格属于层次结构；<br>• 每个 ISP/企业/大学都部署（如 `dns.poly.edu`）；<br>• 用户主机 DNS 请求**首先发给它**；<br>• 起**代理作用**，代表用户向层次结构发起查询 | 学校网络的 DNS、家庭路由器的 DNS 转发器 |

> 🌐 典型查询路径（`www.amazon.com`）：  
> 客户 → **本地 DNS** → **根 DNS** → `.com` **TLD** → `amazon.com` **权威 DNS** → 返回 IP  

---

### 五、DNS 查询方式

| 方式 | 特点 | 图例对应（讲义 p.18） |
|------|------|---------------------|
| **递归查询**（Recursive Query） | 客户请求本地 DNS“全权负责”，本地 DNS 必须返回最终答案（或报错）；<br>• **负担压在被请求服务器**（如本地 DNS） | 图中 **粗箭头（1,2,3）**：客户→本地→根→TLD→权威，再原路返回答案 |
| **迭代查询**（Iterative Query） | 每级服务器“知道多少说多少”：<br>• 若不知答案，仅返回“你应该问谁”（下一个服务器地址）；<br>• **负担分散**，更健壮 | 图中 **虚线箭头（4–8）**：本地→根→TLD→权威，每步返回“下一步去哪” |

> ✅ 实际中：**客户 ↔ 本地 DNS** 通常为 **递归**；**本地 DNS ↔ 上层服务器** 通常为 **迭代**

---

### 六、DNS 缓存（Caching）与更新

- **缓存机制**：
  - 任一 DNS 服务器获知某映射后，**缓存该记录**（如本地 DNS 缓存 `.com` TLD 地址）；
  - 缓存项含 **TTL**（Time-To-Live）字段，过期后自动删除；
  - 显著**减少查询延迟 & 上层服务器负载**（根服务器极少被直接访问）。

- **动态更新**（补充知识）：
  - 标准：**RFC 2136**（DNS UPDATE）；
  - 支持客户端/服务器动态增删改记录（如 DHCP 分配 IP 后更新 DNS）；
  - 应用：云服务弹性 IP 自动注册、私有网络服务发现等。

---

### 七、DNS 资源记录（RR）

格式统一为四元组：  
**`(name, value, type, ttl)`**

| Type | 含义 | `name` 示例 | `value` 示例 |
|------|------|-------------|-------------|
| **A** | 主机地址 | `www.example.com` | `93.184.216.34` |
| **NS** | 域的权威服务器 | `foo.com` | `dns.foo.com` （主机名）|
| **CNAME** | 规范名别名 | `www.example.com` | `server-east.example.net` （真实主机名）|
| **MX** | 邮件服务器 | `example.com` | `mail1.example.net` （邮件服务器名）|

> ✅ 实例（讲义 p.23）：  
> 注册 `networkutopia.com` 需在 `.com` TLD 插入：  
> - `(networkutopia.com, dns1.networkutopia.com, NS)`  
> - `(dns1.networkutopia.com, 212.212.212.1, A)`  
> 再在其权威 DNS 插入：  
> - `(www.networkutopia.com, [IP], A)`  
> - `(mail.networkutopia.com, mx.networkutopia.com, MX)`

---

### 八、DNS 协议与报文结构（p.21–22）

- **两类报文**：查询（Query）、回答（Reply）——**格式相同**。
- **报文头部关键字段**：
  - `16-bit 标识符（ID）`：匹配请求与响应；
  - `标志位（Flags）`：
    - QR（Query/Response bit）
    - RD（Recursion Desired）：客户端是否希望递归
    - RA（Recursion Available）：服务器是否支持递归
    - AA（Authoritative Answer）：响应是否来自权威服务器

- **正文部分**：
  - **问题部分**：查询的域名 + 查询类型（如 A、MX）；
  - **回答部分**：0+ 条资源记录（RR）；
  - **权威部分**：推荐的权威服务器 NS 记录；
  - **附加部分**：辅助信息（如 NS 记录对应的 A 记录 → “glue record”）。

## 6.3 FTP：文件传输协议（File Transfer Protocol）

### 一、基本概述

- **功能**：在**客户机与远程主机之间**上传/下载文件  
- **模型**：**客户机/服务器模型**
  - **客户机**（FTP Client）：发起文件传输请求的一方（上传 or 下载）
  - **服务器**（FTP Server）：远程主机，提供文件服务  
- **标准**：**RFC 959**
- **服务器端口**：**21**（用于控制连接）

> 📌 典型流程：  
> 用户 → 本地文件系统 → FTP 客户机 →（网络）→ FTP 服务器 → 远程文件系统

---

### 二、双连接机制：控制连接 vs 数据连接

FTP **同时维护两个 TCP 连接**，这是其核心特征：

| 连接类型 | 端口 | 方向 | 功能 | 特点 |
|----------|------|------|------|------|
| **控制连接** | TCP 21 | 客户机 → 服务器 | 发送命令与接收响应：<br>• 用户认证（`USER`, `PASS`）<br>• 目录浏览（`LIST`）<br>• 文件传输指令（`RETR`, `STOR`） | • **“带外”（out-of-band）**：命令独立于数据<br>• **持久连接**：在整个会话期间保持打开<br>• 服务器**保持状态**（如当前目录、登录状态） |
| **数据连接** | TCP 20（主动模式） | 服务器 → 客户机 | 传输**实际文件内容**或目录列表 | • **按需建立**：服务器在收到传输命令后主动发起<br>• **用完即关**：一个文件传输完成后即关闭<br>• 下一文件 → 新建连接 |

> ✅ **图示对应（p.26）**：  
> ```
> FTP 客户机 ⇄〔TCP 控制连接，端口21〕⇄ FTP 服务器  
> FTP 客户机 ←〔TCP 数据连接，端口20〕← FTP 服务器
> ```

---

### 三、典型工作流程

1. 客户机向服务器 **21端口** 建立 **TCP 控制连接**
2. 通过控制连接完成：
   - 认证：`USER <username>`，`PASS <password>`
   - 浏览目录：`LIST`
   - 发起传输命令（如 `RETR filename`）
3. 服务器收到文件传输命令后：
   - 向客户机 **主动建立（主动模式）** 一个 **TCP 数据连接**（通常用 **20端口**）
   - 通过**数据连接**传输文件
4. 文件传输完毕 → **关闭数据连接**
5. 后续文件 → 重复步骤3（新建数据连接）
6. 会话结束：客户机发送 `QUIT`，**关闭控制连接**

> ⚠️ 注意：FTP 是**有状态协议**——服务器需维护用户登录状态、当前工作目录等。

---

### 四、常用 FTP 命令与响应（ASCII 文本格式）

#### 常用命令（经控制连接发送）：

| 命令 | 说明 |
|------|------|
| `USER <username>` | 提交用户名 |
| `PASS <password>` | 提交密码 |
| `LIST` | 获取当前目录文件列表 |
| `RETR <filename>` | **下载**（Retrieve）文件 |
| `STOR <filename>` | **上传**（Store）文件 |
| `QUIT` | 终止会话 |

#### 常见响应码（类比 HTTP，3位数字 + 可读短语）：

| 响应码 | 含义 |
|--------|------|
| `331` | Username OK, password required |
| `125` | Data connection already open; transfer starting |
| `425` | Can’t open data connection |
| `452` | Error writing file |
| `226` | Closing data connection; transfer successful |
| `221` | Service closing control connection |

> 📌 提示：可使用 `telnet <ftp-server> 21` 手动测试 FTP 命令交互（p.27）。

---

### 五、补充说明：传输模式与安全

- **传输格式**：
  - 支持 **ASCII 模式**（文本文件，自动换行转换）
  - **二进制模式**（`TYPE I`）：用于图片、程序等不可转换数据  
- **安全缺陷**：
  - 用户名/密码**明文传输**
  - 控制/数据连接均未加密 → 易被嗅探  
- **现代替代方案**：
  - **FTPS**（FTP over SSL/TLS）
  - **SFTP**（SSH File Transfer Protocol，**非 FTP 扩展**，基于 SSH）

## 6.4 电子邮件

### 一、电子邮件系统三大组成部分

| 组件 | 功能说明 | 举例 |
|------|----------|------|
| **用户代理**（User Agent, UA） | 又称“邮件阅读器”（Mail Reader）<br>• 撰写、编辑、阅读邮件<br>• 与邮件服务器交互（发送/接收） | Outlook、Thunderbird、Eudora、elm、Netscape Messenger |
| **邮件服务器**（Mail Server） | 每用户必备；包含：<br>• **邮箱**（Mailbox）：接收并存储进来的邮件<br>• **报文队列**（Message Queue）：暂存待发送的邮件 | Microsoft Exchange、Postfix、Sendmail |
| **SMTP**（Simple Mail Transfer Protocol） | 邮件服务器之间 **传输邮件的标准协议**（RFC 2821）<br>• 客户端：发送方邮件服务器<br>• 服务端：接收方邮件服务器 | 基于 TCP 端口 25 |

> 📌 典型流程（p.32，Alice → Bob）：
> 1. Alice 用 UA 写信发往 `bob@someschool.edu`  
> 2. UA 将信提交至 Alice 的邮件服务器，放入报文队列  
> 3. Alice 的服务器（SMTP client）→ Bob 的服务器（SMTP server）建立 TCP 连接  
> 4. 通过 SMTP 发送邮件  
> 5. Bob 的服务器将邮件存入其邮箱  
> 6. Bob 用 UA 读取邮件  

```
[ UA_A ] → [ Mail Server_A ]  
     \          ↓ SMTP (TCP/25)  
      \         [ Mail Server_B ] → [ UA_B ]
```

---

### 二、SMTP 协议详解（RFC 2821）

#### ✅ 基本特性
- **传输层**：TCP（端口 25），可靠传输  
- **工作模式**：**推协议**（Push）——发送方主动推给接收方  
  > 对比 HTTP：**拉协议**（Pull）  
- **报文格式限制**：
  - 只允许 **7-bit ASCII** 编码（头部 + 主体）  
  - 以 `CRLF.CRLF`（即 `\r\n.\r\n`）标志报文结束  

#### ✅ 通信三阶段（p.31）
1. **握手阶段**（HELO/EHLO）
2. **报文传输阶段**（MAIL FROM → RCPT TO → DATA → `.`）
3. **关闭阶段**（QUIT）

#### ✅ 交互示例（p.33）

```text
S: 220 hamburger.edu                         ← 服务器欢迎
C: HELO crepes.fr                            ← 客户机宣告域名
S: 250 Hello crepes.fr, pleased to meet you
C: MAIL FROM:<alice@crepes.fr>              ← 发件人
S: 250 alice@crepes.fr... Sender ok
C: RCPT TO:<bob@hamburger.edu>              ← 收件人（可多个）
S: 250 bob@hamburger.edu... Recipient ok
C: DATA                                      ← 开始传输正文
S: 354 Enter mail, end with "." on a line by itself
C: Subject: Hello
C: 
C: Do you like ketchup?
C: How about pickles?
C: .                                         ← 结束标记
S: 250 Message accepted for delivery
C: QUIT
S: 221 hamburger.edu closing connection
```

#### ✅ 常见响应码（p.33）
| 码 | 含义 |
|----|------|
| 220 | 服务就绪（Server ready）|
| 250 | 请求完成成功 |
| 354 | 开始输入报文 |
| 550 | 用户邮箱不可用（如 `550 No such user`）|

> 🛠 实验方法：  
> `telnet mail.example.com 25` 手动输入命令，可绕过邮件客户端直接发信（p.34）

#### ✅ 其他特性
- **持久连接**：多个邮件可复用同一 TCP 连接（非每个邮件新建）  
- **无加密/无身份认证**（原生 SMTP）→ 易伪造、易嗅探  
  > 现代增强：**SMTP AUTH**（扩展认证）、**STARTTLS**（加密）

---

### 三、邮件报文格式与扩展

#### 1. 基本文本报文格式（RFC 822）
```

Subject: Hello ketchup lovers!
Date: Thu, 21 Dec 2023 10:00:00 +0800

Do you like ketchup?

```
- **头部**（Header）：以 `:` 分隔的字段（如 From/To/Subject/Date）  
- **空行**：CRLF 分隔头与体  
- **主体**（Body）：纯 ASCII 文本  

> ⚠️ 注意：`From:` 是**报文头部字段**；`MAIL FROM:` 是**SMTP 命令**，二者不同！

#### 2. MIME 扩展（RFC 2045, 2046）  
支持多媒体：图片、音频、附件等  
关键头部字段：
| 字段 | 作用 |
|------|------|
| `MIME-Version: 1.0` | 声明 MIME 版本 |
| `Content-Type: type/subtype` | 指明媒体类型（如 `image/jpeg`, `text/html`, `multipart/mixed`）|
| `Content-Transfer-Encoding:` | 指定编码方式（如 `base64`, `quoted-printable`）|

> ✅ 示例（图片邮件，p.37）：
> ```text
> MIME-Version: 1.0
> Content-Type: image/jpeg
> Content-Transfer-Encoding: base64
> 
> /9j/4AAQSkZJRgABAQEAYABgAAD/2wBDAAg...
> ```

### 四、邮件访问协议（从服务器到 UA）

| 协议 | 全称 | 特点 | 状态性 |
|------|------|------|--------|
| **POP3** | Post Office Protocol v3 (RFC 1939) | • 下载邮件到本地<br>• 默认“下载后删除”（可选“下载并保留”）<br>• 命令简单（`USER`, `PASS`, `LIST`, `RETR`, `DELE`, `QUIT`） | **无状态**：会话间不保留状态；换设备无法同步 |
| **IMAP** | Internet Message Access Protocol (RFC 1730/3501) | • 邮件**保留在服务器**<br>• 支持文件夹管理、部分下载、搜索<br>• 多设备同步 | **有状态**：会话间保持用户状态（如文件夹结构、已读标记）|
| **HTTP** | Webmail（如 Gmail、Hotmail、QQ 邮箱） | 通过浏览器访问，后端使用 POP/IMAP/自定义协议 | 依赖 Web 会话（Cookie） |

#### ✅ POP3 交互示例（p.39）
```text
C: USER bob
S: +OK
C: PASS hungry
S: +OK user successfully logged on
C: LIST
S: 1 498   ← 邮件#1，498字节
S: 2 912
S: .
C: RETR 1
S: <message 1 content>
S: .
C: DELE 1   ← 删除邮件#1
C: QUIT
S: +OK POP3 server signing off
```

#### ✅ POP3 vs IMAP 关键对比（p.40）
| 场景 | POP3 | IMAP |
|------|------|------|
| 多设备访问 | ❌ 困难（邮件已下载本地） | ✅ 所有设备看到相同状态 |
| 服务器存储 | 尽快清空（默认） | 长期保留（主流做法） |
| 网络流量 | 一次性下载 | 按需加载（节省流量） |
| 功能丰富性 | 低 | 高（搜索、标记、文件夹） |

## 6.5 WWW 与 HTTP

### 一、WWW 基本概念

| 术语 | 说明 |
|------|------|
| **Web页（Web Page）** | 由若干**对象**（Objects）组成 |
| **对象（Object）** | 可寻址的数据单元，如：<br>• HTML 文件（基本文件）<br>• 图片（JPEG/GIF）<br>• Java 小程序（Applet）<br>• 音频/视频文件等 |
| **URL**（Uniform Resource Locator） | 唯一标识对象的地址：<br>`http://www.someschool.edu/someDept/pic.gif`<br>→ `主机名` + `路径名` |

> 📌 典型 Web 页结构：  
> 一个基础 HTML 文件 + 若干 **引用对象**（如 10 张图片）

---

### 二、HTTP 协议概览（RFC 1945 / RFC 2616）

| 属性 | 说明 |
|------|------|
| **全称** | 超文本传输协议（HyperText Transfer Protocol） |
| **模型** | 客户/服务器（Client/Server）<br>• 客户：浏览器（Explorer, Chrome, Safari, Navigator）<br>• 服务器：Web 服务器（如 Apache） |
| **传输层** | TCP（端口 **80**）——保证可靠传输 |
| **连接过程** | 1. 建立 TCP 连接<br>2. 浏览器发送 HTTP 请求报文<br>3. 服务器响应 HTTP 报文<br>4. 关闭 TCP 连接 |
| **状态性** | ✅ **无状态**（Stateless）<br>• 服务器不保存客户请求历史<br>• 简化设计，但无法直接支持会话 |

> ⚠️ 为何无状态？  
> • 维护状态会增加复杂性（崩溃后状态不一致需重建）  
> • 但实际应用需要状态 → 引入 **Cookie**（见后文）

---

### 三、HTTP 连接类型

| 类型 | 特点 | 使用版本 | 缺点/优点 |
|------|------|----------|-----------|
| **非持久连接**（Non-persistent） | • 每个对象单独使用一个 TCP 连接<br>• 服务器响应后立即关闭连接 | HTTP/1.0（默认） | ❌ 每对象需 2×RTT：<br> – 1 RTT 建连<br> – 1 RTT 发请求+收响应首部<br>❌ 高连接开销（OS 资源分配） |
| **持久连接**（Persistent） | • **单个 TCP 连接传输多个对象**<br>• 服务器响应后保持连接开放 | HTTP/1.1（默认） | ✅ 减少 RTT 与连接开销<br>• **非流水线**：前一响应收完再发下一请求 → 每对象 1 RTT<br>• **流水线**（默认）：遇到引用立即发请求 → **所有对象 ≈ 1 RTT** |

> 📊 **响应时间建模（非持久连接）**：  
> 总时间 = `2 × RTT + 传输时间`  
> → 请求 1 个 HTML + 10 张图 = `22 × RTT + Σ传输时间`

---

### 四、HTTP 报文格式

#### 1. 请求报文（Request）

```http
GET /somedir/page.html HTTP/1.1
Host: www.someschool.edu
Connection: close
User-agent: Mozilla/4.0
Accept-language: fr

```
- **结构**：
  - **请求行**：`方法 URL 版本`
  - **首部行（Header）**：`字段名: 值`
  - **空行**（`\r\n\r\n`）：标志报文头结束
  - **实体主体**（可选）：如 POST 表单数据

- **常用方法**：
  | 方法 | 用途 |
  |------|------|
  | `GET` | 请求指定资源（URL 或表单参数） |
  | `POST` | 提交数据（如表单），数据放在**实体主体**中 |
  | `HEAD` | 只请求首部（如检测资源最后修改时间） |
  | `PUT` | 上传文件到服务器指定路径（需权限） |
  | `DELETE` | 删除指定资源（需权限） |

> 🛠 实验：`telnet cis.poly.edu 80`，手动输入：
> ```
> GET /~ross/ HTTP/1.1
> Host: cis.poly.edu
> 
> ```

#### 2. 响应报文（Response）

```http
HTTP/1.1 200 OK
Connection: close
Date: Thu, 06 Aug 1998 12:00:15 GMT
Server: Apache/1.3.0 (Unix)
Last-Modified: Mon, 22 Jun 1998 ...
Content-Length: 6821
Content-Type: text/html

<html>...实际HTML内容...</html>
```
- **结构**：
  - **状态行**：`版本 状态码 短语`
  - **首部行**
  - **空行**
  - **实体主体**（如 HTML 文件内容）

- **常见状态码**：
  | 类别 | 码 | 含义 |
  |------|----|------|
  | 2xx 成功 | `200 OK` | 请求成功，内容在报文体中 |
  | 3xx 重定向 | `301 Moved Permanently` | 资源已永久迁移，新位置在 `Location:` 首部中 |
  | 4xx 客户错 | `400 Bad Request` | 请求语法错误<br>`404 Not Found`：资源不存在 |
  | 5xx 服务错 | `505 HTTP Version Not Supported` | 服务器不支持请求的 HTTP 版本 |

### 五、Cookie：实现“有状态”交互

尽管 HTTP 无状态，但**Cookie 机制**可模拟会话状态：

#### ✅ Cookie 四要素
1. 响应中的 `Set-Cookie:` 首部（服务器 → 客户）  
2. 请求中的 `Cookie:` 首部（客户 → 服务器）  
3. 客户本地的 **Cookie 文件**（浏览器管理）  
4. 服务器后端的 **用户状态数据库**

#### ✅ 工作流程（p.58）
```
首次访问：
  Client → GET → Server
  Server → 200 OK + Set-Cookie: ID=1678 → Client
  Client 保存 {ID=1678}
下次访问：
  Client → GET + Cookie: ID=1678 → Server
  Server 查数据库：ID 1678 → Susan
```

#### ✅ 应用场景
- 用户身份识别（登录态）
- 购物车内容保持
- 个性化推荐
- 会话跟踪（Web 邮箱、后台管理）

#### ⚠️ 隐私问题
- Cookie 可被用于**跨站追踪**
- 第三方 Cookie（广告商）易泄露用户行为
- GDPR/CCPA 等法规要求明确告知与用户授权

### 六、Web 缓存（代理服务器 / Proxy Cache）

#### ✅ 原理
- 用户浏览器配置使用**缓存服务器**
- 请求流程：
  1. 浏览器 → 缓存服务器
  2. 若缓存命中 → 直接返回（**低延迟**）
  3. 若未命中 → 缓存 → 起始服务器取回 → 返回客户 + 本地缓存副本

#### ✅ 部署位置
- ISP（大学、公司、住宅宽带出口）

#### ✅ 好处
| 优势 | 说明 |
|------|------|
| ✅ 降低用户响应时间 | 命中时响应在毫秒级 |
| ✅ 减少接入链路流量 | 缓存 40% 命中率 → 接入链路负载降 40%（p.64） |
| ✅ 减轻源站压力 | 热门内容可完全由缓存服务 |

#### ✅ 缓存一致性：**条件 GET**（Conditional GET）
> 避免传输未修改对象，节省带宽

- 客户/缓存发请求时加首部：  
  `If-Modified-Since: <上次缓存日期>`
- 服务器响应：
  - 未修改 → `304 Not Modified`（无实体体）
  - 已修改 → `200 OK` + 新内容

```http
GET /image.jpg HTTP/1.1
If-Modified-Since: Wed, 10 Jan 2024 08:00:00 GMT
```
→ 若未改：`HTTP/1.1 304 Not Modified` ✅

## 6.6 套接字编程：创建网络应用

### 一、核心概念：什么是套接字（Socket）？

- **定义**：套接字是**进程与网络之间的接口**，是应用层与传输层的“门”（door）  
- **类比**：  
  > 应用进程 ↔ **门（套接字）** ↔ 运输层（TCP/UDP）  
  > 门的“门牌号” = **IP 地址 + 端口号**
- **作用**：  
  • 发送数据：→ 将数据**推入**门（由运输层封装发送）  
  • 接收数据：← 从门**拉出**数据（接收运输层交付的数据）

> ✅ **关键接口函数**（BSD Socket API）：  
> `socket()`, `bind()`, `listen()`, `accept()`, `connect()`, `send()`, `recv()`, `close()`

---

### 二、UDP vs TCP 编程模型对比

| 特性 | UDP | TCP |
|------|-----|-----|
| **连接** | ❌ 无连接 | ✅ 面向连接（需三次握手） |
| **可靠性** | ❌ 不可靠（可能丢包、乱序） | ✅ 可靠、有序、无差错 |
| **服务器状态** | ❌ 无连接状态 | ✅ 维护每个客户连接状态 |
| **套接字类型** | `SOCK_DGRAM`（数据报套接字） | `SOCK_STREAM`（流式套接字） |
| **典型应用** | DNS 查询、流媒体、VoIP | Web（HTTP）、邮件（SMTP）、文件传输（FTP） |

---

### 三、UDP 套接字编程详解（p.68–73）

#### 1. 核心函数（Python `socket` 模块）

| 函数 | 用途 |
|------|------|
| `socket.socket(socket.AF_INET, socket.SOCK_DGRAM)` | 创建 UDP 套接字 |
| `sendto(data, (host, port))` | 向指定地址发送数据（含目标 IP + Port） |
| `recvfrom(buffer_size)` | 接收数据，返回 `(data, (sender_IP, sender_port))` |

#### 2. UDP 服务器/客户端工作流程

```
          UDP Server                        UDP Client
        ┌──────────────┐                 ┌──────────────┐
        │ socket()     │                 │ socket()     │
        │ bind()       │                 │              │
        │              │                 │              │
        │ loop:        │                 │              │
        │   recvfrom() ◄────── Datagram ─── sendto()    │
        │   …process… │                 │              │
        │   sendto() ────── Datagram ───► recvfrom()   │
        │              │                 │              │
        └──────────────┘                 └──────────────┘
```

#### 3. 示例：简单回显服务器（p.69）

**UDP Server**（无连接，可处理多个客户）
```python
from socket import *
serverPort = 12000
serverSocket = socket(AF_INET, SOCK_DGRAM)
serverSocket.bind(('', serverPort))             # 绑定本地所有接口

print("UDP 服务器就绪…")
while True:
    message, clientAddress = serverSocket.recvfrom(2048)
    modified = message.decode().upper()
    serverSocket.sendto(modified.encode(), clientAddress)
```

**UDP Client**（每次需指定服务器地址）
```python
from socket import *
serverName = 'localhost'
serverPort = 12000
clientSocket = socket(AF_INET, SOCK_DGRAM)

message = input('输入小写句子：')
clientSocket.sendto(message.encode(), (serverName, serverPort))
response, serverAddress = clientSocket.recvfrom(2048)
print("来自服务器：", response.decode())
clientSocket.close()
```

#### ✅ UDP 特点总结
- 服务器**不维护连接状态**
- 同一服务器套接字可与**多个客户端**交替通信
- `sendto()` / `recvfrom()` **显式携带地址信息**

---

### 四、TCP 套接字编程详解（p.73–81）

#### 1. 核心函数（Python）

| 函数 | 用途 |
|------|------|
| `socket.socket(socket.AF_INET, socket.SOCK_STREAM)` | 创建 TCP 套接字 |
| `bind((host, port))` | 绑定本地地址（服务器必需） |
| `listen(backlog)` | 开始监听连接请求（`backlog`：最大等待队列长度） |
| `accept()` | **阻塞等待**客户连接；返回 `(conn_socket, client_addr)` |
| `connect((server, port))` | 客户端主动发起连接 |
| `send(data)` / `recv(bufsize)` | 在**已连接套接字**上读写（✔️ 无需地址） |

> ⚠️ **关键区分**：
> - **欢迎套接字**（`serverSocket`）：用于接收新连接（`accept()`）
> - **连接套接字**（`connSocket`）：用于与**特定客户**通信（`send()`/`recv()`）

#### 2. TCP 服务器/客户端工作流程

```
          TCP Server                          TCP Client
        ┌─────────────────┐                 ┌─────────────────┐
        │ socket()        │                 │ socket()        │
        │ bind()          │                 │                 │
        │ listen()        │                 │                 │
        │ accept() ◄──────┼─ TCP 握手 ───────┼─ connect()      │
        │   → connSocket  │                 │                 │
        │ loop:           │                 │ loop:           │
        │   recv() ◄──────┼──── Data ───────┼─ send()         │
        │   send() ───────┼──── Data ───────┼► recv()         │
        │   …             │                 │   …             │
        └─────────────────┘                 └─────────────────┘
```

#### 3. 示例：TCP 回显服务器（p.74）

**TCP Server**
```python
from socket import *
serverPort = 12000
serverSocket = socket(AF_INET, SOCK_STREAM)
serverSocket.bind(('', serverPort))
serverSocket.listen(1)          # 最多1个等待连接
print("TCP 服务器启动…")

while True:
    connSocket, addr = serverSocket.accept()     # 创建新连接套接字
    sentence = connSocket.recv(1024).decode()
    capitalized = sentence.upper()
    connSocket.send(capitalized.encode())
    connSocket.close()            # 关闭本次连接（非服务器套接字！）
```

**TCP Client**
```python
from socket import *
serverName = 'localhost'
serverPort = 12000
clientSocket = socket(AF_INET, SOCK_STREAM)
clientSocket.connect((serverName, serverPort))  # 主动握手

sentence = input('输入句子：')
clientSocket.send(sentence.encode())
response = clientSocket.recv(1024)
print("来自服务器：", response.decode())
clientSocket.close()
```

#### ✅ TCP 特点总结
- `accept()` **阻塞**直至客户连接到达
- 每客户对应**一个独立连接套接字**（可并发扩展为多线程/多进程）
- `send()`/`recv()` 操作对象是**字节流**（无消息边界）
- 连接关闭：客户/服务器均可 `close()`，触发 TCP 四次挥手

### 五、并发服务器模型（p.78–79 拓展）

讲义虽未深入，但指出“一个客户一个套接字”的 TCP 模型天然支持并发：

| 模型 | 实现方式 | 适用场景 |
|------|---------|---------|
| **多进程** | `fork()` 创建子进程处理 `connSocket` | Unix/Linux，简单但开销大 |
| **多线程** | `threading.Thread` 启动新线程 | 跨平台，轻量，需注意同步 |
| **I/O 多路复用** | `select()` / `poll()` / `epoll()` | 高并发（C10K+）|
| **异步 I/O** | `asyncio`（Python） / `libevent` | 高性能服务器（如 Nginx） |

> 📌 简单并发服务器示例（伪代码）：
> ```python
> while True:
>     conn, addr = welcome_socket.accept()
>     start_thread(handle_client, conn)   # 新线程处理 conn
> ```

### 六、套接字编程注意事项

| 项目 | 说明 |
|------|------|
| **端口号** | • 0–1023：Well-known ports（需 root）<br>• >1023：用户进程可用（如 12000） |
| **字节编码** | Python 3 中 `send()`/`recv()` 处理 `bytes`，需 `.encode()` / `.decode('utf-8')` |
| **阻塞 I/O** | 默认函数为阻塞式（`accept`, `recv`, `connect`） |
| **错误处理** | 必加 `try/except`（如连接拒绝、超时） |
| **资源释放** | 务必 `close()` 套接字，避免端口泄漏 |
