## 3.1 数据链路层的基本功能

### 1. 基本概念辨析

| 术语                       | 定义                                                     |
| ------------------------ | ------------------------------------------------------ |
| **链路 (Link)**            | 一条无源（无中间交换节点）的点到点物理线路段，即物理信道。                          |
| **数据链路 (Data Link)**     | 协议 + 链路，即在物理链路上加上协议控制，实现可靠通信的逻辑通道。                     |
| **结点 (Node)**            | 网络中参与通信的实体，包括**主机 (Host)**、**交换机**、**路由器 (Router)** 等。 |
| **点到点 (Point-to-Point)** | 在**相邻两个结点之间**的一条链路上进行的通信。                              |
| **端到端 (End-to-End)**     | 从**源结点**到**目的结点**的通信，路径可能经过多个链路与结点。                    |

> 📌 **图示示意**：
> ```
> 端到端路径： 源结点 —[点到点]— 中间结点 —[点到点]— … —[点到点]— 目的结点
> ```

---

### 2. 为网络层提供的服务类型

数据链路层向网络层提供三种典型服务：

| 服务类型 | 特点 | 适用场景 |
|----------|------|----------|
| **无确认的无连接服务** | - 不建立连接<br>- 发送独立帧，**无需确认**<br>- 不检测/恢复丢帧 | - 误码率极低的信道<br>- 实时性要求高、容忍少量丢包（如语音传输）<br>- **绝大多数 LAN 默认采用** |
| **有确认的无连接服务** | - 不建立连接<br>- 每帧需**单独确认**<br>- 超时未确认 → 重传 | - 高误码率、不可靠信道（如无线通信） |
| **有确认的面向连接的服务** | - 三阶段：**连接建立** → **数据传输** → **连接释放**<br>- 提供可靠、有序传输 | - 高可靠性 & 实时性要求高的场景 |

---

### 关键总结

- 数据链路层解决的是**相邻结点之间**的通信问题，核心是帧的**可靠、有效传输**。
- “**无确认无连接**”是局域网主流服务，因介质好、效率高；
- 无线等高误码环境倾向于“**有确认无连接**”或“**面向连接**”；
- 数据链路层的可靠性 ≠ 网络层（如 IP）或传输层（如 TCP）的可靠性——各层职责不同。

## 3.2 组帧（Framing）

### 1. 基本概念

- **组帧**：将物理层传输的原始比特流，按一定规则**切分为离散的数据单元**，这些单元称为**帧（Frame）**。
- 数据链路层在相邻结点之间的逻辑信道（“数字管道”）上传输的基本单位是**帧**。

---

### 2. 组帧的主要方法

| 方法 | 原理 | 优点 | 缺点/注意事项 |
|------|------|------|---------------|
| **字符计数法** | 帧头含一个字段表示**整个帧的字符数（含计数字段本身）** | 实现简单 | **计数出错将导致本帧及后续帧全部错位**；可靠性差 |
| **首尾字符定界法**（带字符填充） | 使用**特殊字符**标记帧起始/结束；若数据中出现该字符或控制字符，则**插入转义字符**（如 `ESC`）进行填充 | 解决定界问题；兼容字符数据 | 依赖特定字符集（如 ASCII）；实现略复杂 |
| **首尾标记定界法**（带位填充） | 使用**特殊比特串**（如 `01111110`，称为 **Flag**）标记帧边界；发送端在连续 5 个 `1` 后**插入 1 个 `0`**（**0 比特插入**）；接收端删除该 `0`（**0 比特删除**） | 适用于任意比特数据；通用性强（如 HDLC、PPP） | 需硬件/软件支持位填充/删除 |
| **物理层编码违例法** | 利用物理层编码的**冗余特性**（如曼彻斯特编码中 `high-high` / `low-low` 不合法）作为帧定界符 | 无需额外开销（无填充）；高效 | 仅适用于特定物理编码（如 IEEE 802.3/4/5 LAN） |

---

### 3. 关键细节补充

#### 3.1 字符计数法（示例与问题）

- 帧头第 1 字节为长度字段（值含该字节）  
  例如：`[6][A][B][C][D][E]` 为一帧（6 字节）；下一帧从 `[6]` 开始。  
- ❗ 若长度值被噪声篡改为 `7`，则本帧将多读 1 字节，导致**本帧解析错误 + 后续帧全部错位**。

#### 3.2 带字符填充的首尾字符定界法  
- 起止符：`SOH`（Start of Header）、`EOT`（End of Transmission）  
- 填充规则：
  - 数据中出现 `SOH` / `EOT` / `ESC` → 插入 `ESC`（转义）
  - 实际发送：`ESC + X`，其中 `X` 为 `SOH/EOT/ESC` 的替换字符（如 `ESC SOH` → `ESC x`）

#### 3.3 带位填充的首尾标记定界法（重点！）

- **标志字段 Flag = `01111110`**
- **发送方**：扫描数据 → 每遇 `5` 个连续 `1`，立即插入 `1` 个 `0`
  - 例：数据为 `011011111111111111110010`  
    填充后：  
    `011011111` **`0`** `11111` **`0`** `11111` **`0`** `10010`  
    再前后加 Flag → `01111110 [上述比特] 01111110`
- **接收方**：扫描比特流 → 每遇 `5` 个连续 `1` 后的 `0`，立即**删除该 `0`**
  - 可还原原始数据

> ⚠️ 注意：Flag 本身也有 `6` 个连续 `1`，**不会触发填充**（因是前导/后缀，非用户数据）

#### 3.4 物理层编码违例法（IEEE 802 LAN）

- **曼彻斯特编码**：  
  - `1` = `高→低` 跳变  
  - `0` = `低→高` 跳变  
  - 非法组合：`高→高` / `低→低`  
- 用这些**非法跳变**作为帧起始/结束定界符，**无需额外比特**。

---

### 4. 小结：组帧的核心目标

| 目标 | 说明 |
|------|------|
| **透明性** | 无论用户数据内容如何（含 Flag/控制符/特殊比特），都能正确切分帧 |
| **自同步** | 接收方可从比特流中识别帧边界（靠 Flag / 编码违例等） |
| **高效性** | 尽量减少填充/额外开销（如编码违例法零开销） |
| **容错性** | 单点错误不应导致连续多帧解析失败（字符计数法最差） |

---

✅ **最常用方法**：**带位填充的首尾标记定界法**（PPP、HDLC 等广泛采用）。

## 3.3 差错检测和纠正

### 1. 差错的基本概念

| 项目 | 说明 |
|------|------|
| **差错** | 接收的数据 ≠ 发送的数据 |
| **随机差错** | 码元错误相互独立，与前后无关<br>→ **主要由热噪声**（导体电子热运动）引起 |
| **突发差错** | 相邻多个数据位同时出错（成串）<br>→ **主要由冲击噪声**（外界电磁干扰）引起<br>⚠️ **冲击噪声是引起差错的最主要原因** |

---

### 2. 差错评价指标

| 指标 | 定义 | 公式 | 备注 |
|------|------|------|------|
| **误码率（Pe）** | 错传的码元数 / 所传码元总数 | $ P_e = \frac{N_e}{N} $ | 一般指 **误比特率**（二进制情况下两者等价） |
| **典型值** | | | - 普通电话线：$10^{-4} \sim 10^{-6}$<br>- 多数通信系统：$10^{-5} \sim 10^{-9}$ |

> 🔍 注：对二进制系统，误比特率 = 误码率。

---

### 3. 差错控制的基本方式

| 方式 | 原理 | 适用场景 | 特点 |
|------|------|----------|------|
| **反馈纠错（ARQ）** | 接收端发现差错但**不能定位**，请求重发，直到确认为止 | 双工通信、非实时系统（如文件传输） | ✅ 简单；❌ 有延迟、开销大 |
| **前向纠错（FEC）** | 接收端能**定位并纠正**少量错误，无需重传 | 单工通信、实时系统（如电话、直播） | ✅ 无重传延迟；❌ 纠错码冗余高 |
| **混合纠错（HEC）** | 少量错误自动纠正；超出能力时请求重传 | 高可靠 & 实时兼顾的场景（如现代无线） | ✅ 折中方案；综合性能优 |

---

### 4. 误码控制基本原理

- **核心思想**：通过添加**冗余码**（监督码）构造校验关系  
  → 使得：
  - 无误时：信息码 + 冗余码 满足特定数学关系；
  - 出错时：该关系被破坏 → 可检测/定位/纠正。

- **编译码模型**：
  - 码字长度：$ n $ 位  
  - 信息码长度：$ m $ 位  
  - 冗余码长度：$ r = n - m $ 位  
  - 码字 = 信息码 + 冗余码

---

### 5. 检错与纠错能力：汉明距离（Hamming Distance）

- **定义**：两个等长码字之间**对应位不同**的位数。  
  例：`010110` 与 `110010` 汉明距离 = 3（第1、4、5位不同）

- **关键关系**：
  - 若编码的**最小汉明距离为 $ d_{\min} $**，则：
    - 最多可**检错**：$ d_{\min} - 1 $ 位  
    - 最多可**纠错**：$ \left\lfloor \dfrac{d_{\min} - 1}{2} \right\rfloor $ 位  

  > 📌 例：$ d_{\min} = 3 $ ⇒ 可检 2 位错，或纠 1 位错。

---

### 6. 常见检错码概述（后续章节展开）

| 方法 | 原理简述 | 特性 |
|------|----------|------|
| 奇偶校验码 | 每帧加 1 bit，使 `1` 的总数为奇（或偶） | ✅ 简单；❌ 只能检**奇数位**错；无法纠 |
| 循环冗余校验 **（CRC）** | 基于模2除法，生成校验码（FCS） | ✅ **高效检突发错**；广泛用于 MAC 层（如以太网）；❌ 不可纠 |
| 海明码 | 多组奇偶校验位+特定位置安排 | ✅ 可纠 1 位错（$ d_{\min}=3 $）；常用于内存 ECC |

> ⚠️ 注：教材后续（如 3.6.2 以太网）会重点讲解 **CRC 的应用**（FCS 字段）。

---

### 📌 小结

| 要点 | 说明 |
|------|------|
| **差错根源** | 随机差错（热噪声） vs 突发差错（冲击噪声）；后者更严重 |
| **控制策略** | ARQ（重传）、FEC（纠）、HEC（混合）按需选用 |
| **核心资源** | **冗余比特**——以带宽/存储换可靠性 |
| **能力界限** | 由**最小汉明距离 $ d_{\min} $** 决定检错/纠错上限 |
| **实际应用** | 数据链路层普遍采用 **检错 + ARQ重传**（如以太网+FCS+CRC）；FEC多用于物理层或高误码无线链路 |

## 3.4 可靠传输机制

### 一、设计目标与基本思路

可靠传输机制的核心目标是：  
✅ **在不可靠信道上实现无差错、不丢失、不重复、按序交付** 的数据帧传输。  
➡️ 主要手段：**确认（ACK）＋ 重传（Retransmission）＋ 序号（Seq）＋ 定时器（Timer）**

> 📌 注：链路层的可靠传输主要用于**高误码率链路**（如无线链路），而多数有线局域网（如以太网）**仅检错、不重传**（依赖上层如 TCP 实现端到端可靠）。

---

### 3.4.1 完全理想化的单工传输

#### 🧱 基本假设（两个理想化条件）：
1. **信道完全可靠**：无差错、无丢失、无重复、无乱序；  
2. **接收方无限缓冲**：可无限快接收，**无需流量控制**。

#### 📐 模型示意：
```
主机 A（发送方）──[数据帧]──► 主机 B（接收方）
           单工、无反馈、无确认
```

> ⚠️ 实际中不存在——仅用于协议演进的起点。

---

### 3.4.2 停-等协议（Stop-and-Wait, S&W）

#### ✅ 保留条件1：**信道无差错**  
#### ❌ 放弃条件2：**接收方速率可能低于发送方** → 需**流量控制**

#### 🔧 改进方式：
- 发送一帧后**暂停**，等待接收方返回 **ACK（确认帧）**；  
- 收到 ACK 后，才发下一帧。

#### ⏱️ 时间线对比：
| 无流量控制（错误） | 有流量控制（S&W） |
|------------------|------------------|
| A 持续发帧 → B 缓冲溢出 | A 发一帧 → 等 ACK → 再发 |

#### ✅ 优点：
- 实现简单；  
- 自然实现**流量控制**（接收方控制发送节奏）；  
- 帧序可隐含（因每次只发一帧）。

#### ❌ 缺点：
- **信道利用率低**  
  $$
  \text{利用率} = \frac{L / b}{L / b + R} = \frac{L}{L + Rb}
  $$
  - $L$：帧长（bit）  
  - $b$：信道速率（bps）  
  - $R$：往返时延 RTT（s）  
  > 📌 例：卫星链路 $L=1000$, $b=50\,\text{kbps}$, $R=500\,\text{ms}$ → 利用率 ≈ 4%

---

### 3.4.3 可靠的停-等协议（ARQ 版）

#### ❌ 放弃理想条件1：**信道可能出错/丢帧**

#### ➕ 新增机制：
| 机制 | 作用 | 实现方式 |
|------|------|----------|
| **序号（Seq）** | 区分新帧与重传帧 | 1-bit 序号（0/1 轮转） |
| **确认（ACK）** | 告知帧已正确接收 | ACK 中带序号 $n$（对应刚收的帧） |
| **定时器（Timer）** | 检测丢失/超时 | 发帧后启动；超时 ⇒ 重传 |
| **校验（如 CRC）** | 检出帧错 | 出错帧直接丢弃（不 ACK） |

#### 🔄 四种典型情况：

| 情况 | 示意图 | 处理 |
|------|--------|------|
| **(a) 正常** | A→(0)→B；B→ACK0→A | A 收 ACK0 后发下一帧（Seq=1） |
| **(b) 帧出错** | A→(0, 出错)→B（丢弃） | A 超时 ⇒ 重传帧0 |
| **(c) 帧丢失** | A→(0)→×（丢） | A 超时 ⇒ 重传帧0 |
| **(d) ACK 丢失** | A→(0)→B；B→ACK0→× | A 超时 ⇒ 重传帧0；B 收重复帧0 ⇒ 丢弃 + **再发 ACK0** |

> ✅ **为何不用 NAK（否认帧）？**  
> ——因超时重传本身已隐含“未确认”，**ACK + 超时** 足矣；简化设计。

#### 📉 缺陷：
- 仍为**单帧流水线**，效率低；  
- 未解决**突发高误码下大量重传**问题。

---

### 3.4.4 滑动窗口协议（Sliding Window Protocol）

#### 💡 核心思想：**连续发送多帧** → **流水线（pipelining）**

#### 🔄 关键概念：

| 项目 | 说明 |
|------|------|
| **发送窗口 $W_S$** | 允许已发但**未确认**的最大帧数；控制发送流量 |
| **接收窗口 $W_R$** | 允许接收的帧序号范围；控制接收顺序与去重 |
| **窗口滑动** | 收到 ACK 后，窗口向前滑动（释放已确认帧） |

#### 🖼️ 窗口示意图（发送端）：
```
[已发已确认][已发未确认][可发][不可发]
     ↑         ↑        ↑
   LAR       LFS      LAR+W_S
（Last ACK Rcvd）（Last Frame Sent）
```

#### ⚙️ 重要特性：
- 全双工通信 ⇒ 可**捎带确认**（Piggybacking）：ACK 附在反向数据帧中；  
- $W_S = 1, W_R = 1$ ⇒ **停-等协议**（单帧滑动窗口）；  
- $W_S > 1$ ⇒ 支持并发传输。

---

### 3.4.5 后退 N 帧协议（Go-Back-N, GBN）

#### 🎯 目标：提高信道利用率（解决 S&W 效率低）

#### 📏 参数：
- 固定发送窗口大小 $W_S$（例：$W_S = 7$）  
- **接收窗口 $W_R = 1$**（只收期望序号帧）

#### 📥 接收端行为：
- 正确收到**期望帧** ⇒ 送交上层 + 发 **累积 ACK**（确认到该序号为止所有帧）；  
- 收到**非期望帧** ⇒ **直接丢弃**（即使后续帧正确也不缓存）。

#### 📤 发送端行为：
- 收到 ACK $n$ ⇒ 窗口滑动至 $n+1$；  
- **超时 ⇒ 重传窗口内所有未确认帧**（从最早未确认帧开始）。

#### ⚠️ 缺点：
- **1 帧出错 ⇒ 后续正确帧全丢 + 全重传** ⇒ 高误码下效率骤降；  
- 接收端不能缓存失序帧 ⇒ 浪费链路能力。

---

### 3.4.6 选择重传协议（Selective Repeat, SR）

#### 🎯 改进 GBN 缺陷：**仅重传出错/丢失帧**

#### 📏 参数：
- $W_S > 1$, $W_R > 1$  
- **关键约束**：  
  $$
  W_S + W_R \leq 2^n,\quad W_S \geq W_R \quad\Rightarrow\quad W_S, W_R \leq 2^{n-1}
  $$
  > 📌 例：2-bit 序号（0~3），最大 $W_S = W_R = 2$；若 $W_S=3$，会出现**新帧 vs 重传帧歧义**。

#### 📥 接收端行为：
- 正确收到**任一窗口内帧** ⇒ 缓存 + 发 **单帧 ACK**；  
- 按序交付上层（等前序帧补齐后）。

#### 📤 发送端行为：
- 收到 ACK $n$ ⇒ 标记帧 $n$ 为已确认；  
- **超时 ⇒ 仅重传该帧**；  
- 窗口滑动：当**最小未确认帧被确认**时滑动。

#### ✅ 优点：
- 高误码下仍高效；  
- 充分利用带宽。

#### ❌ 缺点：
- 实现复杂（需帧缓存、个体 ACK/Timer）；  
- 窗口大小受序号空间限制。

---

### ✅ 可靠传输机制总结表

| 机制        | 用途 | 说明 |
|-------------|------|------|
| **差错检测** | 检错 | CRC/FCS 等，发现比特错误 |
| **定时器**   | 防丢失 | 超时 ⇒ 重传（应对丢帧/ACK 丢失） |
| **序号**     | 防重复/乱序 | 区分新帧、重传帧；检出丢失/重复 |
| **确认（ACK）** | 反馈 | 告知接收成功；可累积或逐帧 |
| **窗口/流水线** | 提效率 | 并发发多帧；窗口大小 = 并发度 |

---

### 📌 最终小结

- **停-等**：简单，但效率低 → 适用于低速/高延迟链路；  
- **GBN**：实现较简单，但重传代价高 → 适用于**误码率较低**环境；  
- **SR**：效率最高，但复杂 → 适用于**高误码率**链路（如无线）；  
- 数据链路层是否实现可靠传输，**取决于链路特性**（以太网不做，PPP/HDLC 做）。

## 3.5 介质访问控制（MAC）

### 一、MAC 层的核心问题

| 对比项 | ARQ 协议 | MAC 协议 |
|--------|----------|----------|
| **目标** | 可靠传输帧（解决丢帧、延时、重排序） | 共享介质协调（解决多用户冲突） |
| **参与者** | 仅两个对等实体 | 同一广播域内的**所有站点协同动作** |
| **关注点** | 信道误码、丢包 | 其他**用户干扰**（冲突） |

> ✅ **核心问题**：多个站点共享同一广播信道时，如何**有序、高效、公平**地接入媒体？

---

### 3.5.1 信道划分协议（Channel Partitioning Protocols）

#### 📌 思想：**静态划分信道资源**，避免冲突  
> ⚠️ 缺点：低负载时资源浪费严重（如仅 1 个活跃站，也只能用 1/N 带宽）

| 方法 | 原理 | 应用举例 |
|------|------|---------|
| **频分复用 FDM** | 将频带划分为多个子频带，各站独占一频段 | 有线电视、早期同轴以太网 |
| **时分复用 TDM** | 将时间划为固定时隙，各站轮流使用固定时隙 | 数字程控交换、同步卫星通信 |
| **波分复用 WDM** | 光频下的 FDM（不同波长 = 不同信道） | 光纤骨干网（8×2.5 Gb/s → 20 Gb/s） |
| **码分复用 CDMA** | 所有站共享频率/时间，靠**正交编码**区分用户 | 蜂窝网（3G WCDMA、CDMA2000） |

#### 🔍 CDMA 关键原理：
- 每个用户分配唯一**码片序列** $c_m$（如 `[+1, −1, −1, +1]`）
- 发送比特 $d_i$ → 编码为 $Z_{i,m} = d_i \cdot c_m$
- 接收端用**内积**解码：$\hat{d_i} = \sum_{m=1}^M Z_{i,m} \cdot c_m$
- ✅ 多用户可并发；  
- ❌ 编码非正交 ⇒ 干扰；需功率控制。

---

### 3.5.2 随机访问协议（Random Access Protocols）

#### 📌 思想：**站点自主决定发送时机**；冲突 ⇒ 重传  
> ✅ 低负载高效；❌ 高负载冲突开销大

##### （1）ALOHA 协议
| 类型 | 特点 | 效率 |
|------|------|------|
| **纯 ALOHA** | 帧产生即发；冲突窗口 = **2 帧时长** | 最大吞吐率 $S = \frac{1}{2e} \approx 18.4\%$ |
| **时隙 ALOHA** | 仅在时隙起点发帧 ⇒ 冲突窗口 = **1 帧时长** | 最大吞吐率 $S = \frac{1}{e} \approx 36.8\%$ |

##### （2）载波监听多路访问 CSMA
| 类型                 | 行为                                  | 冲突可能             |
| ------------------ | ----------------------------------- | ---------------- |
| **1-坚持 CSMA**      | 闲？→ 立发；忙？→ 持续监听                     | 高（传播时延 ⇒ “隐蔽冲突”） |
| **非坚持 CSMA**       | 闲？→ 立发；忙？→ 随机等待再试                   | 较低（但信道利用率↓）      |
| **p-坚持 CSMA**（时隙化） | 闲？→ 以概率 $p$ 发，$(1-p)$ 延迟；忙？→ 下一时隙再试 | 折中               |

> 🌐 典型应用：**有线以太网（CSMA/CD）**、**无线局域网（CSMA/CA）**

##### （3）CSMA/CD（Collision Detection）  
**用于有线局域网（如传统以太网）**

- **CD = 冲突检测**：边发边监听；冲突 ⇒ 立停 + 发 **48 bit 干扰信号（Jamming）**
- **争用期**（Collision Window）：  
  - 长度 = $2\tau$（端到端往返传播时延）
  - 若发帧 **≤ 64 字节** 前无冲突 ⇒ 本次发送成功
- **最小帧长 = 64 字节**：确保冲突发生在发送期内
- ⚠️ **无法用于无线**：
  - ① 全双工收发器成本高；
  - ② “**隐蔽站/暴露站**” 问题使检测无效。

##### （4）CSMA/CA（Collision Avoidance）  
**用于无线局域网（IEEE 802.11）**

- **CA = 冲突避免**：无法检测 ⇒ 主动避免
- **核心机制**：
  - **帧间间隔 IFS**：高优先级帧用更短 IFS（SIFS < PIFS < DIFS）
  - **虚拟载波监听**：通过 NAV（Network Allocation Vector）预约信道
  - **随机退避**：信道空闲后仍需退避（二进制指数退避）
  - **RTS/CTS 握手**（可选）：解决隐蔽站问题  
    `A → RTS → B → CTS → A → Data → B → ACK`

---

### 3.5.3 轮流协议（Taking-Turns MAC Protocols）

#### 📌 思想：**有序轮流发送**（牺牲低负载效率，换取高负载公平性）

| 协议 | 原理 | 缺点 |
|------|------|------|
| **轮询（Polling）** | 主站轮询从站：“可发？” | 1. 轮询开销<br>2. 主站单点故障 |
| **令牌传递（Token Passing）** | 站点持“令牌”方可发；发完传给下一站 | 1. 令牌开销<br>2. 令牌丢失/损坏需维护<br>3. 单点故障（令牌/站点） |

> ✅ 典型应用：**令牌环网（IEEE 802.5）**、**令牌总线（IEEE 802.4）**

---

### 📊 三类 MAC 协议对比总结

| 特性 | 信道划分 | 随机访问 | 轮流访问 |
|------|----------|----------|----------|
| **低负载效率** | ❌ 差（资源固定分配） | ✅ 高（单站可占满信道） | ❌ 中（需等待轮询/令牌） |
| **高负载效率** | ✅ 高（无冲突） | ❌ 差（冲突多） | ✅ 高（公平无冲突） |
| **延迟确定性** | ✅ 高（固定时隙/频段） | ❌ 低（竞争不确定） | ✅ 中（最坏等一圈） |
| **实现复杂度** | 中 | 低（CSMA）→ 高（CSMA/CA） | 高（需协调逻辑） |
| **典型应用** | 光纤骨干、卫星 | 以太网（CSMA/CD）、Wi-Fi（CSMA/CA） | 令牌环、工业控制网 |
