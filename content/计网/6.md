# 计算机网络与工程 - 第6章 应用层笔记

## 6.1 网络应用模型

### 概述
- 应用层是网络协议栈的最上层，直接面向用户提供各种网络服务
- 本章涵盖的主要网络应用包括：E-mail、Web、即时消息、远程登录、P2P文件共享、多用户网络游戏、流式视频、因特网电话、实时视频会议、大规模并行计算等
- 网络应用开发原理：编写在不同端系统上运行并通过网络通信的程序（如Web服务器软件与浏览器软件通信）
- 网络核心设备不处理应用层功能，这种设计使应用开发更快速

### 客户/服务器模型
- **服务器特点**：
  - 始终在线的主机
  - 拥有永久性IP地址
  - 可扩展为服务器集群
- **客户端特点**：
  - 与服务器通信
  - 可能间歇性连接
  - 可能具有动态IP地址
  - 客户端之间不直接通信

### 对等(P2P)模型
- **纯P2P特点**：
  - 无始终在线的服务器
  - 任意端系统直接通信
  - 对等节点间歇性连接，IP地址经常变化
  - 例如：Gnutella
  - 优势：高度可扩展
  - 劣势：难以管理
- **P2P文件共享实例**：
  - 用户(Alice)在笔记本上运行P2P客户端
  - 间歇连接网络，每次获得新的IP地址
  - 寻找资源时，应用程序显示拥有该资源的其他对等节点
  - 选择对等节点(Bob)后，通过HTTP直接从Bob的PC下载文件
  - 下载同时也在向其他用户提供上传
  - 所有对等节点既是客户端也是临时服务器，实现高度可扩展性

### 混合模型
- **Napster模型**：
  - 文件传输采用P2P方式
  - 集中式文件搜索：对等节点在中心服务器注册内容，并查询同一中心服务器定位内容
- **即时消息模型**：
  - 用户间聊天采用P2P
  - 集中式在线状态检测/定位：
    - 用户在线时向中心服务器注册IP地址
    - 用户联系中心服务器发现伙伴的IP地址

## 6.2 DNS系统

### DNS概述
- **问题背景**：人类使用名字(如www.yahoo.com)而网络使用IP地址(32位)，需要两者间的映射
- **DNS定义**：
  - 分布式数据库，由层次化的多台名字服务器实现
  - 应用层协议，主机、路由器和名字服务器通过它进行名字解析(地址/名字转换)
  - 是因特网核心功能，但实现为应用层协议
- **集中式DNS不可行原因**：
  - 单点故障风险
  - 通信量过大
  - 远程集中式数据库延迟高
  - 维护困难
  - 缺乏扩展性

### DNS服务功能
- 主机名到IP地址转换
- 主机别名处理（规范名和别名）
- 邮件服务器别名
- 负载分配（例如为一个规范名配置多个IP地址的冗余Web服务器）

### DNS层次结构
![[Pasted image 20260110100854.png]]
- **根DNS服务器**：
  - 当本地DNS无法解析名字时被联系
  - 若未知名字映射，会联系下层DNS服务器获取映射并返回给本地DNS服务器
- **顶级域(TLD)服务器**：
  - 负责com, org, net, edu等通用顶级域以及国家顶级域如cn, uk, fr, ca, jp
  - 例如：Network Solutions维护com顶级域服务器，Educause维护edu顶级域服务器
- **权威DNS服务器**：
  - 组织的DNS服务器，为组织的服务器(Web和电子邮件)提供IP地址映射
  - 可由组织自己或服务提供商维护
- **本地DNS服务器**：
  - 不严格属于等级结构
  - 每个ISP(住宅、公司、大学)都有一个，也称为"默认名字服务器"
  - 当主机发出DNS请求时，请求被发送到本地DNS服务器
  - 作为代理，将请求转发到DNS层次结构

### DNS解析过程
- **递归查询**：
  - 将名字解析负担放在被联系的名字服务器
  - 服务器负责完成全部解析工作
- **迭代查询**：
  - 被联系的服务器用要联系的下一个名字服务器地址回应
  - 例如回复："我不知道该名字，但你可以询问这个服务器"
- **解析流程示例**：
  1. 位于cis.poly.edu的主机想查询gaia.cs.umass.edu的IP地址
  2. 请求发送至本地DNS服务器
  3. 本地DNS服务器联系根DNS服务器
  4. 根DNS服务器指引到edu TLD服务器
  5. TLD服务器指引到umass.edu权威服务器
  6. 权威DNS服务器提供gaia.cs.umass.edu的IP地址
  7. 信息按原路返回给请求主机

### DNS优化机制
- **缓存机制**：
  - DNS服务器知道映射后会缓存该信息
  - 缓存项在一定时间后超时失效
  - 顶级域服务器信息通常缓存在本地DNS服务器中，减少对根DNS服务器的访问
- **记录更新**：
  - 正在由IETF设计更新/通知机制(RFC 2136)

### DNS记录格式
- **资源记录(RR)格式**：(name, value, type, ttl)
- **主要记录类型**：
  - **Type=A**：name是主机名，value是IP地址
  - **Type=NS**：name是域名(如foo.com)，value是该域的权威DNS服务器主机名
  - **Type=CNAME**：name是别名，value是规范主机名（例如www.ibm.com实际是服务器east.backup2.ibm.com）
  - **Type=MX**：name是邮件域，value是邮件服务器主机名

### DNS协议与报文
- **报文结构**：
  - 查询和回答报文具有相同格式
  - **报文首部**包含：
    - 标识符(16位)：用于匹配查询和回答
    - 标志位(flags)：标明查询/回答、是否需要递归、是否提供递归、回答是否权威等
  - **查询区域**：包含请求查询的名字和类型
  - **回答区域**：包含对查询的资源记录响应
  - **权威区域**：包含权威服务器的记录
  - **附加信息区域**：包含额外有用的"有帮助的"信息

### DNS记录插入流程
- 例如创建新公司Network Utopia：
  1. 在注册机构注册域名networkutopia.com，提供权威DNS服务器名和IP
  2. 注册机构在com TLD服务器插入两条RR：
     - (networkutopia.com, dns1.networkutopia.com, NS)
     - (dns1.networkutopia.com, 212.212.212.1, A)
  3. 在权威服务器插入：
     - www.networkutopia.com的Type A记录
     - mail.networkutopia.com的Type MX记录

## 6.4 电子邮件

### 电子邮件系统组成
- **三大部分**：
  1. **用户代理（UA）**：
     - 也称为"邮件阅读器"
     - 用于写作、编辑、阅读邮件报文
     - 例如：Eudora, Outlook, elm, Netscape Messenger
     - 负责处理存储在服务器上的报文的到达和离开
  2. **邮件服务器**：
     - 邮箱：包含用户的到达报文
     - 输出报文队列：离开(将发送)邮件报文的队列
  3. **简单邮件传输协议（SMTP）**：
     - 用于在发送电子邮件报文的邮件服务器之间进行通信
     - 客户机：发送邮件服务器
     - 服务器：接收邮件的服务器

### SMTP协议（RFC 2821）
- **传输特点**：
  - 使用TCP从客户机到服务器可靠地传输电子邮件报文
  - 用端口25
  - 直接传输：发送服务器到接收服务器
- **传输的三个阶段**：
  1. 握手(欢迎)
  2. 报文的传输
  3. 关闭
- **报文格式要求**：**首部和主体**必须以7比特ASCII格式
- **SMTP vs HTTP**：
  - HTTP：拉协议
  - SMTP：推协议
  - 两者都有ASCII命令/响应交互，使用状态码
- **其他特点**：
  - 使用持久连接
  - SMTP客户机使用CRLF.CRLF以决定报文结束

### 邮件发送流程示例
1. Alice使用UA写作报文并向bob@someschool.edu发送
2. Alice的UA向其邮件服务器发送报文；报文放置在报文队列中
3. SMTP的客户机侧打开与Bob的邮件服务器的TCP连接
4. SMTP通过TCP连接发送Alice的报文
5. Bob的邮件服务器将该报文放入Bob的邮箱
6. Bob调用其用户代理来读报文

### SMTP交互示例
```
S: 220 hamburger.edu
C: HELO crepes.fr
S: 250 Hello crepes.fr, pleased to meet you
C: MAIL FROM:<alice@crepes.fr>
S: 250 alice@crepes.fr... Sender ok
C: RCPT TO:<bob@hamburger.edu>
S: 250 bob@hamburger.edu... Recipient ok
C: DATA
S: 354 Enter mail, end with"." on a line by itself
C: Do you like ketchup?
C: How about pickles?
C:.
S: 250 Message accepted for delivery
C: QUIT
S: 221 hamburger.edu closing connection
```

### 邮件报文格式
- **RFC 822**：文本报文格式的标准
  - **首部行**，如：
    - From:
    - To:
    - Subject:
  - **主体**："报文"，均为ASCII字符
  - 首部与主体间有空行

- **MIME扩展**（多用途Internet邮件扩展，RFC 2045, 2056）：
  - 用于支持多媒体内容
  - 在报文**首部**的附加行声明MIME内容类型
  - 包含字段：
    - MIME-Version: 1.0
    - Content-Type: image/jpeg（多媒体数据类型，子类型，声明参数）
    - Content-Transfer-Encoding: base64（使用编码数据的方法）
    - base64编码数据

### 邮件访问协议
![[Pasted image 20260110102457.png]]
- **目的**：从服务器获取邮件（SMTP只负责交付/存储到接收方服务器）
- **主要协议**：
  1. **POP3（邮局协议，RFC 1939）**：
     - 授权(代理<-->服务器)并下载
     - 三个阶段：
       - 特许阶段：user/pass命令
       - 事务阶段：list/retr/dele命令
       - 更新阶段：quit命令
     - "download and delete"模式：如果用户改变客户机，不能重读电子邮件
     - "Download-and-keep"模式：在不同客户机上的报文拷贝
     - 跨越会话是无状态的

## 6.5 WWW（万维网）

### Web和HTTP基础
- **Web页组成**：
  - 由对象组成
  - 对象可以是HTML文件，JPEG图片，Java小程序，音频文件等
  - Web页由基本的HTML文件组成，包括几个引用对象
  - 每个对象可由URL寻址
  - URL例子：www.someschool.edu/someDept/pic.gif

- **HTTP（超文本传送协议）**：
  - Web的应用层协议
  - 客户机/服务器模式：
    - 客户机：请求、接收，"显示"Web对象（浏览器）
    - 服务器：Web服务器响应请求而发送对象
  - 版本：
    - HTTP 1.0: RFC 1945
    - HTTP 1.1: RFC 2616

### HTTP工作原理
- **使用TCP**：
  - 客户机向服务器发起TCP连接(产生套接字)，端口80
  - 服务器从客户机接受TCP连接
  - 在浏览器(HTTP客户机)和Web服务器(HTTP服务器)之间交换HTTP报文
  - 关闭TCP连接

- **无状态特性**：
  - 服务器不保留有关客户机过去请求的任何信息
  - 维护"状态"的协议复杂：
    - 过去历史（状态）必须维护
    - 如果服务器/客户机崩溃，"状态"的视图可能不一致，必须要重新建立

### HTTP连接类型
- **非持久HTTP**：
  - 至多一个对象经过一个TCP连接发送
  - HTTP/1.0使用非持久HTTP
  - 每个对象要求2 RTT（往返时间）
  - 操作系统必须为每个TCP连接工作并分配资源
  - 总响应时间 = 2RTT + 传输文件时间
![[Pasted image 20260110103845.png]]
- **持久HTTP**：
  - 多个对象能够经过客户机和服务器之间的单个TCP连接发送
  - HTTP/1.1以默认模式使用持久连接
  - 在发送响应后，服务器让连接打开
  - 在相同的客户机/服务器之间的后继HTTP报文通过该连接发送
  - 两种工作模式：
    1. **非流水线方式**：
       - 仅当前面的响应已经收到，客户机发出新的请求
       - 对每个引用对象一个RTT
    2. **流水线方式**（HTTP/1.1默认）：
       - 只要客户机遇到一个引用对象，它就立即发送请求
       - 对于所有引用的对象可能只花费一个RTT时间

### HTTP报文格式
- **HTTP请求报文**：
  - 两类HTTP报文：请求，响应
  - ASCII格式(人可读)
  - 包含：
    - 请求行（GET, POST, HEAD命令）
    - 首部行
    - 空行（指示报文结束）
  - 示例：
    ```
    GET/somedir/page.html HTTP/1.1
    Host: www.someschool.edu
    Connection: close
    User-agent: Mozilla/4.0
    Accept-language:fr
    ```

- **请求方法类型**：
  - HTTP/1.0：
    - GET
    - POST
    - HEAD（请求服务器许可响应请求的对象）
  - HTTP/1.1增加：
    - PUT（向URL字段中定义的路径，上载在实体主体中文件）
    - DELETE（删除在URL字段中定义的文件）

- **HTTP响应报文**：
  - 包含：
    - 状态行（协议状态码状态短语）
    - 首部行
    - 数据（如请求的HTML文件）
  - 示例：
    ```
    HTTP/1.1 200 OK
    Connection close
    Date: Thu, 06 Aug 1998 12:00:15 GMT
    Server: Apache/1.3.0(Unix)
    Last-Modified: Mon, 22 Jun 1998......
    Content-Length: 6821
    Content-Type: text/html
    
    data data data data data...
    ```

- **HTTP响应状态码**：
  - 200 OK：请求成功，请求的对象在这个报文后面
  - 301 Moved Permanently：请求的对象已转移，新的URL在响应报文的Location:首部行中指定
  - 400 Bad Request：请求报文不为服务器理解
  - 404 Not Found：请求的文档没有在该服务器上发现
  - 505 HTTP Version Not Supported

### 用户-服务器状态：Cookies
- **Cookie组成**（四个部分）：
  1. 在HTTP响应报文中的cookie首部行
  2. 在HTTP请求报文中的cookie首部行
  3. 保持在用户主机中的cookie文件并由用户浏览器管理
  4. 位于Web站点的后端数据库

- **工作流程**：
  1. 用户首次访问网站，服务器产生唯一ID，在后端数据库生成表项
  2. 服务器在HTTP响应中使用Set-cookie首部发送ID
  3. 浏览器在本地存储cookie
  4. 后续请求中，浏览器在HTTP请求中包含cookie首部

- **Cookie用途**：
  - 鉴别
  - 购物车
  - 推荐
  - 用户会话状态(Web电子邮件)

- **隐私问题**：
  - Cookies使站点了解用户很多信息
  - 用户可能向站点提供名字和电子邮件
  - 搜索引擎使用重定向和cookies来获取更多信息
  - 跨越站点获得信息，用于定向广告

### Web缓存（代理服务器）
- **工作原理**：
  - 用户设置浏览器：经过缓存访问
  - 浏览器向缓存发送所有HTTP请求
    - 对象在缓存中：缓存返回对象
    - 否则缓存向起始服务器请求对象，然后向客户机返回对象，并在本地存储一份拷贝
  - 目标：满足客户机请求而不涉及起始服务器

- **优点**：
  - 减小客户机请求的响应时间
  - 减小机构访问链路的流量

- **条件GET方法**：
  - 目的：如果缓存中有最新缓存版本，就不发送该对象
  - 缓存器：在HTTP请求If-modified-since: <date>中，指定缓存版本的日期
  - 服务器：如果缓存的拷贝是最新的，响应不包含对象：HTTP/1.0 304 Not Modified
