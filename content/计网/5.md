# 📘 第5章 传输层笔记整理

## 一、传输层概述

### 1. 基本功能
- **逻辑通信**：为**不同主机上的应用进程**提供逻辑通信（vs. 网络层是主机间逻辑通信）；
- 运行于**端系统**（主机），不在网络中间设备（路由器）；
- **发送方**：将应用层报文分段 → 交给网络层；
- **接收方**：将段重组成报文 → 交付给应用层。

### 2. 传输层 vs. 网络层
| 项目   | 网络层               | 传输层             |
| ---- | ----------------- | --------------- |
| 通信粒度 | 主机 ↔ 主机           | **进程 ↔ 进程**     |
| 服务类型 | 尽力而为（Best-effort） | 可靠/不可靠（TCP/UDP） |
| 依赖关系 | 传输层依赖并强化网络层服务     | —               |

> 🧒 **家庭类比**：12个孩子（进程）通过邮政服务（网络层） + Ann/Bill（传输层）通信。

### 3. 因特网传输层协议
| 协议 | 特性 | 适用场景 |
|------|------|---------|
| **TCP** | 面向连接、可靠、按序、拥塞/流量控制、连接建立/释放 | Web、邮件、文件传输（FTP）、远程登录（Telnet） |
| **UDP** | 无连接、不可靠、不保证顺序、无拥塞控制 | DNS、SNMP、流媒体、实时游戏、VoIP |

> ❗ 不提供：时延保证、带宽保证（因特网是“尽力而为”网络）。

---

## 二、传输层寻址与端口

### 1. 端口概念
- 16位无符号整数，用于标识应用进程；
- 端口是**传输层的服务访问点（SAP）**。

### 2. 端口分类
| 类型 | 范围 | 说明 |
|------|------|------|
| **熟知端口**（Well-known） | `0 ~ 1023` | 系统/标准服务（如 HTTP 80, SSH 22） |
| **登记端口**（Registered） | `1024 ~ 49151` | 用户应用登记使用（如 MySQL 3306） |
| **动态/私有端口**（Ephemeral） | `49152 ~ 65535` | 客户端临时使用（动态分配） |

> 📌 通信结束后，客户端端口可被复用。

### 3. 复用（Multiplexing）与分解（Demultiplexing）
- **复用**：发送端多个应用进程 → 单个传输层 → 单个网络层；
- **分解**：接收端依据 IP + 端口号 → 正确交付给目标进程（套接字）。

#### 无连接分解（UDP）
- 套接字标识：**`(目的IP, 目的端口)`**
- 多个源（不同IP/端口）可发往同一个 UDP 套接字。

#### 面向连接分解（TCP）
- 套接字标识：**四元组 `(源IP, 源端口, 目的IP, 目的端口)`**
- 同一服务器可支持大量并发连接（如 Web 服务器每客户一线程/套接字）。

## 三、UDP 协议（用户数据报协议）

### 1. 特点
- RFC 768；
- 尽力而为
- 段首部小
- **无连接**：无需握手，无状态；
- **轻量**：首部仅 8 字节；
- **无拥塞控制**：可全速发送；
- **不可靠**：不保证到达、不保序；
- 可在**应用层实现可靠性**（如 QUIC、DNS retry）。

### 2. 适用场景
- 容忍丢包（如视频/音频流）；
- 要求低延迟（实时交互）；
- 简单请求–响应（DNS、SNMP）。

### 3. UDP 报文格式（8 字节首部）
```
 0      7 8     15 16    23 24    31
+--------+--------+--------+--------+
| 源端口 | 目的端口 |   长度   | 校验和 |
+--------+--------+--------+--------+
|           应用数据（报文）         |
+------------------------------------+
```
- **长度**：UDP 段总长（含首部，单位：字节）；
- **校验和**：可选（但通常开启），覆盖：UDP首部 + 数据 + 伪首部（含IP地址等）。

## 四、TCP 协议（传输控制协议）

RFCs: 793, 1122, 1323, 2018, 2581

### 1. 核心特性
- 全双工、点到点、可靠、按序、面向字节流（无报文边界）；
- **面向连接**：三次握手建连，四次挥手释放；
- **流量控制**：防止接收方缓存溢出；
- **拥塞控制**：防止网络过载；
- **滑动窗口 + 流水线传输**。
![[Pasted image 20260109221642.png]]
> 🌀 TCP 是**面向字节流**：应用写入的“报文”被 TCP 视为连续字节流，无边界。

---
![[Pasted image 20260109221858.png]]
TCP 连接是一条**虚连接**，而不是一条真正的物理连接  
TCP 对应用进程一次把多长的报文发送到TCP 的 缓存中是不关心的  
TCP 根据对方给出的窗口值和当前网络拥塞的程 度来决定一个报文段应包含多少个字节（UDP 发 送的报文长度是应用进程给出的）。  
TCP 可把太长的数据块划分短一些再传送。TCP 也可等待积累有足够多的字节后再构成报文段发 送出去  
### 2. TCP 报文段格式（20 字节固定首部 + 可变选项）

![[Pasted image 20260109222003.png]]
#### 首部字段详解

| 字段       | 长度   | 含义                                                                                                    |
| -------- | ---- | ----------------------------------------------------------------------------------------------------- |
| 源/目的端口   | 各 2B | 标识进程                                                                                                  |
| 序号（seq）  | 4B   | **本报文段第一个数据字节**在字节流中的编号（按字节编号）                                                                        |
| 确认号（ack） | 4B   | **期望收到的下一个字节序号**（即已成功接收 `ack-1` 及以前所有字节）→ **累计确认**                                                    |
| 数据偏移     | 4bit | 首部长度（单位：32bit = 4B），最小为 5（20B），最大 15（60B）                                                             |
| URG      | 1bit | 紧急指针有效（高优先级数据）                                                                                        |
| ACK      | 1bit | 确认号字段有效（除 SYN 报文外，一般都为 1）                                                                             |
| PSH      | 1bit | 推送：要求接收方立即交付应用（而非等缓存满）                                                                                |
| RST      | 1bit | 复位：连接出错，立即断开重建                                                                                        |
| SYN      | 1bit | 同步：用于连接建立（SYN=1 的报文不携带数据）                                                                             |
| FIN      | 1bit | 终止：发送方数据发完，请求释放连接                                                                                     |
| 窗口       | 2B   | 通知对方自己的**接收窗口大小（rwnd）**，单位：字节                                                                         |
| 校验和      | 2B   | 覆盖：TCP首部 + 数据 + **伪首部**（源/目的IP+协议+TCP长度）                                                              |
| 紧急指针     | 2B   | 指出本报文段中**紧急数据末尾**距首字节的偏移（仅当 URG=1 有效）                                                                 |
| 选项       | 可变   | 常见：<br>• `MSS`（最大报文段长度，默认 536B）<br>• `窗口扩大因子`（支持 >64KB 窗口）<br>• `时间戳`（RTT测量/防序号回绕）<br>• `SACK`（选择性确认） |

> ✅ **MSS（Max Segment Size）** ≠ MTU；MSS = MTU − IP首部(20) − TCP首部(20) ≈ 1460B（以太网）。

## 五、TCP 序号与确认机制

### 1. 序号（Sequence Number）
- TCP 将**整个连接上的数据流视为一个连续的字节流**，每个字节都有唯一序号；
- **报文段的序号 = 该段中第一个数据字节的序号**；
- 初始序号（ISN）在三次握手时随机生成，防止旧连接干扰新连接。

### 2. 确认号（Acknowledgment Number）
- 表示**期望收到的下一个字节的序号**；
- 采用**累计确认（Cumulative ACK）**：确认号为 `N`，表示序号 `< N` 的所有字节均已正确接收；
- 例如：若接收方已收到字节 0~299，则确认号为 300。

### 3. 失序报文段处理
- **TCP 规范未强制规定**如何处理失序段；
- 实际实现通常：
  - **缓存失序段**，等待缺失字节到达后按序交付；
  - 或直接丢弃（效率较低，现代实现多采用缓存）；
- **选择性确认（SACK）** 可显式告知发送方哪些非连续块已收到，提升效率。

### 4. 捎带确认（Piggybacking）
- 在有数据要发送时，将 ACK 附带在数据报文段中一并发送；
- 减少纯 ACK 报文数量，提高带宽利用率；
- 若无数据可发，则单独发送 ACK（通常延迟 ≤ 200ms，受 Nagle 算法等影响）。

## 六、TCP 超时与重传机制

### 1. 往返时延（RTT）估计
- **SampleRTT**：从发送报文段到收到其 ACK 的时间（**不包含重传样本**）；
- **EstimatedRTT**：使用**指数加权移动平均（EWMA）** 平滑 SampleRTT：
$$
  \text{EstimatedRTT} = (1 - \alpha) \cdot \text{EstimatedRTT} + \alpha \cdot \text{SampleRTT}
$$
  - 典型值：$\alpha = 0.125$

### 2. 超时间隔（TimeoutInterval）计算
- 引入**偏差（DevRTT）** 反映 RTT 波动：
$$
  \text{DevRTT} = (1 - \beta) \cdot \text{DevRTT} + \beta \cdot |\text{SampleRTT} - \text{EstimatedRTT}|
$$
  - 典型值：$\beta = 0.25$
- 最终超时值：
$$
  \text{TimeoutInterval} = \text{EstimatedRTT} + 4 \cdot \text{DevRTT}
$$
- 此机制使超时值能适应网络变化：RTT 波动大 → 超时更保守；波动小 → 更激进。

## 七、TCP 连接管理

### 1. 三次握手（Three-Way Handshake）
目的：同步初始序号、确认双方收发能力、协商参数（如 MSS）。

| 步骤 | 发送方 | 报文标志 | 序号/确认号 | 状态变化 |
|------|--------|----------|--------------|----------|
| 1 | 客户 → 服务器 | SYN=1 | seq=x | 客户: CLOSED → SYN_SENT |
| 2 | 服务器 → 客户 | SYN=1, ACK=1 | seq=y, ack=x+1 | 服务器: LISTEN → SYN_RCVD |
| 3 | 客户 → 服务器 | ACK=1 | seq=x+1, ack=y+1 | 双方: ESTABLISHED |
![[Pasted image 20260109224401.png]]
> ✅ **为何需要三次？**  
> 防止历史重复连接请求突然到达服务器造成资源浪费（两次握手无法区分新旧请求）。

### 2. 四次挥手（Connection Release）
TCP 连接是全双工的，需分别关闭两个方向。

| 步骤  | 发起方（A）→ 被动方（B） | 报文标志  | 序号/确认号  | 状态变化                                                      |
| --- | -------------- | ----- | ------- | --------------------------------------------------------- |
| 1   | A → B          | FIN=1 | seq=u   | A: ESTAB → FIN_WAIT_1                                     |
| 2   | B → A          | ACK=1 | ack=u+1 | A: FIN_WAIT_1 → FIN_WAIT_2<br>B: ESTAB → CLOSE_WAIT       |
| 3   | B → A          | FIN=1 | seq=v   | B: CLOSE_WAIT → LAST_ACK                                  |
| 4   | A → B          | ACK=1 | ack=v+1 | B: LAST_ACK → CLOSED<br>（2MSL 后）<br>A: TIME_WAIT → CLOSED |
![[Pasted image 20260109224346.png]]
#### 关键点：TIME_WAIT 状态
- **持续 2MSL（Maximum Segment Lifetime）**：
  1. 确保最后一个 ACK 能到达对方（若丢失，B 会重传 FIN，A 可再次 ACK）；
  2. 使本连接所有报文在网络中彻底消失，避免“失效请求”干扰新连接。
- MSL 通常为 30 秒 → TIME_WAIT 持续 60 秒。

## 八、TCP 可靠传输实现：滑动窗口

### 1. 基本原理
- **发送窗口**：允许发送但未确认的字节范围；
- **接收窗口（rwnd）**：接收方通告的可用缓存大小；
- **发送方实际窗口 = min(cwnd, rwnd)**；
- 窗口滑动条件：收到新的确认（ACK）。

### 2. 窗口字段说明（以字节为单位）
设发送窗口 `[P1, P3)`，当前已发送至 P2：
- `P1`：已确认的最高序号 + 1（窗口后沿）；
- `P2`：已发送但未确认的最高序号 + 1；
- `P3`：窗口前沿 = P1 + rwnd；
- **可用窗口 = P3 − P2**（可继续发送的字节数）。
![[Pasted image 20260109225028.png]]
### 3. 接收方行为
- **按序交付**：即使缓存中有高序号数据，也必须等缺失字节到达；
- **累积确认**：只确认连续字节流的末尾；
- **缓存失序段**：等待空缺填补后一并交付。

> ⚠️ **窗口收缩不被推荐**：TCP 标准强烈反对接收方缩小窗口前沿（可能导致死锁）。
	A 的发送窗口并不总是和 B 的接收窗口一样大

## 九、TCP 流量控制

### 1. 目的
- 防止发送方速率过快导致接收方缓存溢出；
- 通过 **rwnd（接收窗口）** 动态调节发送速率。

### 2. 零窗口问题与持续计时器
- 若接收方通告 `rwnd = 0`，发送方暂停发送；
- 启动 **持续计时器（Persistence Timer）**；
- 超时后发送 **1 字节探测段**，触发接收方返回当前 rwnd；
- 避免因 ACK 丢失导致永久死锁。

### 3. 发送时机优化（提升效率）
- **MSS 触发**：缓存 ≥ MSS 时立即发送；
- **Push 操作**：应用层强制立即发送（PSH=1）；
- **计时器触发**：防止长时间无数据发送（如 Nagle 算法 vs. 延迟 ACK 的交互问题）。

## 十、TCP 拥塞控制

### 1. 基本概念
- **拥塞窗口（cwnd）**：反映网络承载能力，动态调整；
- **发送窗口上限 = min(rwnd, cwnd)**；
- **慢开始门限（ssthresh）**：切换算法的阈值。
拥塞控制所要做的都有一个前提，就是**网络能够承受现有的网络负荷**  
拥塞控制是一个全局性的过程，涉及到所有的主 机、所有的路由器，以及与降低网络传输性能有 关的所有因素  
流量控制往往指在给定的发送端和接收端之间的 点对点通信量的控制  
流量控制所要做的就是抑制**发送端**发送数据的速 率，以便使接收端来得及接收  
### 2. 核心算法组合
**开环控制方法** 就是在设计网络时**事先**将有关发生拥塞的因素考虑周到，力求网络在工作时不产生拥塞  
闭环控制是基于**反馈环路**的概念。属于闭环控制的有以下几种措施
- 监测网络系统以便检测到拥塞在何时、何处发 生
- 将拥塞发生的信息传送到可采取行动的地方
- 调整网络系统的运行以解决出现的问题
#### (1) 慢开始（Slow Start）
- 初始：`cwnd = 1 MSS`；
- 每收到一个 ACK → `cwnd += 1 MSS`；
- **每 RTT 翻倍**（指数增长）；
- 适用阶段：连接初期或超时后。

#### (2) 拥塞避免（Congestion Avoidance）
- 当 `cwnd ≥ ssthresh` 时启用；
- 每 RTT → `cwnd += 1 MSS`（线性增长，“加法增大”）；
- 目标：缓慢探测网络容量，避免突增引发拥塞。

#### (3) 乘法减小（Multiplicative Decrease）
- **检测到拥塞**（超时或 3 个重复 ACK）：
  - `ssthresh = max(cwnd / 2, 2 MSS)`
  - 若因**超时**：`cwnd = 1 MSS` → 重启慢开始（TCP Tahoe）；
  - 若因**3 个重复 ACK**：进入快恢复（TCP Reno）。
#### (4) 加法增大
每 RTT → `cwnd += 1 MSS`

### 3.快重传（Fast Retransmit）
- 接收方对每个**失序段**立即发送**重复 ACK**；
- 发送方收到 **3 个重复 ACK** → 立即重传缺失段（无需等待超时）。

### 4.快恢复（Fast Recovery）
- 执行乘法减小后，**不回到慢开始**,将门限 `ssthresh` 减半；
- `cwnd = ssthresh`；
- 进入拥塞避免阶段（线性增长）；
- 假设网络未严重拥塞（仅个别丢包）。

发送窗口的上限值 = Min \[rwnd, cwnd\]
