# 📘 第5章 传输层笔记整理

## 一、传输层概述

### 1. 基本功能
- **逻辑通信**：为**不同主机上的应用进程**提供逻辑通信（vs. 网络层是主机间逻辑通信）；
- 运行于**端系统**（主机），不在网络中间设备（路由器）；
- **发送方**：将应用层报文分段 → 交给网络层；
- **接收方**：将段重组成报文 → 交付给应用层。

### 2. 传输层 vs. 网络层
| 项目 | 网络层 | 传输层 |
|------|--------|--------|
| 通信粒度 | 主机 ↔ 主机 | **进程 ↔ 进程** |
| 服务类型 | 尽力而为（Best-effort） | 可靠/不可靠（TCP/UDP） |
| 依赖关系 | 传输层依赖并强化网络层服务 | — |

> 🧒 **家庭类比**：12个孩子（进程）通过邮政服务（网络层） + Ann/Bill（传输层）通信。

### 3. 因特网传输层协议
| 协议 | 特性 | 适用场景 |
|------|------|---------|
| **TCP** | 面向连接、可靠、按序、拥塞/流量控制、连接建立/释放 | Web、邮件、文件传输（FTP）、远程登录（Telnet） |
| **UDP** | 无连接、不可靠、不保证顺序、无拥塞控制 | DNS、SNMP、流媒体、实时游戏、VoIP |

> ❗ 不提供：时延保证、带宽保证（因特网是“尽力而为”网络）。

---

## 二、传输层寻址与端口

### 1. 端口概念
- 16位无符号整数，用于标识应用进程；
- 端口是**传输层的服务访问点（SAP）**。

### 2. 端口分类
| 类型 | 范围 | 说明 |
|------|------|------|
| **熟知端口**（Well-known） | `0 ~ 1023` | 系统/标准服务（如 HTTP 80, SSH 22） |
| **登记端口**（Registered） | `1024 ~ 49151` | 用户应用登记使用（如 MySQL 3306） |
| **动态/私有端口**（Ephemeral） | `49152 ~ 65535` | 客户端临时使用（动态分配） |

> 📌 通信结束后，客户端端口可被复用。

### 3. 复用（Multiplexing）与分解（Demultiplexing）
- **复用**：发送端多个应用进程 → 单个传输层 → 单个网络层；
- **分解**：接收端依据 IP + 端口号 → 正确交付给目标进程（套接字）。

#### 无连接分解（UDP）
- 套接字标识：**`(目的IP, 目的端口)`**
- 多个源（不同IP/端口）可发往同一个 UDP 套接字。

#### 面向连接分解（TCP）
- 套接字标识：**四元组 `(源IP, 源端口, 目的IP, 目的端口)`**
- 同一服务器可支持大量并发连接（如 Web 服务器每客户一线程/套接字）。

---

## 三、UDP 协议（用户数据报协议）

### 1. 特点
- RFC 768；
- **无连接**：无需握手，无状态；
- **轻量**：首部仅 8 字节；
- **无拥塞控制**：可全速发送；
- **不可靠**：不保证到达、不保序；
- 可在**应用层实现可靠性**（如 QUIC、DNS retry）。

### 2. 适用场景
- 容忍丢包（如视频/音频流）；
- 要求低延迟（实时交互）；
- 简单请求–响应（DNS、SNMP）。

### 3. UDP 报文格式（8 字节首部）
```
 0      7 8     15 16    23 24    31
+--------+--------+--------+--------+
| 源端口 | 目的端口 |   长度   | 校验和 |
+--------+--------+--------+--------+
|           应用数据（报文）         |
+------------------------------------+
```
- **长度**：UDP 段总长（含首部，单位：字节）；
- **校验和**：可选（但通常开启），覆盖：UDP首部 + 数据 + 伪首部（含IP地址等）。

### 4. UDP 校验和计算（反码和）
1. 将段内容视为 16-bit 字的序列；
2. 全部相加（回卷进位：最高位进位加到低位）；
3. 取反码 → 得校验和；
4. 接收方重算，若 ≠ 校验和字段 ⇒ 差错。

> ⚠️ 校验和**不能 100% 检错**（如两位同时翻转可能抵消）。

## 四、TCP 协议（传输控制协议）

RFCs: 793, 1122, 1323, 2018, 2581

### 1. 核心特性
- 全双工、点到点、可靠、按序、面向字节流（无报文边界）；
- **面向连接**：三次握手建连，四次挥手释放；
- **流量控制**：防止接收方缓存溢出；
- **拥塞控制**：防止网络过载；
- **滑动窗口 + 流水线传输**。

> 🌀 TCP 是**面向字节流**：应用写入的“报文”被 TCP 视为连续字节流，无边界。

---

### 2. TCP 报文段格式（20 字节固定首部 + 可变选项）

```
 0      7 8     15 16    23 24    31
+--------+--------+--------+--------+
|        源端口        |     目的端口     |
+--------+--------+--------+--------+
|                序号（seq）              |
+--------+--------+--------+--------+
|              确认号（ack）              |
+--------+--------+--------+--------+
| 数据 |        |U|A|P|R|S|F|           |
| 偏移 | 保留  |R|C|S|S|Y|I|    窗口     |
| (4bit)| (6bit)|G|K|H|T|N|N|           |
+--------+--------+--------+--------+
|     校验和     |    紧急指针     |
+--------+--------+--------+--------+
|               选项（可变长）             |
+--------+--------+--------+--------+
|                 填充（使首部为4字节倍数）        |
+--------+--------+--------+--------+
|                 应用数据                 |
+------------------------------------------+
```

#### 首部字段详解

| 字段 | 长度 | 含义 |
|------|------|------|
| 源/目的端口 | 各 2B | 标识进程 |
| 序号（seq） | 4B | **本报文段第一个数据字节**在字节流中的编号（按字节编号） |
| 确认号（ack） | 4B | **期望收到的下一个字节序号**（即已成功接收 `ack-1` 及以前所有字节）→ **累计确认** |
| 数据偏移 | 4bit | 首部长度（单位：32bit = 4B），最小为 5（20B），最大 15（60B） |
| URG | 1bit | 紧急指针有效（高优先级数据） |
| ACK | 1bit | 确认号字段有效（除 SYN 报文外，一般都为 1） |
| PSH | 1bit | 推送：要求接收方立即交付应用（而非等缓存满） |
| RST | 1bit | 复位：连接出错，立即断开重建 |
| SYN | 1bit | 同步：用于连接建立（SYN=1 的报文不携带数据） |
| FIN | 1bit | 终止：发送方数据发完，请求释放连接 |
| 窗口 | 2B | 通知对方自己的**接收窗口大小（rwnd）**，单位：字节 |
| 校验和 | 2B | 覆盖：TCP首部 + 数据 + **伪首部**（源/目的IP+协议+TCP长度） |
| 紧急指针 | 2B | 指出本报文段中**紧急数据末尾**距首字节的偏移（仅当 URG=1 有效） |
| 选项 | 可变 | 常见：<br>• `MSS`（最大报文段长度，默认 536B）<br>• `窗口扩大因子`（支持 >64KB 窗口）<br>• `时间戳`（RTT测量/防序号回绕）<br>• `SACK`（选择性确认） |

> ✅ **MSS（Max Segment Size）** ≠ MTU；MSS = MTU − IP首部(20) − TCP首部(20) ≈ 1460B（以太网）。

---

### 3. TCP 连接管理

#### ① 三次握手（建连）

| 步骤 | 方向 | 报文标志 | seq / ack |
|------|------|---------|-----------|
| 1 | A → B | SYN=1 | seq = x |
| 2 | B → A | SYN=1, ACK=1 | seq = y, ack = x+1 |
| 3 | A → B | ACK=1 | seq = x+1, ack = y+1 |

- A：主动打开 → `SYN_SENT` → `ESTABLISHED`
- B：被动打开（`LISTEN`）→ `SYN_RCVD` → `ESTABLISHED`

> 🌟 目的：同步初始序号 + 防止旧连接请求“复活”造成混乱。

#### ② 四次挥手（释放连接）

| 步骤 | 方向 | 报文标志 | seq / ack | 说明 |
|------|------|---------|-----------|------|
| 1 | A → B | FIN=1 | seq = u | A 主动关闭，不再发数据 |
| 2 | B → A | ACK=1 | ack = u+1 | B 确认；A→B 连接**半关闭** |
| 3 | B → A | FIN=1 | seq = w | B 也关闭（若无数据待发） |
| 4 | A → B | ACK=1 | ack = w+1 | A 确认；进入 `TIME_WAIT` |

- A：主动关闭；B：被动关闭；
- A 最终需等待 **2MSL**（Maximum Segment Lifetime，如 30~120s）：
  1. 确保最后 ACK 到达 B；
  2. 让网络中旧报文“自然消亡”，防新连接受污染。

#### ③ TCP 有限状态机（重点状态）
- `CLOSED`：初始/终止；
- `LISTEN`：服务器等待连接；
- `SYN_SENT` / `SYN_RCVD`：握手阶段；
- `ESTABLISHED`：数据传输；
- `FIN_WAIT_1/2`, `CLOSE_WAIT`, `LAST_ACK`, `TIME_WAIT`：释放阶段。

> ⏱️ `TIME_WAIT` 是主动关闭方独有，持续 2MSL。

### 4. TCP 可靠传输实现：滑动窗口机制

#### 核心思想
- 基于**字节序号**的可靠控制（非报文段）；
- 发送方 & 接收方各维护：
  - 发送窗口（`cwnd` + `rwnd` 限制）
  - 接收窗口（`rwnd`）

#### 以字节为单位的滑动窗口
- **窗口 = [已确认, 可发送)**；
- 窗口三指针：
  - `P1`：已确认末尾（前沿左边界）；
  - `P2`：已发送未确认末尾；
  - `P3`：可发送右边界（`P1 + window_size`）；
- **可用窗口 = P3 − P2**

> 📉 TCP 标准**禁止窗口收缩**（即 P3 不能左移），但允许通告更小新窗口（靠右移 P1 实现）。

#### 接收方策略（对失序段）
- TCP 规范未规定 → 实现可选：
  - 丢弃失序段；
  - **缓存失序段**，等缺失段到达后按序交付（主流做法）；
- 必须支持**累计确认**（重复确认相同 ack 可触发快重传）。

#### 发送/接收缓存作用
| 缓存 | 内容 |
|------|------|
| 发送缓存 | ① 应用待发送数据；② 已发未确认数据 |
| 接收缓存 | ① 按序已到未读数据；② 不按序到达数据 |

---

### 5. TCP 流量控制（Flow Control）

- 目标：匹配**发送速率**与**接收能力**；
- 机制：接收方通过 `rwnd` 字段动态告知发送方可发字节数；
- 发送窗口 = min(cwnd, rwnd)

#### 零窗口问题 & 持续计时器（Persistence Timer）
- 若接收方通告 `rwnd = 0` → 发送方停止发送；
- 为防死锁：启动**持续计时器** → 超时后发**1字节探测段**；
- 接收方 ACK 中带回当前真实窗口 → 解除僵局。

#### 发送时机策略（兼顾效率）
1. 达到 **MSS**；
2. 应用请求 **PUSH**；
3. **计时器到期**（即使未满 MSS）。

---

### 6. TCP 拥塞控制（Congestion Control）

> 🔍 区别于**流量控制**（点对点 vs 全局网络）

#### ① 拥塞窗口 `cwnd`
- 发送方维护，反映**网络容量**；
- 实际发送窗口 = min(cwnd, rwnd)

#### ② 四种算法（经典 Reno）

| 算法 | 触发条件 | 行为 |
|------|---------|------|
| **慢开始**（Slow Start） | cwnd < ssthresh 或刚建连/重置 | cwnd 指数增长：每 RTT **翻倍**（每收到 1 ACK ⇒ cwnd += 1 MSS） |
| **拥塞避免**（Congestion Avoidance） | cwnd ≥ ssthresh | cwnd 线性增长：每 RTT **+1 MSS**（加法增大） |
| **快重传**（Fast Retransmit） | 收到 **3 个重复 ACK** | 立即重传丢失段（不必等超时） |
| **快恢复**（Fast Recovery） | 快重传后触发 | ssthresh = cwnd / 2；cwnd = ssthresh；进入拥塞避免（非慢开始） |

#### 关键参数
- **ssthresh**（慢开始门限）：
  - 初始通常设为 65535B 或接收方通告窗口；
  - 拥塞时：`ssthresh = max(2, cwnd / 2)`（乘法减小）；

#### 拥塞检测方式
- **超时（Timeout）** ⇒ 严重拥塞：ssthresh = cwnd/2；cwnd = 1；重走慢开始；
- **3 重复 ACK** ⇒ 轻微拥塞（包失序/个别丢包）：快重传 + 快恢复。

#### 图解（经典拥塞窗口变化）

```
cwnd
 ↑
24|               ●（超时：TCP Tahoe）
  |              / \
22|             /   \         ●（3 重复 ACK → 快恢复：TCP Reno）
  |            /     \       / \
16|-----●-----/       \-----/   \-------
  |    / \   /         \   /     \
  |   /   \ /           \ /       \
 0+------------------------------------→ 传输轮次
    慢开始 拥塞避免     快恢复 拥塞避免
```

> 📌 **TCP Tahoe vs Reno**  
> - Tahoe：3 重复 ACK ⇒ 仍慢开始（cwnd=1）——性能较差，已淘汰；  
> - Reno：3 重复 ACK ⇒ 快恢复（cwnd = ssthresh）——主流实现。

## 五、总结对比

| 特性 | UDP | TCP |
|------|-----|-----|
| 连接 | 无连接 | 面向连接（三次握手/四次挥手） |
| 可靠性 | 不可靠 | 可靠（ACK/重传/序号） |
| 顺序 | 不保证 | 保证按序 |
| 流量控制 | 无 | 有（接收窗口 rwnd） |
| 拥塞控制 | 无 | 有（cwnd + 四算法） |
| 首部大小 | 8 字节 | ≥20 字节 |
| 开销/延迟 | 低 | 高 |
| 适用场景 | 实时、容忍丢包 | 可靠传输、关键数据 |
