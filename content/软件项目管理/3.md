## 一、项目启动与招投标流程（乙方视角）

### 1. 乙方参与项目的全流程
1.  **项目分析**：获取招标信息后，首先分析项目可行性（能否干、能否收回钱）。
    *   *风险提示*：避免“陪标”或陷入无法回款的陷阱。
    *   *合规性*：国家招投标管理日益严格，存在“黑名单”机制（如串标会被禁止投标）。
2.  **竞标（投标）**：
    *   购买招标文件（需付费）。
    *   准备投标文件（技术标 + 商务标）。
    *   **报价策略**：需平衡利润与中标率（过高或过低均不利），部分项目设有“合理价格区间”。
3.  **开标与评标**：
    *   **时间观念**：必须严格遵守截止时间（如 9:00 前递交，逾期废标）。
    *   **公开唱标**：宣读报价。
    *   **专家评标**：技术 + 商务打分，排名公示。
    *   **异议处理**：未中标方可申诉，但需避免恶意申诉。
4.  **合同签订**：中标后协商签订制式合同。

### 2. 合同管理核心要点
*   **合同类型与税率**：
    *   **技术开发合同**：软件类常用，税率约 **3%**，且可能享受退税政策。
    *   **货物采购合同**：若被认定为非软件（如硬件集成），税率可能高达 **13%**。
    *   *案例*：智能体项目被财务认定为非软件，导致税率差异巨大，需提前明确合同类型。
*   **合同执行与跟踪**：
    *   甲乙双方按约履行（甲方给钱，乙方干活）。
    *   **变更管理**：合同可修改，但必须**双方协商一致**。
    *   **一致性原则**：招标、投标、合同内容需保持一致（类似毕设开题、中期、论文的一致性）。
*   **违约处理**：
    *   常见违约：甲方不给钱、乙方交付质量差。
    *   处理难点：如恒大案例，资金链断裂导致连锁违约。
*   **验收流程**：
    *   提交验收申请 -> 第三方评测 -> 测试大纲/报告评审 -> **预验收**（专家提问题整改） -> **正式验收**。
    *   *注意*：验收可能涉及多层级审批（如党委会通过），需预留时间。
*   **运维与维护**：
    *   软件交付后进入生产环境，问题易爆发。
    *   建议单独签订运维合同，明确服务范围和响应标准，避免“重硬轻软”。
*   **合同终止**：
    *   正常终止（履行完毕）或提前终止（需处理遗留问题）。
    *   内部项目（企业内部部门间）流程相对简化。

## 二、项目授权与项目经理（PM）

### 1. 项目章程（Project Charter）
*   **定义**：确认项目存在的文件，标志项目正式立项。
*   **作用**：
    *   明确项目目标。
    *   **任命项目经理**（“名正言顺”）。
    *   授权项目经理调动资源。
*   **启动会**：通常由公司高层召开，宣布项目经理及团队，要求各部门配合。

### 2. 项目经理的角色与能力
*   **角色定位**：
    *   **不是单纯写代码**：技术好是加分项，但核心是**管理**。
    *   **职责**：制定计划、组织实施、项目控制、团队协调。
    *   **类比**：如同班长，不一定学习最好，但管理能力需最强。
*   **核心能力要求**：
    1.  **技术能力**：理解开发、新技术（需保持自我更新，如关注技术公众号）。
    2.  **沟通协调能力**：最关键能力，需协调团队、客户、高层。
    3.  **领导力与人格魅力**：凝聚团队。
    4.  **商务与战略能力**：理解项目商业价值。
*   **敏捷环境下的 PM**：
    *   强调服务型领导（Servant Leadership）。
    *   快速响应客户需求变更。

## 三、项目生命周期（Project Life Cycle）

### 1. 生命周期特征
*   **阶段划分**：启动 -> 计划 -> 实施 -> 收尾（每个阶段末有可交付成果和评审）。
*   **资源投入曲线**：随时间呈“低 - 高 - 低”趋势（实施阶段人力/成本最高）。
*   **风险与不确定性**：随项目进展**逐渐下降**。
*   **干系人影响力**：随项目进展**逐渐下降**（前期影响最大，后期修改成本极高）。
*   **成功可能性**：随项目进展**逐渐上升**。

### 2. 项目生命周期 vs 产品生命周期
*   **项目生命周期**：从立项到交付验收，结束即终止。
*   **产品生命周期**：包含项目生命周期，后续还包括运维、升级、直到被淘汰的全过程。

### 3. 开发模型选择
需根据**项目特点**（需求明确度、风险、发布频率）选择模型：

| 模型类型 | 特点 | 适用场景 |
| :--- | :--- | :--- |
| **预测型 (瀑布)** | 计划驱动，一次性执行，阶段间严格评审。 | 需求明确、短期项目、国家大型项目。 |
| **V 模型** | 强调测试，开发与测试对应。 | 质量要求极高（航天、金融、核心系统）。 |
| **迭代型** | 分版本循环开发，逐步明确需求。 | 需求变化大、高风险项目。 |
| **增量型** | 分批次交付可独立使用的功能。 | 核心需求明确，非核心需求可后续完善。 |
| **螺旋模型** | 风险驱动，一圈圈迭代评估。 | 高风险、大规模系统。 |
| **原型模型** | 快速构建原型供用户评审。 | 需求模糊，需通过演示明确需求。 |
| **敏捷 (Agile)** | 迭代 + 增量，快速响应变更，价值交付。 | 需求不明确、需快速上线、互联网应用。 |
| **DevOps** | 开发 + 运维一体化，持续集成/部署/交付。 | 大厂主流，强调自动化与快速反馈。 |

### 4. 敏捷开发 (Scrum) 核心
*   **核心理念**：小步快跑，频繁交互，快速交付价值。
*   **关键会议**：
    *   **迭代计划会**：确定本次冲刺（Sprint）要做的事。
    *   **每日站会**：昨天做了什么、今天做什么、有什么问题（限时简短）。
    *   **迭代评审会**：向干系人展示成果。
    *   **迭代回顾会**：总结改进（做得好/不好的地方）。
*   **工件**：产品待办列表（Product Backlog）、冲刺待办列表（Sprint Backlog）。
*   **适用性**：适合短期、需求不明确项目；不适合超大型、强管控项目（可结合混合模型）。

## 四、需求工程（Requirement Engineering）

### 1. 需求的重要性
*   **项目失败主因**：往往不是技术问题，而是**需求不明确**或**无法满足用户需求**。
*   **常见问题**：需求隐含错误、用户不断增加需求、需求变更频繁。
*   **目标**：确保开发出的产品是用户真正想要的（避免“删记录不确认”等体验问题）。

### 2. 需求的层次
1.  **业务需求 (Business)**：组织/客户的高层目标（为什么要做）。
2.  **用户需求 (User)**：用户具体要完成的任务（用例、场景）。
3.  **功能需求 (Functional)**：软件具体具备的功能（菜单、操作逻辑）。
4.  **非功能需求 (Non-Functional)**：性能、安全、可靠性、国产化适配等。
    *   *案例*：国产化替代（华为 CPU/麒麟系统）带来的适配挑战。

### 3. 需求管理过程
*   **需求获取**：访谈、调研、专题讨论会、头脑风暴、**场景串联**（模拟用户操作流程）。
*   **需求分析**：明确输入、输出、处理逻辑。
*   **需求规格说明书 (SRS)**：遵循模板，术语一致，作为设计、编码、测试的依据。
*   **需求验证**：**用户必须参与评审**，测试用例应基于需求文档编写。
*   **需求变更控制**：
    *   变更是**正常**的，但需有效管理。
    *   原则：先小人后君子，一切以合同和确认文档为准。
    *   技巧：引导用户需求，避免天马行空；懂业务才能更好管理需求。

## 💡 助教提示 (Key Takeaways)

1.  **思维转变**：从“程序员思维”转向“项目经理思维”。技术是基础，但管理、沟通、成本控制才是 PM 的核心。
2.  **合同意识**：工作中（尤其是乙方）要重视合同条款、税率差异及验收流程，这直接关系到项目利润和回款。
3.  **需求为王**：不要急于写代码。花时间在需求调研和确认上，使用“场景串联”方法验证需求，能减少后期大量返工。
4.  **模型选择**：没有最好的模型，只有最适合的。
    *   稳字当头选 **V 模型/瀑布**。
    *   快字当头选 **敏捷/Scrum**。
    *   风险大选 **螺旋**。
5.  **合规性**：招投标过程中务必遵守法律法规，避免串标等违规行为导致进入“黑名单”。
