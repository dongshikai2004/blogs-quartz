以下是对《软件项目管理》第四章“进度计划”（闫波，北京理工大学）内容的详尽整理。本章主要涵盖了进度管理的重要性、基本概念与过程、进度估算方法、进度计划的编制以及案例分析。

# 第四章：进度计划 (Schedule Planning)

## 一、 进度管理概述

### 1. 进度管理的重要性
*   **核心挑战**：按时完成项目是项目经理最大的挑战之一。
*   **刚性因素**：时间是项目规划中灵活性最小的因素。
*   **冲突源头**：进度问题是项目冲突的主要原因，尤其在项目的后期阶段（如交付前夕）。
*   **冲突强度变化**：在项目形成期、早期、中期和末期，关于进度（Schedules）、优先级、人力、技术观点等的冲突强度不同，其中进度冲突在后期尤为显著。

### 2. 进度安排的两种前提
1.  **交付日期确定**：先确定最终交付日期，然后反向安排进度计划。
2.  **使用资源确定**：先确定可用资源，然后根据资源情况安排进度计划。

### 3. 进度的定义
*   **进度**：是对执行的活动和里程碑制定的工作计划日期表。
*   **进度管理**：为了确保项目按期完成所需要的过程。

### 4. 软件项目进度管理过程 (6个步骤)
1.  **活动定义 (Activity Definition)**：识别为完成项目交付物必须执行的具体活动。将工作包分解为更小的计划活动。
2.  **活动排序 (Activity Sequencing)**：确定活动之间的依赖关系（强制性、软逻辑、外部、内部依赖）。
3.  **活动资源估计 (Activity Resource Estimating)**：确定所需的人力、设备、材料等资源的种类、数量及使用时间。
4.  **活动历时估计 (Activity Duration Estimating)**：估算完成任务所需的持续时间。
5.  **制定进度计划 (Schedule Development)**：分析活动顺序、历时、资源需求，制定项目进度模型。
6.  **进度控制 (Schedule Control)**：监控项目状态，更新进度，管理变更。

## 二、 进度管理图示工具

### 1. 网络图 (Network Diagrams)
展示活动及其逻辑关系，是活动排序的输出。
*   **PDM (Precedence Diagramming Method，优先图法/单代号网络图)**：
    *   **节点 (Box)** 表示活动。
    *   **箭线** 表示逻辑关系。
    *   可表示四种逻辑关系：结束-开始 (FS)、结束-结束 (FF)、开始-开始 (SS)、开始-结束 (SF)。
    *   在软件项目中比ADM更通用。
*   **ADM (Arrow Diagramming Method，箭线图法/双代号网络图/AOA)**：
    *   **箭线** 表示活动。
    *   **节点 (Circle)** 表示事件（前道工序结束，后道工序开始）。
    *   仅适合表示“结束-开始”关系。
    *   可能需要**虚活动 (Dummy Activity)** 来表示逻辑关系，不消耗资源。
    *   原则：事件代号唯一，节点序号沿箭线方向增大，消除悬点。

### 2. 甘特图 (Gantt Chart)
*   显示任务的基本信息：工期、开始时间、结束时间、资源。
*   **优点**：直观，有时间刻度。
*   **缺点**：不能清晰展示活动间的逻辑关系。

### 3. 里程碑图 (Milestone Chart)
*   标志重大工作的完成或关键事件的发生（如规格说明书颁布、产品交付）。
*   **特点**：里程碑不同于活动，不消耗资源，仅表示时间点。

### 4. 资源图 (Resource Histogram)
*   显示项目进展过程中资源（人力、设备）的分配和使用情况。

### 5. 敏捷进度图
*   **燃尽图 (Burndown Chart)**：显示剩余工作量随时间的变化。
*   **燃起图 (Burnup Chart)**：显示已完成工作量随时间的变化。

## 三、 进度估算的基本方法

### 1. 历时估计的影响因素
实际工作时间、人员规模、生产率、有效工作时间、连续工作时间、人员级别、历史项目参照。

### 2. 基于规模的进度估算

#### A. 定额估算法
*   **公式**：$T = Q / (R \times S)$
    *   $T$: 活动持续时间
    *   $Q$: 活动的工作量
    *   $R$: 人力或设备数量
    *   $S$: 产量定额（单位时间完成的工作量）
*   **适用性**：简单易懂，适合小规模项目（<10,000 LOC 或 <6个月）。

#### B. 经验导出模型
*   **通用公式**：$D = a \times E^b$
    *   $D$: 月进度
    *   $E$: 人月工作量
    *   $a$: 系数 (2-4)
    *   $b$: 指数 (约1/3，依赖项目属性)
*   **常用模型**：
    *   **Walston-Felix (IBM)**: $D = 2.4 \times E^{0.35}$
    *   **基本 COCOMO**: $D = 2.5 \times E^b$
        *   有机式 (Organic): $b=0.38$
        *   半有机式 (Semidetached): $b=0.35$
        *   嵌入式 (Embedded): $b=0.32$

#### C. Jones 的一阶估算准则
*   基于功能点 (FP) 进行估算。
*   **公式**：粗略进度 = $FP^{\text{幂次}}$
*   **幂次表参考**：
    *   系统软件：最优0.43 / 平均0.45 / 最差0.48
    *   商业软件：最优0.41 / 平均0.43 / 最差0.46
    *   封装商品软件：最优0.39 / 平均0.42 / 最差0.45

### 3. 基于网络图的估算 (CPM & PERT)

#### A. 关键路径法 (CPM - Critical Path Method)
*   **特点**：基于单一、确定的历时估算。
*   **适用**：活动时间比较确定时。
*   **核心概念**：
    *   **关键路径**：网络图中最长的路径，决定了项目的最短完成时间。关键路径上的浮动时间为0。
    *   **正推法 (Forward Pass)**：计算最早开始时间 (ES) 和最早完成时间 (EF)。
        *   $EF = ES + \text{Duration}$
        *   多前置任务时，取最大 EF 作为后置任务的 ES。
    *   **逆推法 (Backward Pass)**：计算最晚开始时间 (LS) 和最晚完成时间 (LF)。
        *   $LS = LF - \text{Duration}$
        *   多后置任务时，取最小 LS 作为前置任务的 LF。
    *   **浮动时间 (Float/Slack)**：
        *   **总浮动 (Total Float, TF)**：不影响项目总工期的情况下，活动可延迟的时间。$TF = LS - ES = LF - EF$。
        *   **自由浮动 (Free Float, FF)**：不影响紧后活动最早开始时间的情况下，活动可延迟的时间。$FF = ES_{\text{successor}} - EF_{\text{current}} - \text{Lag}$。

#### B. 工程评价技术 (PERT - Program Evaluation and Review Technique)
*   **特点**：处理不确定性，使用加权平均历时。
*   **适用**：活动时间存在很大不确定性时。
*   **三个估算值**：
    *   $O$ (Optimistic)：乐观时间
    *   $M$ (Most Likely)：最可能时间
    *   $P$ (Pessimistic)：悲观时间
*   **期望历时 (E)**：$E = (O + 4M + P) / 6$
*   **标准差 ($\sigma$)**：$\sigma = (P - O) / 6$
*   **方差 ($\sigma^2$)**：$\sigma^2 = [(P - O) / 6]^2$
*   **路径评估**：
    *   路径期望值 = 各活动期望值之和。
    *   路径方差 = 各活动方差之和。
    *   路径标准差 = $\sqrt{\text{路径方差}}$。
*   **概率计算**：利用正态分布理论（±1$\sigma$: 68.3%, ±2$\sigma$: 95.5%, ±3$\sigma$: 99.7%）评估按时完成的概率。

### 4. 其他估算策略

#### A. 基于进度表的估算
*   **可能的最短进度表**：理想状态（顶尖人才、完美管理、先进工具、无需求变更），几乎无法实现。
*   **有效进度表**：“最佳进度”，代表高效团队的实际能力（前25%人才，良好工具和管理）。
*   **普通进度表**：一般项目的实用进度（中等人才，常规工具和管理）。

#### B. 基于承诺的进度估计
*   由开发人员做出进度承诺，而非单纯估算。
*   **优点**：提高开发者关注度和士气。
*   **缺点**：过于乐观，易产生大误差。

#### C. 敏捷估算方法
*   **开发速度稳定前**：计划扑克/举手表决（达成共识）。
*   **开发速度稳定后**：
    *   基于故事点 (Story Point) 生产率：$历时 = \text{总SP} / \text{平均生产率(SP/天)}$。
    *   基于迭代生产率：$历时 = \text{剩余SP} / \text{每迭代SP}$。

#### D. 预留分析
*   **应急预留 (Contingency Reserve)**：应对已识别风险，包含在进度基准中。
*   **管理预留 (Management Reserve)**：应对未知风险，不包含在进度基准中，需管理层批准使用。

## 四、 编制进度计划

### 1. 编制步骤
1.  进度编制
2.  资源调整
3.  成本预算
4.  计划优化调整
5.  确定计划基线

### 2. 进度编制的基本技术

#### A. 超前 (Lead) 与滞后 (Lag)
*   **滞后 (Lag)**：紧前活动结束后，紧后活动需要等待的时间。
*   **超前 (Lead)**：紧前活动结束前，紧后活动可以提前开始的时间（搭接）。
*   **作用**：合理拆分任务，缩短工期。

#### B. 时间压缩法 (Schedule Compression)
在不改变项目范围的前提下缩短工期。
1.  **赶工 (Crashing)**：
    *   通过增加资源（如加班、加人）来压缩关键路径上活动的历时。
    *   **成本考量**：计算“进度压缩单位成本” = $(\text{压缩成本} - \text{正常成本}) / (\text{正常历时} - \text{压缩历时})$。
    *   优先压缩单位成本最低的关键活动。
    *   **Charles Symons 方法**：进度压缩因子 < 0.75 时，工作量急剧增加；通常最多压缩 25%。
2.  **快速跟进 (Fast Tracking)**：
    *   并行开展原本串行的活动（改变逻辑关系）。
    *   **风险**：可能导致返工和质量风险。

#### C. 关键链法 (Critical Chain Method, CCM)
*   **理论基础**：约束理论 (TOC)。
*   **特点**：同时考虑任务逻辑关系和**资源约束**。
*   **缓冲机制**：
    *   去除各活动的安全时间，集中设置缓冲。
    *   **项目缓冲 (Project Buffer)**：放在关键链末端，保护项目完工日期。
    *   **接驳缓冲 (Feeding Buffer)**：放在非关键链汇入关键链处，保护关键链不受非关键链延误影响。
    *   **资源缓冲 (Resource Buffer)**：确保关键资源可用。
*   **优点**：解决资源冲突和不确定性，提高绩效。
*   **缺点**：时间估计简化，局限单一资源视角。

### 3. 资源优化
*   **资源平衡 (Resource Leveling)**：
    *   解决资源冲突或超负荷。
    *   往往导致关键路径改变，工期延长。
*   **资源平滑 (Resource Smoothing)**：
    *   在不改变关键路径和完工日期的前提下，利用浮动时间调整资源需求，使资源使用更平稳。

## 五、 案例分析与工具

### 1. 课堂练习示例 (CPM计算)
*   **任务**：根据PDM网络图，计算ES, EF, LS, LF, TF, FF，确定关键路径。
*   **关键点**：
    *   关键路径是总浮动时间为0的路径。
    *   关键路径长度即为项目最短工期。
    *   非关键活动的浮动时间可用于资源平衡或应对风险。

### 2. MED计划案例 (医疗信息商务平台)
*   **敏捷计划应用**：
    *   **发布计划 (Release Planning)**：远期、粗粒度，基于 Product Backlog。
    *   **迭代计划 (Iteration/Sprint Planning)**：近期、细粒度，基于 Sprint Backlog。
    *   **Scrum 流程**：Sprint (2-4周) -> 每日站会 -> Sprint 评审 -> Sprint 回顾。
    *   **工具输出**：PDM网络图、甘特图、预算曲线。

### 3. 软件项目进度问题 (SPSP) 模型
*   **目标**：在给定任务、关系和资源限制下，寻找人员安排方案，以实现**时间最短、成本最小**。
*   **优化方向**：
    1.  调整资源，解决冲突。
    2.  调整进度，优化项目，缩短工期。
    3.  调整成本预算，减少费用。

### 4. 工具使用
*   **MS Project**：用于编排计划、资源分配、生成甘特图和跟踪进度。
