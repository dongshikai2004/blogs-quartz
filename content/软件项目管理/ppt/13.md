## 二、 范围管理 (Scope Management)

### 1. 范围执行控制定义
*   **定义**：监督项目的范围状态，管理范围基准变更的过程。
*   **范围确认 (Scope Verification)**：
    *   是项目干系人正式验收并接受已完成的项目可交付物的过程。
    *   通过参与者（倡议者、委托人、顾客等）的行为正式确定项目范围。
    *   **关键点**：用户参与；明确的要求说明。
    *   **区别**：
        *   **范围核实**：关心工作结果的“接受 (Acceptance)”。
        *   **质量控制**：关心工作结果的“正确性 (Correctness)”。
    *   **贯穿始终**：若项目提前终止，需对完成程度建立文档。

### 3. 范围确认的输入、工具与输出
*   **输入**：项目范围说明书、WBS和WBS字典、项目范围管理计划、可交付成果。
*   **工具和技术**：检查（审查、评审、产品评审等）。
*   **输出**：已接受的交付物、变更申请、推荐的纠正措施、更新的WBS和WBS字典。

### 4. 项目范围控制与变更
*   **范围控制**：监控项目工作范围和产品范围状态，控制变更后果，确保所有变更通过整体变更控制过程。
*   **范围变更原因**：
    *   外部事件（如政府规定变更）。
    *   产品范围定义的过失或疏忽。
    *   项目范围定义的过失或疏忽。
    *   增值变更（如新技术降低成本）。
    *   应对风险的应急计划。
*   **变更控制关注点**：
    *   确定变更已发生。
    *   对造成变更的因素施加影响，确保一致认可。
    *   管理发生的变更。
*   **防治不合理扩张**：
    *   **范围蔓延 (Scope Creeping)**：项目期间不期望的需求缓慢增加，超出计划目标。
    *   **镀金 (Gold-plating)**：团队主动在定义范围外增加额外工作或功能。

### 5. 敏捷项目范围管理
*   **特点**：需求不断被定义。
*   **Scrum框架**：
    *   **每日站立会议 (Daily Scrum)**：回答三个问题（昨天做了什么？今天打算做什么？遇到什么困难？）。
    *   **冲刺规划会议 (Sprint Plan)**：产品负责人展示高优先级Backlog，团队分解任务。
    *   **冲刺复审会议 (Sprint Review)**：向利益相关人展示成果。
    *   **冲刺回顾会议 (Sprint Retrospective)**：总结改进开发过程。
*   **管理手段**：
    *   把需求列入未完项 (Product Backlog)。
    *   不断构建和评审原型系统。
    *   通过发布多个版本来明确需求。

## 三、 进度与成本管理 (Time & Cost Management)

### 1. 跟踪与控制建议
*   **进度跟踪**：及时更新信息，比较基准与实际差异。
    *   建议：进度张弛有度；关注关键路径（尤其是多条时）；明确检查点。
*   **成本跟踪**：计算任务实际成本；每天更新；对比预算。
*   **资源跟踪**：记录总实际工时；每天更新；对比计划与实际工时差异。

### 2. 性能分析的主要技术
1.  **图解控制法**：
    *   **甘特图 (Gantt Chart)**：直观显示计划与实际进度的对比。
    *   **延迟图**：强调每个活动的相对位置和延迟情况。
    *   **时间线 (Timeline)**：展示特定资源或任务的时间分布。
    *   **累计费用曲线 (S曲线)**：平滑递增的计划成本(BCWS)和实际成本(ACWP)曲线。
    *   **资源载荷图**：显示资源使用的偏差。
*   **偏差分析与控制**：
    *   持续时间偏差(%) = ((实际持续时间 - 计划持续时间) / 计划持续时间) * 100%
    *   进度偏差(%) = ((实际结束时间 - 计划结束时间) / 计划持续时间) * 100%
    *   需进行根因分析。

2.  **挣值分析法 (Earned Value Analysis, EVA)**
    *   **核心概念**：综合范围、成本、进度进行绩效评估。
    *   **基本参数**：
        *   **PV (Planned Value)** / BCWS：计划完成工作的预算成本。
        *   **AC (Actual Cost)** / ACWP：已完成工作的实际成本。
        *   **EV (Earned Value)** / BCWP：已完成工作的预算价值（挣值）。
        *   **BAC (Budget At Completion)**：项目总成本预算，不包含管理储备。
    *   **偏差分析**：
        *   **进度偏差 SV** = EV - PV （<0 落后，>0 超前，=0 正常）
        *   **成本偏差 CV** = EV - AC （<0 超支，>0 节约，=0 正常）
    *   **绩效指数**：
        *   **进度绩效指数 SPI** = EV / PV （<1 落后，>1 超前）
        *   **成本绩效指数 CPI** = EV / AC （<1 超支，>1 节约）
        *   *注：研究表明进度20%左右时CPI趋于稳定。*
    *   **预测指标**：
        *   **ETC (Estimate To Complete)**：完工尚需估计。
            *   非典型偏差：ETC = BAC - EV
            *   典型偏差：ETC = (BAC - EV) / CPI
        *   **EAC (Estimate At Completion)**：完工时估算。
            *   EAC = AC + ETC
            *   使用剩余预算：EAC = AC + BAC - EV
            *   使用CPI：EAC = AC + (BAC - EV) / CPI 或 EAC = BAC / CPI
            *   其他公式：EAC = BAC / (CPI * SPI)
        *   **VAC (Variance At Completion)**：完工成本差异 = BAC - EAC
        *   **SAC (Schedule At Completion)**：预计完工时间 = 原计划时间 / SPI
        *   **TCPI (To-Complete Performance Index)**：完工尚需绩效指数 = (BAC - EV) / (BAC - AC) [若Goal=BAC]
    *   **EV计算方法**：
        *   自下而上（麻烦）。
        *   **50/50规则**：开始算50%，结束算100%。
        *   **0/100规则**：开始算0%，结束算100%。
        *   经验加权法。

3.  **网络图分析**
    *   分析网络图中某任务的进度成本。
    *   **贝叶斯网络 (Bayesian Network)**：
        *   用于解决不确定性问题，模拟因果关系。
        *   结构：有向无环图 (DAG)，节点表示随机变量，箭头表示因果。
        *   **应用步骤**：
            1.  构建项目网络图。
            2.  构建活动的贝叶斯网络（确定影响因素如复杂度、人员经验、需求波动等）。
            3.  数据处理（定义变量状态如H/M/L及概率区间）。
            4.  模型学习（确定先验概率和条件概率分布表）。
            5.  网络更新（根据新证据更新概率）。
            6.  敏感性分析（找出影响进度的关键因素，如编码过程质量、需求波动等）。

4.  **敏捷方法中的进度衡量**
    *   **原则**：交付价值替代预测型衡量指标。
    *   **MVP (Minimum Viable Product)**：最小可行性产品，快速验证构思。
    *   **可视化工具**：
        *   **燃尽图 (Burndown Chart)**：基于迭代，显示剩余Story Points随时间的变化。理想线 vs 实际线。
        *   **燃起图 (Burnup Chart)**：显示已完成Story Points随时间的积累。
        *   **功能燃起/燃尽图**：以功能数量为维度。
    *   **基于流程的指标**：
        *   **交付周期 (Lead Time)**：从需求确认到上线交付的时间 ("From Idea To Launch")。
        *   **周期时间 (Cycle Time)**：处理一个工作项目所需的时间。
        *   **响应时间**：等待开始的时间。
        *   **控制在制品 (WIP)**：限制同时进行的任务数量。
    *   **完成指标**：
        *   **迭代速率 (Velocity)**：每迭代完成的Story Points数。用于预测剩余迭代次数。
        *   **期望值管理**：控制对团队生产力提升的预期。

## 四、 质量管理 (Quality Management)

### 1. 质量保证 (QA) vs 质量控制 (QC)
*   **质量保证 (QA)**：
    *   **定义**：为了提供信用，证明项目将达到质量标准而开展的有计划、有组织的工作活动。
    *   **性质**：管理职能，贯穿始终，预防性。
    *   **活动**：审计质量要求、确立流程、改进体系评估。
    *   **对象**：过程、体系、服务。
*   **质量控制 (QC)**：
    *   **定义**：监控特定项目成果，判定是否符合标准，消除不满意原因。
    *   **性质**：技术性活动，贯穿执行全过程，检测性。
    *   **对象**：可交付成果、管理成果（成本/进度绩效）。
    *   **方法**：测试、检查、技术评审、代码走查、控制图、趋势分析、抽样统计。

### 2. 质量控制的具体技术与工具
*   **质量审计 (Audit)**：
    *   对过程或产品的独立评估，确保遵循规程。
    *   分为过程审计（检查执行过程）和产品审计（检查工作产品如设计、代码）。
*   **技术评审 (Technical Review, TR)**：
    *   **目的**：尽早发现缺陷，验证符合需求/规范，统一开发模式。
    *   **正规技术评审 (Formal Inspection)**：同行审查，有规范步骤。
    *   **流程**：准备 -> 评审会议（讲解、提问、讨论）-> 决策（接受/拒绝/修改后重审）-> 记录报告。
*   **代码评审/走查**：
    *   **代码评审**：小组阅读、讨论、静态分析。
    *   **代码走查**：开发人员自查。
    *   **审核 (Inspection)**：Fagan引入，更正式，角色分明（仲裁人、作者、读者、记录员），步骤包括概述、准备、会议、返工、后续。
*   **软件测试**：
    *   **目的**：发现错误（成功的测试是发现了未发现的错误）。
    *   **类型**：
        *   单元测试：模块逻辑，覆盖分支。
        *   集成测试：模块接口。
        *   系统测试：整体符合需求规格。
        *   压力测试：大数据量/低资源下的健壮性（LoadRunner）。
        *   接收测试：客户参与，验证功能和便捷性。
*   **缺陷跟踪**：
    *   流程：发现 -> 分配 -> 修正 -> 验证 -> 关闭/拒绝。
    *   工具：TrackRecord, Bugzilla, Jira, BugZero等。
    *   静态检查工具：Findbugs（检查代码潜在错误如自赋值）。
*   **老七种工具 (部分提及)**：
    *   **控制图 (Control Chart)**：显示质量随时间变化，识别偏差（如7点规则判断趋势）。
    *   **排列图 (Pareto Chart)**：80/20定律，识别主要影响因素（A类70-80%, B类80-90%, C类90-100%）。
    *   **因果图 (Fishbone/Ishikawa)**：分析人、机、料、法、环 (4M1E) 对结果的影响。
    *   **散点图**：分析两个变量的相关性。
    *   **趋势分析**：依据过去预测未来。
    *   **检查/抽样统计**：基于概率抽取部分检查。
*   **新七种工具 (部分提及)**：
    *   **亲和图 (KJ法)**：整理思路，抓住实质。
    *   **关联图**：分析复杂因果关系（横向/纵向）。
    *   **系统图 (Tree Diagram)**：目的-手段展开，寻找最佳路径。
    *   **矩阵图/矩阵数据分析法**：多因素对应分析，定量分析（主成分分析）。
*   **6σ管理法**：
    *   以顾客为中心，DMAIC流程（定义、测量、分析、改进、控制）。
    *   注重量化数据、流程关注、持续改进短板。

### 3. 敏捷项目质量管理
*   **特征**：不断审核，定期检查效果，小批量工作早期发现问题。
*   **活动**：
    *   结对编程 (Pair Programming)。
    *   测试驱动开发 (TDD)。
    *   持续集成 (CI)。
    *   不同层面测试。
    *   迭代评审与回顾 (Refactor)。
    *   验收测试驱动开发 (ATDD)。

### 4. 敏捷 vs 传统思维对比
| 维度 | 敏捷 (Agile) | 传统 (Traditional) |
| :--- | :--- | :--- |
| **项目交付** | 灵活的解决方案 | 固定的解决方案 |
| **交付时间** | 尽早交付 | 最后交付 |
| **输出形式** | 多种输出 (Incremental) | 单一输出 |
| **功能特性** | 按优先级排序 | 按之前承诺 |
| **团队角色** | 自组织团队 | 项目经理指导团队 |
| **管理方式** | 协同合作 | 命令与控制 |
