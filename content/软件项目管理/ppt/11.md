# 第11章 配置管理计划 (Software Configuration Management Plan)

## 1. 前言与背景

### 1.1 软件开发中的常见问题
在软件项目中，缺乏有效的配置管理常导致以下问题：
*   找不到文件的历史版本。
*   开发人员使用错误的版本修改程序。
*   未经授权修改代码或文档。
*   人员流动导致工作交接不彻底。
*   已修复的Bug在新版本中重现（回归问题）。
*   无法重新编译某个历史版本。
*   协同开发或异地开发中，版本变更混乱导致项目失败。

### 1.2 核心挑战：变化
*   软件开发过程是一个**持续不断变化的过程**。
*   有效的项目管理旨在控制变化，以最有效的手段应对变化，不断命中“移动的目标”。

### 1.3 文档管理基础
*   **定义**：文档是描述、定义、规定、报告或认证活动、需求、过程或结果的书面或图示信息。
*   **分类**：
    *   按正式程度：非正式文档、正式文档。
    *   按生命周期阶段：开发文档、产品文档、管理文档。
    *   常见文档类型：可研报告、项目开发计划、需求说明书、设计说明书（概要/详细/数据库）、用户/操作手册、测试计划/报告、进度月报、总结报告等。
*   **管理规则**: 书写规范、图表编号、目录标准、管理制度。

## 2. 软件配置管理 (SCM) 概述

### 2.1 定义
*   **软件配置管理 (SCM)**：又称软件形态管理或软件构建管理。
*   **核心目标**：界定软件组成项目，对每个项目的变更进行管控（版本控制），维护不同项目间的版本关联，确保软件在开发过程中任一时间的内容都可被追溯。
*   **地位**：贯穿整个软件生命周期，是PMBOK、ISO9000、CMM/CMMI的重要组成部分。

### 2.2 SCM 的三个方面
1.  **Version Control (版本控制)**
2.  **Change Control (变更控制)**
3.  **Process Support (过程支持)**

### 2.3 关键活动
*   配置项标识、工作空间管理、版本控制、变更控制、状态报告、配置审计。

### 2.4 有无SCM的对比
*   **没有SCM的后果**：
    *   工作成果被覆盖（新内容覆盖旧内容，若新内容错误则无法恢复）。
    *   手工管理混乱（如文件名后缀 `-01`, `-02`，难以记忆和管理）。
*   **使用SCM的好处**：
    *   **直接好处**：保留所有历史版本，防止丢失和覆盖。存储成本低廉。
    *   **间接好处**：完整保留企业知识财富，便于分享利用，减少人员离职带来的损失。

### 2.5 配置管理的商业理念
*   **“恰好够用”原则**：不必追求完美的100分配置管理（代价太高），通常以30%的代价获取60-70分（及格）的成绩最划算。
*   **特征**：
    *   全员参与，养成习惯。
    *   规范清晰明了，不给人员添加过多负担。
    *   工具选择综合考虑价格、易用性和功能，不盲目追求最先进。

## 3. 配置管理基本概念

### 3.1 主要术语
*   **配置项 (Configuration Item, CI / SCI)**：
    *   产品生命周期各阶段产生的各种形式和版本的文档、程序、部件及数据的集合。
    *   包括：系统/软件需求规格说明书、设计说明书、源代码、测试规格书等。
    *   **配置识别步骤**：识别CI -> 分配唯一标识号 -> 确定特征 -> 确定纳入时间 -> 确定责任人 -> 填写并审批配置项管理表。
*   **基线 (Baseline)**：
    *   软件生存期中各个开发阶段的一个特定点。
    *   标志着一个阶段的结束和里程碑。
    *   基线一旦建立，其修改必须经过正式程序（SCCB批准）。
    *   **主要基线类型**：
        *   **功能基线**：经评审批准的系统功能特性文档（通常在SRR/SDR后建立）。
        *   **指派基线 (需求基线)**：描述待开发软件功能的文档（通常在SRR后建立）。
        *   **产品基线**：最终批准的描述配置项全部必要功能和物理特性的文档，包括代码、文档、工具等（通常在FCA/PCA后建立）。
        *   **开发配置**：开发进程中不断演化和累积的配置。
*   **SCCB (Software Configuration Control Board, 软件配置控制委员会)**：
    *   **组成**：项目经理及相关人员。
    *   **职责**：评估变更（技术、逻辑、策略、经济、组织角度），批准或拒绝变更申请。
    *   **性质**：决策机构，非作业机构；不提出变更方案，只决定能否变更。
*   **版本 (Version)**：源代码、文档、数据及系统环境的集合。

### 3.2 配置库结构
*   **开发库 (动态库)**：存放开发周期某阶段正在工作的信息，权限较宽松。
*   **受控库 (静态库/配置库)**：存放阶段结束时通过审核的产品及相关信息，由CM严格管理。
*   **产品库**：存放最终发布的产品。

### 3.3 工作空间 (Workspace)
*   **定义**：为完成特定任务，从版本库中选择一组正确版本的文件拷贝到开发环境。
*   **类型**：
    *   **私有空间**：开发人员独立编写和测试代码。
    *   **集成空间**：团队共享，用于集成所有开发成果。
*   **特点**：稳定、一致、透明。

## 4. 软件配置管理过程

### 4.1 基本活动流程
1.  **配置项标识与跟踪**
2.  **配置管理环境建立**
3.  **基线变更管理**
4.  **基线审核**
5.  **配置状态统计**
6.  **配置管理计划制定**

### 4.2 详细过程解析

#### (1) 配置项标识
*   将项目受控部分拆分为SCI，建立唯一标识。
*   **命名规范示例**：`QTD-School–RM–SRS-v1.0` (公司-项目-类型-编号-版本)。

#### (2) 配置管理环境建立
*   建立配置管理库（VOB - Versioned Object Base）。
*   设置目录结构、用户权限。

#### (3) 变更控制过程 (核心)
*   **变更原因**：需求渐进明细、范围定义疏忽、增值变更、风险应对、外部事件等。
*   **基本原则**：建立基准、变更流程、CCB、明确分工、完整体现影响、妥善保存文档。
*   **变更请求 (CR, Change Request)**：
    *   对软件变更需要的描述（增强、缺陷修复等）。
    *   必须通过CR流动来实现对基线的修改。
*   **变更处理流程**：
    1.  **提出变更**：识别需要，提交CR。
    2.  **接受变更**：确认CR清晰、完整、无重复，赋予唯一ID。
    3.  **变更评价 (影响分析)**：
        *   技术影响、接口影响、成本影响、进度影响。
        *   使用核查表确定急迫性、性质。
    4.  **变更处置 (SCCB决策)**：
        *   **批准**：指派实施、验证、更新版本、发布。
        *   **不批准**：没必要、不可行、有更好替代。
        *   **推迟**：需附加信息或进一步分析。
    5.  **变更实施**：
        *   PM管理工作包，安排进程。
        *   责任人检出 (Check out) -> 实施更改 -> 记录信息。
    6.  **变更验证**：
        *   审查、评审、走查、回归测试。
        *   验证通过后检入 (Check in) 受控库。
    7.  **结束变更**：
        *   确认正确实施、无副作用、文档更新。
        *   CMO备档原基线，建立新基线，关闭CR，通知申请人。

#### (4) 配置审计
*   **目的**：确保配置管理活动符合方针规程；保证基线产品的完整性和一致性。
*   **时机**：产品交付/发布前、阶段结束、维护工作中。
*   **实施**：准备检查单 -> 审核文档记录 -> 发现并消除不一致。
*   **类型**：
    *   **功能配置审计 (FCA)**：验证配置项是否达到了需求规格说明中规定的功能特性。
    *   **物理配置审计 (PCA)**：验证配置项的物理存在是否与文档一致（如代码、文档版本对应）。

#### (5) 配置状态统计 (CSR)
*   **目的**：及时、准确报告配置项当前状况。
*   **内容**：变更请求数量、历史报告、存储量增长、SCCB运作异常次数等。

## 5. 软件配置管理计划 (SCMP)

### 5.1 计划制定过程
*   参加项目规划 -> 规划配置管理任务 -> 形成配置管理计划 -> 评审配置计划。

### 5.2 配置管理计划模板要素
1.  **引言**
2.  **软件配置管理**
    *   组织结构与职责 (CMO, SCCB, PM等)。
    *   与软件生命周期的关系。
3.  **软件配置管理活动**
    *   配置标识 (命名规范、主要配置项列表)。
    *   项目基线 (定义基线名称、包含内容、建立时机)。
    *   配置库 (目录结构、存储区域划分)。
    *   配置控制程序 (变更流程、CR表单格式)。
    *   配置状态统计 (报告频率、格式)。
    *   配置审核 (审计计划、检查单)。
4.  **支持**
    *   配置管理工具选择。

### 5.3 配置管理工具
*   **功能要求**：版本管理、变更管理、问题追踪、建立管理、状态统计、配置审计、访问控制。
*   **工具演变**：
    *   **面向文件**：早期工具（如VSS, CVS），以单个文件为对象，缺乏任务概念。
    *   **面向变更集**：引入变更集概念，将代码修改与任务关联。
    *   **面向开发流程**：基于任务的变更控制，定义开发流程，与项目管理结合紧密（如ClearCase + ClearQuest）。
*   **常见工具**：ClearCase, ClearQuest, PVCS, Harvest, CVS/SVN, Git, VSS。

### 5.4 不同规模项目的建议
*   **小项目/小企业**：制定规则 + 版本管理工具。
*   **中小项目/企业**：制定规则 + 配置管理工具。
*   **大项目/大企业/异地开发**：制定规则 + 配置管理工具 + **专职配置管理人员**。

## 6. 案例分析

### 6.1 反面案例：陷入泥潭的项目
*   **背景**：电力公司电网计费系统，瀑布模型，仅有简单版本管理，无系统化SCM支持。
*   **问题**：
    *   需求不明确，边做边看，文档修改未同步。
    *   编码与设计脱节，接口改变导致大量返工。
    *   版本混乱，开发人员用错代码版本。
    *   集成联调耗时过长（2周变2月）。
    *   测试不充分，交付后缺陷频发，一期二期界限模糊。
    *   团队士气低落，人员流失。
*   **教训**：缺乏完善CM系统导致需求界定不清、版本管理失效、并行开发困难、变更失控、进度不可控、测试不规范、交付无依据、复用率低。

### 6.2 正面案例：“校务通系统”配置管理计划
*   **工具**：VSS (因项目小、工期短、团队熟悉)。
*   **组织**：PM为SCCB负责人，设专职配置管理员。
*   **标识**：制定了详细的命名规范（如 `QTD-SCHOOL-TCM-Contract-V1.0`）。
*   **基线**：定义了需求、总体设计、项目实现、系统测试等基线及其建立时间。
*   **权限与目录**：明确了不同角色对不同目录（TCM, RM, SPP等）的访问权限。

### 6.3 InfoSys公司配置管理计划
*   **更规范的示例**：
    *   区分了工作区域、评审区域、基准/发行区域。
    *   详细的版本号规则：
        *   代码：临时1.0，主要修改1.1，次要1.1.1。
        *   文档：初始0.0a，修订0.0.b，基准1.0。
    *   规定了文档在存储区域间的移动流程和备份存档过程。

## 7. 高级主题：ClearCase UCM 模型 (Unified Change Management)

### 7.1 UCM 核心概念
UCM 提升了配置管理的抽象层次，将工件组织化，简化变更管理。
*   **Project (项目)**：包含组件和流。
*   **Component (组件)**：呈现软件结构，是相关目录和文件的集合，可在项目间共享。
*   **Stream (流)**：
    *   **开发流**：开发人员的私有工作区，包含个人活动。
    *   **集成流**：团队共享工作区，记录项目基线。
    *   流决定了视图中元素的版本。
*   **Activity (活动)**：
    *   开发任务（如修复Bug、新增功能）。
    *   包含一个**变更集 (Change Set)**，即该任务修改的所有文件版本的集合。
    *   开发者基于活动工作，完成后交付 (Deliver) 到集成流。
*   **Baseline (基线)**：
    *   组件的一个稳定版本集合。
    *   代表项目里程碑（如创建、测试、发布）。
    *   可升级状态（如从“测试”升级为“发布”）。
*   **View (视图)**：
    *   用户的工作空间，映射流中的特定版本。
    *   **动态视图**：通过网络直接访问VOB，实时同步。
    *   **静态视图 (Snapshot)**：本地拷贝，需定期Update同步，适合离线或远程。

### 7.2 UCM 工作流程
1.  **初始化**：PM创建项目，确定初始基线。
2.  **加入**：开发人员创建私有视图，获取基线。
3.  **开发**：
    *   创建活动 (Activity)。
    *   在视图修改文件，自动记录到活动的变更集。
    *   本地编译调试。
4.  **交付 (Deliver)**：
    *   开发人员将完成的活动交付到集成流。
    *   合并代码，解决冲突。
5.  **集成与基线**：
    *   集成人员在集成流中创建新基线。
    *   构建 (Build) 并运行基本测试。
    *   提升基线状态（如标记为“推荐基线”）。
6.  **重基线 (Rebase)**：
    *   开发人员将私有视图同步到最新的推荐基线，继续下一轮开发。

### 7.3 UCM 的优势
*   **以活动为中心**：任务与代码紧密关联，易于追踪。
*   **并行开发支持**：通过分支和流机制，隔离高风险工作，支持多版本并行。
*   **自动合并**：减少集成时间和冲突。
*   **异地开发支持**：通过MultiSite等技术实现分布式版本库同步。
*   **简化变更管理**：团队整体受益，角色分工明确（PM、开发、集成、测试）。

## 8. 附录：相关国家标准 (GB/T)

*   **基础标准**：GB/T 11457 (术语), GB/T 8566 (生命周期), GB/T 25000.1 (质量模型).
*   **开发与文档**：GB/T 8567 (文档编制规范), GB/T 15853 (支持环境).
*   **测试与质量**：GB/T 15532 (测试规范), GB/T 25000.51 (就绪软件产品质量).
*   **项目管理**：GB/T 30264 (知识体系), GB/T 20157 (软件维护).
*   **安全与可靠性**：GB/T 30998 (安全保障), GB/T 29831 (可靠性).
*   **军用标准**：GJB 5000A (能力成熟度), GJB 438B (军用文档).
