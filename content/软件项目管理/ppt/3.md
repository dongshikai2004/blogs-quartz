# 第三章：范围计划 (Scope Planning)

*   **范围计划**
*   **进度计划**

## 二、 软件需求管理过程

### 1. 需求工程 (Requirement Engineering) 概述
*   **定义：** 应用已证实有效的技术、方法进行需求分析，确定客户需求，帮助分析人员理解问题并定义目标系统的所有外部特征。
*   **组成：**
    *   **需求开发 (Requirement Development)：**
        1.  **需求获取 (Elicitation)：** 利益相关者咨询、语境规范化。
        2.  **需求分析 (Analysis)：** 对系统的理解，结构化文档记录。
        3.  **规格说明 (Specification)：** 识别省略、冗余、不一致性。
        4.  **需求验证 (Verification)：** 跟踪满足需求的地方。
    *   **需求管理 (Requirement Management)：**
        1.  **需求跟踪 (Traceability)：** 跟踪需求实现情况。
        2.  **变更管理 (Change Management)：** 需求维护和传播。
        3.  **实现确认：** 成功实现的结果和确认。

### 2. 软件需求的层次与类型
软件需求分为三个主要层次及非功能性需求：

1.  **业务需求 (Business Requirement)：**
    *   **来源：** 投资人、客户高层、市场部门。
    *   **内容：** 组织为什么要开发该系统？希望达到的宏观目标。
    *   **文档：** 愿景文档 (Vision)、范围文档 (Scope)、项目章程 (Project Charter)、市场需求文档 (MRD)。
2.  **用户需求 (User Requirement)：**
    *   **来源：** 最终用户。
    *   **内容：** 用户利用系统能做什么？完成什么任务？
    *   **表达方式：** 用例 (Use Case)、场景描述、事件-响应表。
3.  **功能需求 (Functional Requirement)：**
    *   **来源：** 开发人员需实现的具体功能。
    *   **内容：** 系统必须执行的行为，以满足业务和用户需求。
    *   **别名：** 行为需求 (Behavioral Requirement)。
    *   **描述方式：** “系统应该...” (The system shall...)。
4.  **非功能性需求 (Non-Functional Requirement) / 系统需求：**
    *   包括：质量属性 (性能、可靠性等)、约束条件、假设、业务规则。

### 3. 需求获取 (Requirement Elicitation)
*   **目的：** 从了解、理解到确定不同用户的需要和限制。
*   **关键：** 沟通和交流。避免交流障碍、沟通不全、意见冲突 a。
*   **主要方法：**
    1.  **访谈和调研：**
        *   分析系统需求规格说明。
        *   市场调查（竞品分析）。
        *   访问用户及领域专家。
        *   考察现场（观察实际操作环境）。
    2.  **专题讨论会 (JAD/Workshops)：** 协调讨论，群体决策，达成共识。
    3.  **头脑风暴 (Brain-storming)：** 自由联想，激发创新设想。
    4.  **场景串联：** 描绘事件触发时的情景和事件流，有助于测试用例设计。
*   **注意事项：** 识别真正客户、正确理解需求、具备忍耐力和清晰思维、说服和教育客户、建立分析小组。

### 4. 需求分析 (Requirement Analysis)
*   **定义：** 对问题进行详细分析，确定输入、输出及处理逻辑。建立系统的概念模型。
*   **特点：**
    *   用户与开发人员存在沟通鸿沟（不懂业务 vs 不懂技术）。
    *   需求动态多变。
    *   变更代价随生命周期呈非线性增长（越后期代价越大）。
    *   遵循 80-20 规则。
*   **建模方法：**
    1.  **原型分析方法 (Prototyping)：**
        *   **思想：** 快速构造初始原型 -> 用户评价 -> 修改完善 -> 螺旋式上升。
        *   **优点：** 符合认识规律，易于消除误解。
        *   **步骤：** 确定基本需求 -> 开发原型 -> 共同评价 -> 修改原型/需求。
    2.  **结构化分析方法 (Structured Method)：**
        *   **基础：** 面向数据流，自顶向下逐步求精。
        *   **工具：** 数据流图 (DFD)、数据字典 (DD)、系统流程图。
        *   **原则：** 分解与抽象、模块独立性、信息隐蔽。
    3.  **基于UML的面向对象分析：**
        *   **核心：** 用例建模 (Use Case Modeling)。
        *   **图表：** 用例图、顺序图、状态图、活动图。
        *   **步骤：** 识别Actor -> 描述Use Case -> 实现视图。
    4.  **敏捷需求分析：**
        *   **来源：** Product Backlog (产品待办列表)。
        *   **细化：** Sprint Backlog -> User Stories (用户故事)。
        *   **用户故事格式：** `As a <type of user>, I want <some goal> so that <some reason>.`
        *   **INVEST原则：** Independent (独立), Negotiable (可协商), Valuable (有价值), Estimatable (可估算), Small (小), Testable (可测试)。
        *   **优先级排序：** MoSCoW法 (Must have, Should have, Could have, Won't have/Want to have)。

### 5. 需求规格说明 (SRS - Software Requirements Specification)
*   **标志：** 形成完整、规范的《需求规格说明书》是需求分析完成的标志 as the basis for development.
*   **编制原则：**
    *   描述“做什么”而非“怎么做”。
    *   使用规格说明语言。
    *   包含系统运行环境。
    *   是一个认识模型。
    *   允许不完备性和扩充。
*   **SRS文档结构参考：
    1. 引言
    2. 系统定义
    3. 应用环境
    4. 功能规格
    5. 性能需求
    6. 产品提交
    7. 实现约束
    8. 质量描述
    9. 其它
    10. 签字认证

### 6. 需求验证 (Requirement Verification)
*   **目的：** 确保需求准确、完整、一致、可行。
*   **验证标准：**
    *   正确吗？
    *   一致吗？
    *   完全吗？
    *   实际可行吗？
    *   必要吗？
    *   可检验吗？ (Testable)
    *   可跟踪吗？ (Traceable)
*   **可测试性需求：** 软件应具有高内聚、弱耦合、接口明确、意图明晰。具备可控制性、可分解性、稳定性、易理解性、可观察性。

### 7. 需求变更管理 (Requirement Change Management)
*   **现状：** 需求变更是常态，也是项目失败主因之一。
*   **变更原因：**
    1.  用户因素（主观变化）。
    2.  系统因素（技术限制/发现新特性）。
    3.  工作环境因素。
    4.  需求开发工作缺陷（前期没做好）。
    *   *策略：重点控制用户因素和工作缺陷导致的变更。*
*   **管理流程：**
    1.  建立基线 (Baseline)。
    2.  建立变更控制委员会 (**SCCB** - Software Configuration Control Board)。
    3.  提交变更申请。
    4.  SCCB评估影响（进度、成本、质量）。
    5.  决策：拒绝 / 接受（当前版本） / 推迟（下一版本）。
    6.  若接受：修改合同、需求文档、项目计划。
    7.  跟踪受影响的工作产品。
    8.  维护变更记录和历史。
*   **工具：** Rational RequisitePro, IBM Rational DOORS, Borland CaliberRM, JIRA, ZenTao (禅道), PingCode等。

## 三、 任务分解定义 (WBS - Work Breakdown Structure)

### 1. WBS 定义
 a   *   **定义：** 面向可交付成果的，对项目元素进行的分组。它组织并定义了**整个项目范围**。
    *   **核心原则：** **不在WBS中的工作，就不是该项目的工作。**
    *   **结构：** 分级的树型结构，由粗到细。
    *   **工作包 (Work Package)：** WBS的最底层要素，是可分配、可估算、可监控的最小可交付成果。应由唯一主体负责。

### 2. WBS 的意义
*   使项目明确、清晰、透明、具体。
*   保证项目结构的系统性和完整性。
*   建立完整的项目保证体系。
*   明确各方责任。
*   是进度计划、成本估算、质量控制的基础。
*   为沟通提供依据。
*   作为范围基准，是范围变更控制的输入。

### 3. 任务分解的方法
*   **类比法：** 参照类似项目的WBS。
*   **模板法：** 使用行业标准或组织内部模板。
*   **自上而下 (Top-Down)：** 从项目整体开始，逐层分解直到工作包。
*   **自下而上 (Bottom-Up)：** 先列出具体任务，再归类汇总（较少用，通常用于补充）。

### 4. WBS 分解步骤
1.  确认并分解项目的组成要素。
2.  确定分解标准（必须统一，不能混合使用多种标准）。
3.  确定分解的详细程度（是否足够详细以进行估算和控制）。
4.  确定项目交付成果。
5.  验证分解的正确性（建立编号体系）。

### 5. 分解标准 (Decomposition Criteria)
*   **按生存期阶段分解：** 规划、需求、设计、编码、测试、提交。
*   **按功能组成分解：** 如招生管理、分班管理、档案管理等。
*   **注意：** **同一层级必须使用统一的分解标准**，严禁混合使用（例如不能在同一层既有“需求阶段”又有“数据库模块”）。

### 6. 检验分解结果的标准
*   最底层要素是否是实现目标的充分必要条件？
*   最底层要素是否有重复？
*   每个要素是否清晰完整定义？
*   最底层要素是否有明确的责任人？
*   最底层要素是否可以进行成本估算和进度安排？

### 7. WBS 指南 (Best Practices)
*   **层级深度：** 最好不要超过 **7层**。
*   **工作包大小：** 软件项目推荐分解到 **40小时** (约1周) 的任务量。
*   **交付物：** 每个Work Package必须有明确的提交物。
*   **完成标准：** 定义任务完成的具体标识。
*   **WBS字典：** 配套编写WBS字典，详细描述每个节点的信息（ID、名称、描述、责任人、验收标准等）。
*   **评审：**最后需与相关人员评审WBS。

### 8. WBS 编号系统示例
*   采用层级编码，如：
    *   1. 软件产品
        *   1.1 功能1
            *   1.1.1 子功能1
            *   1.1.2 子功能2
        *   1.2 功能2

### 9. 敏捷项目中的任务分解
*   **传统项目：** 预先进行完整的自上而下分解。
*   **敏捷项目：** 演化形式。
    *   **Epics (史诗)：** 大型需求。
    *   **Stories (故事)：** Epics分解为较小的User Stories。
    *   **Tasks (任务)：** Stories在Sprint计划中进一步分解为具体的开发任务。
    *   **输出：** Product Backlog -> Sprint Backlog。

## 四、 案例分析与实例

### 1. 学生管理系统 (结构化分析实例)
*   **数据流图 (DFD)：** 展示了学管科、体检科、学籍科与学生管理系统之间的数据流动（如学生基本信息、健康信息、成绩查询等）。
*   **数据字典 (DD)：** 定义了数据流的结构，例如：
    *   `学生基本信息 = 学号 + 姓名`
    *   `学生成绩 = 学号 + {课程名 + 成绩}`

### 2. 医疗信息商务平台 (MED) WBS实例
*   **顶层：** 医疗信息商务平台
*   **第二层 (功能模块)：**
    *   F1 用户管理 (注册、登录、权限、信息管理)
    *   F2 产品信息 (编辑、浏览、查找)
    *   F3 网上交易 (售前、售中、售后、统计分析)
    *   F4 分类广告
    *   F5 协会/学会管理
    *   F6 医院管理
    *   ...等
*   **底层工作包示例：** F1.1 个人注册, F2.1 离线产品编辑, F3.1 询价, F6.3 检索等。

### 3. George and Martha 野餐会 WBS实例
这是一个生活化的WBS例子，展示了如何将一个活动分解为具体任务、持续时间和责任人。
*   **任务示例：**
    *   做冰茶 (George, 15min)
    *   准备三明治 (Martha, 10min)
    *   装车 (George, 4min)
    *   开车去营地 (Martha, 20min)
*   **启示：** WBS不仅适用于大型软件项目，也适用于任何需要计划和管理的项目或活动。

### 4. 网管系统 WBS实例
*   **分解维度：** 按功能模块分解。
    *   F1 配置管理
    *   F2 故障管理
    *   F3 安全管理 (F3.1-F3.4)
    *   F4 性能管理 (F4.1-F4.7)

## 五、 本章小结

1.  **软件需求管理过程：** 获取 -> 分析 -> 规格说明 -> 验证 -> 变更管理。
2.  **需求建模方法：** 原型法、结构化分析 (DFD/DD)、UML面向对象分析 (用例图)、敏捷方法 (User Stories)。
3.  **任务分解 (WBS)：**
    *   是项目管理的基础。
    *   面向可交付成果。
    *   定义了项目范围基准。
    *   分解需遵循统一标准，底层为工作包。
    *   WBS是进度、成本、质量计划的基础。
