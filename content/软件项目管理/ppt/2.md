# 第二章 软件项目初始过程

## 一、 项目立项 (Project Initiation)

### 1. 项目启动背景
*   **核心逻辑**：顾客需求 $\rightarrow$ 输入 $\rightarrow$ 产品实现 $\rightarrow$ 输出（顾客满意）。
*   **涉及方**：甲方（建设方/客户）、乙方（承建方/开发方）。

### 2. 项目建议书 (Project Proposal)
*   **定义**：又称立项申请，是项目建设单位向上级主管部门提交项目申请时所必须的文件，是对拟建项目提出的框架性总体设想。
*   **作用**：是可行性研究的依据。
*   **关注点**：效益和利润。明确目标、时间表、资源、经费，并得到项目经理和项目发起人的认可。
*   **核心内容**：
    *   项目的必要性。
    *   项目的市场预测（产品方案或服务的市场预测）。
    *   项目建设必需的条件。
    *   其他：项目名称、建设依据、目的意义、国内外技术现状、关键技术/方案、研发情况等。

### 3. 可行性研究与分析 (Feasibility Study & Analysis)
*   **定义**：通过技术、经济分析，研究用最小投入获得最佳经济效果的科学方法。
*   **主要内容**：
    *   **可能性**：技术、物资、资金、人员支持可行性。
    *   **效益性**：经济效益、社会效益。
    *   **必要性**：社会环境、领导意愿、人员素质、认知水平。
*   **研究阶段**：机会研究 $\rightarrow$ 初步可行性研究 $\rightarrow$ 详细可行性研究 $\rightarrow$ 可行性研究报告。
*   **7个基本步骤**：
    1.  确定项目规模和目标。
    2.  研究正在运行的系统。
    3.  建立新系统的逻辑模型。
    4.  导出和评价各种方案。
    5.  推荐可行性方案。
    6.  编写可行性研究报告。
    7.  递交可行性研究报告。

#### 可行性研究的具体内容
1.  **技术可行性**：开发风险、人力资源有效性、技术能力可能性、物资可用性。
2.  **经济可行性**：
    *   支出分析（一次性/非一次性）。
    *   收益分析（直接/间接/其他）。
    *   收益投资比、投资回收期、敏感性分析。
3.  **运行环境可行性**：管理体制、规章制度、工作习惯、人员素质、数据积累、硬件平台等。
4.  **其他可行性**：法律可行性、社会可行性。

#### 初步可行性研究 vs 详细可行性研究
*   **初步可研**：
    *   **时机**：立项申请书批准后，介于机会研究和详细可研之间。
    *   **目的**：粗略论证，分析前途，初步估计关键技术、周期、人力财力、功能可实现性及效益。
    *   **结果**：肯定（马上实施/转详细可研/专题研究）或否定。
*   **详细可研**：
    *   **依据**：国家政策法律、主管部门批复、市场调研报告、技术资料等。
    *   **方法**：投资估算法、增量净效益法、经济评价法（财务/国民经济评价）、市场预测法。
    *   **报告结构**：概述、技术背景、现行系统分析、技术方案、实施进度、投资估算、人员培训、风险分析、效益预测、结论与建议。

### 4. 项目论证与项目评估
*   **项目论证 (Project Argumentation)**：
    *   **定义**：对技术先进性/适用性、经济合理性/赢利性、实施可能性/风险性进行全面综合分析。
    *   **核心**：市场是前提，技术是手段，财务经济是核心。
    *   **作用**：决策依据、筹资依据、计划编制依据、风险防范保证。
*   **项目评估 (Project Evaluation)**：
    *   **定义**：在可行性研究基础上，由第三方进行的评估。
    *   **目的**：审查可研报告的可靠性、真实性、客观性。
    *   **依据**：项目建议书及批复、可研报告、申请报告及初审意见、相关协议文件等。

#### 可行性论证与项目评估的对比
| 维度 | 可行性论证 | 项目评估 |
| :--- | :--- | :--- |
| **关系** | 基础工作 | 延伸、深化、再论证 |
| **时点** | 项目建议书批准之后 | 可行性论证批准之后 |
| **性质** | 宏观到微观深入研究 | 微观问题拿到宏观衡量 |
| **行为主体** | 建设单位（投资者）组织委托 | 贷款银行或有关部门组织委托 |
| **立足点** | 投资者角度 | 贷款银行/部门角度 |
| **侧重点** | 必要性与技术论证 | 建设可能性与偿债能力 |
| **作用** | 投资决策和审批依据 | 贷款决策和最终投资决策依据 |
| **相同点** | 均处于建设前期；内容、原理、理论、目标基本相同 | |

### 5. Make or Buy 决策 (自制或外购)
*   **定义**：确定待开发产品的哪些部分应当“采购”、“外包开发”或者“自主研发”。
*   **决策示例计算**：
    *   **自制**：初始成本 \$25,000，维护费 \$2,500/月。
    *   **购买**：初始成本 \$17,000，维护费 \$2,700/月。
    *   **差异**：制造差异 \$8,000，服务差异 \$200/月。
    *   **平衡点**：$8000 / 200 = 40$ 个月。
    *   **结论**：若软件生存期 < 40个月，选购买；若 > 40个月，选自制。

## 二、 合同项目 (Contract Projects)

### 1. 合同管理概述
*   **合同定义**：监督各方履行权利义务、具有法律效力的文件。
*   **法律特征**：民事法律行为；以设立/变更/终止民事权利义务为目的；当事人意思表示一致；平等自愿。
*   **实质要件**：甲方、乙方、标的（产品/货物/服务）。
*   **效力**：依法成立即生效。
*   **无效情况**：欺诈胁迫损害国家利益、合法形式掩盖非法目的、损害公共利益、违反法律等。
*   **形式**：书面（合同书、信件、数据电文）、口头、其他。

### 2. 甲方与乙方的角色
*   **甲方（采购方）**：明确需求、编制招标文件、评标、定标、签合同、变更管理、绩效审核。
*   **乙方（供方）**：准备标书/报价、签合同、执行标的、报告绩效、变更管理。

### 3. 合同类型与风险
| 合同类型 | 子类型 | 特点/公式示例 | 风险承担 |
| :--- | :--- | :--- | :--- |
| **成本补偿类** | CPPC (成本+百分比) | 总价随实际成本增加而增加（如成本20万，总价23万） | 甲方风险大 |
| | CPFF (成本+固定费) | 费用固定，成本实报实销（如成本20万，总价21.5万） | 甲方风险较大 |
| | CPIF (成本+奖金) | 设定目标成本和分享比例，节约共享，超支共担 | 风险共担 |
| **固定价格类** | FPI (固定价+奖励) | 设最高限价，低于目标价分享节约额，高于目标价分担超支 | 乙方风险中等 |
| | FFP (固定总价) | 价格锁定，无论实际成本多少，总价不变 | 乙方风险大 |
| **单价合同** | 工时和材料 | 综合两者优点，用于工作量不确定时 | 风险适中 |

*   **软件项目合同构成**：商务合同 + **技术合同**（核心，包括技术开发、转让、咨询、服务）。

### 4. 合同管理流程

#### 甲方合同环境
1.  **合同准备**：
    *   招标书定义（采购需求/SOW工作任务说明）。
    *   供方选择（收集建议书、评定、确定最终供方）。
    *   合同文本准备（草案制定、评审、修订、确认）。
2.  **合同签署**。
3.  **合同管理**：
    *   验证和检验（验收清单制定/评审/执行/报告）。
    *   违约事件处理（通告、确定处理方式、处理、报告）。
4.  **合同中止/结束**：文档归档、终止通知、项目执行总结。

#### 乙方合同环境
1.  **合同准备**：
    *   项目分析（需求分析、技术方案、规模估算、风险评估、初步规划）。
    *   竞标（能力评估、可行性分析、编写建议书/标书）。
    *   合同文本准备。
2.  **合同签署**：谈判日程、条款协商、文本审阅、正式签署。
3.  **合同管理**：
    *   执行跟踪（计划审批、管理规划、跟踪、责任落实、变更处理、进展报告）。
    *   修改控制（变更评估、策略确定、处理、报告）。
    *   违约处理。
    *   产品提交与维护（提交审查、签字认可、维护需求记录/分类/确认/任务下达）。
4.  **合同中止**：同甲方。

#### 内部项目实施
*   核心是确定任务范围和各方配合。
*   通过“协议”调整，可视作无商业约束力的“合同”，过程参考甲乙方流程。

### 5. 案例：医疗信息商务平台招投标
*   **甲方**：定义招标书（需求、SOW）。
*   **乙方**：项目分析（需求、技术架构SOA、功能模块、风险、规模）、编写标书。
*   **过程**：发布招标 $\rightarrow$ 乙方购标/写标 $\rightarrow$ 竞标/开标 $\rightarrow$ 中标方拟定合同 $\rightarrow$ 双方签署。

## 三、 项目授权 (Project Authorization)

### 1. 项目章程 (Project Charter)
*   **定义**：确认项目存在的文件。
*   **内容**：
    *   项目确认。
    *   项目经理授权。
    *   项目目标概述（如：9个月内升级2000名员工软硬件，预算150万）。
    *   主要干系人及职责（Sponsor, CIO, PM, HR, Purchasing等）。
    *   签字确认（Sign-off）。
*   **敏捷项目章程要素**：项目目标、发布标准、预期的工作流。

### 2. 项目经理 (Project Manager)
*   **角色**：领导者、管理者、决策者、分析者、计划者、控制者、组织者。
*   **职责**：开发计划、组织实施、项目控制。
*   **能力要求**：
    *   计算机及网络应用能力。
    *   新技术接受能力、自我更新能力。
    *   沟通、协调、管控能力。
    *   个人人格魅力。
*   **敏捷视角**：强调“仆人式领导” (Servant Leadership)。

## 四、 项目生存期 (Project Life Cycle)

### 1. 基本概念
*   **项目生命期**：连接项目开始和结束阶段的序列。
*   **阶段转移**：通常涉及技术交接，前一阶段可交付成果验收后进入下一阶段（“快速跟进”除外，但增加风险）。
*   **四个典型阶段**：
    1.  **概念/启动**：确立需求目标，可行性研究，编写章程，选择PM。
    2.  **开发/计划**：任命团队，WBS分解，总体规划，预算，风险评估，获得许可。
    3.  **执行/实施**：团队建设，细化技术，采购，跟踪执行，过程控制，协调。
    4.  **结束/收尾**：验收，移交，文档归档，经验总结，资源释放，团队解散。
*   **阶段特征**：
    *   以可交付成果完成为标志。
    *   阶段末进行审查（阶段关卡/验收站）。
    *   **资源投入曲线**：开始低，中间高，结尾迅速降低。
    *   **风险与不确定性**：随项目执行逐渐下降。
    *   **成功可能性**：随项目执行逐渐上升。
    *   **干系人影响力**：随项目执行逐渐下降。

### 2. 产品生存期 vs 项目生存期
*   **产品生存期**：调研 $\rightarrow$ 批准 $\rightarrow$ 需求 $\rightarrow$ 设计 $\rightarrow$ 开发 $\rightarrow$ 测试 $\rightarrow$ 交付 $\rightarrow$ 生产 $\rightarrow$ 维护 $\rightarrow$ 升级 $\rightarrow$ 淘汰。
*   **项目生存期**：批准 $\rightarrow$ 需求 $\rightarrow$ 设计 $\rightarrow$ 开发 $\rightarrow$ 测试 $\rightarrow$ 交付。

### 3. 软件开发模型变迁
作坊式 $\rightarrow$ 过程控制 $\rightarrow$ 敏捷 $\rightarrow$ DevOps

### 4. 生存期模型分类与选择

#### A. 预测型 (Predictive)
*   **特点**：提前大量计划，一次性执行，连续过程。
*   **瀑布模型 (Waterfall)**：
    *   阶段：需求 $\rightarrow$ 设计 $\rightarrow$ 编码 $\rightarrow$ 测试 $\rightarrow$ 维护。
    *   适用：需求明确、短期项目、类似项目经验丰富。
*   **V模型**：强调测试与开发阶段的对应关系。

#### B. 迭代型 (Iterative)
*   **特点**：允许对未完成工作反馈，改进和修改。
*   **RUP (统一软件开发过程)**：
    *   阶段：初始（范围/构架）、细化（构架/计划）、构造（开发/测试）、交付（集成/发布）。
    *   每次迭代产生一个可发布的子集。
    *   适用：需求早期可能变化、高风险、用户参与度高、面向对象/UML、高素质团队。
*   **螺旋模型 (Spiral)**：
    *   四个象限：制定计划、风险分析、实施工程、客户评估。
    *   特点：风险驱动，多次迭代瀑布，灵活设计，用户早期看到产品。
    *   适用：风险主要制约因素、需求不明确、重大变更可能、大规模、新技术。
*   **原型模型**：适用于需求不明确、复杂性高、变更频繁的项目。

#### C. 增量型 (Incremental)
*   **特点**：向客户提供各个已完成的、可立即使用的可交付成果。
*   **优点**：循序渐进降低风险，关键功能早出现，早期预警缺陷，降低估计失误。
*   **适用**：大部分需求明确但可能变化、市场把握不准需逐步了解、庞大复杂系统分步实施。

#### D. 敏捷型 (Agile)
*   **特点**：既有迭代也有增量，频繁交付，完善工作。
*   **核心价值观**：《敏捷宣言》。
*   **Scrum模型**：
    *   角色：PO, Scrum Master, Team。
    *   事件：迭代计划会、每日站会（昨天/今天/障碍）、迭代评审会、迭代回顾会。
*   **XP (极限编程)**：针对业务需求和开发实践的规则。
*   **Lean (精益)**：持续改进，减少浪费。
*   **Kanban (看板)**：可视化工作流。
*   **规模化敏捷**：Scrum of Scrums, SAFe, LeSS, Spotify模型。

#### E. DevOps
*   **定义**：Development + Operations。
*   **核心**：全程敏捷思维，开发与运维紧密合作，消除目标差异，促进沟通协作整合。
*   **实践**：持续集成 (CI)、持续交付 (CD)、持续部署。

#### F. 混合模型 (Hybrid)
*   结合上述多种模型的特点。

### 5. 生存期模型选择步骤
1.  熟悉各种模型。
2.  评审、分析项目特性（需求清晰度、风险、规模、技术等）。
3.  选择适合的模型。
4.  标识不一致处并进行裁剪。

## 五、 案例分析与作业

*   **案例**：医疗信息商务平台采用敏捷/迭代模型，分为四个迭代。
*   **作业**：
    1.  阅读《合同法》和《招投标法》。
    2.  查阅可行性研究和项目评估方法，查看合同范本。
    3.  针对业务系统招标书撰写投标书。
    4.  学习撰写软件开发合同。
    5.  学习了解敏捷和DevOps。
