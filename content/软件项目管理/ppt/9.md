# 软件项目管理：第9章 风险计划 (Risk Planning)
## 二、 软件项目风险基本概念

### 1. 风险的定义
*   **通用定义**：损失发生的不确定性；对潜在的、未来可能发生损害的一种度量。
*   **软件风险**：指软件开发过程中以及软件产品本身可能造成的伤害和损失（如质量下降、成本超支、进度推迟）。
*   **项目风险三要素**：
    1.  风险是一个事件。
    2.  风险具有事件发生的概率 ($P$)。
    3.  风险事件造成的影响/后果 ($I$)。
*   **PMI定义**：一种不确定事件或状况，一旦发生，会对至少一个项目目标产生积极（机会）或消极（威胁）影响。

### 2. 风险的属性
*   **随机性**：发生与否及时间具有不确定性。
*   **相对性**：不同主体对风险的承受能力不同（取决于收益大小、投入大小、地位及资源）。
*   **可变性**：性质、后果可能变化，也可能产生新风险。
*   **客观性、不利性、同利益的对称性**。

### 3. 风险的分类
| 分类维度 | 类型 | 说明 |
| :--- | :--- | :--- |
| **按后果** | **纯粹风险** | 不能带来机会，无获利可能，只有损失。 |
| | **投机风险** | 可能带来机会/利益，又隐含威胁/损失（如采用新技术）。 |
| **按来源** | **自然风险** | 自然灾害等。 |
| | **人为风险** | 管理、技术、市场等人因导致。 |
| **按可管理性** | **可管理风险** | 可以识别并采取措施控制。 |
| | **不可管理风险** | 难以预测或控制。 |
| **按影响范围** | **局部风险** | 影响项目部分模块或阶段。 |
| | **总体风险** | 影响整个项目目标。 |
| **按可预测性** | **已知风险** (Known knowns) | 如用户需求变更。 |
| | **可预测风险** (Known unknowns) | 如项目评审推迟。 |
| | **不可预测风险** (Unknown unknowns) | 完全未知的突发状况。 |
| **按范围角度** | **项目风险** | 预算、进度、人员、资源、组织问题。 |
| | **技术风险** | 设计、实现、接口、检验、维护、规格多义性、技术陈旧。 |
| | **商业风险** | 市场风险（产品非市场所需）、策略风险、管理风险（失去支持）、预算风险。 |

### 4. 风险成本
*   **构成**：风险事件造成的损失/减少的收益 + 预防风险支付的费用。
*   **有形成本**：直接损失、间接损失。
*   **无形成本**：机会减少、生产率降低、资源分配不当。
*   **预防与控制费用**：保险、购买服务、设备维护等。
*   **负担**：个体负担 vs 社会负担。
*   **原则**：只有当**风险管理成本 << 风险真正发生招致的损失**时，风险管理才有意义。

### 5. 关键术语区分
*   **风险因素 (Risk Factors)**：引起或增加风险事件发生机会、影响损失严重程度的内在或间接原因（如：需求变化、设计错误、人员能力欠缺）。
*   **风险事件 (Risk Events)**：人们不愿意发生或未规划的事件，可能导致无法实现目标（如：项目成本失控）。
*   **关系**：风险因素 $\rightarrow$ 引发 $\rightarrow$ 风险事件 $\rightarrow$ 导致 $\rightarrow$ 损失/实际与预期差异。

### 6. 风险管理的层次
1.  **危机管理 (救火模式)**：风险已造成麻烦后才处理（最差）。
2.  **风险缓解**：事先制定补救措施，但无防范措施。
3.  **着力预防**：将风险识别和预防作为项目一部分规划执行。
4.  **消灭根源**：识别并消灭产生风险的根源（最佳）。

## 三、 软件项目风险管理的过程

风险管理是一个连续的过程，包含四个主要步骤：**标识 $\rightarrow$ 评估 $\rightarrow$ 规划 $\rightarrow$ 控制**。

### 1. 风险识别 (Risk Identification)
*   **目标**：系统化地确定对项目计划的威胁，识别已知和可预测的风险。
*   **主要内容**：
    1.  识别潜在风险。
    2.  识别引起风险的主要因素。
    3.  识别风险可能引起的后果。
*   **输入**：WBS（工作分解结构）、SOW（工作说明书）、历史数据、其他项目经验、进度计划、评审报告、公司目标等。
*   **方法及工具**：
    *   **德尔菲方法 (Delphi)**：专家匿名背对背打分/意见。
    *   **头脑风暴法 (Brainstorming)**。
    *   **情景分析法**。
    *   **面谈法**。
    *   **风险条目检查表 (Checklist)**：核对单
        *   基于SEI（软件工程研究所）的检查表领域：产品工程（需求、设计、代码等）、开发环境、程序约束（资源、进度、合同等）。
        *   基于生存期的检查表：初始、设计、实施、收尾阶段。
    *   **SWOT分析法**：
        *   S (Strengths) 优势, W (Weaknesses) 劣势, O (Opportunities) 机会, T (Threats) 威胁。
        *   组合策略：SO, WO, ST, WT。
*   **常见软件风险**：缺乏培训人力、需求变更/不明确、人员流失、外部强加决策、绩效不达标、进度不切实际、新技术应用、业务知识不足、效率低下。

### 2. 风险评估 (Risk Assessment)
*   **目标**：确定风险发生的概率 ($P$)、后果严重程度 ($I$)、影响范围及发生时间。
*   **核心公式**：风险值 $R = F(P, I)$。
*   **定性评估**：
    *   **概率度量**：高/中/低；极高/中/低/极低；0-1之间数值。
    *   **后果度量**：灾难/严重/轻微/可忽略；高/中/低。
    *   **工具**：**概率-影响矩阵图 (Risk Matrix)**。根据P和I的组合确定风险等级（L/M/H）。
    *   **输出**：确定优先次序，找出 **Top 10 风险**。
*   **定量评估**：
    *   **访谈**：采集专家意见，确定概率分布模型。
    *   **盈亏平衡分析**：平衡点越低，盈利机会越大，亏损风险越小。
    *   **决策树分析与 EMV (Expected Monetary Value)**：
        *   $EMV = \sum (Outcome_i \times Probability_i)$
        *   用于比较不同行动方案的期望损益，选择最佳方案。
    *   **灵敏度 (敏感性) 分析**：考察单个要素不确定性对目标的影响程度，常用**龙卷风图**展示。
    *   **建模与仿真**：常用**蒙特卡罗技术 (Monte Carlo)**，模拟费用或进度的潜在影响。
    *   **量化检查表**：核查表
        *   **Standish Group 成功潜力评分表**：用户参与(19分)、高层支持(16分)、需求清晰(15分)等，总分100。
        *   **McFarlan 风险问卷**：基于时间、人天、部门数、硬件要求等打分评估风险等级。

### 3. 风险规划 (Risk Planning / Response)
*   **目标**：制定应对策略和措施，提高成功机会，降低负面影响。
*   **原则**：措施需适合风险重要性，经济有效，及时可行。遵循 **80/20 规律**（80%的风险由20%的关键风险引起）。
*   **四大主要策略**：
    1.  **回避风险 (Avoid)**：
        *   主动放弃或拒绝使用导致风险的方案（如：放弃采用不成熟的新技术）。
        *   注意：可能产生新风险，并非所有情况适用。
    2.  **转移风险 (Transfer)**：
        *   将损失或财务后果转嫁出去。
        *   手段：出售、分包、开脱责任合同、购买保险。
    3.  **损失控制 (Control/Mitigate)**：
        *   **损失预防**：降低发生概率。
        *   **损失抑制**：降低发生后的损失程度。
        *   *案例*：针对人员流动风险 $\rightarrow$ 建立文档标准、交叉培训、加强沟通、细致评审。
    4.  **自留风险 (Accept/Retain)**：
        *   自己承担损失。
        *   类型：主动自留（有计划） vs 被动自留（无计划）；全部自留 vs 部分自留。
        *   通常用于低风险或处理成本高于损失的情况。
*   **相关术语**：应急预案、备用计划、应急补助、储备技术。

### 4. 风险控制 (Risk Control)
*   **内容**：项目跟踪控制。
*   **活动**：监控风险触发器、执行应对计划、评估措施有效性、识别新风险。
*   *(注：课件中此部分标题下内容为问号，但通常指持续监控和纠偏)*

## 四、 软件项目风险管理计划

### 1. 计划内容
风险管理计划是项目管理计划的一部分，应包含：
*   **风险应对计划**：特别是 **Top 10 风险清单**。
*   **岗位职责**：指定风险管理负责人。
*   **时间与预算**：风险管理活动所需资源和时间。
*   **追踪机制**：如何监控和汇报风险。
*   **沟通渠道**：建立匿名风险汇报渠道等。

### 2. 风险管理计划表结构示例
| 任务 | 可能的风险 | 产生阶段 | 产生原因 | 避免的措施 (预防) | 发生后的处理 (应急) |
| :--- | :--- | :--- | :--- | :--- | :--- |
| 系统设计 | 技术风险 | 设计阶段 | 新技术(TeMIP)经验欠缺 | 请专家培训 | 更换其他技术实现 |
| 开发环境确认 | 资源风险 | 设计阶段 | 设备未到位 | 提前采购/租赁 | 调整进度/借用资源 |

### 3. Top 10 风险清单 (核心工具)
*   列出概率最高、影响最大的前10个风险。
*   定期更新和审查。

### 4. 传统 vs 敏捷项目的风险管理
*   **传统项目**：强调前期详细计划，风险识别较全面。
*   **敏捷项目风险特点**：
    *   没有长期详细计划，识别远期风险困难。
    *   存在频繁变更。
*   **敏捷风险应对方法**：
    *   **损失预防与抑制**为主。
    *   **跨职能团队**：多角度识别风险。
    *   **迭代内容选择**：优先处理高风险或高价值项，或先做风险小的以建立信心。
    *   **频繁评审增量产品**：及早发现问题。
    *   **持续测试**：快速反馈。
    *   **客户参与**：减少需求误解和变更风险。
