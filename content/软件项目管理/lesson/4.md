## 2. 需求工程回顾与核心概念

### 2.1 范围与需求的关系
*   **范围 (Scope)**：指项目中需要做的事情。
*   **来源**：主要来自于**需求**，但也包含管理过程支撑性文档（如需求评审、设计方案评审、项目计划评审等）。
*   **结论**：项目范围 > 需求范围。

### 2.2 需求获取的注意事项
1.  **识别真正的客户 (Stakeholders)**
    *   **现象**：对接人不一定是最终用户或真正的决策者。
    *   **案例教训**：曾有甲方派出的“专家”提出苛刻且不切实际的需求。经核实，该专家并非实际使用者，仅是为了展示存在感。
    *   **策略**：把握动力源，识别谁真正使用系统。对于非核心决策者的意见，记录但不盲目执行，避免被带偏。
2.  **正确理解客户需求 (业务理解)**
    *   **核心**：必须懂业务。不懂业务无法深入理解需求。
    *   **案例**：教务系统课表打印，老师最关注的是“时间、地点、周次”，而非课程名称或编号。若设计者不懂业务，可能将关键信息用浅色打印，导致 unusable。
    *   **建议**：积累行业经验，尽量不要频繁更换行业，业务理解是核心竞争力。
3.  **需求引导与控制**
    *   **原则**：开发人员需引导客户，而非被客户带走。
    *   **策略**：面对用户临时提出的变更，不要当场拒绝也不要当场答应。应记录并表示“回去商量”，将决策权收回内部评估。
    *   **忍耐力和思维**：面对难缠的甲方（如某些体制内项目），需具备较强的心理承受力和清晰的逻辑思维，坚持工程化流程。

## 3. 需求分析与建模

### 3.1 分析过程
*   **步骤**：
    1.  抽象当前物理系统 -> 建立**概念模型**。
    2.  建立目标系统模型 -> 生成**物理模型**（如数据库表结构）。
*   **时机**：不仅在项目初期，在**需求变更**时也需要重新进行需求分析。
*   **方法**：头脑风暴、专家评审、焦点会议、原型系统等。
*   **价值**：需求明确后，数据库表结构确定，编程即为“水到渠成”（主要是 CRUD 操作）。

### 3.2 建模方法
1.  **原型法 (Prototyping)**
    *   **适用**：BS 架构系统，用户关注页面展现和业务流程。
    *   **工具**：墨刀 (Modao) 等快速原型工具。
    *   **优点**：直观，用户能快速反馈“是不是想要的”。
    *   **重点**：**客户评审**。需经过多轮迭代（用户看原型 -> 提意见 -> 修改 -> 再评审），直至需求明确。
    *   **缺点**：若用户反复无常，会导致开发团队疲惫（案例：某项目因甲方反复推翻页面设计，耗时 3 个月）。
2.  **结构化方法 (Structured)**
    *   **工具**：数据流图 (DFD)、数据字典、ER 图。
    *   **特点**：对编程人员友好，逻辑清晰；对用户不够直观。
3.  **面向对象方法 (UML)**
    *   **工具**：用例图 (Use Case)、状态图、活动图、类图、顺序图。
    *   **用例图**：描述系统向用户提供的有价值的功能（做什么，而非怎么做）。
    *   **现状问题**：学生常将“怎么做”（技术实现）混入需求分析中，导致设计与需求脱节。

### 3.3 需求规格说明书 (SRS)
*   **定义**：Software Requirements Specification，经过评审和签字的正式文档。
*   **内容**：
    *   **功能需求**：软件具备哪些功能模块。
    *   **非功能需求**：性能、安全性、运行环境（硬件、软件、网络、接口等）。
    *   **原则**：描述 **“做什么 (What)"**，而不是 **“怎么做 (How)"**。
*   **重要性**：
    *   是开发和验收的依据（优于合同）。
    *   **签字确认**：签字意味着甲方认可，后续责任划分有据可依。**切忌随意签字**，尤其是空白文件。
*   **不完美性**：允许存在一定不完美，但需双方共识。

## 4. 需求变更管理 (核心重点)

### 4.1 变更的代价
*   **非线性增长**：变更发生得越晚，代价越高。
    *   **需求阶段**：仅改文档。
    *   **编码/测试阶段**：需改文档、设计、代码、测试用例，且可能引发连锁反应（牵一发动全身）。
*   **80/20 规则**：越往后改越麻烦，越往前改越简单。
*   **建议**：发现问题尽早解决，不要当“鸵鸟”。（类比：挂科后尽早补考，拖到大四成本最高）。

### 4.2 变更管理流程
*   **核心原则**：按工程化流程办事，走层级核查。
*   **基线 (Baseline)**：
    *   **定义**：各方干系人达成一致并正式发布的东西。
    *   **规则**：**基线前的内容可随意修改；基线后的内容修改必须走流程。**
    *   **类比**：成绩发布前找老师改分容易；成绩发布后（成为基线）需走教务处核查流程。
*   **配置变更控制委员会 (CCB)**：
    *   负责评估变更申请（改或不改）。
    *   流程：提交变更单 -> CCB 评估 -> 批准 -> 修改版本 -> 测试 -> 部署 -> 新版本发布。
    *   **类比**：两会提案 -> 审核 -> 职能部门修改 -> 发布新法律。

### 4.3 VUCA 时代与敏捷
*   **VUCA 特征**：碎片化 (Volatility)、有限时 (Uncertainty)、易变化 (Complexity)、强时效 (Ambiguity)。
*   **挑战**：频繁变更导致软件质量下降，文档与代码脱节，维护成本极高（如 Windows XP 停止支持）。
*   **对策**：使用敏捷方法，通过迭代管理变更。

## 5. 敏捷需求管理

### 5.1 用户故事 (User Story)
*   **定义**：敏捷中需求的表述方式，简短、简单的描述。
*   **模板**：`作为一个 <用户角色>，我想 <完成活动>，以便 <实现价值>`。
*   **INVEST 原则** (好的用户故事特征)：
    *   **I**ndependent (独立的)
    *   **N**egotiable (可协商的)
    *   **V**aluable (有价值的)
    *   **E**stimable (可估算的)
    *   **S**mall (小的)
    *   **T**estable (可测试的)
*   **目的**：通过沟通和协作达成一致理解，而非仅仅为了写文档。

### 5.2 需求优先级 (MoSCoW 法则)
| 优先级 | 含义 | 形象比喻 |
| :--- | :--- | :--- |
| **M**ust | 必须完成 | 救命 (没它系统无法运行) |
| **S**hould | 应该完成 | 丢钱 (很重要，但可短暂妥协) |
| **C**ould | 可以完成 | 丢脸 (有了更好，没有也行) |
| **W**on't | 本次不做 | 没事 (以后再说) |

### 5.3 敏捷工具与实践
*   **Product Backlog**：产品待办事项列表，需求来源。
*   **Sprint Backlog**：冲刺待办列表，迭代计划。
*   **故事卡/故事墙**：直观展示任务状态（待办、进行中、完成），便于每日检查。
*   **非功能需求的敏捷处理**：同样转化为用户故事（如兼容性需求：“作为用户，我希望在 Windows XP 上运行..."）。
*   **工具推荐**：
    *   原型：墨刀。
    *   数据库设计：PowerDesigner, ERWin (可生成物理模型和报告)。
    *   性能测试：LoadRunner (自动化并发测试，避免手工笨拙测试)。

## 6. 任务分解 (WBS)

*   **定义**：Work Breakdown Structure，工作分解结构。
*   **目的**：将项目分解为更小、更容易管理和操作的工作项。
*   **结构**：树状结构。
    *   上层：项目/子项目。
    *   最下层：**工作包 (Work Package)** / 业务节点。
*   **特征**：面向可交付成果，可量化。
*   **意义**：任务分解是项目管理的基础，分不出来则无法管理。
*   **类比**：建筑项目的预制件组装（高铁修桥、外墙砖预制）。

## 7. 答疑环节：投标作业注意事项

针对课程相关的投标大作业，老师给出以下建议：

1.  **公司名称**：若招标文件未强制要求，可自编（如“邮政大学”、“腾讯”等）。
2.  **报价文件**：
    *   需包含总报价。
    *   若招标要求分项报价，需按规定提供。
    *   **开标环节**：价格文件需在多家在场的情况下公开宣布。
3.  **文档结构**：
    *   严格按照招标文件的附件顺序编排。
    *   若招标文件要求提供纸质版，需密封（开标函单独信封）。
    *   内容缺失处理：若明确要求提供的信息缺失会扣分；若为附加项目，没有可不提供。
4.  **时间要求**：严格按照截止时间提交（案例中提到因时间写错导致的问题）。

## 8. 课程总结与建议

1.  **需求是基础**：需求不明确，进度、成本、质量、风险均无从谈起。
2.  **变更要管控**：不要害怕变更，但必须走流程（基线管理、CCB 评估）。
3.  **工具提效率**：善用专业工具（原型、数据库设计、测试工具），避免手工低效劳动。
4.  **业务是核心**：编程技术易学，业务理解难积累。工作中尽量保持行业连续性。
5.  **行动建议**：遇到问题尽早解决，不要拖延（避免“鸵鸟心态”）。
