## 1. 质量管理 (Quality Management) - 收尾回顾

### 1.1 质量管理的两大支柱
| 类型 | 关注点 | 方法/手段 | 类比理解 |
| :--- | :--- | :--- | :--- |
| **质量保证 (QA)** | **过程规范**：确保按照标准和规范做事 | **审计 (Audit)**：检查是否合规 | 类似财务审计：检查花钱是否符合预算科目和标准（如劳务费、燃料费等是否按规使用）。 |
| **质量控制 (QC)** | **结果好坏**：可交付成果是否满足指标 | **测试 (Testing)**：验证功能与性能 | 教务系统要求5万人并发，通过压力测试验证是否达标。 |

### 1.2 质量计划与方法论
*   **量化原则**：将模糊的质量特性拆解为可量化的指标（功能性、可靠性、易用性、用户反馈等），赋予权重，计算质量分数。体现“渐进明细”原则。
*   **常用工具**：
    1.  **实验设计 (DOE)**：统计学方法，确定哪些因素影响变量（常用于论文论证方法有效性）。
    2.  **基准对照 (Benchmarking)**：借鉴其他项目（同行或跨行业）的优秀实践。*例：从建筑工地借鉴质量管理方法到软件开发。*
    3.  **质量成本分析**：预防成本 vs 评估成本 vs 缺陷成本。**原则：预防胜于检查**。
    4.  **流程图**：全局视角分析逻辑关系，预测潜在问题点。*例：旅游行程规划中识别宾馆入住时间与到达时间的冲突。*
    5.  **因果图 (鱼骨图/石川图)**：
        *   **5M1E分析法**：人 (Man)、机 (Machine)、料 (Material)、法 (Method)、环 (Environment)。
        *   **应用**：逆向追溯问题根源，直到找到**可量化解决**的因素。*例：挂科 -> 玩游戏 -> 每天玩8小时 -> 减少到4小时（可量化解决方案）。*
        *   **心态**：拒绝“鸵鸟心态”，直面问题并解决。

### 1.3 质量体系 vs 质量计划
*   **质量体系 (如 ISO9000)**：企业长期遵循的标准，具有重复性和标准性。
*   **质量计划**：针对**特定项目**的一次性计划，项目结束后失效。

### 1.4 不同领域的质量策略差异
*   **微软/通用软件**：追求六西格玛，注重用户体验，底层封闭。
*   **摩托罗拉 (案例)**：曾以高质量著称，但因市场和技术迭代慢而倒闭。**启示**：仅靠质量无法生存，需结合技术与商业模式。
*   **航天/军工/银行软件**：**零缺陷**，高可靠性，功能可能简单但绝不能出错。
*   **一般业务系统 (如教务、财务)**：满足合同功能需求为主，对极端稳定性的要求略低于航天，但强调功能丰富度。

### 1.5 测试的独立性
*   测试部门应独立于开发部门，以保证客观性。
*   通常比例：3-4名开发配1名测试。
*   测试人员职业发展：压力相对开发较小，薪资差距不大，适合部分同学（特别是女生）考虑。

## 2. 配置管理 (Configuration Management, CM) - 核心内容

### 2.1 为什么需要配置管理？
软件项目面临的主要问题是**持续不断的变更**（需求不明确、用户变更、技术调整）。
**常见痛点（无CM的后果）**：
1.  **版本丢失**：找不到历史版本。
2.  **错误版本修改**：基于旧版本修改，导致覆盖他人工作或逻辑错误。
3.  **未经授权修改**：生产环境随意改动，造成严重事故（尤其是金融/银行系统）。
4.  **工作交接混乱**：人员流动导致代码/文档断层。
5.  **Bug回归**：已修复的Bug在新版本中重现（因代码被错误覆盖）。
6.  **环境依赖缺失**：仅有源代码，缺乏编译环境/库文件，导致无法重新构建（*案例：VC6.0老项目只能在特定旧电脑运行*）。

**CM的核心价值**：
*   **管变更**：有效控制变化。
*   **可追溯**：任意时间点的内容可回溯（代码、设计、需求的一致性）。
*   **完整性**：防止工作成果被覆盖或丢失。

### 2.2 配置管理的基本概念

#### A. 配置项 (Configuration Item, CI)
*   **定义**：软件生命周期中产生的所有受控对象。
*   **范围**：不仅是源代码！包括：
    *   **文档**：需求规格说明书、设计报告、测试计划、用户手册、项目计划。
    *   **数据**：数据库脚本、测试数据。
    *   **代码**：源程序、可执行文件。
    *   **环境**：编译器版本、依赖库、容器镜像（Docker）。
*   **标识**：每个CI必须有**唯一ID**，明确负责人，确定进入CM的时间点。

#### B. 基线 (Baseline)
*   **定义**：经过正式评审和批准的配置项集合，标志着某个开发阶段的结束。
*   **特点**：
    *   基线内的内容**不能随意修改**。
    *   修改基线必须走正式的**变更控制流程**。
*   **常见基线**：
    *   **功能基线**：需求规格说明书（SRS）。
    *   **分配基线**：概要设计/详细设计。
    *   **产品基线**：最终发布的软件及文档。
*   **类比**：大学课程表是基线，想改上课时间需通过校代会（正式流程）；平时调课需走教务审批。

#### C. 版本 (Version) & 工作空间 (Workspace)
*   **版本**：不仅指代码版本，而是**代码+文档+数据+环境**的整体集合。
*   **工作空间**：
    *   **私有空间 (Private Workspace)**：开发人员本地环境，可随意修改，“上帝模式”。
    *   **集成空间 (Integration Workspace)**：团队共享，保持稳定。开发完成后提交至此，其他人拉取最新稳定版进行开发，避免环境不一致导致的冲突。

#### D. 配置库 (Repository) 的三库模型
| 库类型 | 别名 | 状态 | 权限/操作规则 |
| :--- | :--- | :--- | :--- |
| **开发库** | 动态库 / 程序员库 | 未形成基线 | **随便改**。开发人员日常工作的区域。 |
| **受控库** | 静态库 / 主库 | 已形成基线 | **走流程改**。阶段成果经评审后存入。修改需Check-out，经CCB批准后Check-in。测试在此库进行。 |
| **产品库** | 备份库 / 发布库 | 最终产品 | **坚决不能改**。正式交付给客户的产品。若需修改，需复制回受控库，走变更流程，测试通过后生成新版本再入库。 |

### 2.3 配置管理角色与组织
*   **CMO (配置管理员)**：负责建立和维护CM系统，管理库权限，执行基线发布。
*   **CCB (Change Control Board, 变更控制委员会)**：
    *   **性质**：**决策机构**，非作业机构。
    *   **组成**：项目经理、客户代表、技术专家等。
    *   **职责**：评估变更影响，批准或驳回变更请求 (CR)。**不负责具体实施**。
    *   **类比**：教务处（CCB）决定是否允许查分/改分，任课老师（作业人员）具体执行核查。

## 3. 变更控制流程 (Change Control Process)

### 3.1 变更的类型
1.  **缺陷修复 (Defect)**：软件有Bug，必须改。
2.  **增强请求 (Enhancement)**：功能可用但不完善，用户希望优化（*最麻烦，易导致范围蔓延*）。
3.  **新需求 (New Feature)**：原有需求未涵盖的新业务场景。

### 3.2 变更处理流程
1.  **提出变更请求 (CR)**：填写变更申请单，描述变更内容、原因。
2.  **变更评估 (Impact Analysis)**：
    *   **综合影响分析**：不仅看技术实现，还要评估对**进度、成本、质量、其他模块接口**的影响。
    *   *案例*：增加一门必修课，需考虑学时增加、教室资源、教师成本、与其他课程冲突等。
3.  **CCB 决策**：
    *   **批准**：指派责任人，纳入计划。
    *   **驳回**：给出合理理由（如进度风险大、性价比低、建议二期实现）。*技巧：适当“拖延”或“强硬”拒绝不合理变更，以保护项目边界。*
    *   **搁置**：暂时不处理，留待后续版本。
4.  **实施变更**：
    *   从**受控库** Check-out 相关配置项。
    *   在本地工作空间修改。
    *   单元测试。
5.  **验证与关闭**：
    *   提交至受控库/集成空间。
    *   测试人员验证（回归测试）。
    *   CCB或相关人员签字确认，关闭CR。
    *   更新基线和文档。

### 3.3 变更集 (Change Set)
*   **变更请求 (CR)**：逻辑上的修改申请（如“字体太小”）。
*   **变更集 (CS)**：物理上被修改的文件集合（如200个页面文件）。
*   CM工具需建立 CR 与 CS 的关联，实现双向追溯。

## 4. 课程大作业说明 (重要！)

### 4.1 作业目标
模拟完整的项目管理过程，每个人都要扮演 **项目经理 (PM)** 的角色。

### 4.2 提交内容
1.  **集成项目计划 (Integrated Project Plan)**：
    *   包含：范围、进度、成本、质量、风险、人力、沟通、采购、配置管理等子计划。
    *   **核心产出**：**甘特图 (Gantt Chart)**。
        *   若图片过大，可单独提交 `.mpp` (Project文件) 或高清截图。
        *   需体现迭代计划（如每2周一个迭代）。
    *   **注意**：只需提交**最终版**的计划文档。
2.  **配置管理历史记录证明**：
    *   使用 Git/SVN 等工具进行管理。
    *   提交**Commit History 截图**，证明计划是经过多次迭代修改的。
    *   **要求**：备注 (Commit Message) 必须清晰，说明每次修改了什么（如“V1.0初稿”、“V1.1根据反馈调整进度”）。
3.  **智能体编程演示 (Agent Programming)**：
    *   **任务**：利用 AI Agent 辅助开发一个简单的软件系统（需有后端数据库，真实可运行）。
    *   **形式**：**不需提交代码包**，但在每周周五的实验课上需**现场演示**给老师看。
    *   **目的**：掌握前沿技术，了解AI如何改变开发效率（10人半年的活 -> 1人3个月）。
    *   *提示*：可参考B站相关教程，学习如何编写Prompt引导Agent生成全栈代码。

### 4.3 关键要求
*   **规范性**：文档格式严格规范（字体、字号、图表编号、目录）。*反面教材：答辩PPT写成“论文答辩”，错别字连篇。*
*   **命名规范**：文件名需包含 `姓名_学号_内容_版本`，方便助教批改。
*   **模拟真实性**：虽然组内不真正分工干活，但计划中必须体现给虚拟团队成员（10人）的任务分配。

## 5. 讲师寄语与建议
1.  **平衡原则**：项目管理追求质量、成本、进度的平衡，而非单一维度的完美。“够用就行” (Good Enough)，过犹不及。
2.  **大学生活建议**：趁年轻多出去走走（读万卷书，行万里路）。工作后的旅行心境完全不同。利用短假去西藏等难以抵达的地方。
3.  **技术趋势**：AI Agent 编程是未来趋势，务必尽早入门，不要只停留在应用层，要理解其原理和工作流。
4.  **职场心态**：
    *   不要当“鸵鸟”，遇到问题正视并解决。
    *   在银行/金融等严谨行业，**授权**和**流程**高于一切，严禁私自修改生产环境。
    *   学会合理的“扯皮”（谈判），对于不合理的变更要有理有据地拒绝或延期，保护项目团队。
