## 一、测试点的确定  

### 1.1 依据标准  
- **ISO 质量体系**  
  - 在**概要设计或详细设计文档**中，**必须明确指出**每个单元/模块的：  
    - 测试要点  
    - 测试指标  
    - 测试方法  

- **CMM 质量体系**  
  - 在**系统用例模型**中应明确：  
    - 每个用例的**优先级**  
    - 每个用例的**工作流程**  
    - **每个用例模型即为一个测试点**  
    - 每个测试需求**至少应有两个测试用例**（正面 + 负面）

### 1.2 责任与时间安排  
- ✅ **测试点**由**分析设计员**确立，**与测试人员无关**；  
- ✅ **测试用例**由**测试设计员**或**分析设计员**制定，**非普通测试员**；  
- ✅ **测试工作起始时间**：**项目立项后即开始**，**非代码完成之后**；  
- ✅ **测试对象范围**（全生命周期）：  
  - 需求分析文档  
  - 需求规格说明书  
  - 概要设计 & 说明书  
  - 详细设计 & 说明书  
  - 源代码  
  - 用户手册 / 使用手册等

## 二、测试用例设计概述  

### 2.1 定义  
> **测试用例**：为特定目的（如验证路径、覆盖需求）而设计的**测试数据 + 相关规程**的集合；是**有效发现缺陷的最小执行单元**。

### 2.2 重要性  
- 是测试执行的**核心依据** ⇒ 不同测试员执行同一用例应得一致结果  
- 具有**复用性** ⇒ 节约时间，提升效率  
- 是组织**知识资产** ⇒ 积累经验、考核新人、跟踪进度  
- **质量量化指标**：  
  - 测试用例**通过率** ⇄ 代码质量  
  - 缺陷（bug）**数量** ⇄ 测试充分性  
- 撰写要求高：需深入理解**用户场景、功能规格、系统/模块设计**

### 2.3 测试用例数与软件规模关系  

| 层级 | 关系 | 说明 |
|------|------|------|
| **单元测试** | ≈ 线性正比 | 例：代码行数 10k → 用例数 ~ 几千 |
| **集成/系统测试** | 非线性增长 | 模块组合爆炸 ⇒ 用例数随规模**超线性增长** |
| **质量要求** | 正相关 | 航天软件 > 民用软件（用例密度更高） |

> 📉 图表说明（PDF p.3）：横轴软件规模，纵轴测试用例数；集成/系统测试曲线斜率 > 单元测试。

### 2.4 测试用例设计步骤（5步）

#### 2.4.1 为测试需求确定测试用例  
- 来源：  
  - 用例说明书  
  - 补充规约  
  - 设计说明书  
- 编号格式：`TR_XXXX_XX`  
- **每需求 ≥ 2 用例**：1 正面 + 1 负面

#### 2.4.2 为测试用例确定输入与输出  
- **输入**：用户输入/操作/生成状态  
- **输出**：预期结果  
- **设计原则**（关键！）：  
  1. **必须用边界值分析法**（发现错误能力最强）  
  2. **必要时补等价类划分**  
  3. **用错误推测法追加**  
  4. **对照逻辑**：检查逻辑覆盖是否达标，不足则补  
  5. **多条件组合** → 首选**因果图法**

#### 2.4.3 编写测试用例  
- 编号：`TC_<TR_id>`，如 `TC_TR_001_01`  
- 必含字段：  
  - 测试目标状态 & 数据状态（前置条件）  
  - 输入（操作）  
  - 预期结果（输出）

#### 2.4.4 评审测试用例  
- **Checklist**：  
  - 每需求/设计元素是否有对应用例？  
  - 每用例是否明确预期结果？  
  - 是否定义初始状态？  
  - 是否覆盖所有**单一边界**？  
  - 是否覆盖所有**业务数据流**？  
  - 命名是否符合规范？  
- **评审人**：项目经理、系统分析员、测试设计员、测试员

#### 2.4.5 跟踪测试用例  
- **需求追踪**：需求 → TR → TC  
- **执行指标**：  
  - 执行率 = 已执行用例 / 总用例  
  - 通过率 = 通过用例 / 已执行用例  
- **闭环管理**：TC → 执行 → 缺陷 → 修复 → 回归

---

### 2.5 测试用例书写规范（6大要素）

| 字段 | 说明 |
|------|------|
| **标识符** | 全局唯一，引用基础 |
| **测试项** | 被测对象 + 特征；如 UI 测试：窗口缩放、菜单、布局 |
| **测试环境要求** | OS/DB/硬件/网络/浏览器等 |
| **输入数据** | 执行所需的具体输入 |
| **对应输出数据** | 期望结果（状态或数据） |
| **测试用例间关联** | 依赖/前置关系（如 TC02 前提是 TC01 通过） |

## 三、测试用例设计技术  

### 3.1 黑盒测试用例设计技术  

#### 3.1.1 概念  
- 又称**功能测试**/**行为测试**  
- **不关心内部结构**，仅通过接口验证功能是否符合规格  
- 依据：需求规格说明书 / 功能描述  
- “黑盒”：外部可见输入/输出，内部不可见  

#### 3.1.2 作用（查哪类错误）  
- ❌ 功能遗漏 / 错误  
- ❌ 输入/输出处理错误  
- ❌ 数据结构 / 外部文件访问错误  
- ❌ 性能不达标  
- ❌ 初始化 / 终止错误  

#### 3.1.3 局限  
- ❗ **穷举不可能**：32位整数 ×2 → $2^{64}$ 组数据，需 5 亿年（1ms/组）  
- ❗ **依赖规格正确性**：规格本身遗漏/多余功能 ⇒ 黑盒无法发现  
- ❗ 外购/参数化软件 ⇒ **只能黑盒**  
- ❗ 第三方测试 ⇒ 多采用黑盒  

#### 3.1.4 静态黑盒测试  
- 测试对象：**产品说明书**（非可执行）  
- 方法：  
  - **高级审查**：找根本性疏漏（非抠字眼）  
  - **设身处地**：从客户视角出发  
  - **对照标准**：  
    - 公司约定  
    - 行业/国家标准  
    - GUI / 硬件/网络规范  
  - **竞品分析**：规模、复杂性、可测试性、质量  
- ✍️ **说明书属性检查清单（8项）**：  
  > 完整、准确、精确、一致、贴切、合理、代码无关、可测试  
- ✍️ **说明书用语检查清单**（警惕！）：  
  > “总是/所有/从不”、“显然/当然”、“某些/常常/通常”、“良好/高效”、“已处理”、“如果…那么…（无否则）”

#### 3.1.5 动态黑盒测试  
- 程序运行中测试 ⇒ **行为测试**  
- 测试员不知内部实现  

#### 3.1.6 黑盒测试用例设计技术（5类）

##### ① 等价类划分（Equivalence Class Partitioning）

- **定义**：输入域划分为若干**等价子集**；子集中任一值测试 ⇒ 代表整类  
- **等价类特性**：完备性、无冗余性、等价性  
- **划分类型**：
  - **有效等价类**：符合规格
  - **无效等价类**：违反规格

- **划分原则**（5条）：

| 条件类型 | 有效类 | 无效类 |
|----------|--------|--------|
| 取值范围（如 1~999） | 1 类：`1 ≤ x ≤ 999` | 2 类：`x < 1`；`x > 999` |
| 必须满足某条件（如首字符为数字） | 1 类：是数字 | 1 类：非数字 |
| 布尔量 | 1 类：True | 1 类：False |
| 一组特定值（教授/副教授/讲师/助教） | n 类（按值个数） | 1 类：其他值 |
| 遵守某规则（Pascal 语句以 `;` 结尾） | 1 类：以 `;` 结尾 | 多类：以 `:` `,` `空格` `LF` 等结尾 |

- **用例选择策略**：  
  1️⃣ 编号每一等价类；  
  2️⃣ 新用例 → **覆盖尽可能多未覆盖的有效类**；  
  3️⃣ 新用例 → **仅覆盖 1 个未覆盖的无效类**（避免掩盖）

- ✅ **Pascal 标识符实例**（PDF p.10–11）：  
  - 条件：字母开头；字母/数字组合；≤80字符；编译器看前8字符；先声明再用；说明语句≥1标识符  
  - ⇒ **15 类**（7有效 + 8无效），**9 个用例**覆盖全部  
  - 例：  
    - TC1：`VAR x, T1234567: REAL; x:=3.414; ...`  
    - TC5：`VAR T12345...(>80字符): REAL;`  
    - TC9：`VAR PAR: REAL;`（未声明即用）

- ✅ **三角形判断练习**（课堂练习，PDF p.11–12）：  
  - 输入：3 整数 A/B/C  
  - 有效三角形：`A>0 ∧ B>0 ∧ C>0 ∧ A+B>C ∧ B+C>A ∧ A+C>B`  
  - 类型：不等边 / 等腰（A=B∨B=C∨A=C） / 等边（A=B=C）  
  - 等价类：  
    - 有效：不等边 / 等腰 / 等边 / 非三角形（不满足三角不等式）  
    - 无效：边≤0、非整数、超范围（如>200）  
  - ⇒ 设计 6+ 用例覆盖全部

##### ② 边界值分析（Boundary Value Analysis, BVA）

- **原理**：错误多发于**极值附近**（CECOM 研究证实）  
- **基础**：等价类 + **加测边界点**  
- **单故障假设** ⇒ 每次只变 1 个变量，其余取 nom（正常值）

- **5 点法**（对变量 x）：  
  `min`, `min+`, `nom`, `max−`, `max`  
  → n 变量 ⇒ **4n + 1** 个用例  

- **扩展方法**：  
  - **健壮性测试**（7点法）：加 `min−`, `max+` ⇒ 测异常处理  
  - **最坏情况测试**：`5^n` 组（笛卡尔积）  
  - **健壮最坏测试**：`7^n`  

- ✅ **三角形实例**（PDF p.14–15）：  
  - 条件：`1 ≤ a,b,c ≤ 200` 且 `a < b+c` 等  
  - 边界值：  
    - 数值边界：`1`, `0`, `200`, `201`  
    - 逻辑边界：`a = b + c − 1`（是三角形）, `a = b + c`（退化）, `a = b + c + 1`（非三角形）  
  - ⇒ **19 组测试用例**，含有效边界与无效越界

- 📌 **常见边界**：  
  - 16-bit int：`-32768`, `32767`  
  - 数组 `A[10]`：`A[0]`, `A[9]`  
  - 循环：0次 / 1次 / n−1次 / n次  
  - 屏幕：左上角 / 右下角  
  - 报表：首行 / 末行  

- **设计原则**（4条）：  
  ① 输入范围 ⇒ 边界内 / 外值；  
  ② 个数限制 ⇒ max/min/±1；  
  ③ 输出同样适用；  
  ④ 有序集合 ⇒ 首 / 末元素。

##### ③ 错误猜测法（Error Guessing）

- **核心**：**经验驱动**，猜测易错点  
- 用于**补充**：边界值/等价类未覆盖点  
- **依据来源**：  
  - 历史缺陷（本项目/旧版本）  
  - 语言/平台特性（如 C++ 资源释放）  
  - 汉化常见问题：日期格式（年-月-日 vs 月/日/年）  
  - DB 导入导出错误  

- ✅ **EDGE 4.0 中文版案例**（PDF p.17）：  
  - 英文日期习惯：`MM/DD/YYYY` / `DD-MMM-YYYY`  
  - 中文应为：`YYYY-MM-DD`  
  - 汉化遗漏 ⇒ 日期格式未改 ⇒ 导入导出时**日期转换错误**  
  - ⇒ 重点测试所有涉及日期输入/输出/转换的功能

- ✅ 建议：维护**常见错误检查表**，持续更新

##### ④ 判定表法（Decision Table）

- 适用：**多条件组合 → 多动作**，逻辑复杂  
- **组成**：  
  - 条件桩（Condition Stub）：输入条件  
  - 动作桩（Action Stub）：输出动作  
  - 条件项：T/F（各条件取值）  
  - 动作项：Y/N（执行哪些动作）  
  - 规则列 = 1 条测试用例  

- **构建步骤**：  
  1️⃣ 规则数 = $2^n$（n 条件）  
  2️⃣ 列桩  
  3️⃣ 填条件项（笛卡尔积）  
  4️⃣ 填动作项  
  5️⃣ **简化**（合并相似规则）

- ✅ **维修机器例**（PDF p.19）：  
  - 条件：功率>50？记录全？>10年？  
  - 动作：优先维修 / 其他  
  - 8 规则 → 简化为 **5 条用例**

- ✅ **三角形判定表**（PDF p.20）：  
  - 条件：等边？等腰？不等边？非三角形？  
  - 含“—”表示忽略（don’t care）  
  - 补充 3 边界用例：`a = b + c` 等

##### ⑤ 因果图法（Cause-Effect Graphing）

- 目标：处理**输入组合 + 相互制约**  
- 步骤：  
  1️⃣ 找因（输入/等价类）果（输出/状态变化），编号；  
  2️⃣ 画因果图（含 ∧/∨/¬ 逻辑）；  
  3️⃣ 标注**约束**（E/I/O/R/M）；  
  4️⃣ 转判定表；  
  5️⃣ 每列 → 1 测试用例  

- 🚧 **约束符号**：  
  - **E**（互斥）：至多1因成立  
  - **I**（包含）：至少1因成立  
  - **O**（唯一）：有且仅有1因成立  
  - **R**（要求）：因1真 → 因2必真  
  - **M**（屏蔽）：因1真 → 果必假  

- ✅ **薪金计算例**（PDF p.22）：  
  - 因：年薪制？严重过失？  
  - 果：扣4%/2%年终 或 8%/4%月薪  
  - ⇒ 因果图 → 判定表 → **4 条用例**

- ✅ **自动售货机**（PDF p.23）：  
  - 因：投1.5元 / 2元；按可乐/雪碧/红茶  
  - 中间：已投币、已按钮  
  - 果：退0.5元；出饮料  
  - ⇒ 用例：`TC3：投2元+按雪碧 → 退0.5元+出雪碧`

#### 3.1.7 黑盒测试总结  
- 测试人员**起点**；  
- 依**软件类型**灵活选用；  
- 目标：**提效 + 提高覆盖率**；  
- 核心能力：**原理掌握 + 经验积累**

### 3.2 白盒测试用例设计技术  

#### 3.2.1 定义  
- 知晓内部结构（代码/设计）→ 测**内部路径是否按预期执行**  
- 依赖：代码、规格说明、其他源资料  
- 重点：每条通路正确性  

#### 3.2.2 作用  
- ✅ 发现**黑盒测不出的隐患**：  
  - 内存泄漏  
  - 误差累积  
  - 错误算法但输出巧合正确（如负负得正）  
- 检查模块：  
  - 所有独立路径 ≥ 1次  
  - 逻辑判定真/假各 ≥ 1次  
  - 循环边界/内部  
  - 内部数据结构有效性  

#### 3.2.3 局限  
- 路径爆炸：20层循环 ⇒ $5^{20}$ 路径 ⇒ 需 3170 年测完（1ms/路径）  
- 需权衡成本与覆盖

#### 3.2.4 黑盒 vs 白盒 对比  

| 维度 | 黑盒测试 | 白盒测试 |
|------|----------|----------|
| **视角** | 外部行为 | 内部结构 |
| **用例来源** | 需求规格 | 代码/设计 |
| **覆盖不足** | 某些代码未测 | 未实现需求无法测 |
| **适用阶段** | 单元→系统 | 单元/集成为主 |
| **优点** | 易入手；功能导向；适用广 | 针对性强；有度量；工具支持多 |
| **缺点** | 依赖规格；难充分度量 | 生成数据难；工作量大；局限性 |
| **本质** | 确认：**做正确的事？** | 验证：**正确地做事？** |

#### 3.2.5 白盒测试内容（7项）  
1. 代码检查  
2. 静态结构分析  
3. 代码质量度量  
4. 功能确认与接口分析  
5. 逻辑覆盖率分析  
6. 性能与效率分析  
7. 内存分析  

#### 3.2.6 如何安排白盒测试（分阶段）

##### ▶ 单元测试（Unit Testing）  
- **定义**：  
  - 传统：函数/子程序  
  - OO：类 / 方法（复杂类可细分）  
- **目标**（4项）：  
  ① 代码 ↔ 设计一致  
  ② 跟踪需求实现  
  ③ 发现设计/需求缺陷  
  ④ 发现编码错误  
- **责任**：**程序员**（最懂逻辑）  
- **环境**：需辅助模块  
  - **驱动模块**（Driver）：模拟上层调用  
  - **桩模块**（Stub）：模拟被调用子模块  
- **策略**：  

| 类型 | 描述 | 优缺点 |
|------|------|--------|
| **自顶向下** | 顶层→下层，用桩 | ✅ 早验控制逻辑；❌ 桩成本高 |
| **自底向上** | 底层→上层，用驱动 | ✅ 驱动少；❌ 高层验证晚 |
| **孤立测试** | 各模块独立测 | ✅ 简单/高覆盖/并行；❌ 集成路径晚 |
| **混合策略** | 底层用自底向上 + 上层孤立 | ✅ 降低桩成本（如 `max` 作桩测 `funcA`） |

##### ▶ 集成测试（Integration Testing）  
- **目标**：验证模块**接口** + **组合功能**  
- **检查点**：  
  - 接口数据丢失？  
  - 模块功能相互干扰？  
  - 子功能 ⇒ 父功能？  
  - 全局数据结构？  
  - 误差累积？  
- **策略**：  
  - 一次性集成：全组装后测（快但难排错）  
  - 增量式：  
    - 自顶向下（广度/深度优先）  
    - 自底向上  
    - **三明治（混合）**：上→↓ + 下→↑ ⇒ 中间会合  
  - 分层集成：按架构分层测（适合层次系统）  
- **环境搭建**：硬件/OS/DB/网络/Web服务器/浏览器  

##### ▶ 系统测试（System Testing）  
- 依据：需求规格说明书  
- 环境：**模拟真实生产环境**（多版本组合 ⇒ 建环境库复用）  

##### ▶ 验收测试（Acceptance Testing）  
- 依据：软件任务书  
- 模式：**黑盒为主，白盒为辅**

##### ▶ 白盒综合策略（推荐）  
1️⃣ 先静后动：静态分析 + 代码检查 ⇒ 引导动态测试  
2️⃣ 覆盖率为核心：
   - 单元：语句/判定覆盖（基本路径法）  
   - 重点模块：多标准覆盖（条件/组合/路径）  
3️⃣ 阶段侧重：  
   - 单元：代码检查 + 逻辑覆盖  
   - 集成：加静态结构 + 质量度量  
   - 系统：根据黑盒结果补白盒  

#### 3.2.7 白盒测试用例设计技术  

##### 3.2.7.1 静态白盒测试  

- **代码检查**（人工+工具）  
  - 方式：桌面检查 / 代码审查 / 走查（Walkthrough）  
  - 检查项：命名/初值/作用域/逻辑/语法/结构/标准符合性  
  - 输入：需求/设计文档/源码/编码标准/缺陷检查表  
  - 编码标准参考：  
    - ISO C++ / MISRA  
    - 《Effective C++》  

- ✅ **课堂练习**（PDF p.38）：  
  ```c
  main() {
    int a[10], i, j, tmp;
    for(i=0; i<=10; i++) scanf("%d",&a[i]);       // ❌ 越界！
    for(i=0; i<=10; i++)
      for(j=i+1; j<=10; j++)
        if(a[i] < a[j]) { a[i] = a[j]; }           // ❌ 缺tmp，覆盖
    for(i=0; i<=10; i++) printf("%d", a[i]);
  }
  ```
  - 问题：数组越界、算法错（无tmp）、低效（O(n²)）、无返回类型、无缩进/注释  

- **静态结构分析**（工具自动）  
  - 输出：函数调用图 / 控制流图 / 文件调用图 / 类图  
  - ✅ 实例问题（PDF p.41–42）：  
    - 多出口 ⇒ 内存泄漏风险  
    - 死代码  
    - 复杂控制流（>10 判断）⇒ 可靠性↓  

- **代码质量度量**（ISO 9126）  
  - 模型：度量规则 ⇒ 分类标准 ⇒ 质量因素（可维护性等）  
  - 图形：Kiviat 图（蜘蛛图）展示维度分布  

##### 3.2.7.2 动态白盒测试  

- **功能确认与接口分析**  
  - 单元接口：参数个数/类型/单位一致性  
  - 局部数据：初始化/越界/类型  
  - 执行路径：运算优先级/精度/循环终止  
  - 错误处理：描述清晰/及时触发/边界正确  

- **性能与效率分析**：函数耗时/调用频次/并发/响应时间  
- **内存分析**：  
  - 发现：泄漏 / 未初始化使用 / 越界访问 / 释放后使用  
- **代码跟踪**：  
  - 函数级 / 控制流 / 源码级跟踪  

- **逻辑覆盖率分析**（⭐ 核心！）  
  - 公式：覆盖率 = 已测项数 / 总项数 × 100%  
  - 类型（按强度升序）：

| 覆盖类型 | 要求 | 示例用例数（4路径程序） |
|----------|------|------------------------|
| **语句覆盖** | 每条语句 ≥1次 | 1 |
| **判定覆盖**（分支） | 每判定真/假各 ≥1次 | 2 |
| **条件覆盖** | 每条件真/假各 ≥1次 | 2（但可能不满足判定） |
| **判定/条件覆盖** | 每条件真/假 + 每判定真/假 | 2 |
| **条件组合覆盖** | 所有条件取值组合 | $2^n$（n=4条条件 ⇒ 16? 实际8） |
| **路径覆盖** | 所有路径 ≥1次 | 4 |

- ✅ 以 `LogicCover(a,b,c)` 为例详解 6 种覆盖（PDF p.48–53）：  
  ```cpp
  if(a>0 && b>0) c = c/a;
  if(a>1 || c>1) c = c+1;
  c = b + c;
  ```
  - 控制流图：4 条路径  
  - 各覆盖最小用例集：  
    - 语句覆盖：`a=2,b=1,c=6`  
    - 判定覆盖：`a=2,b=1,c=6` + `a=-2,b=1,c=-6`  
    - 条件组合覆盖：4 用例（见下表）

| TC | Input | C1:a>0 | C2:b>0 | C3:a>1 | C4:c>1 | D1:a>0&&b>0 | D2:a>1\|\|c>1 | Path |
|----|-------|--------|--------|--------|--------|--------------|----------------|------|
| 1 | (2,1,6) | T | T | T | T | T | T | P1 |
| 2 | (2,-1,-2) | T | F | T | F | F | T | P3 |
| 3 | (-1,2,3) | F | T | F | T | F | T | P3 |
| 4 | (-1,-2,-3) | F | F | F | F | F | F | P4 |

- ❗ **课堂练习**（PDF p.53）：  
  > 下列关于逻辑覆盖，说法错误的是（ **C** ）  
  > A. 满足条件覆盖并不一定满足判定覆盖 —— ✔  
  > B. 满足条件组合覆盖的测试一定满足判定覆盖、条件覆盖和判定/条件覆盖 —— ✔  
  > **C. 满足路径覆盖也一定满足条件组合覆盖** —— ✘（反例：路径覆盖可能漏某条件组合）  
  > D. 判定/条件覆盖同时满足判定覆盖和条件覆盖 —— ✔  

- **基本路径测试**（Basis Path）  
  - 步骤：  
    1. 画控制流图  
    2. 计算环路复杂度 V(G)  
       - 法1：区域数  
       - 法2：E − N + 2  
       - 法3：判定节点数 + 1  
    3. 确定基本路径集（独立路径数 = V(G)）  
    4. 为每路径设计用例  

  - ✅ `Sort(iRecordNum, iTyep)` 实例（PDF p.58）：  
    - 控制流图 ⇒ V(G) = 4  
    - 4 条基本路径 ⇒ 4 个测试用例（iRecordNum=0/1；iType=0/1/2）  
    | TC | Input | Output | Path |
    |----|-------|--------|------|
    | 1 | (0, 0) | 0 | 4→14 |
    | 2 | (1, 0) | 2 | 4→6→7→14 |
    | 3 | (1, 1) | 10 | 4→6→9→10→13→4→14 |
    | 4 | (1, 2) | 20 | 4→6→9→12→13→4→14 |

- 📌 **覆盖率使用原则**：  
  - 是**手段非目的**  
  - 不追求 100%，避免形式主义  
  - 结合风险：高风险模块高覆盖

### 3.3 面向对象测试用例设计技术  

#### 3.3.1 面向对象测试概述  
- ✅ 优势：代码复用、开发提速  
- ❗ 挑战：  
  - 类修改 ⇒ 所有子类重测  
  - 继承链深 ⇒ 影响范围广  
  - 文档缺失 ⇒ 难追踪依赖  
- ⇒ **测试更迫切 + 需新方法**

#### 3.3.2 面向对象技术基础  
- 程序 = 对象 + 消息  
- 类 = 对象模板；对象 = 类实例  
- 三大特性：  
  - **封装**：数据+操作一体；信息隐藏  
  - **继承**：代码复用；子类特化  
  - **多态**：同名方法多实现（动态绑定）  

##### ▶ OO 测试难点（PDF p.60–62）  
| 特性 | 测试挑战 |
|------|----------|
| **封装** | ① 信息隐藏 ⇒ 难直接测内部；需构造子类/修改代码做桩<br>② 操作依赖实例状态 ⇒ 需设计**调用序列**（非单操作） |
| **继承** | ① 测子类 ⇒ 需覆盖父类所有属性/方法<br>② 构造函数链易错（祖先类属性未正确初始化） |
| **多态** | 函数名 = switch ⇒ 需覆盖**所有绑定实现**才达语句覆盖 |
| **继承+多态** | **YO-YO 问题**：1 操作调用 58 方法 ⇒ 误用难发现 |
| **抽象类** | 需构造子类实现抽象方法 ⇒ 驱动更复杂 |
| **结构** | 类互依赖 ⇒ 难独立单元测试 |

#### 3.3.3 面向对象测试模型（6层）  

| 阶段 | 测试对象 | 依据 |
|------|----------|------|
| OOA Test | 分析模型（文本） | 需求 |
| OOD Test | 设计模型（文本） | OOA + 设计文档 |
| OOP Test | 编码风格/实现 | 代码 |
| **单元测试** | 类/成员函数 | 详细设计 |
| **集成测试** | 类间服务/消息 | OOD / 设计 |
| 确认/系统测试 | 用户需求满足度 | OOA / 需求 |

#### 3.3.4 OO 测试策略（分阶段）  

##### ▶ OOA 测试（5方面）  
1. **对象**：是否全面？是否多属性/服务？命名准确？  
2. **结构**（分类/组装）：  
   - 分类：上层是否概括下层？特性是否共性+特化？  
   - 组装：整体/部件是否符合现实？有无遗漏？  
3. **主题**：≤7个（7±2 原则）；是否抽象合理？主题间消息完整？  
4. **属性/实例关联**：属性是否普适/独立？关联是否完整（尤其 1:N, N:M）？  
5. **服务/消息**：状态是否全覆盖？消息是否引导正确服务？线程合理？  

##### ▶ OOD 测试（3方面）  
1. **类**：是否覆盖 OOA 对象/属性/服务？是否基础/低耦合/方法单职责？  
2. **类层次**：是否涵盖所有类？是否实现关联？子类是否新增特性/上层概括共性？  
3. **类库支持**：同功能接口一致？方法 ≤30 行？结构“深而窄”？  

##### ▶ OOP 测试（2焦点，C++ 为例）  
- 数据成员是否封装（private/protected）？  
- 类是否实现全部要求功能？  

##### ▶ 单元测试（成员函数级）  
- 假设：同类输入/同复杂度输入 ⇒ 行为一致  
- 继承影响：  
  - 若子类**重写**父类方法 → **必须重测**  
  - 若方法**调用重写方法** → 必须重测  
  - 多态方法：加测**输入类型多样性**  

##### ▶ 集成测试  
- 基于**黑盒**（因动态绑定，控制流难预知）  
- 步骤：  
  1. 选定类 → 确定状态/行为/消息/IO  
  2. 定覆盖标准  
  3. 用结构图找关联类  
  4. 构造对象 → 设计用例（输入→状态→服务→行为）  

##### ▶ 确认/系统测试  
- 内容同传统：功能/强度/性能/安全/恢复/易用性/安装卸载  

#### 3.3.5 OO 测试用例设计方法  

| 方法 | 说明 | 适用场景 |
|------|------|----------|
| **传统方法** | 等价类/边界值/判定表 仍适用 | 功能测试 |
| **基于故障** | 从分析模型出发，预设故障 | 防规格错误/子系统交互错 |
| **基于场景**（⭐ 推荐） | 从用例 → 基本流 + 备选流 → 场景 → 用例矩阵 | 用户任务导向 |
| **类随机测试** | 随机操作序列测对象生存史 | 多方法类（如 Account） |
| **类层次分割** | 按状态/属性/类型划分等价类 | 降测试冗余 |
| **行为模型导出** | 用状态图/活动图 → 路径测试 | 动态行为类 |

##### 🔍 基于场景测试详解（ATM 提款为例）  
- **事件流**：  
  - 基本流：插卡→输密→选提款→输入金额→取钱→退卡  
  - 备选流：密码错/卡无效/余额不足/超限额/取消  
- **场景生成**（PDF p.71–72）：  
  - 场景1：基本流  
  - 场景2：基本流 + 备选流1（密码错）  
  - …  
  - 场景8：基本流 + 备选流3 + 4（超限+取消）  
- **用例矩阵**（PDF p.72）：  
  - 行：测试用例（TC1 正面；TC2–6 负面）  
  - 列：条件（用 **V/I/n/a** 标记）  
    - V：Valid（有效）  
    - I：Invalid（触发备选流）  
    - n/a：not applicable  
  - 字段：ID、场景、数据、预期结果  
- **数据确定**：评审后设定具体值（如卡号、密码、金额）

##### 🔍 分割测试（Account 类例）  
- 按**状态**：  
  - 改变状态：`deposit` / `withdraw`  
  - 不改变状态：`balance` / `summarize`  
- 按**属性**：  
  - 用 creditlimit / 改 creditlimit / 不用  
- 按**类型**：  
  - 初始：`open` / `setup`  
  - 计算：`deposit` / `withdraw`  
  - 查询：`balance` / `summarize` / `creditlimit`  
  - 终止：`close`

##### 🔍 行为模型导出（状态图→测试路径）  
- 状态图解决有限状态机“状态爆炸”问题  
- 路径测试：如 `open → setup → deposit → withdraw → close`  
- ✅ 实例（PDF p.75）：  
  - TC1：`Open + setupAcct + deposit(initial) + withdraw(final) + close`  
  - TC2：`Open + setupAcct + deposit(initial) + deposit + balance + credit + withdraw(final) + close`  

#### 3.3.6 结论  
- 目标一致：**最小工作量 ⇒ 最多错误**  
- 视角扩展：复审分析/设计模型  
- 焦点转移：**模块 → 类**
