## 🔍 **第3–4页：1. 测试点的确定**

### ✅ 来源依据：
| 标准体系 | 要求 |
|---------|------|
| **ISO 质量体系** | 在概要/详细设计中应明确**每个单元/模块的测试要点、指标与方法** |
| **CMM 质量体系** | 用例模型中每个用例为一个测试点；每个测试需求 ≥ 2 个测试用例（正 + 负） |

### ✅ 关键认知（颠覆传统）：
- ❌ 错误观念：“测试点由测试员确定”  
- ✅ 正确认知：
  - **测试点由分析/设计人员确立**（因其最理解需求与架构）  
  - **测试用例由测试设计员/分析设计员制定**，非普通测试执行员  
  - 测试活动始于**项目立项后**（非编码完成后）  
  - 测试对象包括：**需求→设计→代码→文档全生命周期制品**  
    > ✔ 需求规格说明书、概要/详细设计说明书、用户手册等

> 📌 实践意义：体现“测试左移”，将质量保障前移至需求与设计阶段。

## 📄 **第5–10页：2. 测试用例设计概述**

### 2.1 测试用例的定义（第5页）
> **测试用例** = 为特定目的（如验证某路径/需求）而设计的：  
> - 一组**测试数据** + 相关**测试规程**  
> - 或者说：**能有效发现缺陷的最小测试执行单元**

### 2.2 重要性（第5–6页）
| 维度 | 说明 |
|------|------|
| **质量保障基石** | 软件质量 ≈ 测试用例数量与质量 × 执行有效性 |
| **执行一致性保障** | 不同测试人员依同一用例应得相同结果 → 可重复、可追溯 |
| **复用价值** | 良好用例可跨版本/项目复用，大幅提效 |
| **量化管理依据** | 用例通过率 = 代码质量直接证据；也是人员绩效考核指标 |
| **专业门槛** | 编写者需精通：用户场景、规格说明、系统设计、代码结构 |

### 2.3 用例数与软件规模关系（第6–7页）
| 类型 | 关系特征 | 说明 |
|------|-----------|------|
| **单元测试** | ≈ 正比 | 模块数↑ → 用例数↑线性 |
| **集成/系统测试** | **非线性增长** | 模块交互组合爆炸 → 用例数增速 > 规模增速 |
| **质量等级影响** | 航天 > 工业 > 民用 | 高可靠系统：用例/代码行比值更高（如 10:1 vs 1:1） |

> 📊 图中示意：曲线斜率随规模增大而陡峭（组合爆炸效应）

### 2.4 测试用例设计步骤（第7–10页）
标准五步法：
1. **为测试需求确定测试用例**  
   - 每需求编号（TR_XXXX）  
   - 每需求 ≥ 2 用例（**正向 + 负向**）  
2. **为用例确定输入 & 输出**  
   - 输入 = 用户输入/触发事件/初始状态  
   - 输出 = 预期结果/终态  
   - ⭐️ **设计原则**（重点！）：  
     - 必用**边界值分析**（错误发现能力最强）  
     - 补充**等价类划分**  
     - 追加**错误推测法**  
     - 检查**逻辑覆盖率**（未达标则补）  
     - 多条件组合 → 优先**因果图法**
3. **编写测试用例**（模板化）
   - 编号：`TC_TR_XXXX`  
   - 内容要素：
     | 字段 | 说明 |
     |------|------|
     | 测试目标状态 | 执行前系统应处状态 |
     | 输入（操作） | 数据 + 操作序列 |
     | 预期输出 | 结果状态/数据 |
     | 测试环境 | 硬件/OS/版本依赖 |
     | 用例间关联 | 依赖前置用例？ |
4. **评审测试用例**（第9页）
   - **Checklist**：
     - 需求/设计元素是否全覆盖？  
     - 每用例是否有**明确预期结果**？  
     - 是否定义**初始状态 & 数据状态**？  
     - 是否含**所有单一边界**？  
     - 是否覆盖**所有业务数据流**？  
     - 命名是否符合规约？
   - **参与人**：项目经理、系统分析员、测试设计员、测试员
5. **跟踪测试用例**（第10页）
   - 需求 → 测试需求 → 测试用例（**可追溯性矩阵**）  
   - 监控：**用例执行率、通过率、缺陷密度**

### 3.1 黑盒测试用例设计技术（第11–32页）

#### 3.1.1 概念
- **定义**：不看内部结构，只依据**规格说明**验证功能行为  
- **前提**：将程序视为“黑盒”，仅知输入→输出映射

#### 3.1.2 作用（发现5类错误）
1. 功能缺失/多余  
2. 接口输入接收/输出生成错误  
3. 数据结构/外部文件访问错  
4. 性能不能达标  
5. 初始化/终止性错误

#### 3.1.3 局限
- ⚠️ **穷举不可能**：32位整型输入×2变量 → $2^{64}$组合，需5亿年测完  
- ⚠️ 依赖规格说明 → 若需求本身有误，无法发现  
- ⚠️ 仅适用于外部行为测试（如第三方测试）

#### 3.1.4 静态黑盒测试（审查说明书）
- **目的**：早期发现需求缺陷（成本最低阶段）  
- **方法**：
  - **高级审查**：宏观审视完整性、一致性、可行性  
  - **设身处地**：以用户视角验证可用性  
  - **对标标准**：国标/行业/公司/竞品规范  
  - **竞品分析**：研究同类系统（iOS vs Android）  
  - **属性检查表**（8项）：  
    `完整｜准确｜精确｜一致｜贴切｜合理｜代码无关｜可测试`  
  - **用语危险词**：  
    `总是/所有/没有｜当然/显然｜某些/常常｜等等｜良好/高效｜已处理/已忽略｜如果…那么…（无否则）`

#### 3.1.5 动态黑盒测试
- 定义：运行程序 + 黑盒视角 → 行为测试（Behavior Testing）

#### 3.1.6 五大黑盒设计技术（重点！）

##### （1）等价类划分（第13–20页）
- **核心思想**：将输入域划分为若干等价子集，**代表值测试 ≡ 全子集测试**
- **分类**：
  - 有效等价类（valid）  
  - 无效等价类（invalid）
- **划分原则**（6条，必须掌握）：
  1. 值域/个数限定 → 1有效 + 2无效  
     `例：1≤项数≤999 → [1,999]有效；<1、>999无效`
  2. 集合/必须条件 → 1有效 + 1无效  
     `例：首字符为数字 → 是/否`
  3. 布尔量 → 1有效 + 1无效  
  4. 枚举值 → 每值1有效 + 1无效集合  
     `例：教授/副教授/讲师/助教 → 4有效 + 1“其他”无效`
  5. 规则约束 → 1有效 + 多角度无效  
     `例：Pascal语句必以;结尾 → 以: / , / 空格 / LF结尾等`
  6. 组合值 → 每组合1有效 + 1无效（全非法集）
- **选例策略**：
  1. 每有效等价类 → 尽量**多覆盖**（1用例含多个有效类）  
  2. 每无效等价类 → **单个覆盖**（1用例仅1个无效）  
     > ⚠️ 避免“污染”：多个无效同时出现可能掩盖单个错误
- **经典案例**：Pascal标识符生成（9用例覆盖15类）

> ✅ **三角形问题课堂练习**（第21页）  
输入：3整数A,B,C  
输出：不等边 / 等腰 / 等边 / 非三角形  
等价类划分：
| 类别 | 条件 | 示例 |
|------|------|------|
| 无效 | ≤0 或 不满足三角不等式 | (1,1,3), (0,1,1) |
| 有效-不等边 | A,B,C>0, 两两不等, 满足三角不等式 | (3,4,5) |
| 有效-等腰 | 仅两等 | (3,3,4) |
| 有效-等边 | 三边相等 | (3,3,3) |

##### （2）边界值分析（第22–27页）
- **原理**：50%+缺陷出现在边界（CECOM研究证实）  
- **与等价类关系**：是其**扩展**，关注等价类**边界点**
- **取值策略**（5点法）：
  - min, min+, nom（典型值）, max-, max  
  - 例：1~999 → 用例：1, 2, 500, 998, 999
- **多变量情形**：固定n-1个为nom，遍历1个变量边界 → 共 **4n+1** 用例
- **扩展形式**：
  - **健壮性测试**：+ min-, max+ → 检查异常处理  
  - **最坏情况测试**：笛卡尔积（5ⁿ）→ 测试极值组合  
  - **健壮最坏情况**：7ⁿ（min-/min/min+/nom/max-/max/max+）
- **应用原则**：
  - 输入/输出域有界 → 测边界内外  
  - 个数限制 → 测 max/max±1/min/min±1  
  - 有序集合（数组/文件）→ 测首/尾元素  
- **常见边界举例**：
  - 16位整数：32767, -32768  
  - 数组 `int A[10]`：A[0]～A[9] → 用例含 A[-1], A[10]  
  - 循环：0次、1次、倒数第2次、最后一次

> ✅ **三角形边界实例**（第27页）  
条件：1 ≤ a,b,c ≤ 200  
边界用例：  
- (1,1,1), (1,1,2), (200,200,200), (200,200,199)  
- 超界：(0,1,1), (201,1,1), (100,100,201)

##### （3）错误猜测法（第28–29页）
- **本质**：经验驱动的“直觉测试”  
- **原则**：
  - 历史缺陷复现（前版本高频Bug）  
  - 语言/平台特有陷阱（如C++析构、JS闭包）  
  - 模块关联推测（A改→B可能错）  
  - 汉化特有问题（中英文差异：日期格式 YYYY-MM-DD vs MM/DD/YYYY）
- **实践**：建立**常见错误检查表**（Checklist），持续积累
- **案例**：EDGE 4.0中文版  
  - 问题：日期格式未转换 → 导出/导入出错  
  - 测试点：所有含日期输入/显示/转换处

##### （4）判定表法（第30–33页）
- **适用**：多条件组合 → 多动作逻辑（if-else嵌套复杂）
- **组成**：
  | 部分 | 说明 |
  |------|------|
  | 条件桩 | 所有输入条件 |
  | 动作桩 | 所有可能输出动作 |
  | 条件项 | 各条件T/F组合（2ⁿ种） |
  | 动作项 | 对应动作应执行与否 |
  | 规则 | 1列 = 1条完整逻辑路径 |
- **建表步骤**：
  1. 确定规则数（2ⁿ）  
  2. 列条件/动作桩  
  3. 填条件项（笛卡尔积）  
  4. 填动作项  
  5. **简化**：合并相同动作列、无关条件（“—”表示）
- **适用条件**（Beizer）：
  - 规格易转判定表  
  - 条件顺序无关  
  - 规则顺序无关  
  - 无短路依赖（即某规则触发后无需再判其他）
- **案例1**：维修优先级  
  条件：功率>50HP？维修记录全？运行>10年？  
  动作：优先维修 / 其他处理  
  → 8规则 → 简化后4条
- **案例2**：三角形分类  
  条件：a=b?, b=c?, a=c?, 三角不等式？  
  → 含“不可能”规则（如a=b且b=c但a≠c → 矛盾）

##### （5）因果图法（第34–36页）
- **动机**：解决等价类/边界值**忽略条件组合与约束**问题
- **核心**：将自然语言规格 → 因果图 → 判定表 → 用例
- **步骤**：
  1. 识别**原因**（输入/等价类）与**结果**（输出/状态变）  
  2. 画因果图（含逻辑关系：∧/∨/¬）  
  3. 添加**约束**（E/I/O/R/M）  
     - E（互斥）：a,b不能同为1  
     - I（包含）：a,b,c至少1个为1  
     - O（唯一）：a,b有且仅1个为1  
     - R（要求）：a=1 ⇒ b=1  
     - M（屏蔽）：a=1 ⇒ b=0  
  4. 转判定表  
  5. 每列 → 1测试用例
- **案例1**：薪金扣罚  
  原因：年薪制？严重过失？  
  结果：扣年终金4%/2% 或 扣月薪8%/4%  
  → 因果图 → 判定表 → 4用例
- **案例2**：自动售货机  
  - 投币：1.5元 / 2元  
  - 按钮：可乐 / 雪碧 / 红茶  
  - 结果：退5角？出饮料？  
  → 中间状态：已投币、已按钮  
  → 用例：投2元+按可乐 → 出可乐+退5角

#### 3.1.7 结论
> 黑盒测试是起点，需结合软件特性**灵活选用+组合技术**，方能高效覆盖。

### 3.2 白盒测试用例设计技术（第37–69页）

#### 3.2.1 定义
- 已知内部结构（代码/设计）→ 检查路径/逻辑是否符合规格  
- 前提：可访问源码、设计文档

#### 3.2.2 作用
- 发现黑盒无法查的“**负负得正**”类错误（错误逻辑得正确结果）  
- 检测隐患：内存泄漏、精度累积误差、并发竞争  
- 检查：  
  - 所有独立路径  
  - 判定真/假分支  
  - 循环边界  
  - 数据结构有效性

#### 3.2.3 局限
- 路径爆炸：20层循环 → $5^{20}$路径 ≈ 3170年测完  
- 无法发现规格遗漏功能

#### 3.2.4 黑盒 vs 白盒对比（第48页表格）
| 维度 | 黑盒测试 | 白盒测试 |
|------|----------|----------|
| **基础** | 规格说明 | 代码结构 |
| **阶段** | 全阶段（尤其系统/验收） | 单元/集成为主 |
| **优点** | 易上手、功能视角、适用广 | 覆盖可度量、工具支持强 |
| **缺点** | 覆盖难量化、规格误则漏检 | 数据难生成、仅测已实现部分 |
| **本质** | **确认**（Are we building the right product?） | **验证**（Are we building the product right?） |

#### 3.2.5 白盒测试内容
- 代码检查｜静态结构分析｜质量度量｜功能/接口确认｜逻辑覆盖｜性能｜内存分析

#### 3.2.6 各阶段白盒测试安排
| 阶段 | 重点内容 | 依据文档 |
|------|----------|----------|
| **单元测试** | 代码检查、覆盖率、内存 | 详细设计 |
| **集成测试** | 接口、结构分析、性能 | 概要设计 |
| **系统测试** | 接口、覆盖率（补黑盒遗漏） | 需求规格 |
| **验收测试** | 以黑盒为主，白盒辅助 | 任务书 |

##### ▶ 单元测试（详，第48–55页）
- **定义**：对最小单元（函数/类/方法）隔离测试  
- **策略**：
  - **自顶向下**：桩模块多，早验主控流  
  - **自底向上**：驱动模块少，早验底层  
  - **孤立测试**：桩/驱动独立，易并行，但集成晚  
  - **综合策略**：如先底向上测`max()`，再用其桩测`funcA()`
- **环境**：驱动模块（Driver） + 桩模块（Stub）  
  > ❗ 桩 ≠ 空函数！需模拟子模块关键行为

##### ▶ 集成测试（第55–67页）
- **目标**：验证模块间接口、全局数据、误差累积  
- **策略**：
  | 类型 | 说明 | 优缺点 |
  |------|------|--------|
  | 一次性集成 | 先单测→全拼装 | 简单；但错难定位 |
  | 自顶向下 | 顶层→逐层加 | 早验主干；桩多 |
  | 自底向上 | 底层→逐层加 | 驱动少；高层错晚发现 |
  | 三明治 | 上（顶向下）+ 下（底向上）→ 中间汇合 | 平衡；需选好“中间层” |
  | 分层集成 | 按架构层集成（如MVC） | 适合分层系统 |

##### ▶ 系统/验收测试
- 系统：结合黑盒结果补白盒（如高风险路径再覆盖）  
- 验收：用户主导，白盒仅辅助查核心路径

#### 3.2.7 白盒测试技术

##### （1）静态白盒测试（第68–74页）
- **代码检查**（人工+工具）：
  - 方式：桌面检查｜代码审查（Review）｜走查（Walkthrough）  
  - 检查项：命名/初值/作用域/逻辑/语法/结构/可移植性  
  - 工具标准：MISRA C/C++｜Effective C++  
- **静态结构分析**（工具自动）：
  - 产出：调用图｜控制流图｜文件依赖图｜类图  
  - 查问题：死代码、多出口（内存泄漏风险）、高圈复杂度  
    > ✅ 实例：某函数8出口+死代码 → 必改
- **质量度量**（ISO 9126）：
  - 模型：规则 → 分类标准 → 质量因素（可维护性等）  
  - 工具：Kiviat图可视化复杂度/耦合/内聚等

##### （2）动态白盒测试（第74–84页）
| 技术 | 目标 | 方法 |
|------|------|------|
| **功能/接口分析** | 验模块功能 & 接口正确性 | 查参数个数/类型/单位、全局变量一致性、I/O匹配 |
| **性能分析** | 找瓶颈 | 插桩测函数耗时/调用频次 |
| **内存分析** | 防泄漏/越界 | 工具监控：分配未初始化｜越界读写｜释放后使用 |
| **代码跟踪** | 观执行路径 | 函数级/语句级跟踪，录调用栈 |
| **逻辑覆盖率分析**（核心！） | 量化测试充分性 | 见下表 |

###### 🔹 逻辑覆盖率层级（第78–89页，含完整实例）
以函数 `LogicCover(a,b,c)` 为例：
```c
if(a>0 && b>0) c = c/a;
if(a>1 || c>1) c = c+1;
c = b + c;
```
| 覆盖类型 | 定义 | 用例数 | 示例输入 | 覆盖率 | 局限 |
|----------|------|--------|----------|--------|------|
| **语句覆盖** | 每行代码执行 ≥1次 | 1 | (2,1,6) | 100% | 不测分支 |
| **判定覆盖**（分支） | 每判定T/F ≥1次 | 2 | (2,1,6), (-2,1,-6) | 100% | 不测条件组合 |
| **条件覆盖** | 每条件T/F ≥1次 | 2 | (-1,2,3), (2,-1,-2) | 100% | 可能漏判定组合（例：仅覆盖D1=F，未覆盖D1=F且D2=F） |
| **判定/条件覆盖** | 条件+判定均覆盖 | 2 | (-1,-2,-3), (2,1,6) | 100% | 非全部组合 |
| **条件组合覆盖** | 所有**条件组合** ≥1次 | 4 | (2,1,6), (2,-1,-2), (-1,2,3), (-1,-2,-3) | 100% | 接近完备 |
| **路径覆盖** | 所有**路径** ≥1次 | 4 | + (1,1,-3) 覆盖 P2 | 100% | 路径爆炸不可行 |

> 📌 **课堂练习**（第89页）：  
> 问：路径覆盖 ⇒ 条件组合覆盖？  
> 答：❌ **不一定**！路径覆盖关注**执行路径**，但若某路径中条件组合未完全展开（如短路逻辑），仍可能漏组合。  
> ✅ 正确关系：  
> `条件组合 > 判定/条件 = 判定+条件 > 判定 > 语句`  
> `条件组合 > 条件`  
> `路径 > 判定`

##### ▶ 基本路径测试（第90–93页）
- **理论**：McCabe圈复杂度 $V(G)$ = 独立路径数上限  
- **计算**：
  - $V(G) =$ 区域数  
  - $= E - N + 2$（E边，N节点）  
  - $= $ 判定节点数 + 1
- **步骤**：
  1. 画控制流图  
  2. 算 $V(G)$  
  3. 导出基本路径集  
  4. 设计用例覆盖每路径
- **实例**：`Sort()` 函数 → $V(G)=4$ → 4路径 → 4用例

> 💡 覆盖率原则：  
> - 覆盖率是**手段**，非目标  
> - 不追求100%，重**高风险路径**  
> - 结合黑盒用例 + 用户场景 + 故障模式

### 3.3 面向对象测试用例设计技术（第94–114页）

#### 3.3.1 概述
- **优势**：代码复用、开发加速  
- **挑战**：
  - 类修改 → 所有子类需重测  
  - 文档缺失 → 继承链难追溯  
  - 错误易扩散（复用即复错）  
> ✅ 结论：OO更需严格测试！

#### 3.3.2 OO特性对测试影响
| 特性 | 测试难点 |
|------|----------|
| **封装** | 信息隐藏 → 需构造复杂驱动/桩；状态依赖 → 需设计调用序列 |
| **继承** | 子类需测：自身+继承部分；覆盖准则失效（如仅测子类新方法不够） |
| **多态** | 同名方法多实现 → 需覆盖所有绑定（动态绑定 = 隐式switch） |
| **组合** | YO-YO问题：1操作 → 58方法调用 → 误用难防 |

#### 3.3.3 OO测试模型（6阶段）
1. OOA测试（分析模型）  
2. OOD测试（设计模型）  
3. OOP测试（编码风格）  
4. 单元测试（类/方法）  
5. 集成测试（消息/协作）  
6. 系统/确认测试（用户需求）

#### 3.3.4 OO测试策略分述
##### （1）OOA测试（5方面）
- 对象：是否全面？属性≥2？命名准？  
- 结构（分类/组装）：是否反映一般-特殊/整体-部分？  
- 主题：≤7±2？是否合理抽象？  
- 属性/关联：是否适用？是否完备（尤其1:N, N:M）？  
- 服务/消息：状态覆盖？线程合理？重用？

##### （2）OOD测试（3方面）
- 类：是否覆盖OOA对象？实现服务？数据抽象？低耦合？  
- 类层次：是否覆盖OOA关联？子类有新特性？父类含共性？  
- 类库：同类操作接口一致？方法≤30行？深度>宽度？

##### （3）OOP测试（2方面）
- 数据封装：成员是否private？访问是否经方法？  
- 功能实现：是否满足需求？

##### （4）OO单元测试（重点！第103页）
- 测**类/方法**，非传统函数  
- 继承下策略（Brian Marick）：
  - 父类测过 ≠ 子类免测！重写方法 & 调用重写方法的部分需重测  
  - 多态方法：增加**输入类型**维度设计用例

##### （5）OO集成测试
- 基于**黑盒**（因动态绑定难静态分析）  
- 步骤：选类 → 定状态/行为 → 定覆盖标准 → 构对象 → 设输入/期望

##### （6）系统测试
- 同传统：功能/性能/安全/易用性等

#### 3.3.5 OO测试用例设计方法
| 方法 | 说明 | 案例 |
|------|------|------|
| **传统方法可用性** | 黑盒（等价/边界）+ 白盒（路径/覆盖）仍适用 | 方法参数测试 |
| **基于故障测试** | 从模型找潜在故障：功能多余/缺失、子系统交互错 | 检查用例遗漏分支 |
| **基于场景测试**（推荐！） | 从**用例场景**（基本流+备选流）导出用例 | ATM取款：7场景→7用例（1正+6负） |
| **OO随机测试** | 随机排列操作序列，测对象生存史 | Account: open→setup→deposit×2→withdraw→close |
| **类层次分割测试** | 按状态/属性/类型分组测 | Account：<br>– 状态变：deposit/withdraw<br>– 属性用：creditlimit相关操作<br>– 类型：初始化/计算/查询/终止 |
| **行为模型导出** | 从状态图/活动图导路径 | 账户状态图 → 开户→存款→取款→销户 |

> ✅ **场景测试实例**（ATM取款，第109–111页）：  
> - 基本流：插卡→输密→选取款→输额→取钱→退卡  
> - 备选流1：密码错 → 重输（3次锁卡）  
> - 备选流2：余额不足  
> - 备选流3：取款机无钱  
> - 备选流4：取消交易  
> → 组合出8场景 → 用例矩阵（V=valid, I=invalid, n/a=不适用）

#### 3.3.6 结论
> OO测试目标不变（最少工作→最多缺陷），但**视角扩大**：  
> - 从“测模块” → “测类+协作”  
> - 从“测代码” → “复审分析/设计模型”  
>
> 🌟 **测试箴言（小故事）**：  
> *醉汉在路灯下找钥匙，因“这里更亮”——但钥匙在停车场！*  
> → **测最可能出错处**，而非“最容易测处”。
