## 🔹 1. **测试过程模型**

### 1.1 软件生命周期（Software Life Cycle）
从规划开始到废弃为止，包括：
- 概念阶段  
- 需求阶段  
- 设计阶段  
- 实现阶段  
- **测试阶段**  
- 试运行阶段  
- 运行与维护阶段  

> 📌 测试贯穿全生命周期，非仅“最后阶段”。

---

### 1.2 软件开发模型（与测试关联分析）

| 模型 | 特点 | 对测试的影响 | 局限 |
|------|------|--------------|------|
| **瀑布模型** | 线性、阶段分明 | 测试在编码后进行 | 测试滞后，问题后期暴露 |
| **原型模型** | 快速构建原型，反复迭代 | 需配合探索性测试 | 文档不全，用例难设计 |
| **螺旋模型** | 风险驱动 + 原型迭代 | 支持早期测试介入 | 复杂，管理成本高 |
| **增量模型** | 分阶段交付功能模块 | 支持增量测试 + 回归测试 | 模块接口易出错 |
| **RUP**（统一过程） | 4阶段 + 迭代：Inception, Elaboration, Construction, Transition | 测试活动嵌入各迭代 | 重型过程，学习曲线陡 |
| **敏捷开发** | 用户故事驱动、短周期迭代 | ✅ **测试左移**：SM/PO/Dev 全员参与<br>✅ 增量+回归测试结合<br>❌ 无详尽需求/设计文档 | 依赖自动化测试；用例需动态调整 |

> ⚠️ **共性问题**：传统模型未充分重视测试价值，缺乏对测试活动的系统指导。

---

### 1.3 软件测试模型（核心内容）

#### ✅ **1.3.1 V模型**
- 提出者：Paul Rook（1980s），英国NCC发布  
- 本质：瀑布模型的变种，测试与开发活动**镜像对应**  
- 流程：
  ```
  需求 → 高设 → 详设 → 编码  
    ↑      ↑      ↑      ↑  
  验收 ← 系统 ← 集成 ← 单元
  ```
- **优点**：结构清晰、级别对应明确  
- **局限**：
  - 测试是开发“后续阶段”，易误解为仅测代码  
  - 需求缺陷到验收阶段才暴露  
  - 不支持迭代/变更

#### ✅ **1.3.2 W模型**
- 提出者：Evolutif公司  
- 依据：IEEE Std 1012-1998（V&V标准）  
- 核心：**测试与开发同步进行**，每个开发阶段都有对应验证与确认  
  ```
  需求分析 ──→ 需求测试  
    ↓              ↓  
  系统设计 ──→ 设计测试  
    ↓              ↓  
  编码      ──→ 代码测试
  ```
- **优点**：
  - 测试对象 = 需求/设计/代码  
  - 测试尽早介入（如需求一完成即可测试）  
- **局限**：
  - 仍为线性串行模型  
  - 不适应无文档/敏捷场景  
  - 缺乏测试流程独立建模

#### ✅ **1.3.3 H模型**
- 核心思想：**测试是独立流程**，可与其他活动并发、迭代、增量  
- 测试流程两阶段：
  - **准备阶段**：需求分析、测试计划、测试设计、测试编码、测试验证  
  - **执行阶段**：测试运行、测试报告、测试分析  
- **揭示原则**：
  1. 测试 ≠ 仅执行，含大量前期活动  
  2. 测试流程独立于开发，贯穿全生命周期  
  3. 测试应**尽早准备、尽早执行**  
  4. 测试层次可交叉、反复触发（非严格顺序）  
- **优势**：
  - 资源灵活调配  
  - 反映测试过程复杂性  
  - 便于跟踪测试投入流向

#### ✅ **1.3.4 X模型**
- 特点：
  - 左侧：对**程序片段**分别编码与测试 → 频繁集成  
  - 右侧：对**集成后系统**进行整体测试  
  - 支持**探索性测试**（Exploratory Testing）  
  - 缺陷修复可跨团队协作  
- **优点**：
  - 强调单元/集成测试重要性  
  - 引入探索测试，更贴近实际  
- **缺点**：
  - 未覆盖需求测试、验收测试  
  - 仅聚焦测试执行部分，忽略流程管理

---

#### ✅ **1.3.5 测试模型比较总结**

| 模型 | 是否覆盖需求/设计测试 | 是否支持并行/迭代 | 是否独立流程 | 适用场景 |
|------|------------------------|------------------|--------------|----------|
| V    | ❌（隐含但未强调）     | ❌               | ❌           | 小型、需求稳定的项目 |
| W    | ✅                     | ❌（线性同步）   | ❌           | 中大型、文档齐全项目 |
| H    | ✅                     | ✅（完全支持）   | ✅           | 复杂系统、需过程独立的组织 |
| X    | ❌（仅代码/集成）      | ✅               | ⚠️局部独立   | 快速原型、探索型项目 |

> ✅ **最佳实践**：**W模型为框架 + H模型为思想** → 独立测试 + 尽早介入 + 迭代反馈

---

#### ✅ **1.3.6 测试左移 & 测试右移**
- **测试左移**（Shift Left）：
  - 需求评审、设计评审  
  - 测试计划/设计前置  
  - TDD（测试驱动开发）、ATDD（验收测试驱动开发）
- **测试右移**（Shift Right）：
  - **TiP**（Test in Production）：在线测试  
  - A/B测试、日志分析、性能监控  
  - 与DevOps深度结合

> ✅ 现代测试 = **左移预防 + 中测验证 + 右移监控**

## 🔹 2. **测试过程改进模型**

### 2.1 **TMM**（Testing Maturity Model）
- 源自CMM思想 + 测试最佳实践  
- **5级成熟度**（讲义简略，典型5级如下）：
  1. 初始级（Initial）  
  2. 定义级（Definition）  
  3. 集成级（Integration）  
  4. 管理与度量级（Management & Measurement）  
  5. 优化级（Optimization）  
- 配套：成熟度问卷、评估程序、团队培训指南

### 2.2 **TPI**（Test Process Improvement）
- 连续式改进模型，关注**20个关键域**（Key Areas）：

| 类别 | 关键域示例 |
|------|-----------|
| 策略类 | 测试策略、生命周期模型、介入时间 |
| 技术类 | 测试规格技术、静态测试技术、测试自动化 |
| 管理类 | 缺陷管理、测试环境、度量、报告、测试过程管理 |
| 组织类 | 承诺与动力、测试功能与培训、沟通 |

- **4个级别**（A最低 → D最高）：
  - A：临时性、反应式  
  - B：基本可重复  
  - C：定义良好  
  - D：可管理 & 优化  
- 工具：**检查点**（Checkpoints）+ **改进建议**（Improvement Suggestions）  
- 输出：**TPI成熟度矩阵**（可视化各域能力水平）

### 2.3 **CTP**（Critical Test Process）
- 12个关键过程，围绕 **4P 模型**：
  - **Plan**（计划）：上下文建立、风险评估、估算、计划、团队建设  
  - **Prepare**（准备）：测试系统开发、测试件准备  
  - **Perform**（执行）：测试执行、缺陷报告、结果报告  
  - **Perfect**（完善）：测试发布管理、变更管理  

### 2.4 **STEP**（Systematic Test and Evaluation Process）
- 定位：内容参考模型（非阶段强制）  
- 特点：
  - 测试始于需求明确，止于系统退役  
  - 可与TPI结合使用  
- 核心理念：
  - 基于需求的测试策略  
  - 测试驱动设计  
  - 全员协作（测试↔开发）  
  - 系统化缺陷分析与预防  

> ✅ **对比**：TMM/TPI重**过程成熟度**；CTP/STEP重**过程内容与实践**

## 🔹 3. **软件测试标准和规范**

### 3.1 概述

#### 3.1.1 标准定义
> 由公认机构制定的文件，规定活动/结果的**规则、导则或特性值**，实现**最佳秩序与效益**

#### 3.1.2 标准层次（由高→低）

| 层级 | 代表机构/标准 |
|------|---------------|
| **国际标准** | ISO, IEC, ITU |
| **区域标准** | CEN（欧标委）、ETSI（欧电联）、ARSO（非标组） |
| **国家标准** | GB（中国）、ANSI（美国） |
| **行业标准** | IEEE, MIL-STD（美军标）、GJB（国军标）、SJ（电子行业） |
| **地方标准** | DB（如DB11） |
| **企业标准** | Boeing、Siemens 内部规范 |
| **项目规范** | CIMS软件工程规范等 |

#### 3.1.3 标准作用
- 保障质量 & 规范市场  
- **评测核心依据**  
- 搭建用户–开发方–评测机构三方桥梁  
- 促进国际接轨（如ISO认证）

---

### 3.2 ISO/GB 软件质量体系标准

#### 3.2.1 ISO 标准结构
- **ISO 9001**：设计/开发/生产/安装/服务全周期质量保证  
- **ISO 9002**：生产+安装  
- **ISO 9003**：最终检验+测试  
- **ISO 9004**：质量管理导则（组织级）  

#### 3.2.2 **ISO 9000-3**
- 定位：**ISO 9001 在软件领域的应用指南**  
- ❗ 不用于认证，仅为实施参考  
- 依据：ISO 9001 + ISO 8402（术语）  
- 核心过程：
  ```
  合同评审 → 需求规格 → 开发计划 → 质量计划  
    → 设计实现 → 测试确认 → 验收  
    → 复制交付 → 安装 → 维护
  ```

---

### 3.3 软件测试规范

#### 定义
对测试流程**过程化、元素化、标准化**的完整体系。

#### 完整测试规范应含：
- 目的、范围、术语表  
- 可追溯性设计  
- 方针与过程规范  
- 模板（用例/报告/计划）  
- 检查表（Checklist）  
- 工具与培训要求  
- 参考资料  

#### 规范制定要素（每个过程）：
| 要素 | 说明 |
|------|------|
| 角色 | 谁负责？（测试经理、工程师等） |
| 输入 | 本阶段依赖什么？（如需求文档） |
| 准入准则 | 什么条件下可开始？ |
| 活动 | 具体做什么？ |
| 输出 | 交付物是什么？（测试用例、报告） |
| 验证与确认 | 如何检查输出质量？ |
| 出口准则 | 什么条件下可结束/移交？ |
| 度量 | 如何量化过程效果？（用例通过率、缺陷密度等） |

## 🔹 4. 软件测试管理和评判体系

### 4.1 必要性
- 监视 & 测量软件产品  
- 识别 & 控制不合格品  
- 验证设计开发正确性  
- 监控软件过程效能  

### 4.2 发展现状
- **国际**：
  - 越早发现问题 → 成本越低  
    - 编码后修复成本 = 编码前 × 10  
    - 交付后修复成本 = 交付前 × 10  
  - 测试投入 ≥ 研发总费用 50%  
- **国内差距**：
  - 重开发、轻测试  
  - 管理随意，无规范体系  
  - 自动化工具缺失  
  - 评测体系尚未建立
