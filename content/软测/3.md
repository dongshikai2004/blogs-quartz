## 📝 **第3页：1.1 软件生命周期（1）**
### 定义：
- 软件生命周期（Software Life Cycle）：从“规划”到“退役”的全过程时间跨度。

### 典型 7 个阶段：
| 阶段 | 关键活动 |
|------|----------|
| 概念阶段 | 项目立项、可行性分析、初步需求设想 |
| 需求阶段 | 需求获取、分析、规格说明（SRS） |
| 设计阶段 | 概要设计、详细设计（架构、模块、接口） |
| 实现阶段 | 编码、单元测试、代码审查 |
| **测试阶段** | 集成测试、系统测试、验收测试 |
| 试运行阶段 | 用户试用、Beta 测试、上线前验证 |
| 运行与维护 | Bug 修复、功能增强、版本迭代、最终退役 |

> 📌 **关键点**：测试并非孤立阶段，而是**横跨多个生命周期阶段**的活动——例如需求评审=早期测试，运维监控=上线后测试（测试右移）。

## 📝 **第5页：1.2 软件开发模型续**
### 重点内容：
- **增量 vs 迭代**（对比说明）：
  - **增量**：功能逐步叠加，每次交付一个可用子集（如 v1=登录+主页，v2=+购物车…）  
  - **迭代**：每次循环都改进全系统，深度渐进（如 v1=粗糙MVP，v2=优化体验，v3=性能调优）

- **RUP**（3 张子页）：
  - 特点：用例驱动、架构为中心、**4 阶段**（初始 Inception → 细化 Elaboration → 构造 Construction → 移交 Transition）  
  - 每阶段含核心工作流：需求、分析设计、实现、测试、部署  
  - **强调风险控制与早期架构验证**

> ⚠️ **讲义批注**（见第6页开头）：  
> > “遗憾：在这些过程方法中，并没有充分强调测试的价值……无法更好指导测试实践”  
> → 引出：**必须独立构建‘测试模型’来弥补开发模型的测试盲区**

## 📝 **第7–8页：1.3.1 V 模型（共4页内容压缩于此）**
### 起源：
- Paul Rook，1980s后期；英国国家计算中心发布；瀑布模型变种

### 结构（左右对称“V”）：
| 左侧（开发） | 右侧（测试） | 对应关系 |
|--------------|--------------|----------|
| 需求分析 | 验收测试 | 验证是否满足用户需求 |
| 概要设计 | 系统测试 | 验证系统功能/性能/安全性 |
| 详细设计 | 集成测试 | 验证模块间接口与协作 |
| 编码 | 单元测试 | 验证代码逻辑正确性 |

### ✅ 优势：
- 清晰体现测试级别与开发阶段的**映射关系**  
- 强调“测试设计应尽早”——如需求阶段就设计验收用例

### ❌ 局限：
1. **线性顺序**：隐含“测试是开发之后的事”，易忽略早期验证（如需求评审）  
2. **缺陷滞后暴露**：需求问题可能直到验收测试才被发现  
3. **无法支持迭代/敏捷**：变更成本高，不适应现代开发节奏  

> 📌 **实际案例**：某文档格式转换项目，在验收阶段才发现需求中“保留表格样式”被遗漏 → 成本剧增。

## 📝 **第9–10页：1.3.2 W 模型**
### 提出者：Evolutif 公司  
### 理论依据：IEEE Std 1012-1998《软件验证与确认（V&V）》

### 核心思想：
- “双V”结构：**开发是一个V，测试是并行的另一个V**  
- 测试对象 = 全生命周期产出物：需求、设计、代码  
- **测试与开发同步**：需求一完成 → 立即可启动需求测试（而非等到最后）

### ✅ 优势 vs V 模型：
- 打破“测试后置”误区，支持**测试左移**  
- 强调各阶段“验证（Verification）”与“确认（Validation）”

### ❌ 局限：
- 仍为**串行假设**：依赖严格阶段边界，不支持频繁变更/迭代开发  
- 对“无文档”文化（如敏捷）适应性差  
- 未定义独立测试流程（测试仍依附开发节奏）

> 🔁 **关键进步**：测试≠找Bug，而是**质量保障活动**，从源头介入。

## 📝 **第11–13页：1.3.3 H 模型**
### 动机：应对现实——阶段模糊、反复迭代、增量交付

### 核心理念：
- **测试是独立流程**，可与其他活动**并发**进行  
- 测试 = 准备 + 执行 两大阶段（非一次性活动）

#### 测试准备活动（Test Preparation）：
| 活动 | 说明 |
|------|------|
| 需求分析 | 识别可测性、风险点 |
| 测试计划 | 范围、策略、资源、进度 |
| 测试分析 | 设计测试条件/用例大纲 |
| 测试编码 | 编写自动化脚本/测试数据 |
| 测试验证 | 评审用例/脚本有效性 |

#### 测试执行活动（Test Execution）：
| 活动 | 说明 |
|------|------|
| 测试运行 | 执行测试（手动/自动） |
| 测试报告 | 缺陷统计、通过率、风险评估 |
| 测试分析 | 根因分析、质量趋势判断 |

### ✅ 优势：
- 体现测试**复杂性与独立性**  
- 支持**尽早测试、反复测试**（如：代码改了 → 重跑相关测试）  
- **资源灵活调配**：准备与执行可由不同团队并行

### 🎯 应用意义：
- 为测试团队提供**流程自主权**  
- 便于度量投入产出（如：准备工时 vs 执行工时）  
- 与 DevOps/CI-CD 天然契合（持续测试）

> 🌐 **图示理解**：“H”形 = 两个独立竖线（开发流 + 测试流），中间多条横线（触发点）

## 📝 **第14–15页：1.3.4 X 模型**
### 核心流程：
1. 对**独立代码片段**分别进行编码与测试（左下分支）  
2. 频繁集成 → 形成可执行程序  
3. 对**集成后系统**进行整体测试（右上分支）  
4. 支持多路并行变更（多曲线示意）

### ✅ 优点：
- 强调**单元测试 + 集成测试**基础地位  
- 引入**探索性测试（Exploratory Testing）**：无脚本、基于经验的实时测试  
- 缺陷修复可跨团队协作（“不受项目组内部人员限制”）

### ❌ 缺点：
- 忽略**需求/设计/验收测试** → 不完整  
- 未说明测试计划、管理等支撑活动  
- 实践中难落地（过于理想化）

> 🧪 **探索性测试补充**：由 James Bach 提倡，强调“同时设计+执行”，适合复杂/未知系统。

## 📝 **第16–17页：1.3.5 测试模型比较**
| 模型 | 核心贡献 | 主要不足 |
|------|----------|----------|
| **V** | 明确测试级别与开发对应 | 忽略非代码测试；线性滞后 |
| **W** | 支持同步测试；扩展测试对象 | 仍串行；无独立测试流程 |
| **H** | 测试完全独立；支持并发/迭代 | 图形抽象，实操指导弱 |
| **X** | 重视单元/集成；引入探索测试 | 忽略高层测试；不系统 |

### 🧭 综合结论：
- 无“最佳模型”，需**组合使用**  
- 实践中推荐：**在W模型框架下，用H模型思想做独立测试，结合X模型的探索性测试**

## 📝 **第18页：1.3.6 模型使用 + 测试左移/右移**
### 实践原则：
> “灵活运用各模型优点……寻找恰当就绪点开始测试并反复迭代”

### 🔀 测试左移（Shift-Left Testing）：
- **内涵**：测试活动向生命周期**前端迁移**
- **实践**：
  - 需求/设计评审（静态测试）  
  - 验收测试驱动开发（ATDD）  
  - 测试计划/用例设计提前（需求阶段即启动）  
  - 开发自测（TDD / 单元测试）

### 🔀 测试右移（Shift-Right Testing）：
- **内涵**：测试延伸至**生产环境**
- **实践**：
  - **TiP（Test in Production）**：  
    - A/B测试（功能效果验证）  
    - 实时性能监控（APM工具）  
    - 日志/异常分析（ELK, Sentry）  
    - 影子流量测试（Shadow Testing）  
  - 与 **DevOps / SRE** 紧密结合

> 🌉 **趋势**：左移防缺陷，右移快反馈 → 构建质量闭环。

## 📝 **第19–22页：2. 测试过程改进模型**
### 共性目标：提升测试过程可控性、可预测性、效率

#### 2.1 TMM（Test Maturity Model）
- **借鉴 CMM**（能力成熟度模型）  
- **5 级成熟度**（简化版）：
  1. 初始级（Initial）  
  2. 阶段定义级（Definition）  
  3. 集成级（Integration）  
  4. 管理与度量级（Management & Measurement）  
  5. 优化级（Optimization）  
- 每级含：目标、子目标、活动、职责、评估方法  
- 配套：成熟度问卷 + 评估程序 + 培训指南

#### 2.2 TPI（Test Process Improvement）
- **连续式改进模型**（非等级制，按关键域评估）
- **20 个关键域**（Key Areas）分 4 类：

| 类别 | 关键域举例 |
|------|------------|
| **策略与计划** | 测试策略、介入时间、估计与计划 |
| **技术与方法** | 测试规格技术、静态测试、测试自动化、测试环境 |
| **组织与管理** | 承诺与动力、测试功能与培训、沟通、报告、缺陷管理 |
| **生命周期覆盖** | 生命周期模型、底层测试、测试件管理 |

- **4 级成熟度**：A（初始）→ B（可重复）→ C（定义）→ D（管理与优化）  
- **检查点（Checkpoint）**：每级具体达标项  
- **建议（Improvement Suggestion）**：给出落地改进步骤

#### 2.3 CTP（Critical Test Process）
- **上下文驱动**，可裁剪  
- **4P 流程**：
  - **P**lan（计划）  
  - **P**repare（准备）  
  - **P**erform（执行）  
  - **P**erfect（完善）  
- **12 个关键过程**：  
  质量风险评估、测试估算、测试计划、测试团队开发、测试执行、缺陷报告、变更管理等  
- 强调“评估现状 → 识别强弱 → 定制改进”

#### 2.4 STEP（Systematic Test and Evaluation Process）
- 将测试视为**全生命周期活动**（从需求到退役）  
- **核心理念**：
  - 需求即测试起点  
  - 测试驱动设计（TDD思想前置）  
  - 缺陷系统分析（根本原因分析 RCA）  
  - 开发与测试协作  
- 可与 TPI 结合使用（STEP 提供策略，TPI 提供评估框架）

> 🎯 **总结**：  
> - TMM：适合组织级成熟度认证  
> - TPI：适合快速诊断+按需改进  
> - CTP/STEP：适合项目级定制化改进

## 📝 **第23–26页：3. 软件测试标准和规范**
### 3.1 概述
#### 3.1.1 标准定义：
- 公认机构制定的文件 → 规定**规则/导则/特性值** → 实现**最佳秩序与效益**

#### 3.1.2 标准层次（金字塔结构）：
| 层级 | 举例 | 说明 |
|------|------|------|
| **国际** | ISO, IEC, ITU | 全球通用，如 ISO/IEC 25010 |
| **区域** | CEN, ETSI | 欧洲/非洲等区域性 |
| **国家** | GB（中国）, ANSI（美） | 强制或推荐性国标 |
| **行业** | IEEE, GJB, SJ | 如 IEEE 829 测试文档标准 |
| **地方** | DB | 省市级标准 |
| **企业** | Boeing, Siemens | 内部质量体系 |
| **项目** | CIMS 软件工程规范 | 特定项目定制 |

#### 3.1.3 标准作用：
- 产品质量保障标尺  
- 评测依据（第三方测试机构）  
- 用户-开发-评测三方沟通桥梁  
- 推动国际化（ISO 认证 = 出海通行证）

---

### 3.2 ISO/GB 软件质量体系标准
#### 3.2.1 ISO 9000 系列结构：
| 标准 | 适用场景 |
|------|----------|
| ISO 9001 | 设计开发+生产+服务 → **最常用认证依据** |
| ISO 9002 | 仅生产/安装 → 无设计环节企业 |
| ISO 9003 | 仅检验/测试 → 简化版 |
| ISO 9004 | 质量管理体系指南 → 内部提升用 |

#### 3.2.2 ISO 9000-3（软件专用指南）
- **定位**：ISO 9001 在软件领域的实施指南（**非认证标准**）  
- **10 大过程域**：
  1. 合同评审  
  2. 需方需求规格说明  
  3. 开发计划  
  4. 质量计划  
  5. 设计和实现  
  6. **测试和确认**（含测试计划、用例、执行、报告）  
  7. 验收  
  8. 复制、交付和安装  
  9. 维护  

> 📄 **注**：我国等同采用为 **GB/T 19000.3**

---

### 3.3 软件测试规范
#### 完整规范应包含：
| 类别 | 内容示例 |
|------|----------|
| **文档结构** | 目的、范围、术语、参考文献 |
| **过程定义** | 角色职责、进入/退出准则、输入/输出 |
| **活动指南** | 测试设计方法、评审流程、缺陷处理流程 |
| **支持资产** | 模板（用例、报告）、检查表、工具列表 |
| **能力建设** | 培训计划、技能矩阵、知识库 |

#### 制定测试规范需明确：
| 要素 | 说明 |
|------|------|
| 角色 | 测试经理、测试工程师、SQA、开发、产品 |
| 进入准则 | 如：需求基线化、测试环境就绪 |
| 输入项 | 需求文档、设计文档、代码包 |
| 活动过程 | 测试分析 → 设计 → 实现 → 执行 → 评估 |
| 输出项 | 测试计划、用例、报告、缺陷列表 |
| 验证与确认 | 用例评审、测试覆盖分析、回归验证 |
| 退出准则 | 缺陷率 ≤ X%，关键路径100%覆盖 |
| 度量 | 用例数、执行率、缺陷密度、逃逸率 |

> 🛠️ **目标**：使测试**可重复、可管理、可度量**

## 📝 **第27–28页：4. 软件测试管理和评判体系**
### 4.1 必要性（Why?）
- 监视测量产品：缺陷分布、质量趋势  
- 控制不合格品：阻断缺陷流入下一阶段  
- 验证设计开发：测试反哺设计合理性  
- 监视测量过程：测试效率、资源投入产出比

### 4.2 发展现状（中外对比）
| 维度 | 国际先进企业 | 国内多数企业 |
|------|--------------|--------------|
| 成本占比 | ≥50% 研发总投入 | <20%，甚至无专岗 |
| 认知 | 质量=测试+开发共同责任 | “测试=找Bug的” |
| 管理 | 规范流程 + 自动化支撑 | 随意、人工、无体系 |
| 工具链 | CI/CD + TestOps 平台 | 手工测试为主 |

> 💡 **数据警示**（美国QA研究所）：
> - 编码后改Bug = 编码前改的 **10倍** 成本  
> - 交付后改Bug = 交付前改的 **100倍** 成本  
> - 质量越高 → 维护成本越低

### 4.3 如何建立体系（How?）
- 参考成熟模型（TMM/TPI/STEP）  
- 结合组织现状定制  
- 分阶段实施：  
  1. 建立基本流程（计划→设计→执行→报告）  
  2. 引入度量与评审机制  
  3. 推进自动化与工具集成  
  4. 文化建设（质量意识全员化）
