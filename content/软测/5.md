## 1️ 兼容性测试（Compatibility Testing）

### 1.1 概述  
- **定义**：验证软件与其依赖环境（硬件、平台软件、其他软件等）的依赖程度。  
- **目标**：确保软件在各种指定环境组合下功能与性能均满足设计要求。

### 1.2 测试环境准备  
- **硬件环境**：服务器、客户端、网络设备、打印机、扫描仪等。  
- **软件环境**：OS、DB、中间件、浏览器、共享软件等。  
- **解决资源问题途径**：
  - 向厂商租用/借用设备  
  - 使用试用版软件  
  - 借助专业测试实验室  

### 1.3 硬件兼容性测试  
| 类型         | 内容要点                               |
| ---------- | ---------------------------------- |
| **整机兼容性**  | 验证最低/推荐配置合理性；CPU/内存/硬盘/RAID/SCSI支持 |
| **板卡与配件**  | 独立板卡、主板芯片组、驱动中自由软件                 |
| **打印机兼容性** | 安装、测试页、纸张幅面/方向/精度、多份/双面/网络打印、维护功能  |
| **其他外设**   | 鼠标/键盘（红外）、扫描仪、Jog轮、光驱等，依实际需求测试     |

### 1.4 软件兼容性测试  
| 类别 | 说明 |
|------|------|
| 操作系统 | 前后端分别测试（多层架构）；验证跨OS支持能力 |
| 数据库 | 支持异构迁移（e.g. Sybase → Oracle）、版本升级；测试：完整性、应用功能、性能 |
| 中间件 | 不同版本/补丁兼容（e.g. WebSphere SP2→SP3）；关注功能正确性、性能 |
| 浏览器 | 构建**兼容性矩阵**：厂商×版本×OS×特性支持 |
| 其他软件 | 同类软件互操作性；非同类软件共存性；支持软件依赖性 |

### 1.5 数据兼容性测试  
- **核心**：验证软件间能否**正确交互与共享信息**  
- **测试场景**：
  - 剪贴板复制（文字/格式/表格/图形）
  - 新旧版本文件互开性（新特性解释、旧版兼容）
  - 与同系列/同类软件数据交换格式支持
  - 是否符合行业标准（如XML Schema）

#### 1.5.2 XML符合性测试  
1. 建立标准模板（Schema）  
2. 被测软件导出XML  
3. 与模板比对（工具：XMLSPY、自研解析器）  
4. 输出合规性结果  

### 1.6 平台化软件兼容性策略  
- 跨硬件平台  
- 跨操作系统  
- 多数据库支持  
- 客户端兼容性  
- 数据兼容性  

### 1.7 新旧系统数据迁移测试  
**关键措施**：
- ✅ 搭建**模拟环境**，正向+反向迁移测试  
- ✅ **双重备份** + 数据校验  
- ✅ 先升级**备份系统**，试运行（如1周），再升级主系统  
- ✅ 全程记录问题 → 风险预测 + 应急预案  

### 1.8 小结  
- 兼容性测试投入大 → 需**合理取舍**，追求**性价比最优**  
- 定制系统应**尽早测试**，避免数据积累后迁移风险剧增

## 2️ 易用性测试（Usability Testing）

### 2.1 概述  
- **定义**：软件被理解、学习、使用和吸引用户的能力  
- 维度：易理解性、易学习性、美观性、一致性、业务符合性  
- 特点：强主观性 + 依赖经验 + 需用户参与（尤其复杂业务系统）

### 2.2 安装测试  
| 测试点 | 内容 |
|--------|------|
| 安装手册 | 内容准确性、清晰性 |
| 自动化程度 | 是否支持静默/一键安装 |
| 安装选项 | 自定义路径、组件选择 |
| 中断测试 | 断电/强制退出后恢复能力 |
| 多环境安装 | Win/Linux/macOS 等 |
| 正确性 | 安装后功能完整、无残留 |
| 修复/卸载 | 修复不破坏数据；卸载干净彻底 |

### 2.3 功能易用性测试  
- 业务符合性 ✅  
- 功能定制性（个性化配置）  
- 模块集成度（无缝衔接）  
- 数据共享能力（跨模块调用）  
- 约束提示（输入合法性校验）  
- 交互流畅性（操作反馈及时）  
- **错误提示**：明确、友好、可操作  

### 2.4 用户界面测试  

#### 2.4.1 界面整体测试  
| 类型 | 关键点 |
|------|--------|
| 规范性 | 遵循IBM/MS/Apple等行业UI标准；特殊群体适配（老年/视障/低配硬件） |
| 合理性 | 色彩协调、布局均衡、元素大小匹配、窗口比例得当 |
| 一致性 | 字体/标签/颜色/术语/错误信息/快捷键语义统一 |
| 定制性 | 支持多角色界面（如财务总监 vs 出纳）；工具栏/报表/检索可配置 |

#### 2.4.2 界面元素测试  
- 窗口（缩放/最小化/焦点丢失）  
- 菜单（层级合理、无死链）  
- 图标（表意清晰、风格统一）  
- 鼠标（指针变化合理、右键支持）  
- 文字（无错别字、无硬编码、语言风格一致）

### 2.5 辅助系统测试  

| 类型 | 测试要点 |
|------|----------|
| 帮助系统 | 一致性、完整性、可理解性、易检索性（搜索/目录/索引） |
| 向导（Wizard） | 步骤≥3；流程引导清晰；每步均有说明；支持取消/回退 |
| 信息提示 | 语言通俗；破坏性操作需二次确认；统一标记/缩写/颜色规范 |

### 2.6 小结  
> ❗ 不要因主观模糊性而忽视易用性测试——它是用户体验与市场成败的关键。

## 3️ 可靠性测试（Reliability Testing）

### 3.1 什么是软件可靠性？  
- **定义**：在规定**环境** + 规定**时间**内，无故障运行的概率  
- **环境**：  
  - 软件：OS/DB/Compiler  
  - 硬件：CPU/Memory/I/O  
- **时间分类**：
  - **执行时间**：CPU实际处理时间  
  - **日历时间**：自然时间（含停机）  
  - **时钟时间**：程序从启到停的墙上时间  
  - **自然单元**：运行次数、API调用数、事务数等  

> 📌 例：财务软件 9:00–17:00 运行 → 日历时间=24h，时钟时间=8h，CPU时间≈2h

### 3.2 可靠性目标  
- 指标：**可靠度**、**故障强度**（failures/unit time）、**平均无故障时间（MTBF）**  
- 需定义：运行模式、故障等级划分（致命/严重/一般）、目标阈值  

### 3.3 可靠性测试实施  

| 维度 | 内容 |
|------|------|
| **方法** | 黑盒测试 + 基于**操作剖面**（Operational Profile）的随机测试 |
| **目的** | ① 发现缺陷 → 可靠性增长 ② 验证需求 ③ 估计可靠性水平 |
| **环境** | 实验室仿真 or 现场真实环境 |
| **阶段** | 系统测试期 + 交付前验证 |
| **流程** | 1️⃣ 建操作剖面 → 2️⃣ 构建测试环境 → 3️⃣ 执行+量化分析 → 4️⃣ 评估可靠性 |

## 4️ 文档测试（Documentation Testing）

### 4.1 测试范围  
| 类别 | 文档示例 |
|------|----------|
| 用户文档 | 手册、操作指南、维护建议、EULA、安装说明、在线帮助、示例 |
| 开发文档 | 需求/概要/详细设计、DB设计、可行性报告 |
| 管理文档 | 项目计划、测试计划/报告、进度月报、总结报告 |

### 4.2 用户文档内容  
- 包装/标签/广告  
- 授权表、EULA  
- 安装指导、用户手册、CBT向导  
- 示例/模板、错误提示语  

### 4.3 文档作用  
- ✅ 提升安装性、易用性、可靠性  
- ✅ 促进销售 + 降低技术支持成本  

### 4.4 测试要点  
- 静态审查（非代码部分）  
- 动态验证（操作步骤与软件行为一致）  
- **像真实用户一样**：逐字阅读、实操每一步、验图、跑示例  
- 使用简化检查清单（完整性/准确性/一致性/清晰性）  

### 4.5 注意问题  
- 文档常被轻视 → 投入不足  
- 作者 ≠ 功能专家 → 可能存在技术偏差  
- 时间差 → 无法反映最新变更  
- **文档错误 = 软件缺陷**  

### 4.6 小结  
> 📚 文档开发与测试工作量可能**超过代码本身**——务必重视！

## 5️ 负载压力测试（Load & Stress Testing）

### 5.1 概念辨析  
| 术语 | 定义 |
|------|------|
| **负载（Load）** | 并发用户数 / 运行时长 / 数据量 |
| **性能测试** | 广义：效率、资源、稳定性、安全性、扩展性… |
| **负载测试** | 逐步加压 → 观察系统响应（吞吐量、RT、CPU、内存）→ 找稳定边界 |
| **压力测试** | 超负荷运行 → 找系统**瓶颈/失效点**（最大服务能力） |
| **负载压力测试** | 综合：测系统承受**最大并发 × 时间 × 数据量**的能力 |

#### 子类：
- **并发性能测试**（重点）：虚拟用户递增加压 → 找瓶颈  
- **疲劳强度测试**：以最大/日常并发持续运行 → 验长期稳定性  
- **大数据量测试**：
  - 独立型：单业务大数据（如10万条记录查询）
  - 综合型：与负载/压力/疲劳测试结合

### 5.2 实施步骤（9步闭环）  
1. **测试计划**  
   - 系统组件（Client/Network/Server/Middleware）  
   - 配置分析（用户数、客户端软硬件、DB/Web服务器）  
   - 使用模型（用户类型、数量、典型任务）  
   - 任务分布（峰值时间、典型活动）  
   - **目标制定**（可度量！e.g. RT < 2s @ 1000并发）  

2. **需求分析**  
   - 目的：真实环境性能评估、压力预判、瓶颈定位、优化指导  
   - 方法：性能检测 + 性能调优  

3. **测试案例制定**  
   - 协议选择（HTTP/Oracle/ODBC/WebSocket…）  
   - 指标：  
     - 客户端：TPS、RT、吞吐量  
     - 服务器：CPU/Mem/IO（UNIX/Windows）  
     - DB：缓存命中率、锁等待  
     - 中间件：Tuxedo队列深度  

4. **环境/工具/数据准备**
   - 工具选型：
     - 商用：LoadRunner、JMeter、WebLoad、QA Load
     - 开源：JMeter、OpenSTA、TestMaker
     - 自研：特定场景定制
   - ⚠️ 工具盲点：无功能校验、控件支持差、脚本不灵活、日志爆炸  

5. **脚本录制/编写/调试**  
6. **场景设计**（用户数、加压策略、思考时间）  
7. **测试执行**  
8. **结果获取**  
9. **评估与报告**  

#### 核心指标分析（分类）  
| 类别 | 关键指标 |
|------|----------|
| 客户端 | 虚拟用户数、平均RT、TPS、吞吐量（KB/s） |
| Web请求 | 点击率、HTTP状态码分布、页面下载时间分解 |
| 服务器资源 | CPU%、Memory、Swap、Disk I/O、网络带宽/延迟 |
| DB | 缓存命中率、锁争用、慢查询数 |
| 网络 | 吞吐、利用率、延迟抖动、响应时间 |

#### ✅ 典型瓶颈分析经验  
- RT过长 → 查数据库（索引？聚合函数？表连接？）  
- CPU > 95% → 升级CPU或负载均衡  
- DB缓存命中率 < 80% → 增内存  
- 分段测响应：Web - App - DB → 差值定位瓶颈  

#### 🔧 优化方向  
- CPU：多核调度、前置机、并行服务  
- 内存：调DB buffer、减少swap、邻接内存块  
- 磁盘IO：分离日志/DB/索引到不同磁盘；避免镜像写  
- 网络：批量提取（array fetch）、调SDU大小、用共享进程

## 6️ 软件本地化测试（Localization Testing）

### 6.1 基础概念  
| 缩写 | 全称 | 说明 |
|------|------|------|
| **I18N** | Internationalization | 国际化：设计时支持多语言/文化 → **工程化阶段**（开发者负责） |
| **L10N** | Localization | 本地化：针对某地区语言/文化定制 → **可外包** |
| **G11N** | Globalization | 全球化：统筹I18N + L10N的产品策略 |

> ✅ I18N 是 L10N 的**前提**；理想L10N**不引入新缺陷**

#### I18N 关键实践：
- 支持 Unicode / UTF-8  
- 分离代码与资源（.resx/.properties）  
- 消除硬编码（Hard Code）  
- 支持键盘/排序/大小写/时区/货币本地化  
- UI自适应布局（文本扩展预留空间）

### 6.2 翻译验证  
- 内容准确 + 术语统一  
- 符合目标语言**文化心理**（宗教/政治/习俗）  
- 特殊符号处理（©®™）  
- ⚠️ **文本扩展问题**：  
  - 英→德/法常 +30%~50% 长度  
  - 导致UI截断、控件重叠、甚至崩溃  
  - 测试重点：换行/截断/连字符/边界溢出

### 6.3 本地化技术问题  

| 类别 | 常见问题 |
|------|----------|
| **数据格式** | 数字（千分位）、货币（￥/$/€）、日期（MM/DD vs DD/MM）、时间（12h/24h）、度量衡（磅/公斤）、复数规则（英语 vs 中文） |
| **页面布局** | 双字节乱码（GB/BIG5/Shift-JIS）；RTL语言（阿拉伯/希伯来）右对齐；中英文段落缩进差异；连字符规则失效 |
| **配置兼容** | 键盘布局（QWERTY vs AZERTY）；热键冲突；打印机本地化驱动；Non-Unicode路径问题 |

### 6.4 本地化功能测试（L10N Functional）  
- 基础功能 + 安装/升级  
- 多语言环境特定功能：  
  - 时间/日期格式转换  
  - DST（夏令时）自动切换  
  - 文件保存路径编码（避免中文乱码）  
  - 打印内容本地化正确  
  - 排序/索引按本地规则（拼音/笔画/字母）  
  - 联机帮助跳转正常  

### 6.5 小结  
> 🌍 本地化 ≠ 仅翻译！本质是**工程技术活**：字符集、格式、布局、配置、文化适配缺一不可。

## ✅ 总结图：六大测试技术核心目标对比  

| 测试类型 | 核心目标 | 关键输出 |
|----------|----------|----------|
| 兼容性 | “在哪能跑？” | 支持环境清单、兼容矩阵、迁移方案 |
| 易用性 | “好不好用？” | UI改进建议、帮助系统优化、安装流程简化 |
| 可靠性 | “稳不稳？” | MTBF、故障率、操作剖面、可靠性增长报告 |
| 文档 | “会不会用？” | 文档缺陷清单、一致性报告、用户反馈 |
| 负载压力 | “扛不扛得住？” | 性能基线、瓶颈定位、容量规划、调优建议 |
| 本地化 | “当地人爱用吗？” | 多语言包质量、文化合规性、区域功能正确性 |
